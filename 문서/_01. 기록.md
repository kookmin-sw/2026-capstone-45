> # distance, role_confidence, semantic_needs_review 이 점수 기준이 정확도가 높지않은 것 같은데, 이걸 빼고 그냥 llm이 자체적으로 판단하게 하는 방법은 어떤지 판단해봐.

완전히 LLM에게 맡기는 건 가능하지만, 그걸 단독 기준으로 쓰는 건 권장하지 않습니다. 결론부터 말하면:

- `distance + role_confidence + semantic_needs_review`만으로 고르는 현재 방식은 단순해서 한계가 큼
- 그렇다고 점수식을 전부 버리고 “LLM이 알아서 top-k 선택”으로 가면 비용, 속도, 일관성, 재현성이 나빠짐
- 가장 현실적인 해법은 `1차 retrieval는 기계적`, `2차 선택은 LLM`인 하이브리드입니다

왜 pure LLM이 위험하냐면:
- 같은 입력에도 선택이 흔들릴 수 있음
- 왜 골랐는지 숫자로 비교하기 어려움
- 후보가 많아지면 토큰 비용이 급증함
- 잘못하면 그럴듯한 블록을 고르고 진짜 근거 블록을 놓침
- 디버깅 시 “왜 떨어졌는지”가 heuristic보다 덜 명확함

반대로 장점도 있습니다:
- `투자 포인트`, `요약`, `기업 분석 톤` 같은 질의 의도를 더 잘 이해함
- 지금처럼 page-2 fallback 블록이 semantic rationale 블록을 이기는 문제를 줄일 수 있음
- 구조적 role 이름보다 실제 문맥을 더 잘 읽을 수 있음

그래서 추천은 이 방식입니다.

1. 1차 후보 수집
- 벡터/lexical로 넓게 `top 10~20` 뽑기
- 여기서는 recall만 신경쓰기

2. 2차 LLM rerank
- 후보별로 `block text`, `role`, `page`, `짧은 주변문맥`만 넣기
- LLM에게 `top 3`와 이유를 JSON으로 받기
- 예:
```json
{
  "selected": [
    {"record_id": "...", "reason": "...", "used_for": "summary"},
    {"record_id": "...", "reason": "...", "used_for": "financial_table"}
  ]
}
```

3. 최종 fetch
- LLM이 고른 block/page만 fetch

이렇게 하면:
- 지금 heuristic의 단순함은 줄이고
- pure LLM의 불안정성도 막을 수 있습니다

실무 판단:
- `heuristic 완전 제거`는 비추천
- `heuristic을 recall용으로만 축소`하고 `LLM rerank 추가`는 추천

한 줄로 정리하면:
- 검색 엔진이 “후보를 넓게 모으고”
- LLM이 “그중에서 의미적으로 고르는” 구조가 가장 낫습니다

원하면 제가 다음 단계로 바로 해드릴 수 있습니다.
1. 현재 `tool_search_source_document.py`에 `LLM rerank` 2단계 추가
2. 기존 heuristic은 `top 15 후보 수집용`으로만 축소


> # 그러면 1차 후보 수집은 어떻게 어떤 기준으로 하는게 좋을까?

1차 후보 수집은 **정확도보다 재현율(recall)** 중심으로 가는 게 좋습니다.  
즉 “정답만 뽑기”보다 “나중에 LLM이 고를 만한 후보를 빠뜨리지 않기”가 목표입니다.

내 추천은 **단일 점수식 하나**보다 **3개 채널로 후보를 모아서 union** 하는 방식입니다.

**추천 구조**
1. `semantic retrieval`
- block/window/section 단위에서 벡터 검색 `top 5~8`
- 의미적으로 비슷한 후보를 넓게 모음

2. `lexical retrieval`
- BM25나 token overlap으로 `top 5~8`
- 회사명, 티커, 숫자, 표현이 정확히 맞는 블록 확보
- 특히 재무/표/주가 질의에 중요

3. `rule-based prior retrieval`
- 질의 유형에 따라 강제 포함 후보 `top 3~5`
- 예:
  - `재무`, `매출`, `이익`, `주가` 포함
    - table / financial_metrics / consensus / key_data 역할 우선
  - `요약`, `투자포인트`, `분석`
    - thesis / rationale / summary / title 역할 우선
  - 회사명/티커 exact match 블록 우선

그다음:
- 세 채널 결과를 합침
- 같은 anchor/page/block은 dedupe
- 최종 후보를 `10~15개` 정도로 잘라서 LLM rerank로 넘김

**1차 후보에서 써야 할 기준**
지금처럼 `distance + role_confidence + needs_review`만 두면 너무 약합니다.  
1차에서는 최소 아래 5개는 보는 게 좋습니다.

- `semantic similarity`
  - 의미상 가까운가
- `lexical match`
  - 질의 토큰이 실제로 얼마나 겹치는가
- `entity exact match`
  - `KMW`, `032500`, `AT&T` 같은 핵심 엔티티가 정확히 있는가
- `role prior`
  - 질의가 재무형이면 table/metrics 쪽, 요약형이면 thesis/rationale 쪽
- `diversity`
  - 같은 page fallback 블록만 3개 뽑지 않게 제한

**중요한 운영 원칙**
1차 후보 수집에서는 점수를 너무 공격적으로 쓰면 안 됩니다.
- 낮은 precision은 괜찮음
- 낮은 recall은 치명적임

즉:
- “애매하지만 쓸 수 있는 후보”는 넣고
- “절대 아닌 후보”만 빼는 식이 맞습니다

**실무적으로 바로 쓰기 좋은 기본값**
- semantic `top 6`
- lexical `top 6`
- rule-based `top 4`
- union 후 dedupe
- 동일 page 최대 3개
- 동일 role군 최대 2개
- 최종 LLM rerank 입력 `12개`

**지금 프로젝트에 맞춘 예시**
`KMW 기업 분석 보고서` 질의라면 1차 후보는 최소 이런 구성이 들어가야 합니다.
- title/thesis 1~2개
- key data / consensus / financial table 2~4개
- rationale/body 2~3개
- quarterly forecast 1~2개
- analyst/contact/meta 0~1개

즉 “제목만”, “표만”, “fallback page2만” 모이면 안 되고,  
**서술 블록 + 표 블록 + 헤더 블록**이 같이 들어가야 합니다.

한 줄로 정리하면:
- **semantic로 넓게 찾고**
- **lexical로 정확한 키워드 후보를 보강하고**
- **query intent 기반 rule로 빠지면 안 되는 역할 블록을 강제 포함**
- 그 결과를 **LLM이 2차 판단**하게 하는 게 가장 안정적입니다.

원하면 다음으로는 제가 바로:
1. 현재 코드 기준 1차 후보 수집 설계를 구체적인 점수식/쿼터로 써드리거나
2. `tool_search_source_document.py`에 맞는 후보 수집 로직 초안까지 짜드릴 수 있습니다.

> # semantic retrieval, lexical retrieval, rule-based prior retrieval에 대해 좀 더 구체적으로 설명해줘

좋습니다. 이 셋은 서로 대체 관계가 아니라, 서로 다른 실패를 막는 보완 관계로 보면 됩니다.

**1. Semantic retrieval**

이건 “뜻이 비슷한 후보”를 찾는 단계입니다.  
질의와 문서 블록을 임베딩 벡터로 바꿔서, 벡터 공간에서 가까운 블록을 찾습니다.

예를 들어 질의가:
- `KMW 투자 포인트 분석 내용 요약`

이면, 문서 안에 정확히 `투자 포인트`라는 단어가 없어도
- `실적 확인하고 매수하시면 늦습니다`
- `목표주가 상향 조정 사유`
- `미국 통신사 CAPEX 확대 수혜`

같은 블록을 잡을 수 있습니다.  
즉 **표현은 달라도 의미가 비슷한 것**을 찾는 데 강합니다.

장점:
- 동의어, paraphrase, 문장형 질의에 강함
- 사용자가 두루뭉술하게 물어봐도 관련 블록을 찾기 쉬움

약점:
- 숫자, 티커, 고유명사 exact match에는 약할 수 있음
- 가끔 “말은 비슷한데 실제로는 안 중요한 블록”도 끌고 옴
- 임베딩 품질에 따라 흔들릴 수 있음

이 프로젝트에서는 semantic retrieval이 특히 잘 맞는 대상이:
- 제목 블록
- thesis 블록
- summary 블록
- rationale/body 블록
- section purpose가 뚜렷한 블록

입니다.

즉 **“이 문단이 무슨 역할을 하는가”**를 찾는 데 좋습니다.

---

**2. Lexical retrieval**

이건 “글자가 실제로 얼마나 겹치는가”를 보는 단계입니다.  
대표적으로 BM25, tf-idf, token overlap 같은 방식이 여기에 들어갑니다.

예를 들어 질의가:
- `KMW 재무제표 매출 이익 주가 정보`

이면 lexical retrieval은 아래 같은 후보를 잘 찾습니다.
- `KMW`
- `032500`
- `매출액`
- `영업이익`
- `EPS`
- `PBR`
- `목표주가`
- `현재주가`

처럼 **질의와 단어가 직접 겹치는 블록**입니다.

장점:
- 회사명, 티커, 숫자, 표 제목에 강함
- 재무표, Key Data, Consensus Data 같은 블록 찾는 데 매우 좋음
- 빠르고 설명 가능성이 높음

약점:
- 표현이 조금만 바뀌어도 못 찾을 수 있음
- 질의어가 문서 표현과 다르면 약함
- 의미 유사성은 잘 못 잡음

이 프로젝트에서는 lexical retrieval이 특히 중요한 대상이:
- 재무 테이블
- 분기 실적표
- Key Data
- Consensus Data
- 목표주가/현재주가/시가총액 같은 숫자 블록

입니다.

즉 **“정확한 재무/엔티티 단서”**를 찾는 데 좋습니다.

---

**3. Rule-based prior retrieval**

이건 점수 기반 검색이라기보다,  
**“이 질의면 이런 종류의 블록은 기본적으로 후보에 넣어라”**라는 사전 규칙입니다.

쉽게 말하면 retrieval 전에 거는 휴리스틱 prior입니다.

예를 들어 질의에:
- `재무`, `매출`, `이익`, `EPS`, `PER`
가 들어가면
- financial table
- consensus table
- quarterly forecast
- key metrics

역할을 가진 블록을 후보에 우선 포함시킵니다.

반대로 질의에:
- `요약`, `투자포인트`, `분석`, `왜`, `근거`
가 들어가면
- thesis
- summary
- rationale
- body/supporting_argument

역할을 가진 블록을 우선 포함시킵니다.

또:
- `KMW`, `032500` 같은 exact entity가 있는 블록은 무조건 포함
- 첫 페이지 제목/헤더 블록은 항상 1개 정도 포함
- 표 질의면 table이 최소 1~2개는 들어가게 보장

이런 식입니다.

장점:
- semantic/lexical이 놓쳐도 중요한 역할 블록을 강제로 살릴 수 있음
- 검색 결과 다양성을 보장할 수 있음
- 특정 문서 구조에서 매우 실용적임

약점:
- 규칙을 잘못 만들면 편향이 생김
- 도메인 의존적임
- 너무 많아지면 유지보수가 어려움

이 프로젝트에서는 rule-based prior가 특히 중요한 이유가:
지금처럼 검색 결과가 page-2 fallback 블록으로 몰리는 현상을 막기 위해서입니다.

예를 들어:
- top 10 후보 안에 table만 6개 들어오고 rationale이 0개면
- LLM은 투자 thesis를 못 보고 숫자 위주 문서를 만들 가능성이 높습니다

이럴 때 rule-based prior로:
- thesis 1개 이상
- rationale 1개 이상
- table 1개 이상

을 강제하면 훨씬 안정적입니다.

---

**셋의 역할 차이**

아주 짧게 요약하면:

- semantic retrieval
  - “무슨 뜻인가”로 찾음
- lexical retrieval
  - “무슨 단어가 있나”로 찾음
- rule-based prior retrieval
  - “이 질의면 원래 이런 블록이 필요하다”를 강제함

---

**실제 예시**

질의:
- `financial2 문서를 기반으로 KMW 기업 분석 보고서 작성해줘`

이때 이상적인 1차 후보 수집은 이런 식입니다.

`semantic retrieval`이 잡아야 할 것:
- `KMW (032500)`
- `실적 확인하고 매수하시면 늦습니다`
- 목표가 상향 사유
- 미국 CAPEX 확대 논리
- 투자 타이밍 rationale

`lexical retrieval`이 잡아야 할 것:
- `목표주가(12M) 50,000원`
- `현재주가 26,000원`
- `매출액`
- `영업이익`
- `PER`
- `PBR`
- `분기별 수익 예상`

`rule-based prior retrieval`이 보장해야 할 것:
- title/thesis 최소 1개
- summary/rationale 최소 1개
- financial table 최소 1개
- quarterly forecast 최소 1개
- key data/meta 최소 1개

실제로는 이렇게 씁니다.

1. semantic top 6
2. lexical top 6
3. rule-based forced include 3~4
4. dedupe
5. diversity 조정
6. 최종 10~15개를 LLM rerank

이게 가장 안정적입니다.

원하면 다음 답변에서는 제가 바로 이 프로젝트 기준으로:
- 어떤 블록 메타데이터를 semantic용/lexical용/rule-based용으로 써야 하는지
- 그리고 `tool_search_source_document.py`에 넣을 수 있는 구체 규칙
까지 코드 수준으로 적어드릴게요.

> # 1차 후보 수집 설계
좋습니다. 이 프로젝트용 1차 후보 수집은 이렇게 설계하는 게 맞습니다.

**목표**
1차 후보 수집은 `정답 고르기`가 아니라 `놓치지 않고 충분히 모으기`입니다.  
즉 precision보다 recall 우선입니다.

이번 설계는 아래를 전제로 합니다.
- `title/body/table 최소 다양성 보장`은 하지 않음
- 대신 `semantic + lexical + exact entity prior`로 넓게 모음
- 그 결과를 2차 `LLM rerank`로 넘김

---

## 1. 전체 구조

1. query 전처리
2. semantic retrieval
3. lexical retrieval
4. exact entity prior retrieval
5. candidate merge + dedupe
6. lightweight score 정렬
7. top N을 LLM rerank에 전달

---

## 2. query 전처리

질의를 먼저 아래 정보로 분해합니다.

### 추출할 것
- `query_text`
- `query_tokens`
- `query_entities`
- `query_numbers`
- `query_intent_tags`

### 예시
질의:
`financial2 문서를 기반으로 KMW 기업 분석 보고서 작성해줘`

추출:
- `query_tokens`
  - `financial2`, `KMW`, `기업`, `분석`, `보고서`, `작성`
- `query_entities`
  - `KMW`
- `query_numbers`
  - 없음
- `query_intent_tags`
  - `analysis`, `report`

질의:
`KMW 재무제표 매출 이익 주가 정보`

추출:
- `query_entities`
  - `KMW`
- `query_intent_tags`
  - `financial`, `metrics`

### intent tag는 아주 얕게만
이건 강한 rule이 아니라 retrieval query expansion용입니다.
예:
- `재무`, `매출`, `영업이익`, `EPS`, `PER`, `PBR` 포함
  - `financial`
- `요약`, `포인트`, `근거`, `왜`, `전망`
  - `summary`, `thesis`, `rationale`
- `보고서`, `브리핑`, `분석`
  - `report`

---

## 3. semantic retrieval

### 목적
표현이 정확히 안 겹쳐도 의미적으로 관련 있는 블록을 넓게 가져옵니다.

### 입력
- 현재 질의 원문
- 필요하면 expansion query 1~2개 추가

### 추천 방식
- 기존 Chroma 유지
- `query_texts=[원질의 + 확장질의들]`
- 각 질의별 `top_k = 6`
- union

### expansion query 예시
원질의가 `KMW 재무제표 매출 이익 주가 정보`면:
- 원질의
- `KMW financial metrics earnings revenue stock price`
- `KMW 투자 포인트 재무 분석`

확장질의는 1~2개만. 너무 많으면 noise 증가합니다.

### semantic retrieval에서 저장할 것
후보마다:
- `record_id`
- `document`
- `page`
- `display_block_id`
- `distance`
- `generic_role`
- `generated_role_name`
- `section_purpose`
- `role_confidence`
- `semantic_needs_review`
- `retrieved_by = "semantic"`

### 추천 개수
- semantic union 결과 최대 `8~10개`

## 4. lexical retrieval

### 목적
회사명, 숫자, 표 제목, 정확한 표현이 들어간 블록을 잡습니다.

semantic만 쓰면 여기서 자주 놓칩니다.

### 검색 대상 텍스트
각 레코드마다 아래를 합친 lexical text를 만듭니다.
- `embedding_text`
- `generated_role_name`
- `section_purpose`
- `generic_role`
- `display_html`에서 태그 제거한 텍스트
- 필요하면 page title / section title

### 점수
간단히 시작하면 됩니다.

#### 옵션 A: token overlap
```text
lexical_score =
  overlap(query_tokens, record_tokens) / len(query_tokens)
```

#### 옵션 B: BM25
가능하면 BM25가 더 좋습니다.  
하지만 지금 프로젝트는 레코드 수가 아주 크지 않으니 token overlap로도 시작 가능합니다.

### exact boost
다음은 lexical에서 강하게 보정합니다.
- `KMW`, `032500` 같은 entity exact match
- 질의 숫자 exact match
- `매출액`, `영업이익`, `EPS`, `PBR`, `목표주가`, `현재주가` 같은 핵심 토큰 포함

### 저장할 것
- `record_id`
- `lexical_score`
- `entity_match_count`
- `keyword_match_count`
- `retrieved_by = "lexical"`

### 추천 개수
- lexical top `8~10개`

---

## 5. exact entity prior retrieval

이건 강한 도메인 규칙이 아니라, **질의에 나온 엔티티를 실제로 포함하는 후보는 반드시 후보군에 넣자**는 수준입니다.

### 포함 기준
질의에서 추출한:
- 회사명
- 티커
- 인물명
- 제품명
- 주요 숫자

이 후보 텍스트에 exact match하면 포함

### 예시
질의에 `KMW`가 있으면:
- `KMW`
- `케이엠더블유`
- `032500`
같은 alias dictionary로도 확장 가능

### 왜 필요한가
semantic/lexical 둘 다 잘 돌아도, 핵심 엔티티가 들어간 블록이 top-k 밖으로 밀릴 수 있습니다.  
이 prior는 그걸 막는 최소 안전장치입니다.

### 추천 개수
- 강제 포함 최대 `3~5개`

## 6. merge + dedupe

- `display_block_id`

- `lexical_score` 또는 `0`
- `entity_match_count`

- `generic_role`
- `generated_role_name`
- `section_purpose`
- `role_confidence`
- `semantic_needs_review`
- `preview_text`

## 7. lightweight 1차 정렬

여기서는 최종 판단이 아니라, LLM에게 넘길 순서를 정리하는 수준입니다.

복잡한 heuristic 대신 아래 정도만 추천합니다.

```text
first_stage_score =
  semantic_component
  + lexical_component
  + entity_component
  + role_component
  - review_penalty
```

예시:

```text
semantic_component = 
  if semantic_distance exists then (1 - normalized_distance) else 0

lexical_component =
  lexical_score * 0.8

entity_component =
  min(entity_match_count, 2) * 0.2

role_component =
  role_confidence * 0.1

review_penalty =
  0.05 if semantic_needs_review else 0
```

중요:
- 이 점수는 **정답 선택용이 아니라 LLM 입력 후보 정렬용**
- 지금처럼 이 점수로 바로 top-3를 끝내면 안 됨

### 추천 최종 개수
- LLM rerank 입력: `10~12개`

**2. 새 데이터 구조**

현재 `SearchRecord`는 그대로 두고, 아래 2개를 추가하는 게 좋습니다.

```python
@dataclass(slots=True)
class QueryProfile:
    raw_query: str
    normalized_query: str
    query_tokens: list[str]
    entity_candidates: list[str]
    grounded_entities: list[str]
    intent_hints: list[str]
    semantic_queries: list[str]
    lexical_query: str
```

```python
@dataclass(slots=True)
class RetrievalCandidate:
    record: SearchRecord
    channels: set[str]
    semantic_distance: float | None = None
    lexical_score: float = 0.0
    entity_score: float = 0.0
    first_stage_score: float = 0.0
    debug: dict[str, Any] | None = None
```

의도:
- `QueryProfile`
  - 질의 전처리 결과를 한 객체에 묶음
- `RetrievalCandidate`
  - 한 record가 semantic/lexical/entity 여러 채널에서 동시에 잡혔는지 기록

**4. 새 내부 함수 설계**

이 파일 안에 아래 함수들을 추가하는 구조가 가장 자연스럽습니다.

`_preprocess_query(query: str, records: Sequence[SearchRecord]) -> QueryProfile`
- 전체 전처리 entry point
- 내부에서 아래 함수들을 호출

`_extract_query_tokens(query: str) -> list[str]`
- 정규식 tokenization
- lowercase
- stopword 제거
- 중복 제거

`_extract_entity_candidates(query: str) -> list[str]`
- 대문자 토큰, 종목코드, 숫자+단위, 한글 명사구 후보 추출

`_ground_entities(records, entity_candidates) -> list[str]`
- 코퍼스 대조
- 정확히 많이 등장하거나 희귀하게 집중되는 후보만 grounded entity로 승격

`_derive_intent_hints(tokens: Sequence[str]) -> list[str]`
- `summary`, `metrics`, `rationale`, `report` 정도의 약한 힌트만 생성

`_build_semantic_queries(profile: QueryProfile) -> list[str]`
- 원질의 + 짧은 확장 질의 1~2개 생성

`_build_lexical_query(profile: QueryProfile) -> str`
- entity + 주요 토큰을 합친 lexical query 생성

`_collect_semantic_candidates(profile, where_cond) -> dict[str, RetrievalCandidate]`
- Chroma 질의 결과 수집
- `channels={"semantic"}` 부여

`_collect_lexical_candidates(profile) -> dict[str, RetrievalCandidate]`
- 현재는 token overlap 또는 BM25 adapter 사용
- `channels={"lexical"}` 부여

`_collect_entity_prior_candidates(profile) -> dict[str, RetrievalCandidate]`
- grounded entity exact hit 후보 강제 포함
- `channels={"entity_prior"}` 부여

`_merge_candidate_maps(...) -> list[RetrievalCandidate]`
- 동일 `record_id` 기준 dedupe
- semantic/lexical/entity 점수 합침

`_score_first_stage_candidates(candidates, profile) -> list[RetrievalCandidate]`
- 최종 확정이 아니라 “LLM에 보여줄 순서”만 정렬

`_format_candidate_bundle(candidates) -> str`
- tool output 텍스트 생성
- later context 단계에서 읽기 쉽게 `record_id`, `document_id`, `page_id`, `block_id`, `preview` 포함

**5. query 전처리 구체 설계**

이 파일 안에서는 너무 무거운 NLP를 하지 않고 아래 정도만 합니다.

`query_tokens`
- 정규식: `[0-9A-Za-z가-힣]+`
- lowercase
- stopword 제거
- 예:
  - `financial2 문서를 기반으로 KMW 기업 분석 보고서 작성해줘`
  - `["financial2", "kmw", "기업", "분석", "보고서"]`

`entity_candidates`
- 대문자 영문: `KMW`, `AT&T`
- 숫자 코드: `032500`
- 숫자 단위: `50,000원`, `176.0십억원`
- 한글 고유명사형 토큰
- 중요: 여기선 후보만 뽑고 확정은 안 함

`grounded_entities`
- `records` 전체에서 exact string match가 강한 후보만 남김
- 예:
  - `KMW`는 남음
  - `분석`, `정보`는 버림

`intent_hints`
- 아주 약하게만:
  - `metrics`
  - `summary`
  - `rationale`
  - `report`

`semantic_queries`
- 원질의 포함
- 확장 질의는 최대 2개
- 예:
  - `KMW 재무제표 매출 이익 주가 정보`
  - `KMW financial metrics earnings revenue stock price`
  - `KMW 투자 포인트 재무 분석`

`lexical_query`
- entity + 핵심 토큰 묶음
- 예:
  - `KMW 032500 매출 영업이익 EPS PER PBR 목표주가 현재주가`

**6. semantic retrieval 설계**

현재 [invoke_raw()](</c:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/tool_search_source_document.py:687>)는 원질의 1개만 Chroma에 넣습니다.  
이걸 이렇게 바꿉니다.

```python
semantic_queries = profile.semantic_queries

for semantic_query in semantic_queries:

        n_results=SEMANTIC_QUERY_TOP_K,

```

수집 방식:
- 각 query 결과를 union
- 동일 `record_id`면 가장 좋은 distance만 유지
- channel에 `"semantic"` 기록

주의:
- 이 단계에서 top-3로 자르지 않음
- semantic은 recall 확보용

기록할 것:
- `semantic_distance`
- 어떤 semantic query에서 hit됐는지
- metadata role 정보
- later trace용 raw semantic hit 목록

**7. lexical retrieval 설계**

현재 `_fallback_candidates(...)`는 에러 대응용 local lexical 비슷한 역할인데, 이걸 정식 lexical retrieval로 바꿉니다.

내 설계 추천:
- 인터페이스는 BM25 가능하게
- 1차 구현은 token overlap로 시작 가능
- 함수 이름은 `_collect_lexical_candidates(...)`

초기 구현:
```python
lexical_score =
    overlap(query_tokens, record_tokens) / max(1, len(query_tokens))
```

추가 가점:
- grounded entity exact match
- `generated_role_name`, `section_purpose` 안의 token match

검색 대상 텍스트:
- `embedding_text`
- `generated_role_name`
- `section_purpose`
- `generic_role`

later BM25 교체를 위해 scorer interface를 두는 게 좋습니다.

```python
class LexicalScorer(Protocol):
    def score(self, query: QueryProfile, record: SearchRecord) -> float: ...
```

초기엔 `TokenOverlapScorer`, 나중에 `BM25Scorer`로 교체.

**8. entity prior retrieval 설계**

이건 도메인 규칙이 아니라 exact entity recall 보장입니다.

동작:
- grounded entity마다 record 전체를 exact match 검색
- match record는 무조건 후보에 포함
- `entity_score` 부여

예:
- grounded entity가 `KMW`
- `embedding_text`나 `display_html`에 `KMW`, `케이엠더블유`, `032500`가 있으면 포함

가점 예시:
```python
entity_score =
    1.0 if exact_entity_match else 0.0
```

여기서 top 4 정도만 넣으면 충분합니다.

---

**9. merge / 정렬 설계**

merge는 `record_id` 기준입니다.

정렬은 “최종 top-3”가 아니라 “LLM이 보기 좋은 순서”만 만듭니다.

예시 점수:
```python
semantic_component =
    1.0 - normalized_distance if distance is not None else 0.0

first_stage_score =
    (semantic_component * 0.5)
    + (lexical_score * 0.3)
    + (entity_score * 0.15)
    + (record.role_confidence * 0.05)
    - (0.05 if record.semantic_needs_review else 0.0)
```

중요:
- 이 점수는 retrieval ordering용
- selection decision은 나중 단계에서 수행

최종:
- 상위 `12개` 반환
- top 8개만 tool output에 표시해도 됨
- trace에는 12개 전부 남김

1차 후보수집기라면 더 간결하고 구조화된 문자열이 좋습니다.

> # 아쉬운 점: entity grounding 개선

entity grounding이 트럼프, 미국에 과하게 끌렸습니다.
candidate_merge_completed에서 후보가 38개까지 늘었고, entity-only 후보가 많이 들어왔습니다.
matched_via는 analysis에서 거의 전부 entity로 기록됐는데, 실제로는 BM25/semantic 기여도 컸습니다.
즉 analysis의 자기 설명은 실제 retrieval 기여를 과단순화했습니다.

이게 구체적으로 무엇이, 왜 문제인거고 어떻게 해결하면 좋을지 설명해줘

문제는 3개가 연결돼 있습니다.

첫째, `entity grounding`이 너무 넓게 작동했습니다.  
이번 run에서는 `트럼프`, `미국`이 grounded entity가 됐고, 이 두 단어가 기사 전체에 많이 퍼져 있어서 [first_stage.jsonl](</c:/Users/echin/Desktop/ALLLM/llm-to-document/output/17_candidate_search_news1_financial2/trace/20260416T180720157027Z-83454606/search/first_stage.jsonl>)의 `entity_grounding_completed`에서 각각 `23개`, `22개` 매치로 잡혔습니다. 그 결과 `candidate_merge_completed`에서 후보가 `38개`까지 늘었고, 실제로는 별로 중요하지 않은 블록도 `entity` 채널만으로 후보군에 들어왔습니다.

이게 왜 문제냐면, `entity`는 원래 “놓치면 안 되는 핵심 후보를 살려주는 안전장치”여야 하는데, 지금은 반대로 너무 많은 블록을 살려서 분별력을 떨어뜨립니다. `트럼프`, `미국`처럼 기사 전반에 반복되는 단어는 “이 블록이 중요하다”는 신호가 아니라 “이 문서가 그 주제를 다룬다”는 약한 신호에 가깝습니다. 그런데 현재는 이 약한 신호가 candidate ordering에 실제 점수로 들어가니까, BM25나 semantic이 힘들게 끌어올린 진짜 핵심 블록과 비슷한 급으로 취급됩니다.

둘째, 그래서 `entity-only 후보`가 너무 많아졌습니다.  
예를 들어 semantic/BM25로 잡히지 않았는데 `entity`만으로 들어온 블록들이 많이 생겼습니다. 이런 후보는 대체로 “트럼프”나 “미국”이라는 단어만 포함할 뿐, 실제로는 핵심 주장, 요약, 수치, 시장 반응 같은 강한 evidence가 아닐 수 있습니다. 그런데 이 후보들이 merge에 들어오면:
- 후보 수가 불필요하게 커지고
- 상위 10~12개에 noise가 섞일 가능성이 커지고
- 이후 writer LLM이 읽어야 할 bundle도 덜 선명해집니다

셋째, `analysis.json`의 `matched_via`가 현실을 제대로 설명하지 못합니다.  
[analysis.json](</c:/Users/echin/Desktop/ALLLM/llm-to-document/output/17_candidate_search_news1_financial2/trace/20260416T180720157027Z-83454606/analysis.json>)에서는 evidence 대부분이 `matched_via="entity"`로 기록됐는데, 실제 retrieval trace를 보면:
- semantic이 page 1의 핵심 기사 블록을 잡았고
- BM25가 page 2의 시장 반응 블록을 강하게 올렸고
- entity는 그 위에 넓게 덧칠된 보조 채널이었습니다

즉 실제 과정은 `semantic + bm25 + entity` 혼합인데, analysis는 그냥 “entity로 찾았다”고 단순화해버렸습니다. 그러면 나중에 디버깅할 때 잘못된 결론을 내리게 됩니다. 예를 들어 “entity grounding이 잘 먹어서 성공했다”고 착각할 수 있는데, 실제로는 BM25와 semantic이 더 중요한 기여를 했을 수 있습니다.

해결은 이렇게 하는 게 좋습니다.

1. `entity grounding`을 약화  
`entity`는 recall safety net으로만 써야 합니다.
- `entity-only` 후보는 기본적으로 상위 랭킹에 크게 못 오르게 하기
- `bm25`나 `semantic`이 같이 붙은 후보에만 entity 보너스를 주기
- 또는 entity-only 후보는 최대 몇 개만 허용하기

2. 너무 흔한 entity는 grounding 강도를 낮추기  
`트럼프`, `미국`처럼 문서 전체에 넓게 퍼지는 단어는 “핵심 블록 선택” 신호로 약합니다.
- 매치 수가 너무 많은 entity는 점수 축소
- 특정 비율 이상 블록에 등장하면 entity bonus를 감쇠
- 즉 `희소한 entity`와 `문서 전체 공통어`를 구분해야 합니다

3. `matched_via`를 단일값에서 다중값으로 바꾸기  
지금처럼 `entity` 하나만 쓰면 기여가 왜곡됩니다.
예를 들어:
```json
{
  "matched_via": ["semantic", "bm25", "entity"]
}
```
또는
```json
{
  "retrieval_support": {
    "semantic": true,
    "bm25": true,
    "entity": true
  }
}
```
처럼 남기는 게 맞습니다.

4. analysis 생성 시 retrieval trace를 더 직접 반영  
지금 analysis는 모델이 사후적으로 “왜 골랐는지”를 설명하는데, 그 과정에서 단순화가 생깁니다.  
그래서 analysis 프롬프트에 아예 후보의 실제 channel 정보를 같이 넣어 주고,
- 어떤 채널로 처음 발견됐는지
- 어떤 채널이 strongest signal이었는지
를 쓰게 해야 합니다.

5. ordering에서 `entity_score` 가중치를 더 낮추기  
현재는 entity가 너무 쉽게 후보를 밀어 올릴 수 있습니다.  
실무적으로는:
- semantic/BM25 > entity
순서가 돼야 합니다.
즉 entity는 “있으면 약간 플러스” 정도만 되어야지, 독립적인 강한 랭킹 신호가 되면 안 됩니다.

- 지금 문제:
  - `트럼프`, `미국`이 너무 많은 블록을 entity hit로 만들어서 후보군이 부풀고
  - ranking 분별력이 떨어지고
  - analysis는 그걸 과도하게 `entity` 덕분이라고 설명함
- 바람직한 구조:
  - semantic/BM25가 주 채널
  - entity는 보조 채널
  - analysis는 실제 채널 기여를 다중값으로 기록

내 판단으로는 우선순위는 이 순서입니다.
1. `entity-only` 후보 약화
2. `matched_via`를 다중값으로 변경

## Entity Grounding / `matched_via` 개선 계획

### Summary
- `entity-only` 후보는 더 이상 일반 후보와 같은 급으로 경쟁하지 않게 약화한다.
- `matched_via`는 단일 문자열에서 `string[]`로 바꿔 실제 retrieval 기여 채널을 모두 기록한다.
- `트럼프`, `미국`처럼 문서 전반에 넓게 퍼진 흔한 entity는 감쇠해서 entity grounding이 분별력을 망치지 않게 만든다.

### Key Changes
- [tool_search_source_document.py](</c:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/tool_search_source_document.py>)
  - `RetrievalCandidate.entity_score`는 기존처럼 `1.0/2.0` 누적하지 않고, 엔티티별 감쇠 적용 후 합산하되 `0.0~1.0`으로 clamp한다.
  - `entity_grounding_completed` trace payload에 아래 필드를 추가한다.
    - `match_count`
    - `match_ratio`
    - `attenuation_factor`
    - `is_common_entity`
  - 흔한 entity 감쇠 규칙은 아래로 고정한다.
    - `match_ratio = match_count / total_search_records`
    - `match_count > 12` 또는 `match_ratio > 0.30` 이면 `attenuation_factor = 0.0`
    - `match_count > 6` 또는 `match_ratio > 0.15` 이면 `attenuation_factor = 0.5`
    - 그 외는 `attenuation_factor = 1.0`
  - 감쇠 결과 `attenuation_factor = 0.0` 인 entity는 trace에는 남기되, candidate에 entity 채널 점수를 추가하지 않는다.
  - ordering은 두 단계로 분리한다.
    - `signal_candidates`: `semantic` 또는 `bm25` 채널이 하나라도 있는 후보
    - `entity_only_candidates`: `entity`만 있는 후보
  - `signal_candidates`를 먼저 정렬하고, `FIRST_STAGE_LIMIT`를 다 못 채운 경우에만 `entity_only_candidates`를 뒤에 붙인다.
  - ordering score는 아래로 고정한다.
    - `semantic * 0.45`
    - `bm25 * 0.35`
    - `entity * 0.05`
    - `role_confidence * 0.10`
    - `needs_review penalty`
  - `entity_only_candidates`는 별도 정렬 시 `entity_score * 0.20 + role_confidence * 0.05 - penalty`만 쓰고, 최종 상단으로 못 올라오게 한다.
  - `candidate_merge_completed`와 `first_stage_ordered` trace에 `is_entity_only`를 추가한다.

- [create_document.py](</c:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/create_document.py>)
  - `PROMPT_ANALYZE_TRACE`의 evidence 스키마를 아래로 변경한다.
    - `matched_via: array of strings`
  - 프롬프트 규칙에 아래를 명시한다.
    - `matched_via`는 단일값이 아니라 실제 기여한 모든 retrieval 채널을 포함해야 한다.
    - 허용값: `"semantic"`, `"bm25"`, `"entity"`, `"block"`, `"window"`, `"section"`
  - `normalize_analysis_payload()`는 아래를 처리한다.
    - 기존 문자열 응답이면 `[string]`으로 승격
    - 배열이면 string만 남기고 dedupe
    - 비어 있으면 `[]`
  - 저장되는 `analysis.json`의 `evidence[].matched_via`는 항상 `list[str]`로 고정한다.

- 테스트
  - `entity-only` 후보가 `semantic/bm25` 후보보다 앞서지 않는지 테스트 추가
  - 흔한 entity가 감쇠되어 `attenuation_factor=0.0` 또는 `0.5`로 기록되는지 테스트 추가
  - `matched_via`가 문자열/배열 모두 입력돼도 최종적으로 `string[]`로 정규화되는지 테스트 추가
  - 기존 search trace 테스트에 `is_entity_only`와 attenuation 필드 검증 추가
