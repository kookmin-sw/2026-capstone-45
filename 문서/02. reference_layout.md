# Reference Parser Analysis

## 1. 목적

이 문서는 레퍼런스 문서 이미지를 파싱해 문서 구조를 이해시키는 파트에 대한 분석 리포트다.  
핵심 목표는 `OCR 텍스트 추출`이 아니라 `문서 구조를 템플릿으로 추출`하는 것이다.

레퍼런스 파서는 다음 질문에 답할 수 있어야 한다.

- 이 문서는 어떤 레이아웃 계열인가
- 페이지는 몇 개 컬럼으로 구성되는가
- 제목, 본문, 이미지, 캡션, 표, 광고는 어디에 있는가
- 사람은 이 문서를 어떤 순서로 읽는가
- 이후 generation과 rendering이 재사용할 수 있는 구조 규칙은 무엇인가

즉, 레퍼런스 문서는 내용 source가 아니라 형식 조건이다.  
이 단계의 공식 출력은 사람이 읽기 위한 OCR 결과가 아니라 downstream이 사용하는 `ReferenceTemplate`이다.

## 2. 현재 저장소 기준 맥락

현재 저장소에는 reference parsing을 설계할 때 참고할 수 있는 실제 샘플과 실험 결과가 이미 있다.

- `Reference_News paper1.jpg`
- `Reference_News paper2.jpg`
- `OCR_results/dolphin`
- `OCR_results/paddle`
- `OCR_results/surya`

이 샘플들은 다음 특징을 보여준다.

- 신문형 다단 레이아웃
- 큰 메인 사진과 작은 시각 요소가 혼재
- 광고 영역 존재
- 기사 본문과 차트/광고/박스 텍스트가 섞여 있음
- 단순 top-to-bottom 읽기 순서로는 구조를 복원하기 어려움

또한 저장소 내 비교 결과를 보면 parser 후보들의 역할도 구분된다.

- `dolphin`: 구조화 JSON/Markdown까지 출력 가능
- `paddle`: 검출 박스와 타입 확인에 유리
- `surya`: 영역 감지 결과 비교 기준으로 유용

현재 기준 기본 parser는 `dolphin`으로 두되, 출력 계약은 parser-agnostic하게 설계해야 한다.

## 3. 왜 reference parsing이 어려운가

reference parsing은 일반 OCR보다 훨씬 어렵다. 이유는 아래와 같다.

- 문서 종류가 다양하다. 신문, 리포트, 브로슈어, 보고서, 폼은 구조 규칙이 다르다.
- 같은 한국어 문서라도 시각 구조가 다르면 제목/본문 구분 기준이 달라진다.
- 디지털 PDF와 촬영 이미지의 노이즈 양상이 다르다.
- 읽기 순서가 단순한 세로 방향 정렬이 아닐 수 있다.
- 광고, 차트, 배너, 장식 블록이 본문과 섞인다.
- 헤더, 푸터, 페이지 번호는 텍스트지만 생성 대상으로 삼으면 안 된다.
- 이미지와 캡션, 본문과 사이드바처럼 서로 인접한 블록의 관계를 이해해야 한다.

이 때문에 reference 단계는 최소한 아래 다섯 가지를 동시에 해결해야 한다.

1. 블록 검출
2. 블록 분류
3. 읽기 순서 추정
4. 섹션 구조 추정
5. 시각 규칙 추출

## 4. reference 단계가 반드시 이해해야 하는 것

reference 파서는 아래 정보를 추출해야 한다.

- 페이지 구조
- 블록 타입
- 읽기 순서
- 제목 계층
- 이미지 슬롯
- 반복되는 비본문 요소
- 상대적 시각 스타일

좀 더 구체적으로는 다음과 같다.

- 페이지 수, 페이지 크기, 본문 영역
- 컬럼 수와 컬럼 경계
- 문서 제목, 섹션 제목, 부제목
- 일반 문단, 강조 박스, 사이드바
- 이미지, 캡션, 차트, 표
- 광고, 로고, 헤더, 푸터, 페이지 번호
- 이미지가 본문을 감싸는지, 독립 영역인지
- 제목 대비 본문 크기 비율, 섹션 간격, 시각 요소 밀도

## 5. reference 파서가 내야 하는 최소 출력

reference 단계의 출력은 `필수 산출물`, `권장 산출물`, `디버깅 산출물`로 나눈다.

### 5.1 필수 산출물

- `artifacts/<job_id>/01_reference/reference_template.json`
- `artifacts/<job_id>/01_reference/layout_preview/` 내 페이지별 시각화 이미지

### 5.2 권장 산출물

- `artifacts/<job_id>/01_reference/raw_layout.json`
- `artifacts/<job_id>/01_reference/parser_metadata.json`

### 5.3 산출물 원칙

- downstream의 공식 입력은 `reference_template.json` 하나다.
- preview 이미지는 사람 검수용이다.
- raw parser 결과는 디버깅과 성능 비교용이며 공식 계약은 아니다.
- parser를 교체하더라도 `reference_template.json` 스키마는 유지되어야 한다.

## 6. 최소 출력 타입과 필드 설명

## 6.1 `ReferenceTemplate`

reference 단계의 최종 공식 출력이다.  
이 구조가 generation, rendering, review 단계의 기준 입력이 된다.

### 필수 필드

- `template_id`
  - reference 템플릿의 고유 ID
  - 일반적으로 `job_id + reference hash` 조합
  - downstream 캐시 키와 provenance 추적에 사용

- `source_path`
  - 원본 reference 파일 경로
  - 재처리, 디버깅, provenance에 사용

- `document_family`
  - 문서 계열
  - 예: `newspaper`, `report`, `brief`, `brochure`, `unknown`
  - layout rule 적용 분기 기준

- `language`
  - reference 문서의 주 언어
  - OCR 후처리와 renderer 폰트 정책에 사용

- `page_specs`
  - 페이지 단위 메타데이터 배열
  - 페이지 크기, 여백, 컬럼 수, 본문 영역 등을 담음

- `blocks`
  - 모든 구조 블록의 평탄화 리스트
  - 읽기 순서, 블록 타입, 후처리의 기본 자료

- `section_order`
  - 섹션 수준의 순서 정보
  - generation 단계에서 문서 골격을 재구성할 때 사용

- `style_tokens`
  - 절대 픽셀 복제가 아닌 상대 스타일 규칙
  - renderer가 직접 사용

- `image_slots`
  - 이미지 배치 패턴과 위치 규칙
  - generation과 renderer가 이미지 수와 위치를 결정할 때 사용

- `unsupported_blocks`
  - 현재 MVP가 생성 대상으로 다루지 않는 블록 목록
  - 예: 복잡 표, 광고, 복합 차트

- `confidence_summary`
  - reference 파싱 결과의 품질 요약
  - review 단계에서 위험 reference를 식별하는 데 사용

### 권장 필드

- `repeating_elements`
  - 페이지마다 반복되는 헤더, 푸터, 로고

- `template_warnings`
  - 구조 추출의 제한 사항
  - 예: 컬럼 경계 불명확, 캡션 연결 실패

## 6.2 `PageSpec`

페이지 단위 레이아웃 정보를 표현한다.

### 필수 필드

- `page`
  - 1-based 페이지 번호

- `width`
  - 페이지 너비

- `height`
  - 페이지 높이

- `column_count`
  - 추정 컬럼 수
  - 읽기 순서와 본문 폭 계산의 핵심 필드

- `margin_top`
- `margin_right`
- `margin_bottom`
- `margin_left`
  - 본문 유효 영역 추정치

- `content_bbox`
  - 실질 본문 영역 bounding box
  - `[x1, y1, x2, y2]`

- `has_large_visual`
  - 큰 사진, 광고, 차트처럼 시각 요소가 페이지를 크게 차지하는지 여부

### 권장 필드

- `dominant_layout_pattern`
  - 예: `single_column`, `two_column`, `mixed`, `visual_heavy`

## 6.3 `ReferenceBlock`

reference 문서의 최소 구조 단위다.

### 필수 필드

- `block_id`
  - 블록 고유 ID

- `page`
  - 블록이 속한 페이지 번호

- `type`
  - 블록 유형
  - 허용 값:
    - `document_title`
    - `section_heading`
    - `subheading`
    - `paragraph`
    - `image`
    - `caption`
    - `table`
    - `chart`
    - `header`
    - `footer`
    - `advertisement`
    - `decoration`
    - `unknown`

- `bbox`
  - `[x1, y1, x2, y2]`
  - 원본 페이지 좌표 기준

- `reading_order`
  - 문서 전체 또는 페이지 내 읽기 순서 인덱스

- `text`
  - OCR 또는 구조 추출로 확보한 텍스트
  - 비텍스트 블록은 빈 문자열 허용

- `tags`
  - 추가 속성
  - 예: `repeating`, `non_content`, `meta`, `lead`, `sidebar`

- `confidence`
  - 블록 검출/분류 품질 신뢰도

### 권장 필드

- `parent_block_id`
  - 이미지와 캡션 관계처럼 계층 연결이 필요한 경우 사용

- `role_hint`
  - 예: `main_story`, `sidebar`, `market_box`

- `is_supported`
  - 현재 MVP에서 생성 대상으로 직접 사용할 수 있는지 여부

## 6.4 `SectionOrderItem`

문서 골격 생성에 필요한 섹션 수준 요약이다.

### 필수 필드

- `section_id`
  - 섹션 ID

- `title_block_id`
  - 해당 섹션의 대표 제목 블록 ID

- `start_reading_order`
  - 섹션 시작 위치

- `end_reading_order`
  - 섹션 종료 위치

- `page_span`
  - 섹션이 걸친 페이지 범위

- `section_type`
  - 예: `lead_story`, `secondary_story`, `sidebar`, `feature`

### 의미

- generation 단계는 이 정보를 이용해 어떤 순서로 어떤 크기의 섹션을 생성할지 결정한다.
- `section_order`가 부정확하면 reference 구조 모방이 거의 불가능하다.

## 6.5 `StyleTokens`

reference 문서를 픽셀 복제하지 않고 구조적으로 재현하기 위한 규칙 묶음이다.

### 필수 필드

- `column_count`
  - 문서 또는 페이지 평균 컬럼 수

- `title_scale`
  - 제목 크기의 상대 배율

- `subtitle_scale`
  - 부제목 크기의 상대 배율

- `body_scale`
  - 본문 기준 크기

- `line_height_ratio`
  - 본문 line-height 비율

- `paragraph_spacing`
  - 문단 간 간격

- `section_spacing`
  - 섹션 간 간격

- `body_width_ratio`
  - 본문 너비 비율

- `image_alignment_rules`
  - 이미지 정렬 규칙
  - 예: `left_wrap`, `center_full`, `right_column`, `column_span`

- `caption_style`
  - 캡션 크기와 위치 규칙

- `visual_density`
  - 텍스트 대비 시각 요소 밀도

### 의미

- renderer가 직접 해석하는 규칙 집합이다.
- 절대 위치보다 상대 규칙으로 표현해야 문서군 일반화가 가능하다.

## 6.6 `ImageSlot`

이미지가 들어갈 수 있는 위치와 크기 패턴을 요약한다.

### 필수 필드

- `slot_id`
  - 이미지 슬롯 ID

- `page`
  - 슬롯이 속한 페이지

- `bbox`
  - 슬롯 bounding box

- `relative_width`
  - 페이지 또는 컬럼 대비 상대 폭

- `relative_height`
  - 페이지 대비 상대 높이

- `alignment`
  - `left`, `center`, `right`, `full_width`, `column_span`

- `caption_expected`
  - 캡션이 기대되는 구조인지 여부

- `adjacent_text_flow`
  - 텍스트와의 흐름 관계
  - 예: `wrap`, `stacked`, `isolated`

### 의미

- generation 단계가 이미지 수와 위치를 결정할 때 사용하는 핵심 구조다.
- source 이미지가 없어도 slot 정보는 유지되어야 한다.

## 6.7 `UnsupportedBlock`

MVP에서 인식은 하지만 직접 생성 대상으로 삼지 않는 블록을 기록한다.

### 필수 필드

- `block_id`
  - 원래 블록 ID

- `page`
  - 페이지 번호

- `type`
  - 예: `table`, `advertisement`, `complex_chart`, `ornament`

- `reason`
  - 예: `mvp_excluded_table`, `non_content_visual`, `layout_noise`

- `impact`
  - downstream 영향
  - 예: `ignore`, `reserve_space_only`, `manual_review`

- `confidence`
  - unsupported 판정 신뢰도

### 의미

- review 단계에서 특정 시각 요소가 왜 결과에 반영되지 않았는지 설명할 수 있게 해 준다.

## 6.8 `ConfidenceSummary`

reference 파싱 결과의 품질 요약이다.

### 필수 필드

- `overall_confidence`
  - 전체 신뢰도

- `layout_confidence`
  - 블록 검출과 구조 신뢰도

- `reading_order_confidence`
  - 읽기 순서 신뢰도

- `classification_confidence`
  - 블록 타입 분류 신뢰도

- `warnings`
  - 주요 경고 목록
  - 예: `column_ambiguity`, `caption_pairing_failed`, `repeating_header_uncertain`

### 의미

- confidence가 낮을수록 generation에 그대로 쓰지 않고 manual review 비중을 높여야 한다.

## 8. 필수/권장/선택 규칙

### 필수

- `ReferenceTemplate.template_id`
- `ReferenceTemplate.page_specs`
- `ReferenceTemplate.blocks`
- `ReferenceTemplate.section_order`
- `ReferenceTemplate.style_tokens`
- `ReferenceTemplate.confidence_summary`
- `ReferenceBlock.block_id`
- `ReferenceBlock.type`
- `ReferenceBlock.bbox`
- `ReferenceBlock.reading_order`

### 권장

- `image_slots`
- `unsupported_blocks`
- `repeating_elements`

### 선택

- raw OCR 전체 텍스트
- parser 내부 중간 결과

### 기본 규칙

- 필수 필드가 없으면 reference parsing 실패로 간주한다.
- 권장 필드가 없으면 downstream이 fallback 규칙을 사용한다.
- 선택 필드는 디버깅과 비교 실험용이다.

## 9. 필드별 downstream 사용처

- `section_order`
  - content brief와 draft generation에서 사용

- `style_tokens`
  - renderer가 사용

- `image_slots`
  - generation과 renderer가 사용

- `unsupported_blocks`
  - review 단계와 fallback 정책에서 사용

- `confidence_summary`
  - job gating과 manual review 판단에 사용

- `blocks`
  - 디버깅, 후처리, 템플릿 품질 분석에 사용

## 10. 중요한 판단 포인트

reference 단계에서는 다음 판단이 중요하다.

- 광고를 생성 대상이 아니라 비본문 시각 블록으로 분리할 것인지
- 차트를 이미지 슬롯으로 볼지 정보 블록으로 분리할지
- 캡션을 이미지에 종속된 블록으로 연결할지
- 반복 헤더와 푸터를 스타일 신호로 볼지 노이즈로 제거할지
- 컬럼이 불명확할 때 읽기 순서를 어떻게 보수 처리할지

현재 기본 방침은 아래와 같다.

- 광고는 `advertisement` 또는 `unsupported_block`
- 차트는 MVP에서는 `chart` 또는 `image_like`로 분리만 수행
- 캡션은 가능하면 `parent_block_id`로 이미지에 연결
- 헤더와 푸터는 생성 대상에서 제외
- 컬럼이 불명확하면 `1` 또는 `mixed`로 보수 처리

## 11. 실패 유형과 fallback 규칙

### 대표 실패 유형

- 블록 누락
- 블록 오분류
- 읽기 순서 오류
- 광고/장식 오인식
- 표/차트가 본문으로 합쳐짐
- 이미지와 캡션 분리 실패
- 컬럼 병합 오류
- 헤더/푸터가 본문으로 유입

### fallback 규칙

- `section_order` 추출 실패
  - 제목 계층과 reading_order로 단순 골격 재구성

- `image_slots` 추출 실패
  - 큰 `image` block의 bbox에서 유추

- `caption` 연결 실패
  - caption을 독립 블록으로 유지

- `column_count` 불명확
  - `1` 또는 `mixed`로 보수 처리

- `unsupported_blocks` 누락
  - `table`, `advertisement`, `chart` 후보를 기본 unsupported 후보로 재판정

## 12. 평가 기준

reference parsing 품질은 OCR CER만으로 판단하면 안 된다. 구조 보존도가 더 중요하다.

### 주요 평가 지표

- block detection completeness
- block type precision
- reading order accuracy
- section hierarchy accuracy
- image-caption pairing accuracy
- unsupported block leakage rate
- template stability across pages

## 13. 테스트 시나리오

- 신문형 2단 이상 레이아웃에서 `column_count`와 `reading_order`가 추출되는지 확인
- 상단 배너와 페이지 번호가 `header/footer` 또는 `non_content`로 분류되는지 확인
- 큰 광고가 `advertisement` 또는 `unsupported_block`으로 남는지 확인
- 이미지와 짧은 캡션이 `parent_block_id`로 연결되는지 확인
- 차트가 `chart` 또는 `image_like`로 분리되는지 확인
- 표가 `unsupported_blocks`에 기록되는지 확인
- low-confidence reference에서 `confidence_summary.warnings`가 채워지는지 확인

## 14. 결론

reference parsing 단계의 성공은 `텍스트를 얼마나 많이 읽었는가`가 아니라 `문서 구조를 얼마나 안정적으로 템플릿화했는가`로 판단해야 한다.  
MVP가 글과 이미지 중심이더라도, reference 단계는 표, 광고, 차트, 반복 요소를 최소한 식별하고 기록해야 downstream이 안전하게 동작한다.

따라서 이 단계의 공식 계약은 OCR 전문이 아니라 `reference_template.json`이며, 이후 generation과 rendering은 이 구조화 출력만 신뢰하도록 설계하는 것이 맞다.
