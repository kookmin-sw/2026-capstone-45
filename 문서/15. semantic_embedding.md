# Semantic Embedding Design

## 1. 목적

이 문서는 `llm-to-document/llm2doc/tool_search_source_document.py`의 검색을 `semantic_overlay.json` 기반의 semantic-aware retrieval로 확장하기 위한 구현 설계를 정리한다.

현재 검색기는 OCR block의 원문 텍스트만 임베딩해서 검색한다. 이 방식은 구현이 단순하고 fallback이 쉽지만, block의 역할 정보가 검색에 반영되지 않는 한계가 있다. 그 결과 제목, 요약, 본문, 메타데이터, 면책문이 모두 비슷한 수준의 plain text로 취급되고, OCR 노이즈가 심한 block은 검색에서 불리해질 수 있다.

이번 설계의 목표는 `generated_role_name`와 `section_purpose`를 문서 측 임베딩에 반영해서 검색 정렬 품질을 개선하는 것이다. 반면 `generated_role_description`는 v1 범위에서 제외하고, `used_for_generation`은 retrieval relevance 신호로 사용하지 않는다.

## 2. Baseline

현재 baseline은 `docs_v1` 검색이다.

- 인덱싱 입력은 `LayoutAnalyzer`가 추출한 block text이다.
- 인덱싱 대상은 `block.label in {"text", "table"}` 이고, 길이가 너무 짧은 block은 제외한다.
- Chroma metadata에는 `document`, `page`, `block`만 저장한다.
- 검색 시 query text 자체만 임베딩해서 top-k를 찾는다.
- 검색 결과 출력은 block structured HTML만 반환한다.

이 baseline의 장점은 단순성과 안정성이다. 하지만 아래와 같은 한계가 있다.

- semantic role 정보가 검색에 반영되지 않는다.
- OCR 오류가 큰 block은 plain text similarity만으로는 recall이 떨어질 수 있다.
- 제목성 block과 본문 block의 기능 차이가 embedding에 직접 반영되지 않는다.
- metadata, disclaimer, body, summary가 모두 동일한 text channel로만 검색된다.

## 3. 설계 범위와 비범위

이번 문서의 범위는 `llm2doc` 검색 개선에 한정한다.

포함 범위:

- `semantic_overlay.json`를 읽어 검색 인덱싱에 반영하는 방법
- semantic field를 포함한 embedding text 포맷 정의
- Chroma collection version 분리
- rerank 규칙
- artifact 부재 시 fallback 규칙
- 평가 시나리오와 수용 기준

비범위:

- `create_document.py` 프롬프트 개편
- `fetch_source_document`의 응답 형식 변경
- reference pipeline 전체 재설계
- query enrichment 또는 query role classification 도입
- `generated_role_description` 활용

## 4. 입력 계약

### 4.1 기본 입력 소스

검색기는 문서별 semantic artifact가 존재하면 이를 우선 사용한다.

우선 입력:

- `artifacts/<sample-id>/01_reference/semantic_overlay.json`

fallback 입력:

- `LayoutAnalyzer`가 추출한 기존 OCR block

artifact가 없거나, 필요한 필드가 부족하거나, 문서 ID와 artifact 매핑이 불가능하면 기존 `docs_v1` 경로로 fallback한다.

### 4.2 `semantic_overlay.json`에서 사용하는 필드

v1 검색 개선에서 읽는 최소 필드는 아래와 같다.

- `block_id`
- `page`
- `generic_role`
- `domain_role`
- `generated_role_name`
- `section_purpose`
- `role_confidence`
- `semantic_needs_review`

읽을 수는 있지만 검색 점수에 사용하지 않는 필드:

- `used_for_generation`

이번 버전에서는 `generated_role_description`를 읽지 않는다. 이유는 길고 서술적인 설명문이 embedding을 끌고 가면서 query drift를 일으킬 가능성이 있기 때문이다. 먼저 짧고 안정적인 role label과 coarse section intent만으로 효과를 검증한다.

## 5. 인덱싱 레코드 설계

각 indexed block은 최소한 아래 정보를 내부적으로 가져야 한다.

- source text
- embedding text
- retrieval metadata
- display payload

### 5.1 retrieval metadata 스키마

Chroma metadata에는 아래 필드를 저장한다.

- `document`
- `page`
- `block`
- `generic_role`
- `domain_role`
- `generated_role_name`
- `section_purpose`
- `role_confidence`
- `semantic_needs_review`

`used_for_generation`도 diagnostics 용도로 저장할 수 있지만, 검색 점수 계산에는 사용하지 않는다.

### 5.2 block 포함 규칙

이번 설계에서는 `metadata`와 `disclaimer`를 전역 제외하지 않는다. 모든 block type을 동일하게 인덱싱한다. 이유는 이번 retrieval 목표가 "전 블록 균등"이기 때문이다. 날짜, 발행기관, 작성자, compliance 문구 같은 질의에도 대응 가능해야 하므로 특정 role을 전역 필터로 제거하지 않는다.

단, text가 비어 있거나 지나치게 짧은 block은 baseline과 동일하게 제외할 수 있다.

## 6. Embedding Text 포맷

이번 설계의 핵심은 block text 앞에 semantic prefix를 붙여 embedding하는 것이다.

임베딩 원문 포맷은 아래와 같이 고정한다.

```text
document_id: <doc>
page_id: <page>
generic_role: <generic_role or unknown>
domain_role: <domain_role or unknown>
generated_role_name: <generated_role_name or unknown>
section_purpose: <section_purpose or unknown>

content:
<block text>
```

규칙:

- 값이 없으면 `unknown`을 넣는다.
- `generated_role_name`와 `section_purpose`는 짧은 semantic hint로 사용한다.
- 원문 text는 반드시 유지한다. semantic field가 원문을 대체해서는 안 된다.
- query text는 v1에서 변경하지 않는다. 먼저 document-side enrichment만 적용한다.

이 포맷의 목적은 다음과 같다.

- block이 "무슨 역할을 하는지"를 embedding에 반영한다.
- OCR text가 다소 깨져 있어도 role 정보가 보정 신호로 작동하게 한다.
- title, summary, body, metadata 같은 기능 차이를 vector space에 반영한다.

## 7. Collection Versioning

기존 plain-text 검색 컬렉션은 `docs_v1`로 유지한다.

새 semantic-aware 검색 컬렉션은 `docs_v2_semantic`으로 분리한다.

분리 이유:

- baseline과 candidate를 동시에 비교할 수 있어야 한다.
- 기존 캐시를 깨지 않고 rollback이 쉬워야 한다.
- artifact 기반 인덱싱과 OCR fallback 인덱싱을 명확히 구분할 수 있어야 한다.

운영 규칙:

- semantic artifact가 있고 스키마가 유효하면 `docs_v2_semantic`
- artifact가 없거나 읽기 실패하면 `docs_v1` 경로 fallback

## 8. 검색과 재랭킹

검색은 2단계로 설계한다.

### 8.1 1차 retrieval

Chroma에서 semantic-enriched embedding query로 top-k를 찾는다.

규칙:

- query text는 그대로 사용한다.
- 내부 retrieval 후보 수는 `n_results=8`로 늘린다.
- LLM에 반환하는 최종 결과 수는 `3`으로 유지한다.

이 단계의 목적은 semantic prefix가 포함된 문서 벡터 공간에서 넓은 후보군을 먼저 확보하는 것이다.

### 8.2 2차 rerank

1차 후보를 metadata 기반으로 약하게 재정렬한다.

v1 rerank 규칙:

- `role_confidence`는 약한 가점으로 사용한다.
- `semantic_needs_review=true`는 약한 감점으로 사용한다.
- `generic_role`, `domain_role`은 해석 및 선택적 보정용 메타데이터로만 사용한다.
- `used_for_generation`은 rerank에 사용하지 않는다.

이번 버전에서는 `generic_role`을 전역 제외 규칙으로 쓰지 않는다. 또한 query role classifier를 도입하지 않으므로, role 기반 보정은 conservative하게 유지한다.

## 9. 출력 계약

`search_source_document`의 외부 툴 인터페이스는 유지한다.

즉:

- 함수 이름은 그대로 유지한다.
- 입력 파라미터는 `query` 하나만 유지한다.
- 반환 형식은 text 기반 `function_call_output`을 유지한다.

다만 검색 결과 본문에는 기존 식별자 외에 semantic field를 함께 노출한다.

권장 출력 정보:

- `document_id`
- `page_id`
- `block_id`
- `generic_role`
- `generated_role_name`
- `section_purpose`
- structured HTML block

이 변경은 툴 스키마를 깨지 않으면서, LLM이 검색 결과를 해석할 때 block의 기능을 더 잘 파악하게 해준다.

`fetch_source_document`는 이번 설계에서 변경하지 않는다.

## 10. Fallback 정책

semantic artifact는 문서마다 없을 수 있다. 따라서 검색기는 항상 기존 OCR 기반 경로를 유지해야 한다.

fallback 조건:

- 해당 문서의 semantic artifact를 찾지 못함
- JSON 파싱 실패
- 필수 필드 부족
- 문서 ID와 artifact ID 매핑 실패

fallback 동작:

- 기존 `LayoutAnalyzer` block 기반 인덱싱 사용
- 기존 `docs_v1` 컬렉션 사용
- 외부 툴 인터페이스와 호출 흐름은 유지

이 원칙 때문에 `create_document.py`의 tool loop는 수정하지 않아도 된다.

## 11. 평가 계획

baseline은 `docs_v1`, candidate는 `docs_v2_semantic`이다.

평가는 정량 비교와 수동 사례 검토를 함께 한다.

### 11.1 최소 테스트 시나리오

- title 또는 heading 검색
- summary block 검색
- body paragraph 검색
- metadata 검색
- disclaimer 검색
- OCR 품질이 낮은 block 검색

### 11.2 비교 관점

- top-3 relevance
- 같은 query에서 irrelevant hit 비율
- OCR noise가 심한 문서에서 recall 유지 여부
- semantic field가 없는 block을 `unknown` 처리했을 때 안전성

### 11.3 수용 기준

- top-3 relevance가 baseline 이상
- `generated_role_name`와 `section_purpose`가 없는 block도 안전하게 인덱싱 가능
- semantic artifact 부재 시 기존 검색 경로 유지
- `create_document.py`와 tool 호출 계약이 깨지지 않음

## 12. 구현 시 주의사항

- `generated_role_name`와 `section_purpose`는 enrichment field이지 ground truth가 아니다.
- `used_for_generation`은 generation policy 성격이 강하므로 retrieval relevance proxy로 사용하지 않는다.
- `metadata/disclaimer`는 일부 문서에서는 핵심 질의 대상일 수 있으므로 전역 제외하지 않는다.
- `generated_role_description`는 더 풍부한 의미를 주지만, 긴 자연어 설명이 embedding drift를 일으킬 수 있으므로 v1에서는 제외한다.

## 13. 요약

이번 설계의 핵심 결정은 아래와 같다.

- baseline은 `docs_v1` plain-text 검색
- candidate는 `docs_v2_semantic`
- document-side embedding enrichment만 먼저 적용
- `generated_role_name`, `section_purpose`만 v1 semantic field로 사용
- `generated_role_description` 제외
- `used_for_generation`은 diagnostics 전용
- `metadata/disclaimer` 전역 제외 없음
- `search_source_document`의 외부 툴 인터페이스는 유지
