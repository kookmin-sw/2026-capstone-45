# 디버깅 추가

## 문서 목적

이 문서는 `llm2doc`에 추가한 디버그/추적 기능이 어디에 어떻게 구현되었는지, 그리고 구현 과정에서 실제로 어떤 문제가 있었고 어떻게 수정했는지를 설명한다.  
기존 `docs/ai-decision-tracing.md`가 "사용 방법" 중심이라면, 이 문서는 "구현 내부"와 "문제 해결 과정" 중심이다.

---

## 1. 전체 구조

이번 디버그 기능의 핵심 목표는 다음 두 가지였다.

1. 문서 생성 중 AI가 어떤 도구를 어떤 순서로 호출했고, 어떤 근거를 바탕으로 최종 문서를 만들었는지 남기기
2. 실패했을 때 "어느 단계에서 무엇이 잘못되었는지"를 바로 확인할 수 있게 하기

이를 위해 디버그 기능을 아래 네 군데에 나눠 넣었다.

- `llm2doc/debug_trace.py`
  - trace 저장 공통 모듈
- `llm2doc/create_document.py`
  - 기본 문서 생성 파이프라인의 추적/분석/최종문서 추출 로직
- `llm2doc/create_document_ver2.py`
  - ver2 파이프라인에 동일한 추적 구조 적용
- `llm2doc/tool_search_source_document.py`
- `llm2doc/tool_search_source_document_ver2.py`
  - 검색 결과와 재정렬 판단 로그 기록

추적이 켜지면 실행마다 아래 구조가 생성된다.

```text
output/<결과폴더>/trace/<run_id>/
  events.jsonl
  analysis.json
  llm/
    tool_loop_response_1.json
    tool_loop_response_2.json
    ...
    analysis_response.json
    final_document_response.json
  search/
    semantic.jsonl
    contextual_ver2.jsonl
```

## 2. 공통 Trace 모듈 구현

파일: `llm2doc/debug_trace.py`

### 2.1 `resolve_debug_trace()`

`create_document()`와 `create_document_ver2()`에 `debug_trace: bool | None` 인자를 추가했고, 값이 `None`이면 환경변수 `LLM2DOC_DEBUG_TRACE`를 읽도록 했다.

즉 아래 두 방식 모두 가능하다.

- 코드에서 직접 `debug_trace=True` 전달
- 환경변수 `LLM2DOC_DEBUG_TRACE=1`

### 2.2 `DecisionTracer`

`DecisionTracer`는 디버그 추적 전용 객체다.

- `create()`
  - `output_dir/trace/<run_id>` 디렉터리를 만든다.
- `event()`
  - 공통 이벤트를 `events.jsonl`에 한 줄씩 기록한다.
- `save_json()`
  - LLM raw response나 analysis 결과를 JSON 파일로 저장한다.
- `save_text()`
  - 필요시 텍스트 저장
- `append_jsonl()`
  - 검색 로그처럼 여러 건을 누적 기록할 때 사용한다.

이 모듈을 분리한 이유는 다음과 같다.

- 기본 파이프라인과 ver2가 같은 저장 포맷을 쓰게 하기 위해서
- 추적 기능이 꺼져 있을 때 no-op처럼 동작하게 하기 위해서
- 검색기와 문서 생성기가 서로 같은 tracer 객체를 공유하게 하기 위해서

## 3. `create_document.py`의 디버깅 구현

파일: `llm2doc/create_document.py`

### 3.1 검색기 교체

기본 파이프라인은 원래 `ToolSearchSourceDocumentPlain`을 쓰고 있었다.  
이번 작업에서 기본 파이프라인도 판단 로그를 남길 수 있도록 `ToolSearchSourceDocument`로 교체했다.

이 변경 이유는 간단하다.

- `plain` 검색기는 후보 점수/선택 이유 로그가 없다.
- semantic 검색기는 후보 목록, distance, rerank score를 남길 수 있다.

### 3.2 `write_document()`에서 추가한 것

`write_document()`는 실제 도구 호출 루프가 도는 함수다. 여기에 아래 디버그 기능을 넣었다.

- 각 `client.responses.create(...)` 직후 raw response 저장
  - `llm/tool_loop_response_<n>.json`
- tool call 감지 시 `tool_requested` 이벤트 기록
- tool 실행 결과를 `tool_result` 이벤트로 기록
- tool loop가 끝난 뒤 `analysis.json` 생성 단계 추가
- 마지막에 최종 문서만 다시 받는 `request_final_document()` 단계 추가

즉 기존에는:

1. tool loop
2. 모델 마지막 응답을 그대로 최종 출력으로 사용

이 구조였는데, 지금은:

1. tool loop
2. 분석 JSON 생성
3. 최종 문서만 별도 요청

으로 바뀌었다.

이렇게 분리한 이유는 "판단 근거"와 "최종 산출물"을 따로 저장하기 위해서다.

### 3.3 `analysis.json` 생성

`maybe_generate_analysis()`는 tool loop 종료 뒤 별도의 LLM 호출을 한 번 더 해서 구조화된 분석 JSON을 생성한다.

고정 스키마는 다음과 같다.

- `query`
- `user_intent`
- `source_documents`
- `evidence`
- `writing_strategy`
- `uncertainties`

그리고 `normalize_analysis_payload()`에서 모델 응답이 조금 흔들려도 스키마를 강제로 맞춘다.

예를 들어:

- `page_id`가 문자열이어도 정수로 보정
- `used`가 `"true"`여도 boolean으로 보정
- 일부 source document가 빠져도 기본값으로 채움

이 보정 단계가 필요한 이유는 모델이 strict JSON을 요청받아도 실제 응답이 완벽하게 고정되지 않기 때문이다.

### 3.4 최종 문서 추출 강화

`request_final_document()`와 `extract_document_block()`는 최종 `<document>`를 안정적으로 얻기 위한 함수다.

여기서 추가한 내용은 다음과 같다.

- 최종 문서 전용 프롬프트를 별도로 사용
- `<document>`가 없으면 재시도
- 재시도 횟수 증가
- 각 시도의 최종 응답을 `debug_write_output.txt`에 즉시 저장
- `<document>`가 없더라도 아래 경우를 복구
  - JSON 안에 문서 문자열이 들어 있는 경우
  - `<page>`만 있고 `<document>` 래퍼가 없는 경우
  - BeautifulSoup로 파싱 가능한 `<document>` 태그가 있는 경우

즉 원래는 "정확히 `<document>...</document>`가 있어야만 성공"이었는데, 지금은 모델 응답 흔들림을 흡수하는 방향으로 바뀌었다.

### 3.5 실행 단위 이벤트 기록

`create_document()`에는 실행 수준 이벤트를 넣었다.

- `run_started`
- `layout_loaded`
- `render_started`
- `render_completed`
- `run_failed`
- `run_completed`

덕분에 `events.jsonl`만 봐도:

- 실행이 시작됐는지
- 레이아웃 로드까지는 갔는지
- tool loop를 돌았는지
- final document 단계에서 죽었는지
- 렌더링까지 완료됐는지

를 순서대로 볼 수 있다.

## 4. `create_document_ver2.py`의 디버깅 구현

파일: `llm2doc/create_document_ver2.py`

ver2는 기본 파이프라인과 구조는 비슷하지만 검색기가 `ToolSearchSourceDocumentVer2`라는 점이 다르다.  
이번 작업에서는 ver2에도 기본 파이프라인과 동일한 디버그 골격을 맞췄다.

적용한 내용은 다음과 같다.

- `debug_trace` 인자 추가
- `DecisionTracer` 생성 및 전달
- tool loop raw response 저장
- `tool_requested`, `tool_result` 이벤트 기록
- `analysis.json` 생성
- `request_final_document()` 공통 함수 재사용
- `run_started`, `layout_loaded`, `run_failed`, `run_completed` 이벤트 추가

즉 ver2도 이제 base와 같은 방식으로 trace를 남긴다.

## 5. 검색기 디버깅 구현

### 5.1 `tool_search_source_document.py`

기본 semantic 검색기의 목표는 "어떤 후보를 가져왔고, 왜 top-3로 남았는지"를 보이게 하는 것이다.

`search/semantic.jsonl`에는 한 query마다 다음이 기록된다.

- `query`
- `docs`
- `retrieval_mode`
- `fallback_error`
- `candidates`
- `selected`

`candidates` 내부에는 후보별로 아래가 들어간다.

- `record_id`
- `document`
- `page`
- `display_block_id`
- `distance`
- `generic_role`
- `generated_role_name`
- `section_purpose`
- `role_confidence`
- `semantic_needs_review`
- `rerank_score`
- `selected`

또한 `events.jsonl`에는 `search.semantic / search_selected` 이벤트를 남긴다.

### 5.2 `tool_search_source_document_ver2.py`

ver2 검색기는 block/window/section 단위가 섞이기 때문에, 단순 top-k만 남기면 왜 선택됐는지 이해하기 어렵다.  
그래서 `search/contextual_ver2.jsonl`에 더 자세한 구조를 남긴다.

- `retrieved_candidates`
  - 후보 레코드 단위 정보
- `grouped_anchors`
  - 같은 anchor를 묶은 뒤 어떤 kind가 남았는지
- `selected`
  - 최종 top-3

후보별로 기록하는 주요 필드는 다음과 같다.

- `record_kind`
- `anchor_key`
- `distance`
- `lexical_score`
- `metadata_score`
- `role_confidence`
- `multi_kind_count`
- `final_score`
- `selected`

anchor 그룹 기준으로는 다음이 기록된다.

- `best_record`
- `matched_kinds`
- `best_distance`
- `multi_kind_count`
- `final_score`
- `selected`

이 로그를 보면:

- 왜 block보다 window가 선택됐는지
- 같은 블록에서 여러 문맥 후보가 있었는지
- 최종 순위가 lexical overlap 때문인지, metadata 때문인지

를 추적할 수 있다.

## 6. 테스트 구현

파일: `tests/test_debug_trace.py`

이번 작업에서는 tracer 모듈이 최소한 안전하게 동작하는지 확인하기 위해 `unittest` 기반 테스트를 추가했다.

검증 대상은 다음 다섯 가지다.

- 환경변수 true 처리
- 환경변수 미설정 시 기본 false 처리
- enabled tracer가 run 디렉터리 생성하는지
- JSON / JSONL 저장이 되는지
- disabled tracer가 실제 파일을 만들지 않는지

이 테스트는 "모델 호출"을 검증하는 테스트가 아니라, trace 인프라 자체를 검증하는 테스트다.

## 7. 구현 중 실제로 발생한 문제와 수정 내용

### 7.1 `PROMPT_ANALYZE_TRACE.format()`에서 `KeyError('document_id')`

문제:

- analysis 프롬프트 안에 JSON 예시를 `{document_id: ...}` 형태로 써두었는데
- Python의 `str.format()`이 이것을 포맷 슬롯으로 해석했다.
- 그 결과 실제 실행 중 `KeyError('document_id')`가 발생했다.

원인:

- 프롬프트 안의 중괄호를 escape하지 않았기 때문이다.

수정:

- JSON 예시의 `{ ... }`를 `{{ ... }}`로 바꿨다.

의미:

- 이 문제는 디버그 기능 자체보다 "디버그용 analysis 단계"에서만 발생한 버그였다.

### 7.2 최종 문서 단계에서 `<document>`를 못 찾는 문제

문제:

- 모델이 실제로는 거의 완성된 문서를 만들었는데도
- `<document>` 래퍼가 없거나
- reasoning + message 구조 때문에 텍스트를 잘못 읽거나
- `<page>`만 반환하는 경우가 있었다.

기존 문제점:

- 응답을 너무 엄격하게 `<document>...</document>` 하나로만 판단했다.
- 실패 시 마지막 raw 응답이 충분히 남지 않아 원인 확인이 느렸다.

수정:

- `extract_response_text()`를 추가해 Response 객체에서 실제 `output_text`만 안정적으로 추출
- `extract_document_block()`에서 아래 경우를 복구
  - JSON 내부 문자열
  - BeautifulSoup로 파싱 가능한 `<document>`
  - `<page>`만 있는 응답
- `request_final_document()`에서 재시도 횟수 증가
- 각 시도마다 `debug_write_output.txt`를 즉시 갱신
- `final_document_invalid` 이벤트 추가

효과:

- "문서는 사실상 생성됐는데 포맷이 조금 어긋나서 실패"하는 케이스를 상당수 흡수할 수 있게 됐다.

### 7.3 임베딩 질의 실패: `No embedding data received`

문제:

- Chroma가 질의 시 embedding endpoint를 호출했는데
- 응답이 비어 있거나 파서가 기대한 형태가 아니어서
- `ValueError: No embedding data received`가 발생했다.

영향:

- 검색 단계에서 전체 파이프라인이 중단됐다.
- 특히 ver2 검색기에서 이 문제가 더 자주 드러났다.

수정:

- 두 검색기 모두 semantic query 실패 시 로컬 lexical fallback을 추가했다.
- `retrieval_mode`를 `semantic` 또는 `fallback`으로 기록한다.
- 실패 예외 메시지는 `fallback_error`로 로그에 남긴다.
- `query_fallback` 이벤트도 추가했다.

효과:

- 외부 embedding 응답 품질이 흔들려도 검색 단계 자체는 계속 진행할 수 있게 됐다.
- 나중에 로그를 보면 "이 query는 semantic으로 처리됐는지, fallback으로 처리됐는지"를 바로 알 수 있다.

### 7.4 테스트용 임시 디렉터리 권한 문제

문제:

- 처음에는 `tempfile.mkdtemp()` 기반 임시 디렉터리를 테스트에 썼다.
- 그런데 이 환경에서는 해당 디렉터리 아래에 다시 `trace/` 폴더를 만들 때 `PermissionError`가 발생했다.

원인:

- tracer 로직 문제가 아니라, 테스트용 임시 디렉터리 생성 방식/권한 문제였다.

수정:

- 테스트에서 직접 `build/test_debug_trace/case-...` 폴더를 만들도록 바꿨다.

효과:

- tracer 로직을 정상적으로 검증할 수 있게 됐다.

### 7.5 외부 네트워크 불안정

문제:

- 실제 smoke run 중 `APIConnectionError`, `getaddrinfo failed` 같은 네트워크 오류가 간헐적으로 발생했다.

의미:

- 이것은 코드 버그라기보다 외부 응답 API/네트워크 환경 이슈다.
- 다만 이전에는 "왜 실패했는지"가 모호했는데, 지금은 trace에 어디까지 진행됐는지가 남는다.

현재 상태:

- 코드에서 검색 fallback은 넣었지만
- 최종 LLM 호출 단계의 외부 네트워크 실패 자체를 완전히 제거한 것은 아니다.

## 8. 실제로 얻은 디버깅 이점

이번 추가 이후에는 실패했을 때 아래 순서로 확인하면 된다.

1. `events.jsonl`
   - 어느 단계에서 멈췄는지 확인
2. `llm/tool_loop_response_<n>.json`
   - tool loop 중 모델이 무엇을 요구했는지 확인
3. `search/semantic.jsonl` 또는 `search/contextual_ver2.jsonl`
   - 검색 후보와 최종 선택 비교
4. `analysis.json`
   - 모델이 어떤 근거를 사용했다고 스스로 정리했는지 확인
5. `llm/final_document_response.json` + `debug_write_output.txt`
   - 최종 문서 포맷 실패 원인 확인

즉 이전에는 "모델이 왜 저렇게 썼는지"가 거의 보이지 않았는데, 지금은 다음이 보인다.

- 무엇을 검색했는가
- 어떤 후보가 나왔는가
- 무엇이 선택됐는가
- 어떤 근거를 썼다고 정리했는가
- 최종 문서 응답이 왜 실패했는가

## 9. 정리

이번 디버그 추가의 핵심은 단순히 로그를 늘린 것이 아니다.  
문서 생성 과정을 다음 네 단계로 분해해 각각의 상태를 저장하도록 바꾼 것이다.

1. 실행 시작/레이아웃 로드
