# Financial Report v1 Semantic Mapping Spec

## 1. 목적

이 문서는 `financial report v1`에 대해, layout parsing 결과를 semantic interpretation 결과로 변환하는 규칙 초안이다.  
입력은 현재 저장소의 Paddle 결과 형식과 유사한 block list를 가정한다.  
출력은 이후 `reference_template.json`에 결합될 semantic layer다.

이 문서의 목표는 다음과 같다.

- 어떤 block이 어떤 semantic role로 해석되어야 하는지 정의한다.
- role 판별에 사용하는 신호와 우선순위를 정한다.
- ambiguity와 fallback 처리 기준을 정한다.
- v1 구현 범위를 `financial report`에 맞춰 실현 가능한 수준으로 좁힌다.

## 2. 범위

### 2.1 포함 범위

- 한국어 증권/기업 financial report
- 첫 페이지 또는 첫 분석 페이지 중심 레이아웃
- Paddle layout parsing 결과를 입력으로 한 semantic mapping
- 좌측 정보 패널 + 우측 메인 본문 구조
- 보고서 메타, 투자 의견, 수치 패널, 본문 thesis, 근거 블록, analyst 정보, disclaimer 분리

### 2.2 제외 범위

- 표 구조 재복원
- 차트 내용 해석
- 문장 의미 요약
- 다국어 일반화
- 법률/의료 등 다른 문서군 최적화

## 3. 입력 가정

입력 block은 아래 필드를 가진다고 가정한다.

- `block_id`
- `block_label`
- `block_content`
- `block_bbox`
- `block_order`
- `page`

현재 Paddle 예시에서 사용 가능한 주요 `block_label`은 다음과 같다.

- `doc_title`
- `paragraph_title`
- `text`
- `table`
- `chart`
- `image`

현재 샘플 기준 참고 파일:

- `OCR_results/paddle/Paddle_json/financial1/original-00_res.json`
- `OCR_results/paddle/Paddle_json/financial2/original-00_res.json`

## 4. 출력 가정

각 block은 semantic 해석 후 아래 필드를 가진다.

- `block_id`
- `block_type`
- `generic_role`
- `domain_role`
- `role_confidence`
- `section_id`
- `section_purpose`
- `used_for_generation`
- `notes`

여기서 의미는 아래와 같다.

- `block_type`
  - layout parser가 판정한 시각적 타입
- `generic_role`
  - 문서군 공통 역할
- `domain_role`
  - financial report 전용 역할
- `role_confidence`
  - semantic role 추정 신뢰도
- `section_id`
  - 같은 semantic section에 속하는 block 묶음 ID
- `section_purpose`
  - 해당 section의 역할
- `used_for_generation`
  - v1 generation에서 직접 사용하는지 여부

## 7. 판별 전에 해야 하는 전처리

semantic mapping은 raw block을 그대로 받아 바로 role을 붙이면 안 된다. 먼저 아래 전처리를 수행한다.

### 7.1 읽기 순서 보정

- `block_order`가 있는 block은 우선 신뢰한다.
- `block_order`가 없는 `table`, `chart`, `image`는 bbox 기준으로 보정한다.
- 기본 정렬은 `page -> y1 -> x1`.
- 단, 동일 row에 여러 block이 있을 때는 column position을 먼저 본다.

### 7.2 페이지 영역 분할

페이지를 최소한 아래 영역으로 나눈다.

- `top_meta_zone`
- `left_sidebar_zone`
- `main_body_zone`
- `bottom_meta_zone`

기본 휴리스틱:

- `left_sidebar_zone`: `x2 <= 0.32 * page_width`
- `main_body_zone`: `x1 >= 0.32 * page_width`
- `top_meta_zone`: `y2 <= 0.12 * page_height`
- `bottom_meta_zone`: `y1 >= 0.88 * page_height`

이 값은 초기값이며 샘플셋 기반으로 조정 가능하다.

### 7.3 인접 관계 생성

각 block에 대해 아래 관계를 만든다.

- 바로 위 제목 block
- 바로 아래 본문 block
- 동일 열 내 이전/다음 block
- 가까운 표/차트/이미지 block

semantic role은 단일 block이 아니라 주변 block 문맥과 함께 판정한다.

## 8. semantic mapping의 기본 원칙

### 8.1 보수적 해석 우선

- domain role이 확실하지 않으면 generic role까지만 부여한다.
- generic role도 불명확하면 `unknown`.

### 8.2 규칙은 위치 + 키워드 + label 조합으로 판정

- 텍스트만으로 role을 정하지 않는다.
- 위치만으로도 정하지 않는다.
- 최소 2개 이상의 신호가 맞을 때 domain role을 부여한다.

### 8.3 본문과 메타는 강하게 분리

- analyst 정보
- 발간일
- 리서치센터
- disclaimer

는 본문으로 들어가면 안 된다.

### 8.4 표와 차트는 v1에서 evidence 중심

- v1에서는 `table`과 `chart`를 생성 대상으로 직접 사용하지 않는다.
- 대신 `evidence` 또는 `unsupported_evidence`로 역할을 남긴다.

## 9. role mapping 규칙

아래 규칙은 위에서 아래로 우선 적용한다.

## 9.1 `report_header_meta`

### 대상

- `text`

### 판별 조건

- `top_meta_zone`에 위치
- 발간일, 보고서 유형, 기업분석, Report 같은 메타 키워드 포함

### generic/domain

- `generic_role = metadata`
- `domain_role = report_header_meta`

### generation 사용

- `used_for_generation = true`

## 9.2 `report_title`

### 대상

- `doc_title`

### 판별 조건

- `main_body_zone`에 위치
- 회사명, 종목코드, 핵심 제목이 함께 포함된 큰 제목 블록

### generic/domain

- `generic_role = main_title`
- `domain_role = report_title`

### generation 사용

- `used_for_generation = true`

## 9.3 `investment_opinion_box`

### 대상

- `text`

### 판별 조건

- `left_sidebar_zone`에 위치
- `BUY`, `SELL`, `HOLD`, `유지`, `상향`, `하향` 등의 키워드 포함
- 또는 목표주가, 현재주가, 상승여력 관련 텍스트

### generic/domain

- `generic_role = summary`
- `domain_role = investment_opinion_box`

### generation 사용

- `used_for_generation = true`

### 메모

`price_target_box`, `current_price_box`는 이 group 내부 세부 subtype으로 notes에 남길 수 있다.

## 9.4 `key_data_box`

### 대상

- `paragraph_title`
- 그 다음에 오는 `table`

### 판별 조건

- 제목이 `Key Data`
- `left_sidebar_zone`에 위치
- 바로 아래 table이 존재

### generic/domain

- 제목 block: `generic_role = section_heading`, `domain_role = key_data_box`
- table block: `generic_role = evidence`, `domain_role = key_data_box`

### generation 사용

- 제목 block: `true`
- table block: `false`

## 9.5 `consensus_box`

### 대상

- `paragraph_title`
- 그 다음에 오는 `table`

### 판별 조건

- 제목이 `Consensus Data`
- `left_sidebar_zone`에 위치
- 바로 아래 table이 존재

### generic/domain

- 제목 block: `generic_role = section_heading`, `domain_role = consensus_box`
- table block: `generic_role = evidence`, `domain_role = consensus_box`

### generation 사용

- 제목 block: `true`
- table block: `false`

## 9.6 `price_chart_block`

### 대상

- `paragraph_title`
- `chart`

### 판별 조건

- 제목이 `Stock Price`
- 인접한 chart block 존재
- `left_sidebar_zone`에 위치

### generic/domain

- 제목 block: `generic_role = section_heading`, `domain_role = price_chart_block`
- chart block: `generic_role = evidence`, `domain_role = price_chart_block`

### generation 사용

- 제목 block: `true`
- chart block: `false`

## 9.7 `financial_table_block`

### 대상

- `table`

### 판별 조건

- `main_body_zone` 또는 하단 메인 영역에 위치
- 표 제목이나 table content에 `Financial Data`, `매출액`, `영업이익`, `EPS`, `BPS`, `ROE`, `PER` 같은 재무 키워드 포함

### generic/domain

- `generic_role = evidence`
- `domain_role = financial_table_block`

### generation 사용

- `used_for_generation = false`

## 9.8 `thesis_heading`

### 대상

- `paragraph_title`

### 판별 조건

- `main_body_zone`에 위치
- `report_title` 이후 등장
- 메인 본문 문단의 바로 위에 위치
- `Key Data`, `Consensus Data`, `Stock Price` 같은 좌측 패널 제목은 제외

### generic/domain

- `generic_role = section_heading`
- `domain_role = thesis_heading`

### generation 사용

- `used_for_generation = true`

## 9.9 `supporting_argument`

### 대상

- `text`

### 판별 조건

- `main_body_zone`에 위치
- `thesis_heading` 아래에 위치
- analyst/disclaimer/meta 키워드가 아님

### generic/domain

- `generic_role = body`
- `domain_role = supporting_argument`

### generation 사용

- `used_for_generation = true`

## 9.10 `analyst_info`

### 대상

- `text`

### 판별 조건

- 이름, 이메일, `Analyst`, `RA` 포함
- 보통 `bottom_meta_zone` 또는 sidebar 하단에 위치

### generic/domain

- `generic_role = author_info`
- `domain_role = analyst_info`

### generation 사용

- `used_for_generation = true`

## 9.11 `research_center_meta`

### 대상

- `text`

### 판별 조건

- `리서치센터`, `Research Center`, 증권사 부서명 포함
- analyst 정보와 인접

### generic/domain

- `generic_role = metadata`
- `domain_role = research_center_meta`

### generation 사용

- `used_for_generation = true`

## 9.12 `disclaimer_block`

### 대상

- `text`

### 판별 조건

- 페이지 하단에 위치
- 고지, 면책, 중요 내용, 투자판단 관련 주의 문구 포함
- 글자 크기가 작고 길이가 길 가능성 큼

### generic/domain

- `generic_role = disclaimer`
- `domain_role = disclaimer_block`

### generation 사용

- `used_for_generation = true`

## 9.13 `unsupported_evidence`

### 대상

- `image`
- `chart`
- `table`

### 판별 조건

- 위 규칙 어느 쪽에도 안정적으로 속하지 않음
- 그러나 명백히 근거/시각 자료로 보임

### generic/domain

- `generic_role = evidence` 또는 `visual`
- `domain_role = unsupported_evidence`

### generation 사용

- `used_for_generation = false`

## 9.14 `unknown`

### 대상

- 모든 block

### 판별 조건

- 어떤 role 규칙에도 충분히 맞지 않음

### generic/domain

- `generic_role = unknown`
- `domain_role = null`

### generation 사용

- `used_for_generation = false`

## 10. section grouping 규칙

semantic block만으로는 부족하므로 section 단위 grouping이 필요하다.

### 10.1 좌측 패널 그룹

다음은 각각 별도 section으로 묶는다.

- `investment_opinion_box`
- `key_data_box`
- `consensus_box`
- `price_chart_block`

### 10.2 메인 본문 그룹

아래 흐름으로 section을 만든다.

- `report_title`
- `thesis_heading`
- 그 아래 `supporting_argument`

새 `thesis_heading`이 나오면 새 section을 시작한다.

### 10.3 메타 그룹

아래는 본문 section과 분리한다.

- `report_header_meta`
- `analyst_info`
- `research_center_meta`
- `disclaimer_block`

## 11. confidence 규칙

초기 v1은 단순 점수 기반으로 시작한다.

### 11.1 high confidence

아래 3개 이상이 동시에 맞을 때:

- 기대 block_label 일치
- 기대 zone 일치
- 강한 키워드 일치
- 기대 인접 block 존재

### 11.2 medium confidence

아래 2개가 맞을 때:

- label
- zone
- keyword
- adjacency

### 11.3 low confidence

- 1개만 맞거나
- 서로 충돌하는 신호가 있는 경우

### 처리 원칙

- `high`: domain role 부여
- `medium`: domain role 부여 가능, notes에 ambiguity 기록
- `low`: generic role까지만 부여하거나 `unknown`

## 12. 우선순위 충돌 해결

하나의 block이 여러 규칙에 동시에 걸리면 아래 우선순위를 따른다.

1. `disclaimer_block`
2. `analyst_info`
3. `research_center_meta`
4. `report_header_meta`
5. `investment_opinion_box`
6. `key_data_box`
7. `consensus_box`
8. `price_chart_block`
9. `financial_table_block`
10. `report_title`
11. `thesis_heading`
12. `supporting_argument`
13. `unsupported_evidence`
14. `unknown`

이 우선순위는 `메타/주의문구의 본문 오염 방지`를 최우선에 둔 것이다.

## 13. 샘플 기준 기대 해석

### 13.1 financial2

`financial2/original-00_res.json` 기준으로 기대되는 해석은 아래와 같다.

- 좌측 상단 `BUY`, 목표주가, 현재주가 -> `investment_opinion_box`
- `Key Data` 제목 + 아래 표 -> `key_data_box`
- `Consensus Data` 제목 + 아래 표 -> `consensus_box`
- `Stock Price` + chart -> `price_chart_block`
- 우측 `KMW (032500) ...` -> `report_title`
- 우측 각 `paragraph_title` -> `thesis_heading`
- 그 아래 큰 `text` 문단들 -> `supporting_argument`
- 하단 analyst/RA 이메일 -> `analyst_info`
- `하나증권 리서치센터` -> `research_center_meta`

### 13.2 financial1

`financial1/original-00_res.json` 기준으로 기대되는 해석은 아래와 같다.

- 좌측 상단 회사명/브랜드 -> `report_header_meta` 또는 `non_content_visual`
- `Market Insight`, 날짜 -> `report_header_meta`
- AI 감수 안내문 -> `report_header_meta` 또는 `disclaimer_block`
- `AI 데일리 글로벌 마켓 브리핑...` -> `report_title`
- `Summary` -> `thesis_heading` 또는 `summary heading`
- 그 아래 본문 -> `supporting_argument`

## 14. 테스트 시나리오

- `BUY`, 목표주가, 현재주가가 동일 group으로 묶이는지
- `Key Data`, `Consensus Data` 표가 evidence 계열로 들어가는지
- `Stock Price` chart가 본문이 아니라 evidence로 분리되는지
- 메인 `paragraph_title`들이 본문 thesis heading으로 분류되는지
- analyst/email이 본문이 아니라 author_info로 빠지는지
- `리서치센터`가 body가 아니라 metadata로 빠지는지
- 표가 직접 생성 대상이 아니라 `used_for_generation = false`로 남는지
- 애매한 block이 `unknown` 또는 generic fallback으로 가는지

## 15. 구현 순서

1. Paddle block을 내부 공통 스키마로 정규화
2. zone 계산
3. reading order 보정
4. rule-based role assignment
5. section grouping
6. confidence 계산
7. `reference_template.json` semantic layer 결합

## 16. v1 이후 확장 메모

v1 이후에는 아래를 확장한다.

- `summary`와 `supporting_argument` 구분 정밀화
- 표 제목과 표 body 관계 해석
- chart subtype 구분
- multi-page section linking
- financial 외 문서군 domain role 확장

## 17. 결론

v1 semantic mapping의 목표는 financial report를 완전히 이해하는 것이 아니라, 문서 생성에 필요한 수준에서 블록의 역할과 섹션 흐름을 안정적으로 복원하는 것이다.

따라서 초기 구현은 `보수적 role assignment`, `명확한 fallback`, `규칙 기반 해석`, `evidence/metadata 분리`에 집중하는 것이 맞다.
