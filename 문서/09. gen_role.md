# Generated Role Design

이 문서는 현재의 `generic_role` / `domain_role` 기반 closed-set semantic classification을, `Qwen이 역할 이름 자체를 생성하는 open-role 방식`으로 바꾸기 위한 설계 문서다. 목표는 단순히 enum을 제거하는 것이 아니라, 문서 내 block의 기능을 더 자연스럽게 설명하는 role ontology를 모델이 유도하게 만들고, 시스템은 그 결과를 안정적으로 통합해 downstream에서 계속 사용할 수 있게 하는 것이다.

이 문서는 구현 지시 문서로 읽히도록 작성한다. 따라서 `왜 바꾸는지`보다 `무엇을 어떻게 바꾸는지`, `Qwen이 어떤 입력으로 무엇을 생성하는지`, `OCR label을 어떻게 활용하는지`, `문서 내부 통합과 fallback을 어떻게 둘지`를 중심으로 정리한다.

## 1. 목표와 비목표

### 1.1 목표

- `generic_role`, `domain_role` enum 선택을 Qwen의 자유 생성형 role 추론으로 대체한다.
- Qwen이 block마다 `role_name`, `role_description`, `parent_role_name`, `role_confidence`, `needs_review`를 직접 생성하게 한다.
- 생성된 role을 문서 내부에서 통합해 `canonical generated role`로 정규화한다.
- 기존 OCR pipeline이 가진 `raw_label`, `engine_raw_labels`, `secondary_raw_label`도 Qwen 입력에 활용한다.
- 기존 downstream 호환을 위해 필요 시 `generated role -> legacy generic/domain` 브리지를 둔다.

### 1.2 비목표

- bbox 정규화, reading order, block fusion 같은 low-level preprocessing을 모델로 대체하지 않는다.
- `section_id` 부여, artifact 저장, 시각화 렌더링 자체를 전부 모델화하지 않는다.
- OCR engine label을 정답으로 취급하지 않는다.
- 처음부터 전 문서군에 동일 강도로 적용하지 않는다.

## 2. 현재 구조의 한계

현재 구현은 [`allm/semantic_schema.py`](/root/Desktop/workspace/ja/allm/allm/semantic_schema.py) 의 `GENERIC_ROLES`, `DOMAIN_ROLES` enum을 기반으로 작동한다. 이 구조는 다음 장점이 있지만, 동시에 한계가 분명하다.

- 장점
- 출력 형식이 안정적이다.
- downstream 로직이 단순하다.
- 테스트 기준을 고정하기 쉽다.

- 한계
- 문서가 기존 taxonomy에 맞지 않으면 역할을 억지로 끼워 넣게 된다.
- `report_title`, `thesis_heading`, `supporting_argument`처럼 이미 정의된 역할만 고를 수 있어, 실제 문서에서 보이는 더 미세한 기능 차이를 표현하지 못한다.
- `article`, `form`, `mixed-layout` 문서로 갈수록 `domain_role`이 금융 리포트 편향 ontology가 된다.
- 모델이 실제로는 더 적절한 설명을 할 수 있어도, enum 제약 때문에 정보가 버려진다.

따라서 vNext의 핵심은 `분류(classification)`에서 `역할 유도(role induction)`로 중심을 옮기는 것이다.

## 3. 핵심 설계 원칙

### 3.1 Qwen은 역할 이름을 생성한다

Qwen은 더 이상 정해진 후보에서 고르지 않는다. 대신 block의 기능을 가장 잘 설명하는 `짧고 재사용 가능한 role_name`을 생성한다.

예시:

- `report_title`
- `headline_thesis`
- `summary_recommendation_panel`
- `analyst_signature_block`
- `multi_period_financial_table`

### 3.2 시스템은 생성 결과를 통합한다

자유 생성만 허용하면 같은 문서 안에서도 다음처럼 drift가 생길 수 있다.

- `headline`
- `main_headline`
- `report_headline`

따라서 block별 생성 뒤에 `document-level role consolidation` 단계를 둬서, 같은 의미의 role을 하나의 canonical role로 통합해야 한다.

### 3.3 OCR label은 semantic answer가 아니라 관측값이다

OCR 결과에 있는 label은 유용하지만, 그 자체가 최종 semantic role은 아니다.

예를 들어:

- paddle의 `paragraph_title`
- dolphin의 `para`
- canonical label의 `paragraph`

이 정보는 block가 heading-like인지 body-like인지 판단하는 강한 단서가 될 수 있다. 그러나 시스템은 `paragraph_title -> 무조건 section heading`처럼 고정 규칙으로 쓰지 않는다. OCR label은 `Qwen이 추론할 때 참고하는 관측값`으로만 제공한다.

### 3.4 deterministic safety/fallback은 유지한다

자유 생성형이어도 다음은 deterministic하게 남긴다.

- low page quality gating
- backend error fallback
- invalid JSON / schema mismatch 처리
- `analyst_info`, `disclaimer_block`처럼 high-precision safety rule
- 최종 section assignment 및 generation policy 보정

## 4. 목표 아키텍처

```text
canonical blocks + fused OCR signals
-> build block context
-> Qwen open-role proposal per block
-> document/page-level role consolidation
-> canonical generated role assignment
-> optional legacy generic/domain bridge
-> deterministic post-processing
-> diagnostics / artifact export
```

이 흐름에서 `generic_role`, `domain_role`은 더 이상 primary output이 아니다. 필요하면 `bridge output`으로만 유지한다.

## 5. Qwen이 직접 생성할 것

Qwen은 block마다 아래 필드를 직접 생성한다.

- `role_name`
- `role_description`
- `parent_role_name`
- `role_level`
- `role_confidence`
- `needs_review`
- `reason`

권장 의미는 다음과 같다.

- `role_name`
- 문서 내에서 재사용 가능한 짧은 snake_case 이름

- `role_description`
- 그 block의 기능을 한 문장으로 설명

- `parent_role_name`
- 선택값. 더 상위 개념이 있을 때만 사용
- 예: `summary_panel`, `metadata`, `narrative`, `evidence_cluster`

- `role_level`
- `document | section | block`
- 기본값은 `block`

### 5.1 권장 출력 예시

```json
{
  "block_id": "blk-0012",
  "role_name": "summary_recommendation_panel",
  "role_description": "A compact summary block presenting the recommendation and key highlighted numbers.",
  "parent_role_name": "summary_panel",
  "role_level": "block",
  "role_confidence": 0.89,
  "needs_review": false,
  "reason": "The block is short, prominent, and grouped with numeric highlights near the top summary area."
}
```

## 6. 입력 설계

Qwen 입력은 `raw text + OCR labels + raw geometry + generic structural context`로 구성한다.

### 6.1 target block

기존 block payload에 아래 정보를 유지한다.

- `text`
- `canonical_label`
- `raw_label`
- `bbox_norm`
- `reading_order`
- `engine_sources`
- `text_quality_score`
- `tags`
- `flags`

추가로 아래를 반드시 포함한다.

- `center_x`
- `center_y`
- `width_ratio`
- `height_ratio`
- `area_ratio`
- `block_index_on_page`
- `total_blocks_on_page`

### 6.2 OCR label payload

각 block에는 아래 `ocr_labels`를 포함한다.

- `primary_raw_label`
- primary engine 기준 raw label

- `current_raw_label`
- 현재 canonical block의 `raw_label`

- `engine_raw_labels`
- 엔진별 raw label 맵
- 예: `{"dolphin": "para", "paddle": "paragraph_title"}`

- `secondary_raw_label`
- secondary matched block의 label이 있으면 보관

이 값은 이미 [`allm/semantic_context.py`](/root/Desktop/workspace/ja/allm/allm/semantic_context.py) 에서 payload로 전달 가능한 형태이므로, open-role 설계에서도 그대로 활용한다.

### 6.3 OCR label 활용 원칙

Qwen 프롬프트에는 다음 원칙을 명시한다.

- OCR label은 block의 low-level type을 설명하는 단서다.
- OCR label이 서로 다를 수 있으므로, 텍스트와 geometry, neighbors를 함께 보고 해석하라.
- `paragraph_title`, `table_caption`, `figure`, `header`, `footer` 같은 label은 semantic role naming의 참고 신호로 사용하라.
- OCR label을 그대로 최종 role name으로 복사하지 말고, 문서 기능을 설명하는 role로 재해석하라.

예시:

- `paragraph_title`는 `section_heading`, `sub_thesis_heading`, `sidebar_panel_title` 중 무엇인지 문맥으로 결정
- `table`은 `financial_projection_table`, `valuation_table`, `supporting_evidence_table` 중 무엇인지 문맥으로 결정

### 6.4 page context

page-level context는 기존 구조를 유지한다.

- `document_family`
- `page_archetype`
- `quality_score`
- `column_count`
- `dominant_layout_pattern`
- `text_area_ratio`
- `table_area_ratio`
- `visual_area_ratio`

### 6.5 local neighbors / structural relations

주변 block과 상대 관계는 계속 핵심 입력으로 둔다.

- 이전/다음 텍스트 block
- nearest heading
- nearest evidence
- `distance_to_nearest_heading`
- `distance_to_nearest_evidence`
- `same_column_as_nearest_heading`
- `same_column_as_previous_text`
- `same_column_as_next_text`
- `overlap_ratio_with_nearest_heading`
- `overlap_ratio_with_nearest_evidence`
- `relative_order_from_nearest_heading`

neighbor에도 동일하게 `ocr_labels`를 포함해, Qwen이 target뿐 아니라 주변 block의 OCR 해석도 함께 참조하게 한다.

## 7. 출력 스키마

### 7.1 block-level proposal

현재 `RoleDecision`와 별도로 아래 구조를 도입한다.

```python
@dataclass
class OpenRoleProposal:
    block_id: str
    role_name: str
    role_description: str
    parent_role_name: Optional[str]
    role_level: Literal["document", "section", "block"]
    role_confidence: float
    needs_review: bool = False
    reason: Optional[str] = None
```

### 7.2 document-level consolidation result

문서 내부 role 통합 뒤에는 아래 구조를 둔다.

```python
@dataclass
class ConsolidatedRole:
    canonical_role_name: str
    canonical_description: str
    parent_role_name: Optional[str]
    member_role_names: List[str]
```

### 7.3 applied assignment

최종적으로 block에는 아래 정보를 기록한다.

- `generated_role_name`
- `generated_role_description`
- `generated_parent_role_name`
- `generated_role_level`
- `generated_role_confidence`
- `generated_role_source`
- `generated_role_needs_review`
- `raw_generated_role_name`

필요 시 legacy 호환용으로 아래를 추가할 수 있다.

- `legacy_generic_role`
- `legacy_domain_role`

## 8. 검증 규칙

자유 생성형이라고 해서 검증을 없애지 않는다. 다만 enum membership 검증 대신 `format validation`으로 바꾼다.

필수 검증:

- `role_name`은 비어 있지 않아야 한다.
- `role_name`은 snake_case를 권장하고, 허용 문자 집합을 제한한다.
- `role_name` 길이는 예를 들어 3자 이상 64자 이하로 제한한다.
- `role_description`은 비어 있지 않아야 한다.
- `role_confidence`는 0~1 범위여야 한다.
- `role_level`은 `document | section | block` 중 하나여야 한다.
- `block_id`는 요청한 block과 일치해야 한다.

실패 시 fallback:

- invalid JSON -> safety fallback 또는 unknown generated role
- schema mismatch -> safety fallback 또는 unknown generated role
- low confidence -> generic generated role only 또는 review flag 강화

## 9. 문서 내부 role consolidation

이 단계는 open-role 설계의 핵심이다.

### 9.1 왜 필요한가

Qwen은 block별로 role을 독립 생성하므로, 같은 문서 안에서 유사 역할이 여러 이름으로 흩어질 수 있다.

예:

- `main_title`
- `report_headline`
- `document_headline`

이 셋은 사실상 하나의 역할일 수 있다. 따라서 consolidation이 필요하다.

### 9.2 방법

초기 버전은 다음 두 단계로 구현한다.

1. rule/lightweight normalization
- 소문자화
- snake_case 정규화
- trivial suffix 제거

2. Qwen consolidation pass
- 문서 전체에서 생성된 role 후보 목록을 모아
- 중복되거나 거의 같은 역할을 묶고
- canonical role 이름을 다시 제안하게 한다

### 9.3 권장 범위

consolidation은 page별보다 document-level이 좋지만, 비용이 크면 page-level로 시작할 수 있다. 초기 구현은 아래 순서를 권장한다.

1. page-level consolidation
2. document-level consolidation
3. cross-document clustering은 추후 과제로 남김

## 10. legacy bridge 설계

기존 시각화, 평가, section builder, template exporter가 `generic_role`, `domain_role`을 기대할 수 있으므로, 전환기에는 bridge가 필요하다.

권장 방식:

- primary truth는 `generated_role_*`
- secondary compatibility output으로 `legacy_generic_role`, `legacy_domain_role` 유지

이 bridge는 두 방식 중 하나로 구현할 수 있다.

1. deterministic bridge
- generated role name과 description을 보고 기존 taxonomy에 느슨하게 매핑

2. Qwen bridge
- consolidation 뒤 canonical generated role을 입력으로 받아 legacy role을 보조 생성

초기에는 deterministic bridge가 더 단순하고 안전하다.

## 11. 프롬프트 설계

프롬프트는 분류형이 아니라 생성형이어야 한다.

핵심 지시:

- 정해진 role 목록에서 고르지 말 것
- 이 block의 문서 내 기능을 가장 잘 설명하는 짧은 role name을 생성할 것
- role name은 가능한 한 문서 전체에서 재사용 가능해야 함
- OCR label은 low-level hint로만 사용하고, 그대로 복사하지 말 것
- text, neighbors, geometry, page context를 함께 보고 판단할 것
- 확신이 낮으면 `needs_review=true`로 두고 description은 더 명확히 쓸 것

금지 지시:

- `para`, `paragraph_title`, `table` 같은 OCR raw label을 그대로 role name으로 내는 것
- role name을 지나치게 장황한 자연어 문장으로 만드는 것
- block마다 같은 의미를 다른 이름으로 반복 생성하는 것

## 13. 단계적 전환 계획

### Phase 1

- `generated_role_name`, `role_description`만 shadow로 생성
- 기존 `generic_role`, `domain_role`은 그대로 유지
- artifact와 trace에만 generated role을 기록

### Phase 2

- page-level consolidation 도입
- generated role을 primary semantic output으로 저장
- legacy bridge를 추가해 기존 UI/평가와 공존

### Phase 3

- section builder와 visualizer가 generated role을 1순위로 사용
- generic/domain은 compatibility layer로만 유지

### Phase 4

- enum taxonomy 의존 코드를 점진 제거
- 필요 시 cross-document role catalog를 별도 관리

## 14. 테스트 계획

단위 테스트:

- open-role parser가 형식 검증만 수행하는지
- `role_name` snake_case / 길이 제한이 동작하는지
- `ocr_labels`가 target과 neighbors에 모두 포함되는지
- low confidence / invalid JSON / schema mismatch fallback이 유지되는지

통합 테스트:

- 동일 문서에서 유사 role이 consolidation으로 합쳐지는지
- paddle/dolphin label이 다를 때도 안정적으로 canonical generated role이 나오는지
- `article`과 `financial_report` 모두에서 generated role이 생성되는지

회귀 테스트:

- 기존 `reference_visualization.html`이 generated role 필드가 있어도 깨지지 않는지
- legacy bridge를 켠 상태에서 기존 template export가 유지되는지

## 15. Acceptance Criteria

- Qwen이 더 이상 `GENERIC_ROLES`, `DOMAIN_ROLES` enum 중 하나를 고르지 않는다.
- block마다 `role_name`과 `role_description`이 생성된다.
- OCR label 정보가 target과 neighbors 모두에서 prompt에 제공된다.
- 동일 문서의 유사 role은 consolidation 뒤 canonical role로 통합된다.
- low quality / invalid response / low confidence 상황에서 fallback이 유지된다.
- 필요 시 legacy generic/domain bridge를 통해 기존 downstream을 점진 이전할 수 있다.

## 16. 요약

이 설계의 핵심은 `closed taxonomy classification`을 `open role generation + consolidation`으로 바꾸는 것이다.

- Qwen은 역할 이름을 직접 생성한다.
- OCR label은 중요한 입력이지만 정답이 아니라 관측값으로만 쓴다.
- 시스템은 생성 결과를 문서 내부에서 통합한다.
- 기존 `generic_role`, `domain_role`은 필요 시 compatibility bridge로만 유지한다.

즉, 앞으로의 semantic layer는 `정해진 상자에 문서를 끼워 맞추는 구조`가 아니라, `문서가 가진 실제 구조를 모델이 설명하고 시스템이 그것을 안정화하는 구조`로 옮겨간다.
