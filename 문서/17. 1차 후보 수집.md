# 1차 후보 수집

## 문서 목적

이 문서는 `llm-to-document/llm2doc/tool_search_source_document.py`를 **1차 후보 수집 전용 검색기**로 재설계하기 위한 기준 문서다.

이번 문서의 목표는 다음과 같다.

1. 현재 검색기가 맡고 있는 역할을 분해해 책임을 명확히 나눈다.
2. 1차 후보 수집의 입력/출력 명세를 고정한다.
3. semantic retrieval, BM25 retrieval, entity grounding, merge/ordering의 동작 로직을 결정 완료 상태로 정리한다.
4. 후속 단계인 context expansion과 LLM rerank로 자연스럽게 이어질 수 있는 구조를 정의한다.
5. 구현 과정에서 디버깅과 재현이 쉽도록 trace/log 요구사항까지 함께 고정한다.

이 문서는 개념 설명용이 아니라, 다른 개발자가 그대로 구현을 시작할 수 있는 **구현 기준 문서**를 목표로 한다.

---

## 1. 배경과 문제 정의

현재 `tool_search_source_document.py`는 아래 역할을 한 번에 수행하고 있다.

1. semantic 검색
2. 검색 실패 시 local fallback
3. heuristic 기반 재정렬
4. top-3 최종 선택
5. LLM이 바로 읽을 수 있는 HTML-ish 결과 직렬화

이 구조는 다음 문제를 만든다.

- 검색기 내부에서 너무 이른 시점에 top-3가 확정된다.
- `distance + role_confidence + semantic_needs_review`만으로는 recall 확보용 1차 retrieval에 필요한 안정성을 제공하기 어렵다.
- retrieval과 ranking, serialization, debugging이 섞여 있어 원인 분석이 어렵다.
- retrieval 품질이 문제인지, context 부족이 문제인지, LLM 판단이 문제인지 분리가 어렵다.
- `tool_search_source_document_ver2.py`처럼 retrieval과 context expansion이 결합된 구조는 실험 단위를 지나치게 크게 만든다.

따라서 이번 설계에서는 다음을 분리한다.

- **1차 후보 수집**
  - semantic retrieval
  - BM25 retrieval
  - entity grounding
  - merge/dedupe
  - first-stage ordering
- **후속 단계**
  - context expansion
  - LLM rerank
  - fetch 대상 확정

즉 이번 문서의 범위는 “무엇을 볼지 후보를 넓게 모으는 단계”까지다.  
“무엇이 최종 근거인지 고르는 단계”는 후속 단계로 넘긴다.

---

## 2. 목표와 비목표

### 목표

- 1차 후보 수집기는 **정답 선택기**가 아니라 **후속 LLM이 판단할 수 있는 후보 집합 생성기**여야 한다.
- semantic, BM25, entity grounding의 기여가 분리되어 보여야 한다.
- 최종 반환은 top-3가 아니라 `10~12개`의 first-stage candidate bundle이어야 한다.
- 같은 query를 다시 넣었을 때 retrieval 흐름을 trace만으로 재현할 수 있어야 한다.

### 비목표

- 이 문서에서 context expansion까지 구현하지 않는다.
- 이 문서에서 final rerank 프롬프트를 확정하지 않는다.
- `tool_search_source_document_ver2.py`의 block/window/section 구조를 본 설계에 포함하지 않는다.
- `intent_tag` 기반 분기 설계를 사용하지 않는다.
- title/body/table 다양성 보장 규칙을 이번 설계에 넣지 않는다.

---

## 3. 최종 아키텍처

이번 설계의 최종 파이프라인은 아래와 같다.

```text
query preprocessing
  -> LLM query expansion (1회)
  -> semantic retrieval
  -> BM25 retrieval
  -> entity grounding / exact-hit recall 보강
  -> merge / dedupe
  -> first-stage ordering
  -> candidate bundle 반환
  -> (후속 단계) context expansion
  -> (후속 단계) LLM rerank
  -> (후속 단계) fetch 대상 확정
```

### 역할 분리

`tool_search_source_document.py`
- 1차 후보 수집까지 담당
- semantic/BM25/entity 채널 결과를 모아 후보 bundle 반환
- 최종 top-k 확정은 하지 않음

후속 단계
- 문맥 확장
- 실제 의미 판단
- 최종 evidence 선택

이 분리의 의도는 명확하다.

- retrieval은 recall 중심으로 설계
- selection은 의미 판단 중심으로 설계

## 5. 입력 명세

### 5.1 search tool 외부 입력

search tool의 외부 입력은 아래와 같다.

```python
query: str
docs: list[str]
tracer: DecisionTracer | None
```

설명:
- `query`
  - 사용자의 자연어 질의
- `docs`
  - 검색 대상 source document id 목록
- `tracer`
  - 디버깅/추적용 optional context

### 5.2 내부 전처리 입력

전처리 단계가 받는 입력은 아래다.

- 원본 query 문자열
- 현재 검색 대상 `SearchRecord` 집합
- source document id 목록

여기서 `SearchRecord`는 기존 파일의 구조를 유지한다.

### 5.3 LLM query expansion 입력

LLM expansion 단계는 아래 입력을 받는다.

- 원질의
- source document id 목록
- 짧은 instruction prompt
- optional debug/trace handle

중요:
- expansion용 LLM은 최종 답을 쓰는 모델이 아니라, **retrieval helper** 역할만 한다.
- 이 단계는 query preprocessing 직후 **1회만 호출**한다.

### 5.4 BM25 계층 입력

BM25는 외부 검색 계층으로 둔다. BM25 계층이 받는 입력은 아래다.

- `query: str`
- `docs: list[str]`
- `top_k: int`

추가적으로 구현체에 따라 아래가 포함될 수 있다.

- tenant / namespace
- corpus shard id
- timeout

## 6. 출력 명세

## 6.1 QueryProfile

query preprocessing 결과는 `QueryProfile` 형태로 다룬다.

```python
@dataclass(slots=True)
class QueryProfile:
    raw_query: str
    normalized_query: str
    query_tokens: list[str]
    entity_candidates: list[str]
    grounded_entities: list[str]
    semantic_queries: list[str]
    lexical_queries: list[str]
```

필드 설명:

- `raw_query`
  - 원본 질의 문자열
- `normalized_query`
  - trimming, whitespace normalization 정도가 적용된 query
- `query_tokens`
  - tokenization + lowercase + stopword 제거 후 토큰
- `entity_candidates`
  - 패턴 기반으로 뽑은 entity 후보
- `grounded_entities`
  - 코퍼스 기준으로 실제 entity로 보강/확정된 목록
- `semantic_queries`
  - semantic retrieval에 사용할 query 목록
  - 원질의 포함
- `lexical_queries`
  - BM25 retrieval에 사용할 query 목록
  - 원질의 압축 lexical query + LLM lexical query 포함 가능

## 6.2 RetrievalCandidate

1차 후보의 단위는 아래 구조로 다룬다.

```python
@dataclass(slots=True)
class RetrievalCandidate:
    record_id: str
    document_id: str
    page_id: int
    block_id: str
    channels: list[str]
    semantic_distance: float | None
    bm25_score: float | None
    entity_score: float
    role_confidence: float
    semantic_needs_review: bool
    generic_role: str | None
    generated_role_name: str | None
    section_purpose: str | None
    preview_text: str
```

설명:

- `channels`
  - `semantic`, `bm25`, `entity` 중 어떤 채널로 유입됐는지 기록
- `semantic_distance`
  - semantic 채널에서 들어온 경우 best distance
- `bm25_score`
  - BM25 채널에서 들어온 경우 score
- `entity_score`
  - entity exact hit 또는 alias hit 가중치
- `preview_text`
  - 후속 LLM rerank에 넘기기 전 사람이 읽을 수 있는 짧은 요약

## 6.3 search tool 반환 형식

외부 contract는 기존 function-call output 형태를 유지한다.

```python
{
  "type": "function_call_output",
  "call_id": "...",
  "output": "<candidate bundle text>"
}
```

다만 `output` 내용은 더 이상 “top-3 HTML”이 아니다.  
대신 first-stage candidate bundle 요약 텍스트를 반환한다.

권장 직렬화 예:

```text
Candidate result:

Candidate #1
record_id=semantic:financial2:...
document_id=financial2
page_id=1
block_id=page-1-block-15
channels=semantic,entity
semantic_distance=0.31
bm25_score=12.42
preview=KMW (032500) 실적 확인하고 매수하시면 늦습니다

Candidate #2
...
```

최종 반환 후보 수:
- 내부 보유: `10~12개`
- tool output에 표시: 기본적으로 `8~12개`

즉 이 단계는 top-3를 확정하지 않는다.

## 7. 동작 로직

## 7.1 Query preprocessing

### tokenization

토큰화는 아래 패턴을 사용한다.

```python
REGEX_TOKEN = re.compile(r"[0-9A-Za-z가-힣]+")
```

처리 순서:

1. 원질의 trim
2. whitespace normalize
3. regex tokenization
4. lowercase
5. stopword 제거
6. 너무 짧은 토큰 제거
7. 중복 제거

예:

```text
financial2 문서를 기반으로 KMW 기업 분석 보고서 작성해줘
```

결과 예:

```python
["financial2", "kmw", "기업", "분석", "보고서"]
```

### entity candidate 추출

이번 설계에서는 `intent_tag`를 제거하고, 대신 entity 추출만 유지한다.

entity candidate는 다음 패턴에서 뽑는다.

- 영문 대문자 토큰
  - `KMW`, `AT&T`, `EPS`
- 숫자 코드
  - `032500`, `005930`
- 숫자+단위
  - `50,000원`, `176.0십억원`, `5.32%`
- 한글 고유명사형 토큰
  - `삼성전자`, `미래에셋증권`

중요:
- 이 단계에서는 “후보”만 만든다.
- 실제 entity 확정은 grounding 단계에서 수행한다.

### intent_tag 제거

이번 설계에서는 `intent_tag`를 생성하지 않는다.

이유:
- 도메인 다양성이 큰 환경에서 태그 품질이 쉽게 깨진다.
- 1차 후보 수집에서는 semantic/BM25/entity 조합만으로도 충분한 recall 확보가 가능하다.
- 추후 의미 판단은 LLM rerank 단계가 맡는 것이 더 적절하다.

## 7.2 LLM query expansion

### 호출 시점

query preprocessing 직후 **1회만 호출**한다.

### 역할

LLM은 다음을 생성한다.

- semantic retrieval용 자연어 query 1~2개
- BM25 retrieval용 keyword-dense lexical query 1개
- entity 후보 보강용 `entities`

### 출력 형식

JSON only로 제한한다.

예:

```json
{
  "entities": ["KMW", "032500"],
  "semantic_queries": [
    "KMW company analysis and investment thesis",
    "KMW 투자 포인트와 재무 전망"
  ],
  "lexical_queries": [
    "KMW 032500 목표주가 현재주가 매출 영업이익 EPS PER PBR"
  ]
}
```

### 입력 프롬프트 성격

프롬프트는 retrieval helper 용도로 짧고 제한적으로 구성한다.

요구사항:
- 원질의를 바꾸지 않는다.
- query expansion만 생성한다.
- 새로운 entity를 함부로 발명하지 않는다.
- semantic query는 자연어형으로 짧게 생성한다.
- lexical query는 keyword-dense 형태로 생성한다.

### fallback

LLM expansion 실패 시:

- `semantic_queries = [original_query]`
- `lexical_queries = [token 압축 lexical query 1개]`

즉 expansion 실패가 retrieval 실패로 이어지면 안 된다.

---

## 7.3 Semantic retrieval

### 검색 엔진

기존 Chroma 기반 semantic retrieval을 유지한다.

### 입력

semantic retrieval 입력은 아래를 사용한다.

- `original_query`
- `llm semantic queries`

예:

```python
semantic_queries = [
    original_query,
    "KMW company analysis and investment thesis",
    "KMW 투자 포인트와 재무 전망",
]
```

### 검색 방식

각 query에 대해 개별 query를 수행한다.

```python
collection.query(
    query_texts=[semantic_query],
    n_results=4~6,
    where=where_cond,
    include=["metadatas", "distances"],
)
```

### 결과 통합

- query별 top-k 결과를 union
- 동일 `record_id`가 여러 query에서 나오면 best distance만 유지
- semantic 채널 후보로 등록

### 원칙

semantic retrieval은 **recall 확보용**이다.

따라서:
- 이 단계에서 top-3 확정 금지
- best candidate만 남기기보다 넓게 모으는 것이 우선

---

## 7.4 BM25 retrieval

### 왜 BM25를 별도 계층으로 두는가

lexical retrieval은 이번 설계에서 임시 token overlap 점수 함수가 아니라 **외부 BM25 검색 계층**을 전제로 한다.

이유:
- BM25는 lexical retrieval의 전통적 기준선이며 구현 품질이 검증돼 있다.
- retrieval tool 내부에 임시 lexical scorer를 넣으면 실험과 운영 기준이 혼동된다.
- BM25 계층을 분리해야 교체, 비교, 장애 대응, 로깅이 쉬워진다.

### 입력

- `original lexical query`
- `llm lexical query`

예:

```python
lexical_queries = [
    "KMW 재무제표 매출 이익 주가",
    "KMW 032500 목표주가 현재주가 매출 영업이익 EPS PER PBR",
]
```

### BM25 계층 반환값

```python
@dataclass(slots=True)
class BM25Hit:
    record_id: str
    score: float
    matched_terms: list[str]
    document_id: str
    page_id: int
    block_id: str
```

### 앱 내부 처리

앱 내부에서는 다음을 수행한다.

1. BM25 hit 수신
2. `record_id -> SearchRecord` join
3. `RetrievalCandidate`로 변환
4. channel=`bm25` 부여

### 장애 처리

BM25 계층 장애 시:
- search 전체 실패로 만들지 않음
- semantic + entity grounding만으로 계속 진행
- trace에는 BM25 오류를 반드시 남김

## 7.5 Entity grounding

### 목적

entity grounding은 rule-based prior 대체물이 아니라, **exact-hit recall safety net**이다.

즉:
- semantic과 BM25가 놓쳐도
- query에 등장한 핵심 entity가 포함된 record는 후보군에 포함되게 한다.

### grounding 방식

전처리와 LLM expansion에서 얻은 entity 후보를 코퍼스 기준으로 보강한다.

판단 기준:
- exact string match
- alias match
- 반복 출현
- 문서/페이지 집중도

예:
- `KMW`
- `케이엠더블유`
- `032500`

이들이 동일 entity로 묶일 수 있다.

### 원칙

- hard filter 금지
- recall 보강용으로만 사용
- 일반 키워드를 entity로 오인하지 않도록 trace를 남긴다

### 결과

entity grounding으로 추가된 후보는:
- channel=`entity`
- `entity_score > 0`

를 가진다.

---

## 7.6 Merge / dedupe

### 병합 기준

merge 기준은 `record_id`다.

동일 record가:
- semantic
- bm25
- entity

채널 중 여러 곳에서 잡힐 수 있다.

이 경우:
- `channels`를 합치고
- `semantic_distance`, `bm25_score`, `entity_score`를 누적 저장한다.

### 병합 규칙

- semantic distance는 best value 유지
- bm25 score는 best value 유지
- entity score는 exact/alias hit 여부에 따라 갱신
- role metadata는 원본 `SearchRecord` 기준 유지

중요:
- semantic/BM25/entity의 raw hit 정보는 trace에서 각각 유지한다.
- merge는 결과 후보 집합 구성용일 뿐, 채널 기여를 지우면 안 된다.

## 7.7 First-stage ordering

### 목적

first-stage ordering은 **최종 선택**이 아니라 **후속 LLM이 보기 좋은 순서 정렬**이다.

따라서 ordering score는 ranking helper이지, selection 결과가 아니다.

### 예시 신호

- semantic proximity
- bm25 score
- entity score
- role confidence
- semantic_needs_review penalty

예시 ordering score:

```text
first_stage_score =
  semantic_component
  + bm25_component
  + entity_component
  + role_component
  - review_penalty
```

이 식은 구현 예시일 뿐이고, 핵심은 아래 두 가지다.

1. 후보 입력 순서 정렬용이어야 한다.
2. retrieval 단계에서 top-3를 확정하는 용도로 쓰면 안 된다.

### 결과 후보 수

최종 first-stage candidate 수는 `10~12개`로 고정한다.

---

## 8. 외부 BM25 검색 계층 설계

### 인터페이스

외부 BM25 검색 계층은 최소 아래 인터페이스를 제공한다고 가정한다.

```python
class BM25SearchClient(Protocol):
    def search(
        self,
        query: str,
        docs: list[str],
        top_k: int,
    ) -> list[BM25Hit]:
        ...
```

### app 쪽 역할

앱 내부는 아래 역할만 맡는다.

- lexical query 생성
- BM25 adapter 호출
- BM25 결과를 `SearchRecord`와 join
- trace 저장

### 교체 가능성

이 인터페이스를 사용하면 향후:
- 로컬 BM25
- 별도 검색 서비스
- hybrid lexical backend

중 어느 것으로도 교체 가능하다.

## 9. 디버깅 용이성 설계

이번 설계에서 디버깅은 부가 기능이 아니다.  
1차 후보 수집은 채널별 기여가 분리돼 보여야만 운영이 가능하다.

## 9.1 디버깅 기본 원칙

문서에 아래 원칙을 고정한다.

1. retrieval은 채널별 기여가 보여야 한다.
2. raw input/output이 남아야 한다.
3. 최종 후보만이 아니라 intermediate state도 남아야 한다.
4. 동일 query는 재현 가능해야 한다.
5. 실패가 fallback으로 흡수된 경우에도 원인 정보는 보존되어야 한다.

## 9.2 trace 산출물

권장 산출물:

```text
search/
  first_stage.jsonl
  llm_expansion_response.json
  bm25_response_1.json
  bm25_response_2.json
```

또는 run 단위 구조:

```text
trace/<run_id>/
  search/
    first_stage.jsonl
    llm_expansion_response.json
    bm25_response_1.json
```

## 9.3 단계별 로그 요구사항

### Query preprocessing

남길 것:
- 원질의
- normalized query
- query tokens
- entity candidates
- grounded entities

### LLM expansion

남길 것:
- expansion request payload
- raw response
- parsed JSON
- parse 실패 시 fallback reason

### Semantic retrieval

남길 것:
- semantic query list
- query별 top-k hit
- 각 hit의 `record_id`, `distance`, role metadata

### BM25 retrieval

남길 것:
- lexical query list
- query별 BM25 raw hits
- `record_id`, `bm25_score`, `matched_terms`

### Entity grounding

남길 것:
- entity별 exact/alias match 결과
- 어떤 record가 entity 채널로 추가됐는지

### Merge / ordering

남길 것:
- 채널별 hit 수
- 병합 후 후보 수
- candidate별 channels
- ordering score 구성 요소
- 최종 first-stage 10~12개

## 9.4 이벤트 이름

이벤트 이름은 아래로 고정한다.

- `query_preprocessed`
- `llm_expansion_requested`
- `llm_expansion_saved`
- `llm_expansion_failed`
- `semantic_retrieval_completed`
- `bm25_retrieval_completed`
- `bm25_retrieval_failed`
- `entity_grounding_completed`
- `candidate_merge_completed`
- `first_stage_ordered`
- `first_stage_returned`

## 9.5 디버깅 시 확인 순서

운영/개발자는 아래 순서로 trace를 본다.

1. `query_preprocessed`
2. `llm_expansion_saved` 또는 `llm_expansion_failed`
3. `semantic_retrieval_completed`
4. `bm25_retrieval_completed` 또는 `bm25_retrieval_failed`
5. `entity_grounding_completed`
6. `candidate_merge_completed`
7. `first_stage_ordered`
8. `first_stage_returned`

이 순서를 따르면:
- preprocessing 문제인지
- expansion 문제인지
- semantic/BM25 중 어느 채널이 문제인지
- merge/ordering 문제가 있는지

를 분리해서 볼 수 있다.

## 9.6 재현 가능성

반드시 기록할 항목:

- `run_id`
- source doc ids
- original query
- semantic query list
- lexical query list
- semantic top-k
- bm25 top-k
- candidate limit
- external service error message
- fallback 여부

이 정보가 없으면 동일 run을 다시 재현하기 어렵다.

## 9.7 디버깅 acceptance criteria

아래를 만족해야 한다.

- semantic/BM25/entity 채널 기여를 trace만으로 분리 설명할 수 있음
- LLM expansion 실패 여부와 fallback 적용 여부를 바로 확인 가능
- BM25 장애 시 search가 어디까지 진행됐는지 보임
- 최종 후보 1개마다 어떤 채널로 들어왔는지 보임
- 같은 query를 다시 넣었을 때 어떤 후보가 달라졌는지 비교 가능

## 10. 구현 단계와 개발 순서

이번 문서 기준 구현 순서는 아래로 고정한다.

## 단계 1. 구조 정리

- `tool_search_source_document.py`에서 기존 “semantic + fallback + top-3 선택” 흐름 분리
- `SearchRecord` 기반 인덱싱과 collection load/populate는 유지
- `SEARCH_RESULT_LIMIT = 3` 제거
- `FIRST_STAGE_LIMIT = 12` 추가

## 단계 2. query preprocessing 모듈화

추가 함수 예:

- `_preprocess_query(...)`
- `_extract_query_tokens(...)`
- `_extract_entity_candidates(...)`
- `_ground_entities(...)`

결과를 `QueryProfile`로 묶는다.

## 단계 3. LLM expansion 추가

- query preprocessing 직후 1회 호출
- JSON parse / fallback 구현
- semantic/lexical query arrays 생성
- raw request/response trace 저장

## 단계 4. semantic retrieval 채널화

- query expansion 적용
- multi-query union
- `RetrievalCandidate` 생성
- semantic hit trace 저장

## 단계 5. BM25 adapter 연동

- 외부 BM25 client interface 정의
- lexical query 입력
- BM25 hit를 candidate로 변환
- BM25 raw hit trace 저장

## 단계 6. entity grounding 채널 추가

- exact-hit recall 보강
- alias/grounding 결과를 candidate channel로 추가
- grounding trace 저장

## 단계 7. merge / ordering

- `record_id` 기준 dedupe
- first-stage ordering 구현
- `10~12개` 후보 반환
- merge/ordering trace 저장

## 단계 8. tool output / trace 개편

- `_format_search_output(...)`를 candidate bundle 포맷으로 교체
- search trace schema 확장
- semantic/BM25/entity 기여가 분리되어 보이도록 저장

## 단계 9. create_document 연계 검토

- search tool 반환 포맷 변경이 `create_document.py`의 tool loop에 미치는 영향 정리
- 이 문서에서는 구현 세부 코드를 바꾸는 것이 아니라, 후속 연계 포인트만 정의

---

## 11. 개발 중 체크포인트

- semantic-only 상태에서도 candidate recall이 유지되는가
- BM25 추가 후 numeric/table/entity 후보가 실제로 올라오는가
- entity grounding이 일반 키워드를 entity로 오인하지 않는가
- 최종 반환이 top-3가 아니라 10~12 candidates인가
- trace에서 semantic/BM25/entity 채널이 분리되어 보이는가
- LLM expansion 실패 시 fallback이 trace와 함께 확인되는가

---

## 12. 테스트 및 acceptance criteria

## 12.1 테스트 시나리오

- 장황한 보고서 작성 질의
- 숫자/주가/매출 중심 질의
- 회사명 + 티커 질의
- entity가 문서에 없는 질의
- BM25 계층 실패 시나리오
- LLM expansion 실패 시나리오
- trace 비교 시나리오
  - 같은 query 두 run 비교
  - semantic만 성공 / BM25 실패
  - expansion 성공 / expansion 실패

## 12.2 acceptance criteria

- 원질의는 항상 retrieval 입력에 포함됨
- LLM expansion 실패 시에도 검색이 동작함
- BM25 failure 시 semantic + entity만으로 진행 가능
- search tool은 first-stage candidates 10~12개를 반환함
- final selection은 search tool 내부에서 수행하지 않음
- trace 로그만으로 semantic/BM25/entity 기여를 분리해 설명할 수 있음
- expansion / BM25 / merge 단계의 입력과 출력이 각각 남아 재현이 가능함

## 13. 기본 가정

- 기준 구현 대상은 `tool_search_source_document.py`다.
- `tool_search_source_document_ver2.py`는 이번 설계 범위에서 제외한다.
- `intent_tag`는 사용하지 않는다.
- 확장 질의는 query preprocessing 직후 LLM 1회 호출로 생성한다.
- BM25는 외부 검색 계층으로 설계한다.
- repo 내부 임시 lexical scorer는 기준안이 아니라 fallback 또는 초기 실험용으로만 간주한다.
- 디버깅 가능성은 선택 사항이 아니라 1차 후보 수집 구조의 필수 요구사항이다.

## 14. 구현자용 요약

이번 문서의 핵심 결론은 다음 세 줄로 요약할 수 있다.

1. `tool_search_source_document.py`는 더 이상 top-3 선택기가 아니라 **first-stage candidate collector**다.
2. semantic, BM25, entity grounding은 각각 독립된 채널로 수집하고, merge 후 `10~12개` 후보를 반환한다.
3. trace만으로 “왜 이 후보가 들어왔고, 왜 다른 후보가 빠졌는지”를 설명할 수 있어야 한다.
