# Qwen Semantic Role Migration Design v2

이 문서는 `financial_report` 문서군의 semantic role assignment를 기존 rule-first 구조에서 `Qwen-first` 구조로 이전하기 위한 v2 설계 문서다. 목적은 단순히 Qwen 호출 비중을 늘리는 것이 아니라, `semantic meaning`의 최종 판정 책임을 rule에서 모델로 옮기고, rule은 `관측값 추출`과 `high-precision safety fallback`으로 축소하는 것이다.

이 문서는 기존 `semantic_qwen_design.md`를 대체하는 구현 지시 문서로 읽히도록 작성한다. 따라서 배경 설명보다 `무엇을 바꾸는지`, `Qwen이 무엇을 어떻게 추론하는지`, `어떤 입력을 주고 어떤 입력을 주지 않는지`, `fallback과 검증을 어떻게 둘지`를 명시한다.

## 1. 목표와 비목표

### 1.1 목표

- `financial_report`에서 block semantic 판단의 주도권을 rule에서 Qwen으로 이전한다.
- Qwen이 block마다 `generic_role`, `domain_role`, `role_confidence`, `needs_review`를 직접 추론하도록 한다.
- rule은 semantic classifier가 아니라 `safety rule`과 `fallback source`로 축소한다.
- Qwen 입력은 `family-specific lexical hints`가 아니라 `raw text + raw geometry + generic structural context` 중심으로 재구성한다.
- 현재 `rule | qwen | shadow` 모드는 유지하되, `qwen` 모드의 의미를 `Qwen-first with safety fallback`으로 재정의한다.

### 1.2 비목표

- `archetypes.py`의 page archetype 분류를 Qwen으로 바꾸지 않는다.
- `document_family` 판정은 현재 규칙 기반 로직을 유지한다.
- `section_id` 부여, `used_for_generation` 결정, 최종 section 연결은 모델에 맡기지 않는다.
- `article`, `form` 등 다른 문서군까지 이번 설계 범위에 넣지 않는다.
- family-specific lexical hint를 새로 늘리거나 유지하지 않는다.

## 2. 현재 구조의 문제

현재 구현에서 semantic role의 실질적 owner는 Qwen이 아니라 `semantic_rules.py`다.

- `compute_financial_rule_decisions()`가 block의 의미를 대부분 먼저 확정한다.
- `semantic.py`는 이 결과를 baseline과 fallback으로 동시에 사용한다.
- `qwen` 모드에서도 Qwen은 `rule classifier` 위에서 보정하는 역할에 가깝다.
- `shadow` 모드 역시 rule output을 중심으로 비교하도록 짜여 있다.

이 구조의 문제는 다음과 같다.

- rule이 `관측값 추출`과 `semantic decision`을 동시에 담당해 책임이 섞여 있다.
- Qwen이 구조적 단서를 충분히 받지 못하고, 대신 rule이 이미 내린 결론에 끌려간다.
- `report_title_seen` 같은 순차 상태가 rule 내부에만 있어, 제목/섹션 전환을 모델이 직접 학습하거나 추론하기 어렵다.
- `left_sidebar_zone`, `top_meta_zone` 같은 템플릿성 위치 규칙이 semantic 판단의 중심이 되면, 형식이 다른 문서에서 쉽게 깨진다.

따라서 v2의 핵심은 `rule이 답을 정하는 구조`를 끊고, `rule은 관측 또는 안전장치만 담당`하게 만드는 것이다.

## 3. 설계 원칙

### 3.1 의미 판단은 모델이 한다

rule은 semantic label을 직접 정하지 않는다.  
Qwen이 block의 역할을 직접 분류한다.

### 3.2 관측값과 semantic decision을 분리한다

다음은 허용되는 전처리다.

- bbox 정규화
- reading order 계산
- 이웃 block 수집
- heading / evidence와의 상대 거리 계산
- block 크기, 중심점, 상대 순번 계산

다음은 지양해야 하는 semantic rule이다.

- `왼쪽에 있고 BUY가 있으면 investment_opinion_box`
- `상단에 있고 ticker가 있으면 report_title`
- `stock price`가 있으면 무조건 price chart block

즉, 전처리는 `측정`까지만 하고, 그 측정값이 무엇을 의미하는지는 모델이 판단한다.

### 3.3 family-specific hints는 넣지 않는다

다음과 같은 feature는 v2에서 제거한다.

- `has_ticker`
- `has_investment_opinion_keywords`
- `has_key_data_phrase`
- `has_consensus_phrase`
- `has_stock_price_phrase`
- `has_disclaimer_keywords`

이 값들은 결국 `금융 리포트 양식에 맞춘 rule-derived semantic hint`가 되기 쉽고, 다른 형식으로 일반화되기 어렵다. Qwen은 원문 텍스트를 직접 읽을 수 있으므로, 이런 lexical hint는 입력의 필수 구성요소로 두지 않는다.

### 3.4 raw/relative feature를 우선한다

위치 정보는 `left_sidebar_zone` 같은 hard semantic zone보다 아래 같은 값으로 전달한다.

- `bbox_norm`
- `center_x`, `center_y`
- `width_ratio`, `height_ratio`
- `block_index_on_page`
- `total_blocks_on_page`
- `distance_to_nearest_heading`
- `distance_to_nearest_evidence`
- `same_column_as_nearest_heading`
- `same_column_as_previous_text`

이 값들은 특정 금융 템플릿을 가정하지 않고도 문서 구조를 표현할 수 있다.

### 3.5 deterministic post-processing은 유지한다

모델은 `semantic role classification`까지만 맡는다.  
다음은 기존처럼 deterministic post-processing으로 유지한다.

- `section_purpose` 정규화
- `used_for_generation` 정규화
- `section_id` assignment
- `needs_review`의 최종 보강

## 4. Qwen이 직접 추론할 것

Qwen은 block마다 아래 필드를 직접 추론한다.

- `generic_role`
- `domain_role`
- `role_confidence`
- `needs_review`
- `reason`

현재 `RoleDecision` 스키마와의 호환성을 위해 `section_purpose`, `used_for_generation` 필드는 유지할 수 있다. 다만 v2에서는 이 둘을 모델의 authoritative output으로 보지 않는다. 최종값은 후처리 단계에서 role 기반 규칙으로 다시 정한다.

### 4.1 Qwen이 owner가 될 역할

아래 역할은 Qwen이 직접 판정한다.

- `report_title`
- `thesis_heading`
- `supporting_argument`
- `investment_opinion_box`
- `key_data_box`
- `consensus_box`
- `price_chart_block`
- `financial_table_block`
- `unsupported_evidence`

이 역할들은 위치, 주변 문맥, heading-evidence 관계, 문단 길이, 페이지 전개 순서를 함께 봐야 하므로 모델이 맡는 편이 적합하다.

### 4.2 rule에 남길 safety 역할

아래 역할은 high-precision safety rule로 남긴다.

- `analyst_info`
- `disclaimer_block`
- `report_header_meta`
- `research_center_meta`

남기는 기준은 다음과 같다.

- 정규식이나 매우 강한 exact cue로 안정적으로 잡을 수 있다.
- semantic ambiguity가 낮다.
- 잘못 분류될 경우 downstream 영향이 크다.

## 5. Qwen 입력 설계

v2에서 Qwen 입력은 `raw text + raw geometry + generic structural context`로 구성한다.  
입력은 문서군 특화 semantic hint를 포함하지 않는다.

### 5.1 target block

각 block에 대해 아래 값을 전달한다.

- `block_id`
- `text`
- `canonical_label`
- `raw_label`
- `bbox_norm`
- `reading_order`
- `text_quality_score`
- `engine_sources`
- `tags`
- `flags`

추가로 아래 raw/relative geometry를 명시적으로 넣는다.

- `center_x`
- `center_y`
- `width_ratio`
- `height_ratio`
- `area_ratio`
- `block_index_on_page`
- `total_blocks_on_page`

### 5.2 page context

현재 `archetypes.py`가 만든 page-level feature는 유지한다.

- `document_family`
- `page_archetype`
- `quality_score`
- `column_count`
- `dominant_layout_pattern`
- `text_area_ratio`
- `table_area_ratio`
- `visual_area_ratio`

이 값들은 semantic label이 아니라 페이지의 전반적 구조를 설명하는 컨텍스트다.

### 5.3 local neighbors

주변 블록은 semantic 판단의 핵심 입력으로 유지한다.

- 이전 텍스트 2개
- 다음 텍스트 2개
- 가장 가까운 heading 1개
- 가장 가까운 evidence 2개

각 neighbor는 아래 정보를 가진다.

- `relation`
- `block_id`
- `canonical_label`
- `raw_label`
- `text`
- `bbox_norm`
- `reading_order`
- `center_x`
- `center_y`
- `width_ratio`
- `height_ratio`

### 5.4 structural relations

Qwen이 heading-evidence 연결이나 narrative 흐름을 더 잘 읽도록, 다음과 같은 generic structural relation 값을 추가한다.

- `distance_to_nearest_heading`
- `distance_to_nearest_evidence`
- `same_column_as_nearest_heading`
- `same_column_as_previous_text`
- `same_column_as_next_text`
- `overlap_ratio_with_nearest_heading`
- `overlap_ratio_with_nearest_evidence`
- `relative_order_from_nearest_heading`

이 값들은 문서군 특화 rule이 아니라 block 간 관계를 일반적인 형태로 요약한 것이다.

### 5.5 금지할 입력

다음 입력은 v2 payload에서 제외한다.

- `has_ticker`
- `has_email`
- `has_disclaimer_keywords`
- `has_investment_opinion_keywords`
- `has_key_data_phrase`
- `has_consensus_phrase`
- `has_stock_price_phrase`
- `rule_hints`
- `suggested_role`
- `candidate_role_from_rule`

즉, Qwen은 `rule이 생각한 답`이나 `family-specific keyword flag` 없이 원문과 구조만 보고 판단해야 한다.

## 6. 추론 방식

Qwen은 block마다 JSON 하나를 반환한다.  
프롬프트는 다음 원칙을 강제한다.

- 허용된 enum만 사용
- reasoning text를 JSON 밖으로 출력하지 않음
- 불확실하면 `unknown` 또는 `domain_role = null`
- metadata, disclaimer, author, evidence, narrative를 분리
- target block을 주변 context와 상대 위치를 함께 보고 판단

모델이 실제로 하게 될 추론은 다음과 같은 종류다.

- 이 block이 본문 narrative인가, meta/disclaimer인가
- heading이라면 main title 계열인지 subsection/thesis 계열인지
- table/image/chart라면 독립 evidence인지 특정 heading에 종속된 evidence인지
- 짧은 텍스트 블록이 페이지 전개상 제목 역할인지, 요약 패널인지, 일반 메타인지

중요한 점은, 이러한 판단을 hardcoded role rule로 하지 않고 `raw text + relative layout + neighbors`의 조합으로 모델이 수행하도록 하는 것이다.

## 7. 적용 및 fallback 정책

### 7.1 mode 의미 재정의

- `rule`: safety rules만 적용하는 legacy-safe 모드
- `qwen`: Qwen-first + safety fallback
- `shadow`: Qwen 결과를 기록하되 최종 적용은 rule/safety 경로 유지

### 7.2 decision flow

`qwen` 모드의 block decision flow는 아래로 고정한다.

1. safety rule 존재 여부 확인
2. Qwen eligible block이면 context payload 생성
3. Qwen 추론 수행
4. schema validation
5. confidence gating
6. accepted 시 Qwen 결과 적용
7. 실패 시 safety rule 적용
8. safety rule도 없으면 `unknown` fallback 적용
9. deterministic post-processing 수행

### 7.3 fallback 우선순위

- `low_page_quality`: safety rule, 없으면 unknown
- `backend_error`: safety rule, 없으면 unknown
- `invalid_json`: safety rule, 없으면 unknown
- `schema_mismatch`: safety rule, 없으면 unknown
- `low_role_confidence`: safety rule, 없으면 unknown

`confidence_generic_only_threshold <= confidence < confidence_accept_threshold`인 경우는 fallback하지 않고, `domain_role = null`, `needs_review = True`로 낮춰 적용한다.

## 8. 모듈별 변경 설계

### 8.1 `allm/archetypes.py`

- 유지
- page-level 라우팅 및 quality gating feature 제공
- semantic owner가 아님

### 8.2 `allm/semantic_rules.py`

현재의 full rule classifier를 해체한다.

남길 함수:

```python
def compute_financial_safety_rules(
    page: FusedPage,
    analysis: PageAnalysis,
) -> dict[str, RoleDecision]:
    ...
```

이 함수는 아래만 반환한다.

- `analyst_info`
- `disclaimer_block`
- `report_header_meta`
- `research_center_meta`

제거할 책임:

- 보고서 제목 판정
- thesis heading 판정
- body/supporting argument 판정
- key data / consensus / stock price panel 판정
- evidence 연결 판정

### 8.3 `allm/semantic_context.py`

`build_block_context()`는 safety decision을 참고할 수는 있지만, rule-derived semantic hint를 payload에 넣지 않는다.

새 payload 설계 방향:

- `target_block`
- `page_context`
- `local_neighbors`
- `structural_relations`
- `allowed_roles`

삭제할 항목:

- `rule_hints`
- family-specific lexical hint
- semantic label suggestion

### 8.4 `allm/semantic_engine.py`

현재 `rule_decision` 인자를 `fallback_decision` 개념으로 바꾼다.

```python
def resolve_block_decision(
    *,
    payload: BlockContextPayload,
    fallback_decision: RoleDecision,
    config: SemanticConfig,
    backend: Optional[QwenTransformersBackend] = None,
) -> tuple[RoleDecision, Optional[RoleTraceEntry], Optional[str], bool]:
    ...
```

동작 원칙:

- accepted Qwen result가 있으면 그것을 최종 적용
- 실패 시에만 fallback 적용
- fallback이 safety role이 아니고 unknown이면 그대로 unknown 사용

### 8.5 `allm/semantic.py`

오케스트레이션 흐름을 아래로 바꾼다.

- 페이지별 safety rule map 생성
- block별 fallback decision 계산
- block context 생성
- Qwen resolution
- semantic source 기록
- deterministic normalization
- section assignment

`baseline_rule_decision = actual_rule_decision or make_unknown_decision(block)` 같은 full-rule baseline 개념은 제거한다.

### 8.6 `allm/semantic_backends/qwen_transformers.py`

프롬프트를 다음 방향으로 수정한다.

- semantic hint 없이 raw/structural payload만 사용
- heading-evidence 연결과 narrative 흐름을 보라고 명시
- output contract는 enum 기반 JSON 유지
- 불확실하면 generic-only 또는 unknown으로 남기라고 명시

## 9. 단계적 마이그레이션

### Phase 1

- `semantic_rules.py`를 safety-only로 축소
- `semantic_context.py`에 raw/relative structural feature 추가
- 기존 테스트를 깨지 않도록 shadow instrumentation 유지

### Phase 2

아래 역할을 먼저 Qwen-first로 이전한다.

- `supporting_argument`
- `financial_table_block`
- `unsupported_evidence`

이 셋은 구조적 문맥 의존성이 크고, 현재 rule보다 모델 이점이 비교적 크다.

### Phase 3

다음 역할을 이전한다.

- `report_title`
- `thesis_heading`

이 단계에서는 페이지 상단 heading 관계와 narrative 시작점 추론이 핵심이다.

### Phase 4

마지막으로 패널 계열을 이전한다.

- `investment_opinion_box`
- `key_data_box`
- `consensus_box`
- `price_chart_block`

이 단계는 heading과 nearby evidence의 연결 품질이 충분히 확보된 뒤에 수행한다.

## 10. Diagnostics와 artifact 정책

기존 diagnostics 확장은 유지하되 source 의미를 재정의한다.

- `qwen`
- `safety_rule`
- `unknown_fallback`

기존 `rule` 표기는 legacy compatibility 기간에만 남긴다.

`semantic_trace`에는 최소 아래를 남긴다.

- input payload
- raw response
- parsed decision
- applied decision
- fallback reason
- latency

이를 통해 Qwen이 무엇을 보고 어떤 결정을 했는지, fallback이 왜 일어났는지 비교 가능해야 한다.

## 11. 테스트 계획

### 11.1 단위 테스트

- safety rule이 허용된 4개 역할만 반환하는지 검증
- context payload에 family-specific lexical hint가 포함되지 않는지 검증
- context payload에 raw geometry와 structural relation이 포함되는지 검증
- `resolve_block_decision()`이 Qwen-first + safety fallback 정책을 따르는지 검증

### 11.2 회귀 테스트

- `financial2-00` cover summary의 주요 semantic 구조 유지
- `financial3-00`의 narrative section 유지
- `financial3-09`, `financial3-10` evidence page에서 section 미생성 유지
- `financial3-11` compliance page의 disclaimer 검출 유지
- `financial1-00` low quality review 및 fallback 유지

### 11.3 실패 시나리오 테스트

- backend error
- invalid JSON
- schema mismatch
- low page quality
- low role confidence

### 11.4 acceptance criteria

- `qwen` 모드에서 full rule semantic classifier를 baseline으로 사용하지 않는다.
- payload에 family-specific lexical hint가 없다.
- Qwen output이 실패해도 safety rule 또는 unknown으로 안정적으로 수렴한다.
- 현재 CLI 계약은 유지된다.

## 12. 최종 결정 사항

- semantic meaning의 최종 owner는 Qwen이다.
- rule은 `semantic classifier`가 아니라 `safety fallback`이다.
- Qwen 입력은 `raw text + raw geometry + generic structural context`만 사용한다.
- family-specific hints는 넣지 않는다.
- deterministic post-processing은 유지한다.
- migration은 단계적으로 수행한다.

이 문서의 구현이 완료되면, 현재 구조의 핵심 한계였던 `rule이 의미를 먼저 정하고 Qwen이 그 위를 덮는 구조`를 해소하고, 형식이 다른 문서에 대해서도 더 일반화 가능한 semantic pipeline으로 갈 수 있다.
