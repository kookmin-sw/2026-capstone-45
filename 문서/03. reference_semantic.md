# Reference Semantic Interpretation Analysis

## 1. 목적

이 문서는 레퍼런스 문서 해석 과정에서 `semantic interpretation` 단계가 왜 필요한지, 그리고 이 단계를 설계할 때 무엇을 먼저 고민해야 하는지를 정리한 분석 문서다.

여기서 핵심은 `semantic role 목록을 빨리 정하는 것`이 아니다. 더 중요한 것은 아래 질문들에 답하는 것이다.

- layout parsing 이후에 semantic 단계가 왜 별도로 필요한가
- semantic 단계는 어디까지를 목표로 해야 하는가
- 어떤 신호를 근거로 블록의 역할을 판단할 것인가
- 역할 체계를 어떻게 설계해야 나중에 다양한 문서군으로 확장할 수 있는가
- 애매하거나 잘못 해석된 블록이 downstream을 망치지 않게 하려면 무엇을 남겨야 하는가

이 문서는 구현 명세서가 아니라 설계 단계의 사고 틀을 정리하는 문서다.  
즉, 질문은 "무슨 role을 둘까?"보다 "왜 semantic interpretation이 필요한가, 무엇을 먼저 결정해야 하는가, 어떤 순서로 생각해야 하는가?"에 맞춰진다.

## 2. 문제 정의

`layout parsing`은 문서 안에서 `무엇이 어디에 있는가`를 알려준다.  
하지만 문서 생성에 필요한 것은 여기서 한 단계 더 나아간 정보다. 생성기는 각 블록이 `문서 안에서 어떤 역할을 수행하는지` 알아야 한다.

예를 들어 같은 `text` 블록이라도 아래처럼 역할이 다를 수 있다.

- `summary`
- `metadata`
- `body`
- `disclaimer`

같은 `table`도 아래처럼 역할이 달라질 수 있다.

- 핵심 투자지표
- 본문 근거 자료
- 부가 데이터

즉, block type만으로는 충분하지 않다.  
semantic interpretation은 `이 블록이 왜 거기 있는가`와 `문서 전개 안에서 무슨 기능을 하는가`를 해석하는 단계다.

## 3. 왜 이 단계를 별도로 설계해야 하는가

semantic interpretation을 별도 단계로 분리해야 하는 이유는 분명하다.

- 문서 생성은 위치 복제가 아니라 역할 복원이다.
- layout 정보만 있으면 구조는 비슷해도 읽는 흐름이 어색한 문서가 나온다.
- financial report는 특히 `핵심 의견`, `근거`, `메타`, `주의문구`가 분리되어 있어 역할 해석이 중요하다.
- semantic 해석이 없으면 generation은 결국 "왼쪽 박스 하나 + 오른쪽 글 몇 개"를 흉내 내는 수준으로 축소된다.

금융 리포트에서는 같은 위치의 블록이라도 문서군마다 용도가 다르다.  
예를 들어 `BUY`, `목표주가`, `현재주가`는 단순 text가 아니라 `투자 판단 신호`다.  
`Analyst`, `RA`, `리서치센터`는 본문이 아니라 `작성 메타 정보`다.  
`Key Data`, `Consensus Data`, `Stock Price`는 그냥 제목이 아니라 `근거 패널`의 시작점이다.

따라서 semantic 단계는 optional한 장식이 아니라, 실제 문서 생성 품질을 좌우하는 핵심 단계로 봐야 한다.

## 4. 이 단계에서 먼저 고민해야 할 것

이 단계에서 가장 중요한 것은 role 이름보다 설계 경계를 정하는 것이다. 아래 항목들을 순서대로 정리하는 것이 맞다.

### 4.1 semantic interpretation의 범위

먼저 이 단계가 어디까지 해야 하는지 정해야 한다.

고려해야 할 질문:

- 블록 역할 분류까지만 할 것인가
- 섹션 목적까지 볼 것인가
- 문서 전체 흐름까지 볼 것인가
- 텍스트 의미를 깊게 요약할 것인가

현재 v1에서의 기본 결론은 다음이 적절하다.

- `block role`
- `section purpose`
- `content flow`

까지만 다룬다.

즉, v1은 `deep semantic summarization`을 하지 않는다.

이유는 분명하다.

- 너무 깊게 가면 reference parser가 사실상 내용 생성기처럼 변한다.
- 지금 필요한 것은 reference 문서의 `구성 원리`를 복원하는 일이지, 문서 내용을 새로 해석해 재작성하는 일이 아니다.
- v1에서 너무 깊은 의미 이해까지 넣으면 규칙이 불안정해지고 평가도 어려워진다.

### 4.2 semantic role을 어떤 수준에서 정의할 것인가

두 번째로 고민해야 할 것은 taxonomy의 구조다.

고려해야 할 질문:

- 단일 고정 role taxonomy를 만들 것인가
- generic role과 domain role을 분리할 것인가
- unknown 문서군은 어떻게 처리할 것인가

여기서 중요한 결론은 다음이다.

- `block_type`
- `generic_role`
- `domain_role`

을 분리해야 한다.

이유는 다음과 같다.

- 금융 리포트 v1에 필요한 role은 따로 정해도 된다.
- 하지만 그 role을 전체 시스템의 절대 ontology로 만들면 나중에 신문, 의료, 폼 문서가 들어왔을 때 확장성이 깨진다.
- generic layer를 유지하면 새로운 문서군은 domain extension만 추가하면 된다.

즉 semantic role은 처음부터 절대 taxonomy로 굳히면 안 된다.

### 4.3 semantic 해석에 어떤 입력 신호를 사용할 것인가

세 번째로는 해석의 근거를 정해야 한다.

semantic 해석은 텍스트 키워드만으로 하면 안 된다. 반드시 다중 신호 기반이어야 한다.

주요 신호:

- block label
- bbox
- column position
- reading order
- block 크기
- 인접 블록
- 텍스트 키워드
- 반복 여부
- 페이지 내 상대 위치

이유는 다음과 같다.

- `BUY`, `Key Data`, `Analyst` 같은 블록은 텍스트만으로도 강한 단서를 준다.
- 하지만 많은 블록은 위치와 주변 문맥이 없으면 역할을 잘못 해석한다.
- 예를 들어 `table`은 위에 붙은 제목과 위치를 같이 봐야 `key_data_box`인지 `financial_table`인지 판단할 수 있다.
- `paragraph_title`도 메인 thesis heading일 수도 있고, 요약 섹션 제목일 수도 있다.

즉 semantic 해석은 OCR 텍스트 분류가 아니라 문서 문맥 해석이다.

### 4.4 semantic 해석 결과를 downstream이 어떻게 사용할 것인가

semantic 단계는 분석만 하고 끝나면 안 된다. 누가 어떻게 쓸지를 먼저 정해야 한다.

주요 소비처:

- content brief
- draft generation
- review

각각의 쓰임은 다음과 같다.

- content brief: 어떤 역할의 내용이 필수인지 판단
- generation: 어떤 순서와 역할로 문서를 채울지 판단
- review: 필수 역할 누락 여부와 구조적 일관성을 검토

이 연결이 없으면 semantic 단계는 정보는 많지만 아무도 소비하지 않는 죽은 데이터가 된다.

### 4.5 unknown과 ambiguity를 어떻게 다룰 것인가

semantic 설계에서 가장 위험한 것은 `잘못된 확신`이다.

고려해야 할 질문:

- 애매한 블록을 억지로 분류할 것인가
- domain role이 불명확하면 어떻게 할 것인가
- 문서군이 다르면 어떻게 fallback할 것인가

기본 방침은 다음이 맞다.

- domain role이 불확실하면 generic role까지만 부여
- generic role도 불명확하면 `unknown`
- 낮은 confidence는 그대로 남기고 review에 전달

이유:

- 애매한 블록을 억지로 role에 넣으면 generation이 잘못된 구조를 복제한다.
- semantic 단계는 recall보다 precision과 안정성이 더 중요하다.

## 5. 사고의 흐름

이 설계에 도달하는 사고의 흐름은 아래 순서가 자연스럽다.

1. reference 문서를 보면 먼저 위치와 블록이 보인다.
2. 하지만 생성하려면 각 블록의 역할을 알아야 한다.
3. 역할은 텍스트만으로 정해지지 않고 위치, 문맥, 인접성까지 봐야 한다.
4. 문서군마다 역할 체계가 다르므로 generic role과 domain role을 분리해야 한다.
5. 모든 문서군을 처음부터 완성할 수 없으므로 v1은 financial report에 특화한다.
6. 대신 unknown과 fallback 구조를 넣어 확장 가능성을 유지한다.
7. 따라서 semantic interpretation 단계는 `완전한 의미 이해`가 아니라 `문서 생성에 필요한 수준의 역할 해석`으로 정의된다.

이 흐름이 중요한 이유는 설계의 초점을 잃지 않게 해 주기 때문이다.  
semantic 단계는 NLP 태스크가 아니라, reference-conditioned document generation을 위한 `구성 해석 단계`다.

## 6. financial report v1에서 특히 중요하게 볼 것

v1의 주요 타깃 문서군은 `증권/기업 financial report`다.  
따라서 semantic 설계도 이 문서군에서 중요한 역할 그룹을 우선적으로 봐야 한다.

### 6.1 상단 메타 정보

예:

- 발간일
- 보고서 분류
- 기업명과 종목코드

왜 중요한가:

- 본문과 분리되어야 한다.
- 생성 시 상단 메타 블록으로 재구성해야 한다.

### 6.2 투자 의견/목표주가/현재주가 박스

예:

- `BUY (유지)`
- `목표주가`
- `현재주가`

왜 중요한가:

- 금융 리포트의 가장 강한 역할 신호다.
- 본문이 아니라 `investment signal` 영역으로 분리되어야 한다.

### 6.3 Key Data / Consensus Data 같은 수치 패널

왜 중요한가:

- 단순 표가 아니라 핵심 근거 블록이다.
- 메인 논지를 뒷받침하는 증거 패널로 해석되어야 한다.

### 6.4 메인 제목과 thesis heading

왜 중요한가:

- 문서의 주장 흐름을 구성한다.
- generation은 이 흐름을 따라 문단을 채워야 한다.

### 6.5 supporting argument 문단

왜 중요한가:

- 메인 thesis를 뒷받침하는 핵심 서술 블록이다.
- 본문 대부분은 이 역할에 속한다.

### 6.6 price chart / financial table 같은 근거 블록

왜 중요한가:

- v1에서 직접 생성하지 않더라도 evidence role로 분류되어야 한다.
- 그래야 generation과 review가 이 블록의 존재를 설명할 수 있다.

### 6.7 analyst / 리서치센터 / disclaimer

왜 중요한가:

- 본문과 반드시 분리되어야 한다.
- 본문으로 잘못 유입되면 문서가 어색해진다.

### 6.8 현재 기준 기본 방침

- `table`, `chart`는 v1에서 생성 대상이 아니라 `evidence role` 또는 `unsupported`로 우선 해석
- `analyst_info`, `research_center_meta`, `disclaimer`는 반드시 본문과 분리
- `investment_opinion_box` 계열은 별도 역할로 유지

## 7. 설계 단계의 주요 trade-off

semantic 설계는 결국 몇 가지 trade-off를 고르는 일이다.

### 7.1 정확한 domain role 추론 vs 보수적 generic role 유지

v1에서는 보수적 generic fallback이 더 중요하다.

이유:

- 잘못된 금융 특화 role은 downstream을 크게 왜곡한다.
- 애매할 때는 generic role로 남기는 편이 안전하다.

### 7.2 높은 recall vs 낮은 false positive

v1에서는 false positive를 낮추는 쪽이 맞다.

이유:

- 빠진 블록은 review에서 찾을 수 있지만,
- 잘못 분류된 블록은 문서 구조 전체를 흐리게 만든다.

### 7.3 깊은 의미 이해 vs 역할 수준 이해

v1에서는 역할 수준 이해를 우선한다.

이유:

- 의미를 깊게 이해하는 것은 어려울 뿐 아니라 reference 단계의 책임 범위를 넘어서기 쉽다.

### 7.4 금융 v1 최적화 vs 범용 확장성

v1은 금융 리포트 최적화 우선이 맞다.  
대신 generic layer를 분리해 확장 가능성을 보존해야 한다.

### 7.5 규칙 기반 해석 vs 모델 기반 해석

초기 구현은 규칙 기반 + parser output 후처리 중심이 적합하다.

이유:

- failure analysis가 쉽다.
- role 판별 기준을 눈으로 검증하기 좋다.
- 나중에 모델 기반 해석을 추가하더라도 baseline을 유지할 수 있다.

## 8. semantic 해석 결과를 어떤 수준까지 정의할 것인가

이 문서는 코드 스키마를 확정하지는 않지만, semantic 단계가 어떤 개념적 출력을 가져야 하는지는 제안해야 한다.

### 8.1 `SemanticBlockInterpretation`

역할:

- 각 block의 semantic 해석 결과를 담는다.

왜 필요한가:

- layout와 semantic을 분리해 추적할 수 있다.
- block 단위 review와 오류 분석이 가능해진다.

downstream 사용:

- generation이 어떤 block을 어떤 역할로 쓸지 판단

### 8.2 `SectionSemanticGroup`

역할:

- 여러 block을 하나의 섹션 역할로 묶는다.

왜 필요한가:

- 실제 문서 생성은 개별 block보다 섹션 단위로 이뤄진다.

downstream 사용:

- content brief와 draft skeleton 생성

### 8.3 `DocumentSemanticProfile`

역할:

- 문서 전체의 semantic 구성을 요약한다.

왜 필요한가:

- 문서군별 흐름 패턴과 필수 역할을 관리할 수 있다.

downstream 사용:

- 문서 스켈레톤 선택
- missing role 검증

### 8.4 `ContentFlowTemplate`

역할:

- 문서의 전개 순서를 표현한다.

왜 필요한가:

- semantic 단계의 핵심이 결국 `내용 흐름` 복원에 있기 때문이다.

downstream 사용:

- generation order 결정

## 9. 실패 유형을 먼저 생각해야 하는 이유

semantic 해석은 정답을 많이 맞히는 것보다, 틀렸을 때 downstream을 크게 망치지 않게 설계하는 것이 중요하다.

대표 실패 유형:

- `summary`와 `body` 혼동
- `metadata`와 `body` 혼동
- `analyst/disclaimer` 누락
- `table/chart` 역할 오판
- section boundary 오류
- multi-column cross-binding
- unknown 문서군에서 잘못된 domain role 부여

이 실패들이 중요한 이유:

- semantic 오류는 문서의 모양보다 읽는 흐름을 망친다.
- generation은 semantic layer를 신뢰하기 때문에, 잘못된 role은 잘못된 문서 구조로 바로 이어진다.

따라서 이 단계에서는 `fallback`, `unknown`, `role_confidence` 설계가 핵심 고려사항이 된다.

## 10. 현재 저장소 기준으로 얻을 수 있는 힌트

현재 저장소의 Paddle 결과는 semantic 단계에서 참고할 만한 좋은 raw signal을 제공한다.

예시 파일:

- `OCR_results/paddle/Paddle_json/financial1/original-00_res.json`
- `OCR_results/paddle/Paddle_json/financial2/original-00_res.json`

여기서 보이는 특징:

- `doc_title`, `paragraph_title`, `text`, `table`, `chart`, `image`가 분리되어 있다.
- 금융 리포트에서 좌측 패널과 우측 메인 본문 구조가 분명하다.
- 다만 `table`, `image`, `chart`는 `block_order`가 비어 있는 경우가 있어 semantic 해석 전에 후처리가 필요하다.

이 점이 중요한 이유:

- semantic 단계는 parser 결과를 그대로 믿는 단계가 아니라, parser 결과를 문서 맥락 안에서 다시 해석하는 단계여야 한다.

## 11. 이후 단계에서 무엇을 결정해야 하는가

이 문서를 바탕으로 다음 단계에서 구체적으로 결정해야 할 것은 아래와 같다.

- v1 financial report용 generic/domain role taxonomy 확정
- Paddle JSON을 입력으로 한 semantic 해석 규칙 표 작성
- low-confidence fallback 정책 정의
- 5~10개 financial report 샘플에 semantic annotation 수행
- semantic output을 `reference_template.json`에 어떻게 연결할지 결정

즉, 지금 단계에서는 role 이름을 완전히 고정하는 것보다 `무엇을 기준으로 role을 해석할지`와 `애매할 때 어떻게 처리할지`를 먼저 정해야 한다.

## 12. 명시적 가정과 기본값

- 이 문서는 고려사항 중심의 분석 리포트다.
- semantic interpretation은 reference layout parsing 이후 단계다.
- v1의 목적은 완전한 의미 이해가 아니라 `block role + section flow` 해석이다.
- semantic role은 고정 절대 taxonomy가 아니라 `generic + domain extension` 구조를 따른다.
- unknown과 low-confidence fallback은 필수로 둔다.
- 표와 차트는 v1에서 생성 대상이 아니라 semantic evidence 또는 unsupported 대상으로 다룬다.

## 13. 결론

semantic interpretation 단계에서 가장 먼저 고민해야 할 것은 `role taxonomy` 자체가 아니다.  
먼저 정해야 하는 것은 이 단계의 책임 범위, 해석에 사용할 신호, downstream 소비처, fallback 구조, 그리고 확장 가능한 role 계층 구조다.

결국 이 단계의 핵심은 텍스트를 완벽히 이해하는 것이 아니라, `문서 생성에 필요한 수준에서 블록의 역할과 내용 흐름을 안정적으로 해석하는 것`이다.  
이 관점을 놓치지 않아야 v1 financial report에 집중하면서도 나중에 다른 문서군으로 무리 없이 확장할 수 있다.
