# Section 구현 정리

이 문서는 `11-1. section.md`, `11-2. section_artifact.md`, `12. section_generation.md`를 기준으로, 현재 코드베이스에 실제로 반영된 `section` 관련 구현을 정리한 문서다.

목표는 두 가지다.

- section 설계가 코드에서 어디까지 구현되었는지 한 번에 설명하기
- 현재 품질 이슈와 아직 남아 있는 작업을 분리해서 기록하기

## 1. 문서별 역할 요약

### 1.1 `11-1. section.md`

핵심 주제는 `section detection` 자체다.

- 목표는 예쁜 `section_order`를 만드는 것이 아니라, downstream에서 바로 사용할 수 있는 `section artifact`를 만드는 것
- 입력은 `canonical blocks + reading order + bbox/layout + page analysis`이고 semantic은 optional hint
- `rule | qwen | shadow | only_qwen` 모드를 지원하는 공통 section detection 설계를 정의
- 최종 산출물은 `block-to-section assignment -> deterministic assembly -> section_artifact` 흐름으로 만든다

### 1.2 `11-2. section_artifact.md`

핵심 주제는 `section artifact를 reference 파이프라인에 공식 연결하는 방법`이다.

- `section_artifact`를 공식 결과로 승격
- `section_order`는 공식 결과가 아니라 `section_artifact`의 narrative projection으로 재정의
- `build_reference_template()`와 `parse_reference()`에 section detection 단계를 공식 삽입
- reference와 source가 같은 section 계약을 공유하게 만드는 것이 목표

### 1.3 `12. section_generation.md`

핵심 주제는 generation을 `slot-first`에서 `section-first`로 전환하는 것이다.

- 기존: `slot -> fact`
- 목표: `reference section -> source section -> slot -> fact`
- 요약형 section은 `source section bundle`을 정식 경로로 지원
- v1 공식 구조는
  - `rule section-first planner`
  - `optional Qwen section rerank/select`
  - `deterministic slot-to-fact selection`
  - `existing slot writing`
- Qwen은 `slot fact 자유 선택기`가 아니라 `section candidate selector`로만 사용

## 2. 구현된 핵심 변경 사항

### 2.1 section 공식 타입 추가

추가 파일:

- `allm/core/types.py`

추가된 주요 타입:

- `SectionDetectionConfig`
- `SectionBlockAssignment`
- `DetectedSection`
- `SectionArtifact`
- `SectionTraceEntry`
- `SectionDetectionSummary`

의미:

- 이제 section은 임시 heuristic 결과가 아니라, 공통 dataclass 계약을 가진 공식 객체로 다뤄진다.
- reference/source/generation이 같은 구조를 공유할 수 있는 기반이 생겼다.

### 2.2 rule 기반 section detector 구현

추가 파일:

- `allm/section/detector.py`
- `allm/section/__init__.py`

현재 구현 상태:

- `detect_sections(...)`가 rule-only baseline으로 구현됨
- block를 `heading/body/evidence/meta/disclaimer/orphan`로 분류
- reading order, 역할 전환, gap, page change를 기준으로 section boundary를 잡음
- assignment를 모아서 `DetectedSection`과 `SectionArtifact`를 만듦
- block의 `section_id`, `section_purpose`도 업데이트함

중요한 점:

- 현재는 `qwen`, `shadow`, `only_qwen`의 full behavior까지는 구현되지 않았다.
- v1 baseline으로 `rule` 모드만 실제 사용 가능하다.

### 2.3 reference pipeline에 section detection 공식 삽입

수정 파일:

- `allm/pipelines/reference_pipeline.py`

변경 내용:

- `semantic overlay` 다음에 `detect_sections(...)`를 호출하도록 변경
- detector 결과를 기준으로 `section_order`를 생성하도록 변경
- parse 결과로 아래 artifact 저장 지원
  - `section_artifact.json`
  - `section_block_assignments.json`
  - `section_trace.json` (trace 존재 시)
- `parser_diagnostics.json`에 `section` summary 기록

의미:

- section detection이 이제 template 조립의 부수 heuristic이 아니라, parse-reference의 공식 단계가 되었다.

### 2.4 `build_section_order()` 역할 변경

수정 파일:

- `allm/components/template.py`

변경 내용:

- `build_section_order()`는 더 이상 section 탐지 함수가 아님
- 현재 역할은 `section_artifact -> narrative section_order projection`
- 즉 공식 결과는 `section_artifact`이고, `section_order`는 파생 뷰다

이 변경은 `11-2. section_artifact.md`의 핵심 방향과 직접 연결된다.

## 3. generation 레이어의 section-first 전환

### 3.1 generation 타입 확장

수정 파일:

- `allm/generation/generation_types.py`

추가된 핵심 타입:

- `ReferenceSectionPlan`
- `SourceSectionView`
- `SectionMatchDecision`
- `SectionAwareSlotAssignment`

확장된 설정:

- `matching_mode`
- `max_section_candidates`
- `max_synthesis_sections`
- `section_match_threshold`
- `use_qwen_for_section_rerank`
- `allow_cross_section_fallback`
- 기타 section-first planning 관련 필드

의미:

- generation이 이제 flat slot list만 다루는 구조가 아니라, section-aware plan 객체를 공식 입력/출력으로 가질 수 있게 되었다.

### 3.2 reference blueprint를 `sections + slots` 구조로 변경

수정 파일:

- `allm/generation/generation_pipeline.py`

현재 `build_reference_blueprint()`는 아래를 만든다.

- `sections`: `ReferenceSectionPlan[]`
- `slots`: `ReferenceSlot[]`

주요 처리:

- `section_artifact.json`이 있으면 그것을 authoritative source로 사용
- 없으면 기존 `section_order`를 fallback 입력으로 사용
- semantic role이 비어 있는 경우를 대비해 section 기반 fallback role 추론도 수행
- section 없는 slot은 synthetic section으로 묶어서 planner에서 버려지지 않게 함

### 3.3 source fact pack를 `source_sections + facts + global_fact_pool` 구조로 변경

수정 파일:

- `allm/generation/generation_pipeline.py`

현재 `build_source_fact_pack()`는 아래를 만든다.

- `source_sections`: `SourceSectionView[]`
- `facts`: `SourceFact[]`
- `global_fact_pool`
- `global_metadata`
- `global_evidence_cards`

주요 처리:

- block fact 생성
- table row fact 생성
- section-level aggregate fact 생성
- section별 fact ids를 분류해서 `SourceSectionView`를 구성

의미:

- 이제 generation은 source 문서를 단순 fact pool이 아니라 `source section 집합`으로 볼 수 있다.

### 3.4 generation plan을 section-first 구조로 변경

수정 파일:

- `allm/generation/generation_pipeline.py`

현재 `build_generation_plan()`는 아래 순서로 동작한다.

1. reference section 목록을 읽는다
2. source section 후보를 고른다
3. section match decision을 만든다
4. 선택된 source section 내부 fact로 slot assignment를 만든다

즉 현재 공식 구조는:

- `reference_sections`
- `source_sections`
- `section_matches`
- `slot_assignments`
- `summary`

이 구조는 `12. section_generation.md`의 핵심 설계를 직접 반영한다.

## 4. Qwen rerank/select 구현 상태

### 4.1 구현된 내용

수정 파일:

- `allm/generation_backends/qwen_transformers.py`
- `allm/generation/generation_schema.py`
- `allm/generation/generation_pipeline.py`

현재 구현된 것:

- `QwenGenerationBackend.rerank_sections()` 추가
- section rerank용 JSON prompt 구성 추가
- section rerank 응답 파서 `parse_section_match_response()` 추가
- planner에서
  - `rule shortlist`
  - `optional Qwen rerank/select`
  - `fallback to rule`
  구조를 구현

현재 정책:

- Qwen은 shortlist 밖 section을 고를 수 없음
- `local_align`에서는 `qwen_rerank`
- synthesis section에서는 `qwen_bundle_rerank`
- 실패 시 rule top-1 또는 rule bundle로 fallback

### 4.2 아직 남은 부분

현재 구현은 문서 설계의 방향은 맞지만, 아직 다음은 미완성이다.

- Qwen section matching trace를 별도 artifact로 저장하는 것
- document_family, language 등 richer context를 rerank payload에 더 풍부하게 넣는 것
- synthesis bundle 후보 생성과 score를 더 정교하게 하는 것
- fact-level rerank는 여전히 비활성 상태

즉 현재는 `optional Qwen rerank/select`의 골격이 들어간 상태라고 보는 것이 맞다.

## 6. 현재 품질 이슈

설계 구현은 들어갔지만, 품질은 아직 baseline 수준이다.

### 6.1 section detector 과분할

현재 detector는 heading-like block에 매우 민감해서 다음이 독립 section으로 쉽게 분리된다.

- branding
- 날짜
- 감수 라벨
- analyst 이름
- email
- AI 안내문

결과적으로:

- `MIRAE ASSET`
- `2026.3.13`
- `[감수]`
- `김주연`
같은 것들이 narrative section처럼 잡히는 문제가 있다.

### 6.2 matching score가 너무 coarse함

현재 `_source_section_score()`는 주로 아래를 본다.

- purpose match
- heading/body shape
- confidence
- narrative 여부

이 때문에 실제 텍스트 내용은 다르더라도 구조만 비슷하면 같은 source section으로 몰릴 수 있다.

대표 현상:

- 여러 body slot이 `sec-001`로 반복 매칭됨
- 여러 heading slot이 `sec-003` 계열로 반복 매칭됨

### 6.3 block fact + section fact 중복

현재 source fact pack은 같은 내용에 대해 아래 두 개를 동시에 넣는다.

- block fact
- section aggregate fact

이 때문에 `MIRAE ASSET / 미래에셋증권` 같은 짧은 텍스트가 한 slot 안에서 중복 선택되어 반복 출력되기 쉽다.

### 6.4 Qwen 실제 미실행 가능성

`generation-mode qwen`으로 돌려도 환경에 따라 실제로는 Qwen이 안 돌고 rule fallback으로 내려갈 수 있다.

예:

- `transformers`가 checkpoint architecture를 인식하지 못함
- model load 실패
- 따라서 `qwen run`처럼 보이지만 실제 결과는 `rule_top1`일 수 있음

## 8. 현재 코드 흐름 한 장 요약

현재 section 관련 공식 흐름은 아래처럼 정리할 수 있다.

```text
parse-reference
  -> semantic overlay
  -> detect_sections(rule baseline)
  -> section_artifact.json 저장
  -> build_section_order(section_artifact projection)
  -> reference_template.json 저장

generation
  -> build_reference_blueprint(sections + slots)
  -> build_source_fact_pack(source_sections + facts + global_fact_pool)
  -> build_generation_plan(section match + section-aware slot assignment)
  -> optional Qwen section rerank/select
  -> slot writing
  -> render
```

즉 큰 방향 자체는 `11-1`, `11-2`, `12` 문서가 의도한 구조에 맞게 이미 들어와 있다.

## 9. 다음 우선순위

현재 품질을 빠르게 올리려면 아래 순서가 가장 효과적이다.

1. section detector에서 branding/date/email/감수/AI 안내문을 meta 또는 disclaimer로 더 강하게 분리
2. `source_fact_pack`에서 block fact와 section fact의 중복을 줄이기
3. `_source_section_score()`에 lexical similarity를 추가해서 `sec-001`, `sec-003` 쏠림 완화
4. Qwen 실제 실행 환경 정리
   - checkpoint / transformers 호환성 해결
5. synthesis bundle scoring과 trace artifact 확장

## 10. 결론

현재 구현은 설계 문서 기준으로 보면 `뼈대는 들어온 상태`다.

- section은 이제 공식 artifact를 가진다
- template는 artifact projection만 담당한다
- generation은 section-first planning 구조로 옮겨졌다
- Qwen은 section candidate selector로 제한해서 붙였다

반면 품질 측면에서는 아직 baseline 단계다.

- section detector 과분할
- coarse matching
- duplicate facts
- Qwen 환경 의존 fallback

따라서 지금 상태를 가장 정확하게 표현하면 아래와 같다.

> `11-1`, `11-2`, `12`의 구조적 전환은 코드에 반영되었고,
> 현재 남은 일은 detector/matching/fact dedupe/Qwen runtime 품질 개선이다.
