# Section Artifact Type Definition and Reference Pipeline Integration

이 문서는 `11. section.md`에서 정의한 section 설계를 실제 코드베이스에 어떻게 연결할지 설명한다.

핵심 주제는 두 가지다.

1. `section artifact 타입 정의`
2. `reference 파이프라인 연결`

즉 이 문서는 “section 설계 문서가 실제 parse-reference 파이프라인 안에서 어떤 공식 계약과 실행 흐름으로 자리잡는가”를 설명하는 보조 설계 문서다.

## TL;DR

- 지금 코드베이스에는 공식 section artifact가 없다.
- 현재 `section_order`는 `allm/template.py`에서 generation-friendly projection으로 만들어지고, block-level `section_id`는 `allm/semantic.py`에 일부만 기록된다.
- 앞으로는 `section_artifact`를 공식 결과로 만들고, `section_order`는 그 결과의 narrative projection으로 재정의하는 것이 목표다.
- 이를 위해 먼저 공통 타입을 정의하고, 그다음 `build_reference_template()`와 `parse_reference()`에 section detection 단계를 연결해야 한다.
- 이렇게 해야 reference와 source가 같은 section 계약을 공유하게 되고, 이후 `section-first mapping`을 안전하게 구현할 수 있다.

## 1. 왜 별도 section artifact가 필요한가

현재 코드베이스에서 section 관련 정보는 두 층으로 분산되어 있다.

- `allm/semantic.py`
  - 일부 block에 `section_id`, `section_purpose`를 붙인다.
- `allm/template.py`
  - `build_section_order()`가 anchor page와 title-like heuristic으로 `SectionOrderItem` 목록을 만든다.

이 구조의 문제는 아래와 같다.

1. section의 “공식 결과”가 무엇인지 불분명하다.
2. `section_order`는 generation-friendly view인데, 마치 공식 artifact처럼 쓰이고 있다.
3. block-level section 정보와 template-level section 정보가 같은 계약을 공유하지 않는다.
4. Qwen 또는 `only_qwen` 도입 시 trace와 fallback을 담을 공통 구조가 없다.
5. source 문서에도 같은 방식으로 section을 적용하기 어렵다.

따라서 section을 독립 artifact로 승격해야 한다.

## 2. 현재 기준 흐름

현재 `build_reference_template()` 흐름은 대략 아래와 같다.

```text
OCR locate
  -> dolphin/paddle load
  -> fuse_pages
  -> classify_page
  -> detect_document_family / language
  -> semantic overlay
  -> anchor_pages
  -> build_section_order
  -> style_tokens / image_slots / unsupported_blocks
  -> ReferenceTemplate
```

즉 현재는:

- semantic 이후
- section detection 전용 공식 단계 없이
- 곧바로 `build_section_order()`를 호출한다

이 때문에 section은 “템플릿 조립 보조 단계”처럼 취급된다.

## 3. 목표 상태

목표 상태에서는 section이 아래처럼 들어와야 한다.

```text
OCR locate
  -> dolphin/paddle load
  -> fuse_pages
  -> classify_page
  -> detect_document_family / language
  -> semantic overlay (optional hints)
  -> section detection
    -> section_artifact
    -> section_summary
    -> section_trace (optional)
  -> narrative projection
    -> section_order
  -> style_tokens / image_slots / unsupported_blocks
  -> ReferenceTemplate
```

즉 `section detection`이 template 조립 전에 독립된 공식 단계로 들어간다.

## 4. section artifact 타입 정의가 의미하는 것

여기서 “타입 정의”는 단순히 dataclass 몇 개를 추가하는 일이 아니다. 아래 세 가지를 동시에 고정하는 작업이다.

### 4.1 공식 입력 계약

section detection이 어떤 입력을 받는지 고정한다.

최소 입력:

- `pages: List[FusedPage]`
- `analyses: List[PageAnalysis]`
- optional semantic hints already written on `CanonicalBlock`
- `SectionDetectionConfig`

즉 section detection은:

- fuse 이전 raw OCR 결과를 직접 보지 않고
- canonicalized block와 page analysis 위에서 동작하도록 고정한다

### 4.2 공식 출력 계약

section detection의 결과가 무엇인지 고정한다.

필수 출력:

- 문서 전체 section 결과
- block-to-section assignment
- section confidence / needs_review
- optional trace

이 계약이 있어야:

- reference에서도 쓰고
- source에서도 쓰고
- generation에서도 읽고
- visualization/review에서도 보여줄 수 있다

### 4.3 책임 경계 고정

타입 정의는 각 레이어의 책임도 고정한다.

- semantic:
  - block role과 optional section-related hint 제공
- section detection:
  - block들을 section으로 묶고 purpose/confidence를 계산
- template:
  - section artifact에서 narrative projection을 뽑아 `section_order`를 만든다
- generation:
  - reference/source section artifact를 읽고 section-first mapping을 한다

즉 section 타입이 생기면 “누가 어떤 결정을 내려야 하는지”도 자연스럽게 분리된다.

## 5. 어떤 타입을 추가해야 하는가

`11. section.md` 기준으로 최소한 아래 타입들이 필요하다.

### 5.1 `SectionDetectionConfig`

목적:

- 실행 모드 정의
- threshold와 window 정책 정의
- runtime/model 옵션 정의

이 타입이 있어야 section detection을 semantic과 유사한 방식으로 `rule | qwen | shadow | only_qwen`으로 운용할 수 있다.

### 5.2 `SectionBlockAssignment`

목적:

- 각 block이 어떤 section에 속하는지 기록
- section 안에서 heading/body/evidence/meta인지 기록
- boundary와 confidence를 저장

이 타입이 중요한 이유:

- 최종 section artifact는 이 assignment를 기반으로 deterministic assembly할 수 있고
- `only_qwen`에서도 모델 출력이 최종 artifact가 아니라 “assignment 초안”이 되도록 만들 수 있다

### 5.3 `DetectedSection`

목적:

- 최종 section 단위 결과 표현

필수 역할:

- section 대표 title
- section purpose
- 시작/끝 page
- 포함 block 목록
- narrative 여부
- confidence

즉 downstream이 실제로 가장 많이 읽는 타입이다.

### 5.4 `SectionArtifact`

목적:

- 문서 전체 section 결과를 한 객체로 묶기

이 타입은 결국 `section_artifact.json`의 루트 구조가 된다.

### 5.5 `SectionTraceEntry`

목적:

- qwen/shadow/only_qwen에서 모델 입력과 출력을 남기기

semantic에서 `semantic_trace.json`이 있듯, section도 trace artifact를 가져야 한다.

## 6. 타입을 어디에 두는가

권장 방향은 아래 둘 중 하나다.

### 6.1 `allm/types.py` 확장

장점:

- 공통 타입이 한곳에 모인다
- 기존 `ReferenceTemplate`, `CanonicalBlock`, `PageAnalysis`와 나란히 놓이므로 이해가 쉽다

단점:

- 파일이 더 커진다
- semantic/generation/section 타입이 한 파일에 섞일 수 있다

### 6.2 `allm/section_types.py` 신설

장점:

- section 관련 contract를 독립적으로 유지할 수 있다
- 향후 source/generation 연결 시 관련 타입을 모아서 관리하기 쉽다

단점:

- 공통 타입이 파일 여러 개로 분산된다

현재 설계 성격상 `section_types.py`로 분리하는 편이 더 깔끔하다. 다만 `CanonicalBlock` 자체의 필드 확장이 필요하면 `allm/types.py` 수정은 여전히 필요하다.

## 7. ReferenceTemplate와의 관계

가장 중요한 설계 포인트는:

`section_artifact`를 공식 결과로 만들고, `ReferenceTemplate.section_order`는 파생 결과로 낮추는 것

이다.

즉 앞으로는:

- `section_artifact.json`
  - 공식 section 결과
- `reference_template.json`의 `section_order`
  - generation용 narrative projection

이 관계를 분명히 해야 한다.

이렇게 하면:

- review 시 section을 공식 artifact 기준으로 볼 수 있고
- generation은 기존 `section_order`를 당장 깨지 않고 유지할 수 있고
- 점진적으로 `section_order` 의존을 줄일 수 있다

## 8. reference 파이프라인 연결이 의미하는 것

“reference 파이프라인 연결”은 section detection을 `parse-reference`의 공식 단계로 넣는 작업이다.

이는 아래 세 가지를 포함한다.

### 8.1 `build_reference_template()`에 section detection 단계 삽입

현재는 semantic 다음에 곧바로:

- `anchor_pages`
- `build_section_order()`
- `build_style_tokens()`

로 간다.

앞으로는 semantic 다음에:

- `detect_sections(...)`
- `section_artifact`
- `section_summary`
- optional `section_trace`

를 만든 뒤,

- `section_artifact`에서 narrative section만 projection해 `section_order`를 만들고
- style/image/template 단계로 넘겨야 한다.

즉 `build_section_order()`는 독립적인 detector가 아니라 `projection helper`가 되는 방향이 맞다.

### 8.2 diagnostics 확장

현재 diagnostics에는:

- reference_path
- page_analyses
- fusion
- semantic summary/trace

정도가 들어간다.

여기에 section 관련 항목을 넣어야 한다.

예:

- `diagnostics["section"] = section_summary`
- `diagnostics["section_trace"] = ...`

이렇게 해야 리뷰와 디버깅이 쉬워진다.

### 8.3 artifact 저장 확장

현재 `parse_reference()`는 아래를 저장한다.

- `reference_template.json`
- `canonical_pages.json`
- `semantic_overlay.json`
- `parser_diagnostics.json`
- optional `semantic_trace.json`

여기에 아래를 추가해야 한다.

- `section_artifact.json`
- optional `section_block_assignments.json`
- optional `section_trace.json`

즉 section도 semantic처럼 공식 artifact와 trace artifact를 갖게 된다.

## 9. 예상 인터페이스 변화

### 9.1 새 함수

예를 들면 아래 형태가 자연스럽다.

```python
def detect_sections(
    pages: List[FusedPage],
    analyses: List[PageAnalysis],
    config: Optional[SectionDetectionConfig] = None,
    document_family: Optional[str] = None,
) -> Tuple[SectionArtifact, SectionDetectionSummary, List[SectionBlockAssignment], List[SectionTraceEntry]]:
    ...
```

핵심은:

- pages/analyses를 받고
- artifact + summary + assignment + trace를 돌려주는 것

이다.

### 9.2 `build_reference_template()` 반환값

현재 반환은:

- `ReferenceTemplate`
- `diagnostics`
- `canonical_pages`

인데, section artifact를 외부에서도 쉽게 재사용하려면 두 가지 길이 있다.

#### 방법 A

기존 반환은 유지하고, section artifact를 diagnostics와 template에만 녹인다.

장점:

- 기존 호출부 변경이 작다

단점:

- section artifact를 독립 객체로 재사용하기 어렵다

#### 방법 B

section artifact를 추가 반환한다.

예:

```python
return template, diagnostics, canonical_pages, section_artifact
```

장점:

- 이후 source 파이프라인 재사용에 유리하다

단점:

- 호출부 수정이 생긴다

장기적으로는 B가 더 낫지만, 초기 연결은 A로 시작해도 된다.

## 10. `section_order`는 어떻게 바뀌는가

현재 `build_section_order()`는 탐지와 projection을 같이 한다.

새 구조에서는 이 함수의 성격이 바뀐다.

기존:

- pages와 anchor_pages를 받아 heuristic으로 section을 만든다

변경 후:

- `SectionArtifact`를 받아 narrative section만 `SectionOrderItem`으로 projection한다

즉 역할이:

- detector
- projection helper

로 바뀐다.

이 변화가 중요한 이유는 section 탐지 로직을 template 조립 로직과 분리할 수 있기 때문이다.

## 11. CanonicalBlock 필드는 어떻게 볼 것인가

`CanonicalBlock.section_id`와 `CanonicalBlock.section_purpose`는 없어져야 하는 필드는 아니다. 다만 공식 소스가 바뀌어야 한다.

앞으로는:

- semantic이 임시/optional hint로 이 필드를 쓸 수는 있지만
- 최종 authoritative 값은 section detector가 쓴 값이어야 한다

즉 block-level section 필드는:

- semantic output의 부산물
- section artifact assembly 결과의 denormalized copy

로 보는 편이 맞다.

이렇게 해야 block inspect와 section artifact가 서로 일관된다.

## 12. 왜 reference부터 연결하는가

section 설계를 구현할 때 source보다 reference부터 연결하는 게 좋은 이유는 아래와 같다.

1. 기존 `parse-reference` 경로가 이미 안정적이다.
2. `reference_template.section_order`와 직접 연결되므로 효과를 바로 볼 수 있다.
3. visualization/review에 section artifact를 바로 노출할 수 있다.
4. source까지 한 번에 바꾸는 것보다 리스크가 낮다.

즉 reference에서 contract를 먼저 고정하고, source는 같은 contract를 재사용하는 순서가 가장 안전하다.

## 13. 구현 후 기대 효과

이 설계가 들어가면 아래가 가능해진다.

- reference와 source가 동일한 section 계약 사용
- `qwen`과 `only_qwen` section 탐지 결과를 traceable하게 저장
- `section-first mapping`을 공식 artifact 기반으로 구현
- `section_order`를 유지하면서도 점진적으로 교체 가능
- review에서 section 품질을 semantic과 분리해 독립적으로 판단 가능

## 14. 한 줄 요약

`section artifact 타입 정의`는 section 결과의 공식 스키마를 만드는 일이고,  
`reference 파이프라인 연결`은 parse-reference가 그 스키마를 실제로 생성하고 저장하게 만드는 일이다.

이 둘이 끝나야 `11. section.md`에 적은 설계가 문서가 아니라 실제 파이프라인 계약이 된다.
