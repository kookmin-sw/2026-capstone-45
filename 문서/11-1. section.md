# Section Detection Design

이 문서는 `reference parsing`과 `source parsing`에서 공통으로 사용할 수 있는 `section detection` 설계를 정의한다.

목표는 단순히 `section_order`를 더 잘 뽑는 것이 아니다. 이후 `section-first mapping`과 `slot-to-fact generation`에 바로 사용할 수 있는, 안정적이고 추적 가능한 `section representation`을 만드는 것이다.

초기 구현은 `financial_report`를 예시로 설명하되, 설계 자체는 처음부터 범용으로 유지한다. 특히 `document_family`는 hard gate가 아니라 soft prior로만 사용한다.

## TL;DR

- section 탐지의 공식 목표는 `예쁜 section_order`가 아니라 `매핑 가능한 section artifact`를 만드는 것이다.
- 공식 입력은 `canonical blocks + reading order + bbox/layout + page analysis`이며, semantic은 있어도 되고 없어도 되는 optional hint다.
- 기본 권장 모드는 `qwen`이며, 의미는 `rule candidate generation + Qwen refinement`다.
- `qwen` 모드에서 rule은 탐색 공간을 좁히는 역할만 하고, 최종 section 경계와 purpose는 Qwen이 보정한다.
- `only_qwen`은 rule candidate 없이 Qwen이 block-to-section assignment를 직접 만드는 실험 모드다.
- 단, `only_qwen`도 자유 생성 모드로 두지 않는다. 모델은 `per-block assignment`만 반환하고, 최종 section artifact는 deterministic assembler가 만든다.
- `shadow`는 Qwen을 돌리되 적용은 rule로 유지하는 검증 모드다.
- reference와 source 모두 같은 `section_artifact.json` 계약을 사용하고, generation은 `reference section -> source section -> slot/fact` 순서로 연결한다.
- low-confidence 결과는 억지로 narrative section으로 밀지 않고 `needs_review`와 `unknown purpose`로 보수적으로 남긴다.

## 1. 배경과 현재 한계

현재 코드베이스에서 section 관련 로직은 크게 두 군데에 흩어져 있다.

- `allm/template.py`
  - `build_section_order()`가 anchor page 위에서 title-like heuristic으로 narrative skeleton을 만든다.
- `allm/semantic.py`
  - `_assign_section_ids()`가 일부 narrative semantic role에만 `section_id`를 붙인다.

현재 구조의 한계는 아래와 같다.

1. `section_order`가 공식 section artifact라기보다 generation용 projection에 가깝다.
2. section 경계가 `heading`과 `gap`에 지나치게 의존한다.
3. panel heading, side evidence, footer/meta가 narrative section에 섞일 위험이 있다.
4. `section_id`가 일부 block에만 부여되어 section-first mapping 입력으로 쓰기 불안정하다.
5. source 문서가 다양해질수록 `same document_family` 전제를 두기 어렵다.
6. Qwen을 section 탐지에 도입하더라도 현재 구조에는 traceable intermediate contract가 없다.

즉 지금 필요한 것은 `더 공격적인 section splitting`이 아니라, `매핑에 써도 될 만큼 안정적인 section contract`다.

## 2. 목표와 비목표

### 2.1 목표

- reference 문서와 source 문서 모두에 공통으로 적용되는 `section detection contract` 정의
- section 경계, section purpose, section confidence를 포함한 공식 artifact 정의
- `rule | qwen | shadow | only_qwen` 4모드 동시 지원
- `section-first mapping`이 바로 가능하도록 section signature를 구조화
- low-quality 문서나 mixed layout 문서에서도 보수적으로 degrade되는 fallback 정책 정의

### 2.2 비목표

- table/chart 내부 의미 복원
- 최종 generation/rendering 전체 재설계
- 특정 family 전용 hand-crafted parser 최적화
- Qwen이 문서 전체 생성까지 직접 담당하는 구조

## 3. 핵심 원칙

### 3.1 `section id`보다 `section signature`

section 탐지의 목표는 `sec-001`, `sec-002` 같은 번호를 예쁘게 붙이는 것이 아니다. 아래와 같은 `section signature`를 안정적으로 구성하는 것이 우선이다.

- 이 section이 narrative인지 non-narrative인지
- 대표 heading이 무엇인지
- purpose가 thesis인지 summary인지 evidence_panel인지
- 어떤 block들이 포함되는지
- confidence가 충분한지

section 번호는 그 결과물에 붙는 식별자일 뿐이다.

### 3.2 `section-first`는 `section-only`가 아니다

section-first mapping은 아래 순서를 뜻한다.

1. reference section을 본다.
2. source section 후보를 고른다.
3. 그 section 안에서 slot-to-fact selection을 한다.

즉 section은 `검색 공간을 좁히는 상위 가이드`이며, low-confidence 상황에서는 slot-level fallback이 가능해야 한다.

### 3.3 `document_family`는 soft prior

같은 `document_family`이면 가산점은 줄 수 있지만, 다르다고 매칭 대상에서 배제하면 안 된다. 실제 source 문서는 report, IR note, article, memo, deck처럼 섞여 들어올 수 있다.

section detection 자체도 family hard gate 없이 동작해야 한다. 대신 다음을 prior로 활용한다.

- page archetype
- layout pattern
- block role distribution
- heading/body/evidence 비율

### 3.4 confidence-aware fallback

section 탐지는 항상 완벽하지 않다. 따라서 결과는 항상 confidence를 가져야 하고, confidence가 낮으면:

- section purpose를 `unknown`으로 두고
- `needs_review=true`를 세우고
- section-first mapping에서 cross-section fallback을 허용해야 한다.

## 4. 모드 정의

section detection은 아래 4모드를 지원한다.

### 4.1 `rule`

- deterministic heuristic 기반
- heading candidate, layout gap, role transition, page archetype으로 section 경계를 결정
- baseline과 regression 기준으로 사용

### 4.2 `qwen`

- 기본 권장 모드
- 의미는 `hybrid rule + qwen`이다
- rule이 만든 boundary/section candidate를 Qwen에 힌트로 주고
- Qwen은 `merge / split / relabel / purpose refine`만 수행한다
- 최종 결과는 Qwen이 보정한 section result를 적용한다

#### 4.2.1 왜 hybrid로 두는가

`qwen`을 기본 모드로 두되 완전 자유 추론으로 시작하지 않는 이유는 아래와 같다.

- heading, gap, page archetype 같은 구조 신호는 deterministic하게 안정적이다
- model이 매번 문서 전체를 처음부터 분할하게 하면 variance가 커진다
- trace와 regression 비교가 어려워진다
- low-quality 페이지에서 과도한 split/merge가 발생하기 쉽다

따라서 `qwen` 모드는 rule이 만든 candidate를 Qwen이 refinement하는 구조로 고정한다.

#### 4.2.2 입력 단위

`qwen` 모드는 page 단위 또는 작은 page window 단위로 실행한다.

권장 입력 단위:

- `cover_summary`, `body_narrative`는 page 단위
- cross-page section이 의심되는 경우는 2-page window
- `evidence_table`, `compliance`는 필요 시 단일 page

payload에는 아래를 포함한다.

- 문서 컨텍스트
  - `document_family`
  - `language`
  - `page_window`
- 페이지 컨텍스트
  - `page_archetype`
  - `column_count`
  - `dominant_layout_pattern`
  - `quality_score`
- ordered block 목록
  - `block_id`
  - `reading_order`
  - `canonical_label`
  - `text`
  - `bbox_norm`
  - optional semantic hints
- rule candidate
  - proposed boundary 목록
  - provisional section 목록
  - block별 provisional role
  - provisional section purpose

#### 4.2.3 Qwen의 책임

`qwen` 모드에서 Qwen은 아래 5가지만 할 수 있다.

1. boundary 유지 또는 제거
2. provisional section merge
3. provisional section split
4. section purpose relabel
5. orphan/meta/evidence block reassignment

반대로 아래는 Qwen 책임이 아니다.

- reading order 재계산
- OCR 보정
- 임의의 새로운 block 생성
- 문서 전체 생성
- deterministic cleanup 대체

#### 4.2.4 출력 계약

Qwen은 최종 `DetectedSection`을 바로 만들지 않는다. 아래처럼 refinement-friendly output만 반환한다.

```json
{
  "page_window": [3],
  "actions": [
    {
      "type": "merge_sections",
      "source_section_ids": ["cand-002", "cand-003"],
      "target_section_id": "cand-002",
      "reason": "Both belong to the same thesis section."
    },
    {
      "type": "relabel_purpose",
      "section_id": "cand-004",
      "new_purpose": "evidence_panel",
      "reason": "Panel heading and table rows indicate evidence rather than narrative."
    }
  ],
  "confidence": 0.86
}
```

실제 applied result는 deterministic refiner가 위 action을 rule candidate 위에 적용해서 만든다.

#### 4.2.5 후처리와 acceptance

Qwen 결과를 적용한 뒤에도 아래는 deterministic하게 검사한다.

- section이 비어 있지 않은가
- heading/body/evidence 구성에 논리 모순이 없는가
- section purpose와 dominant role이 충돌하지 않는가
- narrative section에 disclaimer/meta만 들어가 있지 않은가

검사를 통과하면 applied result로 사용한다. 통과하지 못하면:

- `shadow`에서는 trace만 남기고 rule 결과 유지
- `qwen`에서는 low-confidence result로 강등하거나 deterministic cleanup 후 적용

#### 4.2.6 언제 `qwen` 모드를 쓰는가

`qwen`은 아래 조건에서 기본 선택이다.

- rule baseline은 어느 정도 작동하지만 panel contamination이 잦은 경우
- heading candidate는 있으나 purpose 구분이 어려운 경우
- thesis/evidence/meta 경계가 문서마다 달라지는 경우
- regression과 traceability를 유지하면서 모델 이득을 얻고 싶은 경우

### 4.3 `shadow`

- rule과 같은 입력으로 Qwen을 돌리되 최종 결과는 rule을 유지
- disagreement, confidence, raw response, fallback reason만 기록
- regression과 평가용

### 4.4 `only_qwen`

- 실험적 모드
- section 탐지의 최종 의사결정에서 rule candidate와 rule fallback을 사용하지 않는다
- 다만 deterministic preprocessing과 deterministic postprocessing은 허용한다
- 즉 아래는 공통 입력으로 사용 가능하다
  - canonical blocks
  - reading order
  - bbox/layout feature
  - page archetype
  - optional semantic hints
- 하지만 section boundary, section assignment, section purpose는 Qwen 출력만으로 결정한다

`only_qwen`은 특히 아래 상황에서 유용하다.

- source 문서가 cross-family로 다양하게 들어오는 경우
- heading heuristic가 약한 문서
- headingless narrative cluster가 많은 경우
- memo/deck/article처럼 layout prior가 약한 경우

기본값은 아니다. v1 기본 권장 모드는 여전히 `qwen`이다.

#### 4.4.1 설계 의도

`only_qwen`은 “Qwen이 다 알아서 한다”는 의미가 아니다. 정확한 의미는 아래와 같다.

- rule candidate generation을 applied path에서 사용하지 않는다
- section boundary와 section assignment의 1차 결정을 모델이 맡는다
- 대신 schema, assembly, validation, cleanup은 deterministic하게 유지한다

즉 `only_qwen`은 `free-form generation`이 아니라 `model-driven structured assignment`다.

#### 4.4.2 입력 단위

`only_qwen`은 block window를 입력 단위로 사용한다.

권장 window 정책:

- 기본은 page 단위
- block 수가 많으면 `max_blocks_per_window` 기준으로 읽는 순서 window 분할
- cross-page narrative가 강하면 인접 page 묶음 허용

payload에는 아래를 포함한다.

- 문서/페이지 컨텍스트
  - `document_family` optional
  - `language`
  - `page_window`
  - `page_archetype`
  - `column_count`
  - `layout stats`
- ordered blocks
  - `block_id`
  - `page`
  - `reading_order`
  - `canonical_label`
  - `raw_label`
  - `text`
  - `bbox_norm`
  - `text_length`
  - optional semantic hints
- instruction
  - block들을 section으로 군집화하라
  - 각 block의 section 내 역할을 정하라
  - non-narrative block은 narrative section에 섞지 마라
  - JSON schema만 반환하라

#### 4.4.3 출력 계약

`only_qwen`의 출력은 반드시 block assignment다.

```json
{
  "page_window": [5],
  "assignments": [
    {
      "block_id": "blk-001",
      "section_local_id": "sec-a",
      "block_role_in_section": "heading",
      "boundary_before": true,
      "boundary_after": false,
      "section_purpose": "thesis",
      "confidence": 0.92,
      "reason": "Heading-like narrative opener."
    },
    {
      "block_id": "blk-002",
      "section_local_id": "sec-a",
      "block_role_in_section": "body",
      "boundary_before": false,
      "boundary_after": false,
      "section_purpose": "thesis",
      "confidence": 0.88,
      "reason": "Supports the same argument."
    }
  ]
}
```

이 모드에서도 Qwen이 최종 `section_artifact.json`을 직접 반환하게 하지 않는다. 그 이유는:

- section id 정규화가 필요하고
- cross-window merge가 필요하며
- malformed output 복구가 필요하고
- 리뷰 가능한 deterministic contract가 필요하기 때문이다

#### 4.4.4 deterministic assembly와 validation

`only_qwen` 결과를 받은 뒤 아래 순서를 반드시 거친다.

1. schema validation
2. block coverage validation
   - 모든 입력 block이 assignment를 가졌는가
   - 존재하지 않는 block id가 포함되지 않았는가
3. section id coherence validation
   - 한 block이 중복 assignment되지 않았는가
   - section_local_id가 지나치게 파편화되지 않았는가
4. deterministic assembly
5. cleanup
   - evidence/meta/disclaimer-only cluster 강등
   - tiny fragment merge 후보 처리
   - mega-section review 표기
6. final confidence 계산

즉 `only_qwen`에서도 모델 출력은 raw truth가 아니라 `assembly input`이다.

#### 4.4.5 실패 정책

`only_qwen`은 rule fallback을 사용하지 않으므로 실패를 숨기면 안 된다.

실패 유형은 최소한 아래로 나눈다.

- `invalid_json`
- `schema_mismatch`
- `missing_block_coverage`
- `duplicate_assignment`
- `incoherent_section_ids`
- `oversegmented_window`
- `undersegmented_window`

처리는 아래로 고정한다.

- 해당 window 결과를 폐기하지 않는다
- low-confidence `unknown_review_required` assignment를 만든다
- deterministic cluster를 임시 section으로 assemble한다
- `needs_review=true`
- trace에 fallback reason과 raw response를 저장한다

즉 only_qwen의 fallback은 `rule 대체`가 아니라 `reviewable degraded artifact`다.

#### 4.4.6 언제 `only_qwen`을 쓰는가

`only_qwen`은 아래 상황에서만 opt-in으로 권장한다.

- rule prior가 거의 없는 cross-family source 문서
- section heading 규칙이 깨지는 memo/deck/article
- semantic taxonomy가 약해서 rule candidate 자체가 오염될 가능성이 큰 경우
- source parsing 실험에서 recall을 먼저 보고 싶은 경우

반대로 reference parsing의 기본 모드로는 권장하지 않는다. reference는 contract 안정성이 더 중요하기 때문이다.

#### 4.4.7 `qwen`과 `only_qwen` 비교

- `qwen`
  - rule이 후보를 만든다
  - Qwen은 refinement를 한다
  - traceability와 안정성이 높다
  - 기본 권장 모드
- `only_qwen`
  - Qwen이 assignment 초안을 만든다
  - deterministic assembler가 artifact를 만든다
  - cross-family 대응력은 좋지만 variance가 크다
  - experimental 모드

## 5. 공통 데이터 모델

공식 contract는 아래 dataclass 수준으로 정의한다.

```python
@dataclass
class SectionDetectionConfig:
    mode: Literal["rule", "qwen", "shadow", "only_qwen"] = "qwen"
    model_name: str = "Qwen/Qwen3.5-9B"
    runtime: Literal["transformers"] = "transformers"
    prompt_version: str = "section-v1"
    quality_threshold: float = 0.55
    boundary_accept_threshold: float = 0.75
    purpose_accept_threshold: float = 0.70
    section_accept_threshold: float = 0.70
    max_blocks_per_window: int = 24
    allow_cross_page_section: bool = True
```

```python
@dataclass
class SectionBlockAssignment:
    block_id: str
    page: int
    section_local_id: str
    block_role_in_section: Literal[
        "heading",
        "body",
        "evidence",
        "meta",
        "disclaimer",
        "orphan",
    ]
    boundary_before: bool
    boundary_after: bool
    section_purpose: Optional[str]
    confidence: float
    needs_review: bool = False
    reason: Optional[str] = None
```

```python
@dataclass
class DetectedSection:
    section_id: str
    title: Optional[str]
    purpose: Optional[str]
    page_start: int
    page_end: int
    block_ids: List[str]
    heading_block_ids: List[str]
    body_block_ids: List[str]
    evidence_block_ids: List[str]
    meta_block_ids: List[str]
    dominant_roles: Dict[str, int]
    is_narrative: bool

```

```python
@dataclass
class SectionArtifact:
    document_id: str

@dataclass
class SectionTraceEntry:

```

class SectionDetectionSummary:

## 6. 파이프라인 개요

공통 파이프라인은 아래로 고정한다.

```text
Canonical blocks + page analyses
  -> block preprocessing
  -> boundary candidate generation
  -> block-to-section assignment
  -> deterministic section assembly
  -> confidence + cleanup
  -> section_artifact.json
  -> narrative projection for section_order / generation
```

### 6.1 block preprocessing

입력 block마다 아래 정보를 준비한다.

- `block_id`
- `page`
- `reading_order`
- `canonical_label`
- `raw_label`
- `text`
- `bbox_norm`, `bbox_px`
- `page_archetype`
- `column_count`
- `layout pattern`
- optional semantic hints
  - `generic_role`
  - `domain_role`
  - `section_purpose`
  - `used_for_generation`

중요한 점은 optional semantic hints가 있어도 section 탐지 contract는 그것 없이 성립해야 한다는 것이다. 즉 semantic은 힌트일 뿐, 필수 입력이 아니다.

### 6.2 boundary candidate generation

rule/hybrid 공통에서 아래 신호를 쓴다.

- heading-like label
- 짧고 title-like한 텍스트
- 이전 block과의 vertical gap
- column shift
- role transition
- page archetype transition
- non-narrative block의 등장

이 단계의 출력은 “여기 section이 시작될 가능성이 있다”는 boundary candidate 집합이다.

### 6.3 block-to-section assignment

최종 section은 항상 block assignment를 통해 구성한다.

이 방식을 택하는 이유:

- traceability가 좋다
- schema validation이 쉽다
- only_qwen에서도 자유 생성보다 안정적이다
- 후처리와 cleanup을 deterministic하게 유지할 수 있다

#### `rule`

- boundary candidate를 바탕으로 읽는 순서대로 section_local_id를 증가시킨다
- block마다 `heading/body/evidence/meta/disclaimer/orphan` 중 하나를 붙인다
- section purpose는 heuristic priority로 결정한다

#### `qwen`

- rule이 만든 boundary candidate와 provisional section candidate를 payload에 넣는다
- Qwen은 아래 중 하나만 하게 한다
  - boundary 유지/삭제
  - section merge
  - section split
  - purpose relabel
  - orphan block reassignment
- Qwen이 자유롭게 새 artifact를 만드는 것이 아니라, rule candidate 위에서 refinement를 한다

#### `shadow`

- rule path로 assignment를 만든 뒤
- 같은 입력으로 Qwen refinement를 수행하고 trace만 저장한다
- 최종 적용은 rule 결과 유지

#### `only_qwen`

- rule candidate를 만들지 않는다
- block window 단위 payload를 직접 Qwen에 넣고
- Qwen이 각 block에 대해 다음을 반환한다
  - `section_local_id`
  - `block_role_in_section`
  - `boundary_before`
  - `boundary_after`
  - `section_purpose`
  - `confidence`
- postprocessing에서만 deterministic assembler가 이 assignment를 `DetectedSection`으로 변환한다

`only_qwen`에서도 Qwen이 직접 완성된 section artifact를 출력하게 하지 않는다. 자유도가 너무 커지고 디버깅이 어려워지기 때문이다.

### 6.4 deterministic section assembly

assembler는 block assignment를 읽는 순서대로 묶어 `DetectedSection`을 만든다.

규칙은 아래로 고정한다.

1. 같은 `section_local_id`를 가진 인접 block은 하나의 section으로 묶는다.
2. 대표 title은 우선순위로 고른다.
   - 첫 번째 heading block
   - 없으면 첫 번째 body block 첫 줄
   - 그래도 없으면 `None`
3. section purpose는 assignment 다수결과 confidence 우선순위로 정한다.
4. narrative 여부는 purpose와 block role 구성으로 결정한다.
5. orphan/meta/disclaimer/evidence-only cluster는 별도 non-narrative section으로 유지한다.

### 6.5 confidence와 cleanup

최종 confidence는 아래를 결합해 계산한다.

- boundary consistency
- heading 존재 여부
- role cohesion
- purpose agreement
- OCR/text quality
- page archetype consistency

confidence가 낮으면 아래 처리한다.

- `needs_review=true`
- purpose를 `unknown` 또는 상위 generic purpose로 낮춤
- section-first mapping에서 cross-section fallback 허용

## 7. `only_qwen`의 실패 정책

`only_qwen`은 applied result에서 rule fallback을 사용하지 않는다.

대신 아래 실패 케이스를 별도 정의한다.

### 7.1 invalid JSON / schema mismatch

- 해당 window 전체를 `unknown_review_required` 상태로 기록
- block들은 deterministic하게 읽는 순서 기반 cluster로 임시 묶되
- purpose는 `unknown`
- `needs_review=true`

### 7.2 incoherent section ids

예:

- 동일 window 안에서 section id가 순서 없이 튀는 경우
- 모든 block이 서로 다른 section으로 분리되는 경우
- heading/body 관계가 완전히 깨지는 경우

처리:

- malformed assignment로 간주
- low-confidence cluster를 만든다
- trace에 `fallback_reason` 기록

### 7.3 over-segmentation / under-segmentation

postprocessing에서 아래를 보정한다.

- evidence/meta/disclaimer만 있는 cluster는 non-narrative section으로 강등
- 너무 짧은 body-only section은 인접 narrative section과 merge 후보로 본다
- 너무 긴 mega-section은 heading boundary가 강한 지점에서 review candidate로 표시한다

중요한 점은 이것이 “rule fallback”이 아니라 “deterministic cleanup”이라는 점이다.

## 8. 아티팩트 정의

공식 산출물은 아래로 고정한다.

### 8.1 `section_artifact.json`

- reference 문서와 source 문서 모두에서 공통 사용
- 공식 section contract
- generation과 review가 이 파일을 기준으로 동작

### 8.2 `section_block_assignments.json`

- qwen/shadow/only_qwen에서 생성
- per-block assignment를 저장
- model decision을 추적할 수 있는 최소 단위

### 8.3 `section_trace.json`

- qwen/shadow/only_qwen에서 생성
- input payload, raw response, parsed assignments, fallback reason, latency 저장

## 9. downstream 연결

### 9.1 reference parsing

reference 쪽에서는 `section_artifact.json`이 공식 section 결과다.

기존 `reference_template.section_order`는 더 이상 독립 휴리스틱 결과가 아니라:

`section_artifact.sections` 중 narrative section만 projection한 결과

로 재정의한다.

즉 `section_order`는 공식 artifact가 아니라 generation 친화적인 narrative skeleton이다.

### 9.2 source parsing

source 문서도 동일한 `section_artifact.json`을 사용한다.

generation은 아래 순서로 section-first mapping을 수행한다.

1. reference section을 본다
2. source section 후보를 찾는다
3. source section 하나 또는 소수 후보를 선택한다
4. 선택된 section 안에서 slot-to-fact mapping을 한다

### 9.3 generation plan 확장

`generation_plan`에는 최소한 아래 필드를 추가하는 방향으로 연동한다.

- `selected_section_id`
- `section_match_confidence`
- `section_match_reason`
- `cross_section_fallback_used`

## 10. 평가 기준

좋은 section 탐지는 아래를 만족해야 한다.

1. thesis heading과 supporting body가 같은 narrative section으로 유지된다.
2. evidence/meta/disclaimer가 narrative skeleton에 섞이지 않는다.
3. section 순서가 사람이 읽는 흐름과 대체로 일치한다.
4. headingless page에서도 body cluster가 과도하게 쪼개지지 않는다.
5. mixed layout에서도 side panel이 main narrative를 오염시키지 않는다.
6. low-quality 페이지는 공격적으로 잘게 나누기보다 `needs_review`로 보수적으로 남긴다.

## 11. 테스트 시나리오

### 11.1 공통 테스트

- heading candidate 검출
- boundary scoring
- purpose classification
- evidence/meta/disclaimer 분리
- multi-column narrative page
- headingless body cluster
- compliance/evidence page 제외

### 11.2 `qwen` / `shadow`

- rule candidate refinement가 trace에 남는가
- merge/split/purpose relabel disagreement가 집계되는가
- invalid response 시 deterministic fallback이 동작하는가

### 11.3 `only_qwen`

- rule candidate 없이 assignment를 assemble할 수 있는가
- cross-family source 문서에서 artifact를 만들 수 있는가
- invalid JSON 시 `unknown_review_required`로 degrade하는가
- semantic hints가 없어도 payload schema가 유지되는가
- section_block_assignments와 section_artifact가 서로 일관되는가

## 12. 권장 롤아웃

1. `rule` baseline 유지
2. `qwen` hybrid를 기본 권장 모드로 도입
3. `shadow`로 안정성 검증
4. source 문서군 일부에 `only_qwen` 실험 적용
5. cross-family source에서 section-first mapping 품질 비교

## 13. 기본 결정 사항

- 기본 권장 모드는 `qwen`
- `only_qwen`은 experimental opt-in
- `document_family`는 hard gate가 아니라 soft prior
- reference와 source 모두 같은 section artifact 계약을 사용
- section 탐지는 family보다 section capability와 structural consistency를 우선한다
