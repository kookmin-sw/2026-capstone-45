# 1차 후보 수집 구현 계획

## 요약
- 기준 구현 대상은 [tool_search_source_document.py](/c:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/tool_search_source_document.py), 연동 조정 대상은 [create_document.py](/c:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/create_document.py)다.
- 검색기는 top-3 확정기가 아니라 `1차 후보 10~12개를 수집해 반환하는 도구`로 바뀐다.
- BM25는 첫 구현에서 로컬 구현으로 넣되, 인터페이스를 분리해 이후 외부 계층으로 교체 가능하게 만든다.
- LLM query expansion은 검색기 내부에서 새 클라이언트를 만들지 않고, `create_document`가 쓰는 OpenAI client를 주입받아 query당 1회만 호출한다.
- 한글 고유명사형 토큰 추출과 도메인 suffix/사전 규칙은 넣지 않는다.

## 핵심 구현 변경
- `ToolSearchSourceDocument.__init__`를 `docs, client, tracer=None, bm25_client=None` 형태로 확장한다.
- `tool_search_source_document.py`에 `QueryProfile`, `RetrievalCandidate`, BM25 어댑터 인터페이스를 추가한다.
- query preprocessing은 tokenization, lowercase, stopword 제거, 길이 필터만 수행한다.
- entity candidate는 `영문 대문자 토큰`, `숫자 코드`, `숫자+단위`만 추출한다.
- LLM expansion은 JSON only로 `entities`, `semantic_queries`, `lexical_queries`를 생성한다.
- semantic retrieval은 `raw_query + llm semantic queries`에 대해 multi-query union을 수행한다.
- BM25 retrieval은 `raw lexical query + llm lexical query`를 대상으로 수행한다.
- entity grounding은 exact-hit recall safety net으로만 사용한다.
- merge/dedupe는 `record_id` 기준으로 수행하고, ordering은 최종 선택이 아니라 LLM 입력 순서 정렬용으로만 사용한다.
- 반환 포맷은 top-3 HTML이 아니라 candidate bundle 텍스트로 바꾼다.
- trace는 `search/first_stage.jsonl`로 통일하고, 단계별 raw snapshot도 저장한다.
- `create_document.py`는 search tool에 client를 주입하고, prompt/tool description을 새 계약에 맞게 조정한다.

## 구현 단계와 보고 방식
### 1단계. 검색기 골격 정리
- `SEARCH_RESULT_LIMIT=3` 전제를 제거하고 `FIRST_STAGE_LIMIT=12`, `TOOL_DISPLAY_LIMIT=12`로 바꾼다.
- `ToolSearchSourceDocument` 생성자에 `client`, `bm25_client`, `tracer`를 받도록 바꾼다.
- 도구 description을 `top-3 page selector`에서 `first-stage candidate collector` 설명으로 수정한다.
- 이 단계 보고:
  - 바뀐 생성자 시그니처
  - 상수/설명문 변경
  - 이후 단계에서 건드릴 함수 경계

### 2단계. Query preprocessing
- `QueryProfile` 추가
- `_extract_query_tokens()`
- `_extract_entity_candidates()`
- `_build_fallback_lexical_query()`
- `_preprocess_query()`
- tokenization, stopword 제거, entity candidate 추출, fallback lexical query 생성을 연결한다.
- `query_preprocessed` trace 이벤트를 추가한다.
- 이 단계 보고:
  - tokenization 규칙
  - entity candidate 추출 범위
  - 한글 고유명사형 토큰을 제외한 이유
  - fallback lexical query 생성 방식
  - 전처리 결과를 trace에 어떻게 남겼는지

### 3단계. LLM expansion
- expansion prompt 상수 추가
- `_request_query_expansion()`
- `_normalize_query_expansion_payload()`
- parse 실패 시 fallback 경로 구현
- `llm_expansion_requested`, `llm_expansion_saved`, `llm_expansion_failed` trace 이벤트와 raw snapshot 저장을 붙인다.
- 이 단계 보고:
  - 사용 모델 선택 규칙
  - JSON parse 실패 처리
  - raw response 저장 경로
  - fallback이 언제 발동하는지
  - preprocessing 결과와 expansion 결과가 어떻게 합쳐지는지

### 4단계. Semantic retrieval 채널 구현
- 기존 단일 query Chroma 호출을 `raw_query + semantic_queries` union으로 바꾼다.
- 동일 `record_id`는 best distance만 유지한다.
- semantic hit payload를 candidate intermediate 형식으로 정리한다.
- `semantic_retrieval_completed` trace 이벤트를 추가한다.
- 이 단계 보고:
  - semantic query 개수 제한
  - union/dedupe 방식
  - 기존 `_rerank_records`와의 차이
  - trace에 남기는 hit 정보

### 5단계. 로컬 BM25 채널 구현
- 새 BM25 모듈 또는 같은 파일 내 인터페이스/구현 추가
- `BM25Hit`
- `BM25SearchClient`
- `LocalBM25SearchClient`
- lexical query별 BM25 검색과 union 구현
- `bm25_retrieval_completed`, `bm25_retrieval_failed` trace 이벤트를 추가한다.
- 이 단계 보고:
  - BM25 문서 단위가 `SearchRecord`인 점
  - score 계산 방식
  - 추후 외부 계층으로 교체 가능한 인터페이스
  - 실패 시 계속 진행 정책

### 6단계. Entity grounding 채널 추가
- preprocessing/expansion에서 나온 entity 후보를 exact-hit 기준으로 grounding
- entity hit를 별도 채널로 candidate에 추가
- `entity_grounding_completed` trace 이벤트를 추가한다.
- 이 단계 보고:
  - grounding 기준
  - hard filter를 쓰지 않는 이유
  - 오탐 방지 기준
  - trace에 남기는 entity hit 정보

### 7단계. Merge / ordering / output 변경
- `record_id` 기준 merge
- `channels` 합치기
- `semantic_distance`, `bm25_score`, `entity_score` 누적
- ordering score 계산
- 최종 `10~12개`를 선택
- `_format_search_output()`을 candidate bundle 전용으로 교체
- `candidate_merge_completed`, `first_stage_ordered`, `first_stage_returned` trace 이벤트를 추가한다.
- 이 단계 보고:

## 가정과 기본값
- BM25는 첫 구현에서 로컬 구현으로 넣고, 인터페이스를 분리해 둔다.
- 새 외부 의존성은 추가하지 않는다.
- expansion 모델은 `OPENAI_QUERY_EXPANSION_MODEL`이 있으면 우선, 없으면 `OPENAI_MODEL`을 사용한다.
- context expansion은 이번 구현 범위에 넣지 않는다.
- 앞으로 구현 제안에 닫힌 도메인 사전, suffix 기반 매칭, 업종명/기관명 휴리스틱이 들어가면 설계 위반 위험으로 먼저 지적한다.
