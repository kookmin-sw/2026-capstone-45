# Section Qwen Detector Implementation Design

이 문서는 `11-1. section.md`, `11-2. section_artifact.md`, `13. section 정리.md`를 이어서, `section detection`에 Qwen을 메인 추론기로 도입하는 구현 설계를 정의한다.

이번 문서의 초점은 일반적인 `qwen hybrid refinement`가 아니라, `only_qwen` 모드를 **Qwen-primary section detector**로 실제 구현 가능한 수준까지 구체화하는 것이다.

핵심 목표는 아래와 같다.

- Qwen이 block grouping과 block role assignment의 메인 결정자가 되도록 한다.
- 하지만 최종 `section artifact`는 여전히 deterministic layer가 조립하고 검증한다.
- section 의미는 block assignment 단계에서 과도하게 결정하지 않고, **section assembly 이후 refinement pass**에서 별도로 생성한다.
- official `section_purpose`는 closed taxonomy를 유지하되, 생성형 purpose 필드로 표현력을 보강한다.

이 문서는 구현자가 추가 의사결정을 하지 않고도 바로 코딩에 들어갈 수 있는 수준의 계약을 정의하는 것을 목표로 한다.

## TL;DR

- `only_qwen`은 `free-form section generator`가 아니라 `Qwen-primary structured assignment detector`다.
- block-level Qwen은 오직 아래만 반환한다.
  - `section_local_id`
  - `block_role_in_section`
  - `boundary_before`
  - `boundary_after`
  - `section_purpose` 후보
  - `confidence`
  - `reason`
- Qwen은 block-level 응답에서 `generated_section_purpose_name/description`를 반환하지 않는다.
- deterministic layer는 schema validation, block coverage validation, contiguous cluster assembly, confidence 계산, cleanup, degraded artifact 생성을 담당한다.
- section 의미의 개방형 표현은 **section assembly 이후 section-level refinement pass**에서 생성한다.
- official `section_purpose`는 closed taxonomy를 유지하고, open meaning은 `generated_section_purpose_*`에 보존한다.
- semantic 결과는 detector payload에 포함하되 truth가 아니라 configurable prior다.
- 이를 위해 `semantic_hint_mode = none | soft | strong`를 section detector 설정에 추가한다.

## 1. 배경과 현재 코드 상태

현재 코드베이스에는 아래 상태가 이미 존재한다.

- `SectionDetectionConfig`는 `rule | qwen | shadow | only_qwen` 모드를 이미 가진다.
- `detect_sections(...)`는 현재 rule baseline만 실제 동작한다.
- `reference_pipeline`은 section detector를 호출하지만 `SectionDetectionConfig(mode="rule")`를 하드코딩한다.
- `SectionArtifact`, `DetectedSection`, `SectionBlockAssignment`, `SectionTraceEntry` 타입은 이미 있다.
- generation은 이미 `section-first planning` 구조로 이동해 있으며, section detector 결과가 downstream quality에 직접 영향을 준다.

즉 지금 필요한 것은 새 artifact 타입을 더 만드는 일이 아니라, **existing section contract 위에 Qwen-primary detection path를 안전하게 올리는 것**이다.

특히 중요한 제약은 아래와 같다.

1. detector 결과는 reference/source 모두에서 공통으로 써야 한다.
2. detector는 malformed model output에도 파이프라인을 깨뜨리면 안 된다.
3. generation은 stable `official purpose`를 필요로 한다.
4. 동시에 새로운 문서군에서는 기존 taxonomy 밖의 의미도 포착할 수 있어야 한다.

따라서 이번 설계의 중심 원칙은 다음 한 줄로 요약된다.

> Qwen은 관계를 추론하고, deterministic layer는 계약을 보장한다.

## 2. 목표와 비목표

### 2.1 목표

- `only_qwen` 모드를 실제 구현 가능한 detector 모드로 구체화
- block grouping은 Qwen이 주도하되 final artifact는 deterministic assembler가 생성
- section 의미는 block assignment와 분리하여 section-level refinement pass에서 생성
- official `section_purpose`는 closed taxonomy 유지
- generated purpose를 병행해서 taxonomy 밖 의미를 보존
- semantic hint를 detector 품질 향상에 활용하되, prior 강도를 설정값으로 제어
- 실패 시 rule 덮어쓰기가 아니라 degraded review artifact를 생성

### 2.2 비목표

- `qwen` hybrid 모드의 의미 재정의
- `shadow` 모드 정책 변경
- generation matching 구조 자체 재설계
- chart/table 내부 구조 복원
- overlapping block window stitching의 full general solution

이번 문서의 구현 대상은 `only_qwen` detector path와 그 supporting contract다.

## 3. 모드 의미 유지 원칙

이번 설계에서는 기존 section 모드 의미를 아래처럼 유지한다.

### 3.1 `rule`

- 현재 baseline detector
- deterministic heuristic 기반
- regression 기준선

### 3.2 `qwen`

- hybrid refinement 모드 유지
- rule이 candidate/boundary를 만들고 Qwen은 refine/merge/split/purpose 보정만 담당
- 이번 문서의 메인 구현 대상 아님

### 3.3 `shadow`

- Qwen을 실행하되 official applied result는 rule 유지
- trace와 비교 분석용

### 3.4 `only_qwen`

- 이번 문서의 메인 구현 대상
- block-to-section assignment는 Qwen이 메인 결정
- deterministic preprocessing/validation/assembly/cleanup은 유지
- failure 시 rule applied fallback 사용 안 함
- 대신 reviewable degraded artifact 생성

즉 이번 문서에서 말하는 “Qwen-primary section detector”는 곧 `only_qwen`의 구현 정의다.

## 4. 전체 데이터 흐름

`only_qwen`의 공식 흐름은 아래와 같다.

```text
preprocess
-> Qwen block assignment
-> validation
-> deterministic assembly
-> deterministic cleanup
-> Qwen section-level refinement
-> final section artifact
-> trace / diagnostics 저장
```

여기서 중요한 해석은 다음과 같다.

- block grouping은 Qwen이 주도한다.
- final section artifact는 deterministic layer가 만든다.
- section-level refinement는 grouping이 끝난 뒤 의미를 더 잘 설명하기 위한 보조 단계다.
- refinement pass는 section merge/split이나 block reassignment를 하지 않는다.

## 5. Qwen 책임과 deterministic 책임

이 절은 구현자가 헷갈리지 않도록 책임 경계를 명확히 고정한다.

### 5.1 block-level Qwen 책임

Qwen은 block window를 입력으로 받아, 각 block에 대한 structured assignment만 생성한다.

block-level Qwen의 공식 책임은 아래로 제한한다.

1. 각 block의 `section_local_id` 제안
2. 각 block의 `block_role_in_section` 추정
3. 각 block의 `boundary_before`, `boundary_after` 추정
4. 각 block이 속한 section의 `section_purpose` 후보 추정
5. assignment 단위 `confidence`와 `reason` 제공

Qwen은 아래를 해서는 안 된다.

- global `section_id` 생성
- 최종 `DetectedSection[]` 직접 생성
- block 삭제 또는 신규 block 생성
- section merge/split action 직접 정의
- `generated_section_purpose_name/description` 생성
- cross-window stitching 결정

즉 block-level Qwen은 어디까지나 **section assembly input producer**다.

### 5.2 deterministic layer 책임

deterministic layer는 model output을 official artifact로 승격시키는 계층이다.

공식 책임은 아래와 같다.

1. preprocessing
  - ordered block sequence 생성
  - page/window 구성
  - detector payload 생성

2. validation
  - JSON parse
  - schema validation
  - enum/type validation
  - block coverage validation
  - duplicate assignment 검증

3. assembly
  - contiguous cluster 생성
  - section title 선택
  - global `section_id` 부여
  - role histogram 계산
  - section confidence 계산
  - `DetectedSection` 조립

4. cleanup
  - meta/disclaimer/evidence-only cluster 정리
  - tiny fragment review flag
  - incoherent section review flag
  - impossible purpose-role combination 강등

5. failure handling
  - degraded artifact 생성
  - fallback reason 기록
  - trace 저장

즉 deterministic layer는 **모델 결과가 틀려도 시스템이 깨지지 않게 하는 계층**이다.

## 6. windowing 정책

v1에서 `only_qwen` detector는 page-based window를 기본으로 한다.

### 6.1 기본 정책

- 기본 단위는 `1 page = 1 window`
- page 내부 block는 reading order 기준으로 정렬
- payload는 page 단위 context를 함께 가진다

### 6.2 cross-page 허용 조건

`allow_cross_page_section=true`이고 아래 조건을 모두 만족할 때만 `2-page window`를 허용한다.

- 두 페이지가 인접함
- 두 페이지 모두 `body_narrative`
- 둘 다 low-quality page가 아님

이 조건을 만족하지 않으면 cross-page only_qwen grouping은 수행하지 않는다.

### 6.3 block 수 초과 정책

window 내 block 수가 `max_blocks_per_window`를 초과하면 v1에서는 일반적인 sliding/overlap stitching을 하지 않는다.

정책은 아래로 고정한다.

- model 호출을 생략하거나, 해당 window를 degraded path로 바로 내린다
- `fallback_reason = "window_too_large"`를 기록한다
- partial chunk merge는 v1 범위에서 제외한다

이 결정은 구현 복잡도와 failure surface를 줄이기 위한 것이다.

## 7. semantic hint 정책

block별 semantic 분석 결과는 detector payload에 포함한다. 다만 이 값은 truth가 아니라 prior다.

이를 위해 `SectionDetectionConfig`에 아래 필드를 추가한다.

```python
semantic_hint_mode: Literal["none", "soft", "strong"] = "soft"
```

### 7.1 `none`

- semantic 관련 필드를 detector payload에 넣지 않는다
- ablation baseline 또는 pure layout/text detector 실험용

### 7.2 `soft`

- semantic 결과를 참고 정보로만 넣는다
- detector는 text/layout/neighbor 관계와 함께 종합 판단한다
- semantic이 틀려도 detector 전체가 semantic에 끌려가지 않도록 프롬프트를 설계한다
- 기본값

### 7.3 `strong`

- semantic 결과를 강한 prior로 넣는다
- detector가 `metadata`, `author_info`, `disclaimer`, `evidence`를 narrative에서 더 강하게 분리하도록 유도한다
- 다만 low-confidence semantic은 deterministic preprocessing에서 약화시킨다

### 7.4 payload에 넣을 semantic 정보

raw semantic 전체를 그대로 넣기보다 detector가 쓰기 쉬운 형태로 요약해서 넣는다.

권장 필드:

- `generic_role`
- `domain_role`
- `generated_role_name`
- `role_confidence`
- `semantic_source`
- `semantic_needs_review`
- `semantic_group_hint`

`semantic_group_hint`는 preprocessing에서 생성하는 coarse summary다.

예시:

- `main_title`, `section_heading`, `summary`, `body` -> `narrative`
- `metadata`, `author_info` -> `meta`
- `evidence` -> `evidence`
- `disclaimer` -> `disclaimer`
- 나머지 -> `unknown`

이 coarse hint는 detector prompt 단순화에 도움이 된다.

## 8. block-level detector payload

`only_qwen` block assignment용 payload는 아래 구조를 공식 계약으로 삼는다.

```json
{
  "task": {
    "prompt_version": "section-v1",
    "mode": "only_qwen",
    "document_family": "article",
    "language": "ko"
  },
  "page_window": [1],
  "page_context": {
    "page_archetype": "body_narrative",
    "column_count": 2,
    "dominant_layout_pattern": "two_column",
    "quality_score": 0.81
  },
  "ordered_blocks": [
    {
      "block_id": "blk-001",
      "page": 1,
      "reading_order": 0,
      "canonical_label": "section_heading",
      "raw_label": "sec_1",
      "text": "Summary",
      "bbox_norm": [0.08, 0.12, 0.42, 0.18],
      "text_quality_score": 0.94,
      "semantic_hint": {
        "mode": "soft",
        "generic_role": "section_heading",
        "domain_role": "thesis_heading",
        "generated_role_name": "summary_heading_block",
        "role_confidence": 0.82,
        "semantic_group_hint": "narrative"
      }
    }
  ],
  "response_schema": {
    "page_window": ["int"],
    "assignments": [
      {
        "block_id": "string",
        "section_local_id": "string",
        "block_role_in_section": "heading|body|evidence|meta|disclaimer|orphan",
        "boundary_before": "boolean",
        "boundary_after": "boolean",
        "section_purpose": "main_argument|thesis|supporting_argument|investment_summary|evidence_panel|metadata|disclaimer|unknown",
        "confidence": "float 0..1",
        "reason": "short sentence"
      }
    ]
  }
}
```

## 9. block-level detector response 계약

block-level detector는 반드시 **모든 입력 block에 대해 정확히 1개 assignment**를 반환해야 한다.

권장 예시는 아래와 같다.

```json
{
  "page_window": [1],
  "assignments": [
    {
      "block_id": "blk-001",
      "section_local_id": "sec_a",
      "block_role_in_section": "heading",
      "boundary_before": true,
      "boundary_after": false,
      "section_purpose": "thesis",
      "confidence": 0.93,
      "reason": "Narrative heading that opens a new section."
    },
    {
      "block_id": "blk-002",
      "section_local_id": "sec_a",
      "block_role_in_section": "body",
      "boundary_before": false,
      "boundary_after": false,
      "section_purpose": "thesis",
      "confidence": 0.88,
      "reason": "Body text continuing the same argument."
    }
  ]
}
```

이 응답에는 `generated_section_purpose_name`, `generated_section_purpose_description`를 포함하지 않는다.

그 이유는 다음과 같다.

- section 의미는 block 단위보다 section assembled context에서 더 안정적으로 추정된다
- detector 응답 계약을 좁게 유지해야 validation이 단순해진다
- block grouping과 section 의미 refinement를 분리하면 failure isolation이 쉬워진다

## 10. validation 정책

block assignment 응답은 적용 전에 반드시 validation을 거친다.

### 10.1 실패 코드

최소 아래 실패 유형을 구분한다.

- `invalid_json`
- `schema_mismatch`
- `missing_block_coverage`
- `duplicate_assignment`
- `unknown_block_id`
- `invalid_role`
- `invalid_purpose`
- `window_too_large`

### 10.2 validation 규칙

1. JSON object 형태여야 한다
2. `page_window`는 요청 window와 일치해야 한다
3. `assignments`는 입력 block 수와 정확히 동일한 길이를 가져야 한다
4. 각 `block_id`는 입력 block 중 하나여야 한다
5. 같은 `block_id`가 두 번 이상 나오면 안 된다
6. `block_role_in_section`는 허용 enum 중 하나여야 한다
7. `section_purpose`는 허용 official taxonomy 중 하나여야 한다
8. `confidence`는 0~1 범위여야 한다

## 11. degraded artifact 정책

`only_qwen`은 applied path에서 rule fallback을 사용하지 않는다.

validation 또는 model execution이 실패하면, 해당 window는 degraded artifact를 생성한다.

### 11.1 degraded path 원칙

- detector를 silent fail 시키지 않는다
- 가능한 한 complete artifact를 남긴다
- 결과는 low-confidence review artifact임을 분명히 표시한다

### 11.2 degraded artifact 생성 방식

degraded artifact는 기존 deterministic baseline을 완전히 대체하지 않는다. 그러나 low-level cluster를 만들기 위한 최소한의 heuristic은 허용한다.

권장 정책:

- block role은 existing heuristic projection을 재사용 가능
- section_local_id는 reading-order cluster 기준으로 임시 생성
- official purpose는 `unknown`
- section confidence는 low fixed value 또는 conservative aggregate
- 모든 assignment/section에 `needs_review=true`

즉 degraded artifact는 “정답 대체”가 아니라 “사람이 검토 가능한 임시 구조”다.

## 12. deterministic assembly

validation을 통과한 assignment는 deterministic assembler가 `DetectedSection[]`로 조립한다.

### 12.1 assembly 규칙

1. reading order 기준으로 assignment를 다시 정렬한다
2. 같은 `section_local_id`라도 **contiguous cluster** 단위로 section을 만든다
3. 비연속 cluster가 반복되면 하나의 section으로 강제 합치지 않는다
4. 각 cluster에 대해 아래를 계산한다
  - `title`
  - `purpose`
  - `page_start`, `page_end`
  - `block_ids`
  - `heading_block_ids`
  - `body_block_ids`
  - `evidence_block_ids`
  - `meta_block_ids`
  - `dominant_roles`
  - `is_narrative`
  - `confidence`
5. assembler가 global `section_id`를 `sec-001`, `sec-002` 형식으로 부여한다

### 12.2 title 선택 규칙

title은 deterministic하게 선택한다.

우선순위:

1. heading block 첫 번째 non-empty text
2. 없으면 first meaningful body/meta text
3. 그래도 없으면 `None`

### 12.3 purpose 집계 규칙

cluster 안의 `section_purpose` 후보를 vote로 집계한다.

원칙:

- 최빈값 사용
- tie면 `unknown`
- low-confidence assignment 비율이 높으면 `needs_review=true`

이 단계의 purpose는 여전히 official purpose이며, generated purpose가 아니다.

## 13. deterministic cleanup

assembly 후에는 structure sanity check를 수행한다.

### 13.1 cleanup 목표

- impossible structure를 조용히 허용하지 않기
- aggressively rewrite 하기보다 review flag를 보수적으로 세우기

### 13.2 규칙

1. meta/disclaimer-only cluster
  - non-narrative 유지
  - purpose가 narrative면 `unknown` 또는 metadata/disclaimer로 강등

2. evidence-only cluster
  - `evidence_panel` 우선

3. tiny fragment
  - 자동 merge보다 `needs_review=true` 우선

4. mega-section
  - 자동 split보다 `needs_review=true` 우선

5. purpose-role 모순
  - 예: `purpose=metadata`인데 body dominant
  - 예: `purpose=disclaimer`인데 heading/body 위주
  - 이런 경우 official purpose를 `unknown`으로 내리고 review flag

cleanup은 section grouping 자체를 크게 재구성하지 않는다.

## 14. official purpose와 generated purpose

이번 설계에서 가장 중요한 결정 중 하나는 **official purpose와 generated purpose를 분리하는 것**이다.

### 14.1 official purpose

official `section_purpose`는 closed taxonomy를 유지한다.

허용값:

- `main_argument`
- `thesis`
- `supporting_argument`
- `investment_summary`
- `evidence_panel`
- `metadata`
- `disclaimer`
- `unknown`

이 값은 downstream matching과 generation에서 안정적 contract로 사용된다.

### 14.2 generated purpose

taxonomy 밖 의미를 보존하기 위해 section-level generated purpose를 추가한다.

필드:

- `generated_section_purpose_name`
- `generated_section_purpose_description`

이 필드는 표현력 보강용이며, official matching contract는 아니다.

즉 원칙은 아래와 같다.

- stable contract는 `section_purpose`
- richer meaning은 `generated_section_purpose_*`

## 15. section-level refinement pass

`generated purpose`는 block-level detector가 아니라 **section assembly 이후 section-level refinement pass**에서 생성한다.

이 단계는 이번 문서의 핵심 추가 사항이다.

### 15.1 refinement pass가 필요한 이유

block assignment 단계에서는 section 의미를 충분히 안정적으로 설명하기 어렵다.

이유:

- block 단위 맥락이 좁다
- section title/body/evidence 조합을 모두 본 뒤 의미를 정하는 것이 더 안정적이다
- official purpose와 generated purpose를 한 번에 생성하면 detector 계약이 과도하게 커진다

따라서 refinement는 section assembled context 위에서 수행한다.

### 15.2 refinement pass 입력

refinement pass 입력은 assembled section packet 목록이다.

권장 입력 필드:

- `section_id`
- `title`
- `purpose`
- `dominant_roles`
- `is_narrative`
- `page_start`, `page_end`
- `heading_block_ids`, `body_block_ids`, `evidence_block_ids`, `meta_block_ids`
- section 대표 text sample
- optional semantic histogram
- optional representative semantic role sample
- section confidence
- current `needs_review`

중요한 점은 refinement pass에 raw block 전체를 다시 넘기지 않는다는 것이다.

refinement는 block regrouping을 위한 단계가 아니라 section meaning refinement 단계다.

### 15.3 refinement pass의 Qwen 책임

Qwen은 아래만 수행한다.

1. current official `section_purpose` 유지/보정 제안
2. `generated_section_purpose_name` 생성
3. `generated_section_purpose_description` 생성
4. `needs_review` 제안

Qwen은 아래를 수행하지 않는다.

- section merge
- section split
- block reassignment
- boundary 수정

즉 refinement pass는 **section meaning refinement only** 로 고정한다.

### 15.4 refinement pass 응답 계약

권장 예시는 아래와 같다.

```json
{
  "sections": [
    {
      "section_id": "sec-003",
      "section_purpose": "thesis",
      "generated_section_purpose_name": "market_wrapup_lead",
      "generated_section_purpose_description": "An opening narrative section summarizing the main market move and framing the rest of the report.",
      "needs_review": false,
      "reason": "The section is narrative and functions as the lead framing argument."
    }
  ]
}
```

### 15.5 refinement acceptance 정책

refinement 결과는 아래 정책으로 적용한다.

- official `section_purpose`가 허용 taxonomy 안이면 유지/보정 허용
- taxonomy 밖이면 official purpose는 기존값 유지 또는 `unknown`
- generated fields는 저장
- low-confidence 또는 ambiguous case면 `needs_review=true`

즉 generated purpose는 저장할 수 있지만, official purpose는 항상 stable contract 범위 안에 있어야 한다.

### 15.6 refinement 실패 정책

refinement pass가 실패해도 block grouping과 assembled section은 유지한다.

정책:

- official purpose는 기존 assembly 결과 유지
- generated purpose 필드는 비움
- `fallback_reason`을 trace에 기록
- 필요 시 `needs_review=true`

refinement는 meaning enhancer이지 artifact survival dependency가 아니다.

## 16. 타입 변경

이번 설계를 위해 `DetectedSection`에 아래 optional 필드를 추가한다.

```python
generated_section_purpose_name: Optional[str] = None
generated_section_purpose_description: Optional[str] = None
```

추가로 `SectionTraceEntry`는 refinement trace를 담을 수 있게 stage를 구분해야 한다.

권장 확장:

```python
stage: Literal["block_assignment", "section_refinement"]
```

또는 기존 구조를 유지한다면 `input_payload`와 `assembled_sections`의 해석 기준을 `stage`에 따라 다르게 정의한다.

핵심은 trace가 아래 두 단계를 분리해서 남길 수 있어야 한다는 점이다.

- block assignment trace
- section refinement trace

## 17. reference/source 파이프라인 통합

이번 설계는 reference/source 공통 detector를 전제로 한다.

### 17.1 `detect_sections(...)`의 공식 구조

`detect_sections(...)`는 mode dispatcher가 된다.

권장 구조:

- `rule` -> existing heuristic path
- `qwen` -> hybrid refine path
- `shadow` -> trace-only compare path
- `only_qwen` -> 이번 문서의 primary detector path

### 17.2 `reference_pipeline` 연결

`build_reference_template()` / `parse_reference()`는 section detector 설정을 외부에서 받을 수 있어야 한다.

현재처럼 `SectionDetectionConfig(mode="rule")`를 하드코딩하지 않고, parse-reference CLI에서도 section detector 설정을 줄 수 있도록 한다.

### 17.3 CLI 변경

`parse-reference`에 아래 인자를 추가한다.

- `--section-mode`
- `--section-model`
- `--section-runtime`
- `--section-device`

기본값은 backward compatibility를 위해 `rule`로 둔다.

refinement pass는 `only_qwen` 내부 공식 단계로 간주하므로, v1에서는 별도 CLI 토글을 추가하지 않는다.

## 18. trace와 diagnostics

`section_trace.json`은 `only_qwen`, `qwen`, `shadow`에서 생성한다.

이번 설계에서는 trace가 최소 아래 두 단계를 담을 수 있어야 한다.

### 18.1 block assignment trace

- detector payload
- raw response
- parsed assignments
- validation 결과
- fallback reason

### 18.2 section refinement trace

- section packet payload
- raw response
- parsed refinement result
- applied refinement
- fallback reason

이 trace가 있어야:

- detector가 왜 해당 grouping을 만들었는지
- refinement가 왜 해당 generated purpose를 만들었는지
- 어떤 단계에서 실패했는지

를 나중에 추적할 수 있다.

## 19. 테스트 기준

### 19.1 block assignment 단계

- 모든 입력 block이 정확히 한 번 assignment되는지
- invalid JSON이면 degraded artifact가 생성되는지
- schema mismatch이면 degraded artifact가 생성되는지
- duplicate assignment이면 degraded artifact가 생성되는지
- unknown block id가 들어오면 degraded artifact가 생성되는지

### 19.2 assembly 단계

- contiguous cluster 기준으로 section이 조립되는지
- 비연속 동일 `section_local_id`가 하나의 section으로 강제 합쳐지지 않는지
- global `section_id`가 deterministic하게 부여되는지

### 19.3 cleanup 단계

- meta/disclaimer-only cluster가 non-narrative로 남는지
- evidence-only cluster가 `evidence_panel`로 정리되는지
- impossible purpose-role 조합이 `unknown + needs_review`로 강등되는지

### 19.4 refinement 단계

- assembled section마다 generated purpose가 생성되는지
- generated purpose가 taxonomy 밖 의미를 표현할 수 있는지
- official purpose가 taxonomy 밖 값으로 오염되지 않는지
- refinement 실패 시 block grouping이 바뀌지 않는지
- refinement 실패 시 generated fields만 비워지고 fallback reason이 trace에 남는지

### 19.5 통합

- `section_trace.json`에 block assignment와 section refinement 두 단계가 모두 기록되는지
- `section_artifact.json`에 official purpose와 generated purpose가 함께 저장되는지
- `rule` 모드 output contract가 변하지 않는지
- `qwen` hybrid와 `shadow` 의미가 유지되는지

## 20. 구현 순서

권장 구현 순서는 아래와 같다.

1. `SectionDetectionConfig`, `DetectedSection`, `SectionTraceEntry` 타입 확장
2. only_qwen block assignment payload/schema/parser 구현
3. validation + degraded artifact path 구현
4. deterministic assembly + cleanup 분리
5. section-level refinement payload/schema/parser 구현
6. trace stage 분리 저장 구현
7. `detect_sections(...)` mode dispatcher 정리
8. `reference_pipeline`과 CLI 설정 연결
9. sample 기반 trace 검증

이 순서를 따르면 detector survivability를 먼저 확보하고 refinement를 나중에 얹을 수 있다.

## 21. 결론

이번 설계의 핵심은 `Qwen이 모든 것을 직접 생성하게 두지 않는다`는 점이다.

- Qwen은 block 관계와 section 의미를 추론한다.
- deterministic layer는 계약을 보장하고 실패를 복구한다.
- official purpose는 stable taxonomy에 묶는다.
- taxonomy 밖 의미는 section-level refinement에서 generated purpose로 보존한다.

즉 `only_qwen`은 자유 생성기가 아니라 아래 구조로 이해해야 한다.

> Qwen-primary structured detector + deterministic artifact assembler + section-level meaning refinement

이 구조를 따르면 detector의 표현력은 넓히면서도, generation과 diagnostics가 요구하는 안정적인 contract는 유지할 수 있다.
