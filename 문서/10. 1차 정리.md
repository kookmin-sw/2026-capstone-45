# 1차 정리

이 문서는 `/root/Desktop/workspace/ja/allm/allm` 기준으로, 지금 실제 구현되어 있는 프로젝트의 전체 프로세스를 코드 중심으로 정리한 문서다.  
중요한 전제는 이 프로젝트가 현재 `완성된 end-user 서비스 전체`를 구현한 상태라기보다, `reference 문서를 OCR 결과로부터 읽어 canonical block 구조와 semantic overlay를 만들고, 그것을 artifact와 HTML 시각화로 저장하는 pipeline`을 구현한 상태라는 점이다.

즉, 현재 구현의 중심은 아래 두 CLI 흐름이다.

- `parse-reference`
- `visualize-reference`

실제 동작의 핵심은 `reference_pipeline.py`에 있고, semantic 확장은 `semantic.py` 계열에, 시각화는 `visualize.py`에 있다.

## 1. 프로젝트가 지금 하는 일 한 줄 요약

현재 프로젝트는 다음 과정을 수행한다.

```text
reference 입력
-> OCR 결과 파일 탐색
-> Dolphin / Paddle OCR block 로딩
-> canonical block 정규화
-> 두 OCR 엔진 결과 융합
-> 페이지 유형 / 문서군 / 언어 분석
-> semantic role 부여(rule 또는 Qwen)
-> reference template 생성
-> artifact JSON 저장
-> 선택적으로 HTML 시각화 생성
```

이 pipeline의 최종 산출물은 대략 아래 5종이다.

- `reference_template.json`
- `canonical_pages.json`
- `semantic_overlay.json`
- `parser_diagnostics.json`
- `reference_visualization.html`

Qwen 또는 shadow 모드를 쓰면 추가로 아래도 생길 수 있다.

- `semantic_trace.json`

## 2. 진입점: CLI 구조

구현된 CLI 진입점은 `allm/cli.py` 하나다.

### 2.1 `parse-reference`

이 명령은 reference 문서 1건을 읽어 artifact를 생성한다.

입력 옵션:

- `--job-id`
- `--reference`
- `--artifacts-root`
- `--ocr-results-root`
- `--semantic-mode`
- `--semantic-model`
- `--semantic-runtime`
- `--semantic-device`

실행 흐름:

1. `SemanticConfig`를 생성한다.
2. `parse_reference()`를 호출한다.
3. 반환된 artifact 경로들을 JSON으로 출력한다.

예시:

```bash
python -m allm.cli parse-reference \
  --job-id financial2-00 \
  --reference OCR_results/surya/financial2-00.png \
  --artifacts-root artifacts \
  --ocr-results-root OCR_results \
  --semantic-mode rule
```

### 2.2 `visualize-reference`

이 명령은 이미 생성된 reference artifact를 읽어 HTML 시각화 파일을 만든다.

입력 옵션:

- `--artifact-dir`
- `--output`
- `--reference-source`

실행 흐름:

1. `render_reference_visualization()`를 호출한다.
2. HTML 파일 경로를 JSON으로 출력한다.

예시:

```bash
python -m allm.cli visualize-reference \
  --artifact-dir artifacts/financial2-00/01_reference \
  --reference-source OCR_results/surya/financial2-00.png
```

## 3. 현재 구현의 책임 범위

현재 코드 기준으로 보면 프로젝트의 책임은 크게 6개로 나뉜다.

### 3.1 OCR 결과 위치 찾기

- 파일: `allm/ocr_locator.py`
- 역할: reference 입력으로부터 대응되는 OCR 결과 파일들을 찾아 `PageSource` 목록을 만든다.

### 3.2 OCR 엔진 결과 로딩

- 파일: `allm/adapters.py`
- 역할: Dolphin / Paddle 결과를 읽어 `EnginePage`, `RawEngineBlock` 구조로 변환한다.

### 3.3 canonical block 생성과 fusion

- 파일: `allm/canonicalize.py`, `allm/fusion.py`
- 역할: 엔진별 raw label을 공통 canonical label로 매핑하고, 서로 겹치는 block들을 합쳐 최종 `FusedPage`를 만든다.

### 3.4 페이지 분석과 템플릿 메타데이터 추출

- 파일: `allm/archetypes.py`, `allm/template.py`
- 역할: 페이지 archetype, 문서군, 언어, 섹션 순서, style token, image slot 등을 만든다.

### 3.5 semantic overlay 부여

- 파일: `allm/semantic.py` 이하
- 역할: 각 block에 `generic_role`, `domain_role`, `generated_role_name`, `section_id`, `used_for_generation` 등을 채운다.

### 3.6 artifact 저장과 시각화

- 파일: `allm/reference_pipeline.py`, `allm/visualize.py`
- 역할: JSON artifact와 preview 리소스를 저장하고, HTML로 확인 가능한 시각화 결과를 만든다.

## 4. 핵심 데이터 구조

전체 흐름을 이해하려면 데이터 구조 변화부터 잡는 것이 가장 쉽다.

### 4.1 low-level OCR 구조

- `RawEngineBlock`
  - OCR 엔진이 준 원시 block
  - `raw_label`, `text`, `bbox_px`, `bbox_norm`, `reading_order` 등을 가진다.

- `EnginePage`
  - 특정 엔진이 해석한 페이지 1장을 나타낸다.
  - `raw_blocks`와 page width/height를 가진다.

### 4.2 canonicalized 구조

- `CanonicalBlock`
  - 엔진별 raw block을 공통 포맷으로 바꾼 block
  - `canonical_label`, `text_quality_score`, `engine_sources`, `semantic_hints`가 추가된다.

- `FusedPage`
  - Dolphin과 Paddle block을 융합한 최종 페이지
  - downstream은 사실상 이 구조를 기준으로 동작한다.

### 4.3 분석 / 템플릿 구조

- `PageAnalysis`
  - 페이지 유형, column 수, OCR 품질, 텍스트/테이블/비주얼 비율 등을 담는다.

- `ReferenceTemplate`
  - 최종 reference artifact의 중심 구조
  - page spec, section order, style token, image slot, warning, confidence summary 등을 포함한다.

### 4.4 semantic 구조

- `SemanticConfig`
  - semantic mode와 모델 설정을 담는다.

- `RoleDecision`
  - block 하나의 semantic 판단 결과다.

- `SemanticRunSummary`
  - 한 번의 semantic overlay 전체 실행 요약이다.

결국 데이터의 큰 흐름은 아래와 같다.

```text
RawEngineBlock
-> EnginePage
-> CanonicalBlock
-> FusedPage
-> PageAnalysis + Semantic Role
-> ReferenceTemplate + Artifact JSON
```

## 5. `parse-reference` 전체 프로세스

실제 end-to-end orchestration은 `allm/reference_pipeline.py`의 `parse_reference()`와 `build_reference_template()`가 담당한다.

### 5.1 1단계: reference 입력으로 OCR 결과 찾기

먼저 `locate_reference_pages(reference_path, ocr_results_root)`를 호출한다.

이 단계의 역할:

- reference 파일명 stem을 기준으로 sample id 후보를 만든다.
- `OCR_results/dolphin/output_json`에서 대응되는 Dolphin JSON을 찾는다.
- 여러 페이지 문서라면 `sample-*.json` 패턴을 모아 페이지 목록으로 본다.
- 각 페이지마다 Paddle JSON, Paddle markdown, layout preview, Surya image 경로를 붙여 `PageSource`를 만든다.

중요한 구현 포인트:

- Dolphin OCR 결과가 기준 축이다.
- Dolphin 결과를 못 찾으면 실패한다.
- Paddle 경로 매핑은 현재 일부 sample에 대해 하드코딩되어 있다.
  - `financial1-00`
  - `financial2-00`
  - `medical1`

즉, 현재 구조는 `OCR_results` 디렉터리 규칙과 샘플 naming convention에 강하게 의존한다.

### 5.2 2단계: OCR 엔진 결과 로딩

페이지별로 아래 두 로더를 호출한다.

- `load_dolphin_page(page_source)`
- `load_paddle_page(page_source)`

이 단계에서 하는 일:

- JSON을 읽는다.
- block text를 `clean_text()`로 정리한다.
- bbox를 pixel / normalized 좌표로 보관한다.
- page width/height를 결정한다.

구현상 차이:

- Dolphin은 block 배열을 직접 읽는다.
- Paddle은 `parsing_res_list`를 읽는다.
- Dolphin 쪽은 page size가 JSON에 직접 없을 수 있어, Surya image나 visualization image 크기로 역추정한다.

### 5.3 3단계: canonical label 정규화

각 `EnginePage`는 `canonicalize_engine_page()`를 거쳐 `CanonicalBlock` 리스트로 바뀐다.

핵심 매핑:

- Dolphin
  - `sec_0 -> document_title`
  - `sec_1 -> section_heading`
  - `para -> paragraph`
  - `fig -> image`
  - `tab -> table`
  - `header/foot/fnote/list/anno -> meta_candidate`

- Paddle
  - `doc_title -> document_title`
  - `paragraph_title -> section_heading_or_panel_title`
  - `text -> paragraph_or_meta`
  - `table -> table`
  - `chart -> chart`
  - `image -> image`
  - `vision_footnote -> footnote`

이 단계에서 추가되는 핵심 값:

- `block_id`
- `canonical_label`
- `text_quality_score`
- `semantic_hints.engine_raw_labels`

즉, 이 단계부터 downstream은 엔진별 label 차이를 덜 신경 쓰고 공통 label 체계로 처리할 수 있다.

### 5.4 4단계: Dolphin + Paddle 결과 융합

페이지 융합은 `fuse_pages()`가 담당한다.

현재 융합 정책은 다음과 같다.

1. Dolphin이 있으면 Dolphin을 primary로 본다.
2. Paddle은 secondary로 본다.
3. primary block마다 secondary block 중 IoU 0.5 이상인 최적 매치를 찾는다.
4. 매치가 있으면 텍스트 품질을 비교해 더 좋은 text를 채택한다.
5. 필요하면 secondary label을 참고해 canonical label을 승격한다.
6. 매칭되지 않은 secondary block도 일부는 추가한다.
7. 최종 block을 reading order 기준으로 재정렬한다.

여기서 중요한 구현 디테일:

- text 교체가 발생하면 `text_replaced_from_secondary` flag가 붙는다.
- secondary에만 있던 block은 `secondary_only` flag가 붙는다.
- low-quality footnote secondary block은 버린다.
- 최종 page diagnostics에 아래 값이 들어간다.
  - `matched_pairs`
  - `text_replacements`
  - `added_secondary_blocks`

즉, 현재 fusion은 `두 엔진을 완전히 대등하게 합친다`기보다 `Dolphin 중심 + Paddle 보강` 구조다.

### 5.5 5단계: 페이지 archetype 분석

융합된 `FusedPage`마다 `classify_page()`를 호출한다.

이 단계에서 계산하는 대표 값:

- `column_count`
- `content_bbox`
- `table_area_ratio`
- `visual_area_ratio`
- `text_area_ratio`
- `quality_score`
- `dominant_layout_pattern`

그리고 rule 기반으로 page archetype을 정한다.

주요 archetype:

- `cover_summary`
- `body_narrative`
- `evidence_table`
- `compliance`
- `unknown`

판정 기준 예:

- 하단 disclaimer / compliance keyword가 강하면 `compliance`
- 금융 signal이 있고 좌측 패널 + 메인 본문 구조가 보이면 `cover_summary`
- 표 비중이 높고 본문이 적으면 `evidence_table`
- 본문 텍스트 비중이 충분하면 `body_narrative`

또한 품질 경고도 붙는다.

- `low_ocr_quality`
- `sparse_text`
- `single_engine_only`

### 5.6 6단계: 문서군과 언어 판정

그 다음 두 가지 전역 판단을 한다.

- `detect_document_family()`
- `detect_language_from_pages()`

현재 `document_family`는 대략 아래 중 하나가 된다.

- `financial_report`
- `form`
- `article`
- `report`

핵심 특징:

- 앞쪽 1~2페이지 텍스트를 중심으로 금융 리포트 여부를 판단한다.
- ticker, 목표주가, buy/sell 같은 signal이 강하면 `financial_report`
- 표만 있고 텍스트가 거의 없으면 `form`
- narrative page나 이미지 중심 article 패턴이면 `article`

언어 판정은 block text들을 합쳐 heuristic하게 결정한다.

### 5.7 7단계: semantic overlay 적용

이 단계가 현재 프로젝트에서 가장 중요한 확장 포인트다.  
함수는 `apply_financial_semantic_overlay()`다.

semantic은 3개 모드가 있다.

- `rule`
- `qwen`
- `shadow`

#### 5.7.1 `rule` 모드

`compute_financial_rule_decisions()`로 block 역할을 직접 부여한다.

대표 판단:

- 상단 메타 정보 -> `report_header_meta`
- analyst / email -> `analyst_info`
- compliance / disclaimer -> `disclaimer_block`
- 좌측 BUY/SELL/HOLD 박스 -> `investment_opinion_box`
- `Key Data` heading + 다음 table -> `key_data_box`
- `Consensus Data` heading + 다음 table -> `consensus_box`
- `Stock Price` heading + 다음 visual -> `price_chart_block`
- 메인 제목 -> `report_title`
- 제목 이후 짧은 heading -> `thesis_heading`
- 긴 본문 문단 -> `supporting_argument`
- table/chart/image -> `unsupported_evidence` 또는 evidence 계열

중요한 점:

- `rule` 모드는 사실상 `financial_report` 문서군에 최적화된 semantic이다.
- `document_family`가 `financial_report`가 아니면 `actual_rule_decision`이 거의 비게 되고, semantic field가 거의 안 채워진다.
- 따라서 현재 rule semantic은 금융 리포트에 강하게 편향되어 있다.

#### 5.7.2 `qwen` 모드

이 모드는 `Qwen-first + fallback` 구조다.

block마다 하는 일:

1. 페이지 품질과 block 자격을 확인한다.
2. `build_block_context()`로 모델 입력 payload를 만든다.
3. `resolve_block_decision()`이 Qwen backend를 호출한다.
4. 응답 JSON을 `parse_role_decision()`으로 검증한다.
5. confidence / schema / fallback 조건을 검사한다.
6. 수용 가능하면 Qwen 결과를 적용한다.
7. 아니면 safety rule 또는 unknown fallback을 적용한다.

Qwen 입력 payload에는 다음이 들어간다.

- target block text / bbox / OCR label / reading order
- page context
- 가까운 이전/이후 text block
- 가장 가까운 heading / evidence block
- 구조적 거리와 column 관계
- 허용 가능한 legacy role 목록

현재 backend 구현은 `QwenTransformersBackend` 하나이며, `transformers` runtime만 구현되어 있다.

#### 5.7.3 `shadow` 모드

이 모드는 Qwen을 실행하되, 최종 block에는 rule 결과를 유지하는 비교 모드다.

즉:

- Qwen 응답과 trace는 남긴다.
- 실제 최종 semantic assignment는 rule 쪽을 유지한다.
- rule과 Qwen의 불일치 개수를 `shadow_disagreement_count`로 센다.

현재 shadow 모드는 모델 migration 검증용 진단 모드에 가깝다.

#### 5.7.4 safety / fallback 규칙

semantic에는 별도의 safety rule 층도 있다.

대표 safety 대상:

- analyst info
- disclaimer
- top metadata
- research center metadata

또한 다음 상황에서는 fallback이 발생한다.

- low page quality
- backend error
- invalid JSON
- schema mismatch
- low role confidence

#### 5.7.5 semantic 후처리

semantic decision이 적용된 뒤에는 deterministic 후처리가 추가된다.

- `_normalize_generation_policy()`
  - `used_for_generation`
  - `section_purpose`
  - 를 role 기준으로 다시 정리한다.

- `_assign_section_ids()`
  - `report_title`, `thesis_heading`을 기준으로 `sec-001`, `sec-002` 같은 `section_id`를 부여한다.

즉, 모델이 역할을 제안하더라도 generation 사용 여부와 section 연결은 여전히 코드가 책임진다.

#### 5.7.6 generated role 필드

현재 semantic layer는 legacy role만 채우는 것이 아니라 generated role 관련 필드도 함께 채운다.

- `generated_role_name`
- `generated_role_description`
- `generated_parent_role_name`
- `generated_role_level`

Qwen이 직접 값을 줄 수도 있고, 없으면 `semantic_schema.py`가 generic/domain role로부터 보수적으로 기본값을 채운다.  
즉, generated role 체계는 이미 현재 코드 안에 일부 반영되어 있다.

### 5.8 8단계: anchor page와 section order 생성

semantic overlay 이후 template metadata를 만든다.

#### 5.8.1 anchor page

`anchor_pages_from_analyses()`는 archetype이 아래인 페이지를 anchor page로 본다.

- `cover_summary`
- `body_narrative`

즉, narrative generation 관점에서 중요한 페이지만 anchor page가 된다.

#### 5.8.2 section order

`build_section_order()`는 anchor page 안에서 narrative block만 골라 section 순서를 만든다.

포함 대상:

- text가 있는 block
- heading / paragraph 계열 block
- semantic role상 metadata / disclaimer / evidence가 아닌 block

섹션 생성 방식:

- title-like block을 만나면 새 section 시작
- 아니면 현재 section에 block을 이어 붙임

결과는 `section-001`, `section-002` 식으로 저장된다.

### 5.9 9단계: style token 생성

`build_style_tokens()`는 anchor page를 기준으로 문서 스타일 힌트를 추출한다.

예:

- `title_font_scale`
- `subtitle_font_scale`
- `body_font_scale`
- `paragraph_spacing`
- `section_spacing`
- `column_count`
- `body_width_ratio`
- `image_alignment_rules`

이 값은 실제 generation 시스템이 아직 구현되어 있지 않더라도, 나중에 reference-aware rendering을 하기 위한 레이아웃 요약값 역할을 한다.

### 5.10 10단계: image slot / unsupported block / repeating element 추출

추가 템플릿 메타데이터도 함께 만든다.

#### 5.10.1 image slot

`build_image_slots()`는 anchor page의 일정 크기 이상 image block을 추출한다.

저장 정보:

- 위치
- 정렬
- width / height 비율
- caption 기대 여부

#### 5.10.2 unsupported block

`collect_unsupported_blocks()`는 현재 generation 대상에서 제외되는 block을 수집한다.

주요 대상:

- `table`
- `chart`
- disclaimer 계열

즉, 현재 구현은 table/chart를 fully supported generation unit으로 취급하지 않는다.

#### 5.10.3 repeating element

`detect_repeating_elements()`는 여러 페이지에서 반복되는 header/footer text를 찾아낸다.

이 값은 추후 template cleaning이나 common chrome 제거에 활용될 수 있다.

### 5.11 11단계: review gate와 confidence summary 생성

`_review_gate()`는 최종 template warning과 confidence summary를 만든다.

체크 항목:

- low OCR quality page 존재 여부
- fusion text replacement가 과도한지
- narrative section이 추출되었는지
- 문서군이 MVP 지원 대상인지

현재 warning 예:

- `low_ocr_quality_pages:...`
- `low_ocr_quality_repaired_via_secondary_engine`
- `no_narrative_sections_extracted`
- `unsupported_for_mvp`

중요한 구현상 판단:

- `review_required`는 아래 중 하나면 true다.
  - low quality page 존재
  - unsupported_for_mvp
  - fusion conflict가 많음

즉, warning과 review_required는 완전히 같은 개념은 아니다.

### 5.12 12단계: `ReferenceTemplate` 조립

이제 앞 단계 결과를 모두 모아 `ReferenceTemplate` dataclass를 만든다.

주요 필드:

- `template_id`
- `source_path`
- `document_family`
- `language`
- `source_engines`
- `page_specs`
- `page_archetypes`
- `anchor_pages`
- `blocks`
- `section_order`
- `style_tokens`
- `image_slots`
- `unsupported_blocks`
- `repeating_elements`
- `template_warnings`
- `confidence_summary`

즉, `ReferenceTemplate`는 단순 block dump가 아니라 `reference 문서의 구조적 요약`이라고 보는 편이 맞다.

### 5.13 13단계: diagnostics와 canonical page snapshot 생성

동시에 별도의 부가 산출물도 만든다.

#### 5.13.1 diagnostics

`parser_diagnostics.json`에 들어가는 대표 정보:

- 원본 reference path
- page source 경로들
- page analysis 결과
- fusion diagnostics
- document family
- language
- semantic summary
- optional semantic trace

#### 5.13.2 canonical pages

`canonical_pages.json`에는 페이지별 최종 block 스냅샷이 들어간다.

각 block에는 다음이 들어갈 수 있다.

- canonical label
- text
- bbox
- engine source
- semantic role
- generated role
- section id
- flags / notes

실무적으로는 template보다 이 파일이 디버깅에 더 직접적으로 유용하다.

### 5.14 14단계: artifact 파일 저장

`parse_reference()`는 최종적으로 아래 파일들을 저장한다.

- `artifacts/<job_id>/01_reference/reference_template.json`
- `artifacts/<job_id>/01_reference/canonical_pages.json`
- `artifacts/<job_id>/01_reference/semantic_overlay.json`
- `artifacts/<job_id>/01_reference/parser_diagnostics.json`
- `artifacts/<job_id>/01_reference/semantic_trace.json` (optional)

추가로 raw OCR 리소스도 복사한다.

- `raw/dolphin/*.json`
- `raw/dolphin/*.md`
- `raw/paddle/*.json`
- `raw/paddle/*.md`
- `layout_preview/*`

즉 artifact 디렉터리 하나만 보면 parse 과정의 입력 근거와 결과물을 함께 추적할 수 있게 되어 있다.

3. preview가 없으면 placeholder 표시

preview 우선순위:

- Surya image

- Paddle visualization

### 7.3 HTML에 들어가는 주요 섹션

- document summary card
- semantic summary card
- section order table
- style token table
- template warning
- confidence summary
- semantic diagnostics
- semantic trace
- semantic overlay raw dump
- page별 preview + overlay box + block table

### 7.4 page overlay

box 색상 기준:

- 없으면 canonical label color 사용

## 8. 현재 구현에서 semantic 판단에 실제로 쓰이는 정보

이 부분은 프로젝트 이해에 중요하다.  
현재 semantic은 단순 keyword classification이 아니라, 다음 레이어들이 함께 쓰인다.

### 8.1 low-level 관측값

- raw text
- raw OCR label
- canonical label
- bbox_norm
- reading order
- page 내 index
- engine source

### 8.2 page-level context

- page archetype
- quality score
- column count
- layout pattern
- text/table/visual area ratio

### 8.3 local structural context

- 이전/이후 text block
- 가장 가까운 heading
- 가장 가까운 evidence
- 같은 column 여부
- heading/evidence와의 거리

즉, Qwen semantic은 이미 `텍스트만 보고 라벨링하는 구조`가 아니라 `구조적 context-aware block role inference`로 설계되어 있다.

## 9. 현재 구현의 강점

### 9.1 OCR 엔진 융합 구조가 분리되어 있다

OCR 로딩, canonicalize, fusion이 분리되어 있어 엔진 추가나 정책 수정이 비교적 쉽다.

### 9.2 semantic 모드가 분리되어 있다

- `rule`
- `qwen`
- `shadow`

가 분리되어 있어 실험, 비교, fallback 운용이 가능하다.

### 9.3 artifact 설계가 디버깅 친화적이다

`reference_template`, `canonical_pages`, `semantic_overlay`, `diagnostics`, `trace`가 분리되어 있어 어느 단계가 틀렸는지 추적하기 쉽다.

### 9.4 시각화가 구현되어 있다

단순 JSON 저장에서 끝나지 않고 사람이 검토할 수 있는 HTML visualization까지 연결되어 있다.

## 10. 현재 구현의 한계와 제약

### 10.1 reference 입력 탐색이 샘플 naming에 의존한다

`ocr_locator.py`는 sample id 패턴과 디렉터리 구조를 많이 가정한다.  
특히 Paddle 매핑은 일부 샘플에 대해 하드코딩이다.

### 10.2 semantic ontology가 금융 리포트에 편향되어 있다

`rule` semantic은 거의 `financial_report` 전용이다.  
article, form, report 일반 문서에는 semantic richness가 크게 떨어진다.

### 10.3 generation 본체는 아직 없다

현재 추출하는 `section_order`, `style_tokens`, `image_slots`, `used_for_generation`은 downstream generation을 위한 준비 정보다.  
하지만 실제 문서 재생성 또는 content generation 엔진은 현재 코드 범위에 없다.

### 10.4 table/chart는 아직 unsupported 취급이 강하다

table/chart는 semantic evidence로는 인식하지만, generation 관점에서는 대부분 unsupported block으로 남긴다.

### 10.5 Qwen backend가 하나뿐이다

현재 semantic backend는 `transformers` 기반 `QwenTransformersBackend`만 구현되어 있다.

### 10.6 low-quality page에서는 Qwen 영향이 제한된다

page quality threshold 아래에서는 Qwen이 건너뛰어지고 fallback이 쓰일 수 있다.  
즉, OCR 품질이 semantic ceiling을 결정하는 구조다.

## 11. 현재 코드 기준의 실제 시스템 그림

지금 구현된 시스템을 가장 현실적으로 표현하면 아래와 같다.

```text
[입력]
reference 이미지/PDF/샘플 경로

-> [OCR source locator]
   Dolphin JSON 기준으로 page source 수집

-> [adapter]
   Dolphin / Paddle 결과를 EnginePage로 로딩

-> [canonicalize]
   raw label -> canonical label

-> [fusion]
   Dolphin primary + Paddle secondary 융합

-> [page analysis]
   archetype / family / language / quality 계산

-> [semantic overlay]
   rule 또는 Qwen으로 block semantic 부여

-> [template assembly]
   section_order / style_tokens / image_slots / warnings 생성

-> [artifact export]
   JSON, raw OCR copy, preview copy

-> [visualization]
   reviewer용 HTML 생성
```

즉, 현재 프로젝트는 `OCR 이후 reference understanding layer`를 꽤 탄탄하게 구현해 둔 상태라고 볼 수 있다.

## 13. 결론

현재 구현 기준으로 이 프로젝트의 본질은 `reference document understanding pipeline`이다.

조금 더 풀어 말하면:

- OCR 결과를 그대로 쓰지 않고 canonical block 구조로 정리한다.
- 두 OCR 엔진 결과를 융합해 품질을 높인다.
- 페이지 유형, 문서군, 스타일, 섹션 구조를 추출한다.
- block마다 semantic role을 부여한다.
- 이를 artifact와 HTML로 저장해 downstream generation 또는 human review의 기반을 만든다.

즉, 이 프로젝트는 아직 `최종 생성 시스템` 전체라기보다, 그 이전 단계인 `reference 해석과 구조화 엔진`이 구현된 상태다.  
하지만 현재 코드만 봐도 `입력 수집 -> 정규화 -> 융합 -> 분석 -> semantic -> artifact -> 시각화`의 end-to-end 흐름은 이미 완성도 있게 연결되어 있다.

다음 문서에서는 이 정리를 바탕으로 아래 둘 중 하나를 더 깊게 파고들 수 있다.

- `semantic layer만 따로 떼어 설계/구현 관점에서 재정리`
- `artifact를 실제 generation pipeline으로 연결하는 후속 설계`
