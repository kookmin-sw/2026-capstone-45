# Section-First Generation Design

이 문서는 현재 `slot-first generation` 구조를 `section-first generation`으로 전환할 때 무엇이 바뀌는지, 왜 필요한지, 어떤 계약과 단계가 추가되어야 하는지를 설명한다.

범위는 `v1 단일 source artifact` 기준이다. 이 문서의 목표는 코드 구현 전에 generation 레이어의 공식 입력, 공식 계획 객체, Qwen의 책임, fallback 정책, 테스트 기준을 먼저 고정하는 것이다.

## TL;DR

- 현재 generation은 `slot -> fact` 중심의 flat planning 구조다.
- 앞으로는 검색 단위를 `fact`보다 한 단계 위인 `source section`으로 올려야 한다.
- 단, 모든 reference section을 `1 reference section -> 1 source section`으로 고정하지 않는다.
- `purpose`와 `composition_strategy`는 분리한다. 같은 `summary`라도 문서에 따라 다른 생성 전략을 가질 수 있다.
- 본문형 section은 `local_align`, 비교/분석형 본문은 `bounded_synthesis`, executive summary류는 `global_synthesis`, 메타/전역 박스는 `global_lookup` 전략을 사용할 수 있다.
- v1 공식 흐름은 `rule section-first planner + Qwen section candidate rerank/select + deterministic slot-to-fact selection + existing slot writing`이다.
- synthesis section에서는 `source section bundle`을 고른 뒤, bundle-aware deterministic selection으로 slot을 채운다.
- v1에서 Qwen은 `section 매핑 선택기` 또는 `section bundle 재정렬기`로만 사용하고, `slot fact 자유 선택기`로는 사용하지 않는다.
- `document_family`는 generation에서도 hard gate가 아니라 ranking prior다.
- `section_artifact.json`이 존재하면 그것이 authoritative section source이고, `section_order`는 narrative projection 또는 fallback input으로만 사용한다.
- v1의 성공 기준은 더 화려한 문장 생성이 아니라 `같은 source section 안에서 더 일관된 slot assignment`와 `요약형 section에서 더 좋은 multi-section coverage`를 만드는 것이다.

## 1. 배경

현재 generation 파이프라인은 대략 아래 순서로 동작한다.

```text
build_reference_blueprint()
  -> build_source_fact_pack()
  -> build_generation_plan()
  -> _resolve_slot_draft()
  -> build_generated_document()
```

이 흐름은 현재 코드의 역할과도 그대로 대응된다.

- `build_reference_blueprint()`
  - reference template에서 generation용 `ReferenceSlot[]`를 만든다.
- `build_source_fact_pack()`
  - source template에서 generation용 `SourceFact[]`를 만든다.
- `build_generation_plan()`
  - 각 slot에 대해 source fact를 고른다.
- `_resolve_slot_draft()`
  - 선택된 fact로 최종 slot 초안을 만든다.

현재 구조에서도 `section_id`, `section_title`은 존재하지만, generation에서의 역할은 보조 신호에 가깝다.

즉 현재 계획 객체의 핵심은 다음이다.

- generation의 상위 단위는 `slot`
- source 검색 단위는 `fact`
- section은 일부 heuristic과 rationale 보조 정보

이 구조는 초기 금융 문서 데모에는 유용하지만, source 문서가 다양해지거나 section 흐름을 더 강하게 유지하고 싶을 때 한계가 드러난다.

## 2. 현재 slot-first 방식의 한계

### 2.1 잘못된 source section에서 fact를 가져오기 쉽다

현재는 slot마다 사실상 전체 fact pool을 대상으로 rule group 또는 score 기반 선택을 수행한다. 이 방식은 role이 비슷하면 다른 section의 fact도 쉽게 후보로 올라오게 만든다.

대표적인 실패는 아래와 같다.

- 본문 body slot이 다른 thesis section의 fact를 가져온다.
- 강조 heading 역할 slot이 직전 body와 무관한 heading fact를 가져온다.
- summary slot이 section 흐름과 무관한 table row를 조합한다.

### 2.2 section 내부 흐름을 유지하기 어렵다

문서 작성은 단순 fact retrieval보다 `어느 section 안에서 어떤 순서로 읽히는가`가 중요하다. 하지만 slot-first 방식에서는 slot 단위 최적화가 section 단위 일관성을 깨기 쉽다.

예를 들어:

- 긴 body slot은 thesis section A에서 채워지고
- 바로 다음 강조 slot은 section B의 heading으로 채워지고
- 그 다음 숫자 박스는 unrelated market snapshot으로 채워질 수 있다

이 경우 개별 slot만 보면 그럴듯해도, section 단위로 보면 문맥이 무너진다.

### 2.3 evidence/meta가 narrative에 섞일 수 있다

현재 `_score_fact_for_slot()`은 evidence panel이나 metadata를 감점하긴 하지만, 검색 공간이 전체 fact pool이라면 여전히 잘못된 section의 fact가 올라올 수 있다.

특히 아래 조합이 위험하다.

- body slot <- table row
- short emphasis slot <- metadata block
- summary slot <- sidebar evidence only

즉 현재 구조는 `무조건 금지`보다 `점수 감점`에 의존하는 경향이 있다.

### 2.4 한 fact가 여러 slot에 과도하게 재사용되기 쉽다

usage penalty가 있더라도, slot-first 구조에서는 대표 fact 몇 개가 많은 slot에 반복 사용되기 쉽다. section 단위 검색이 없으면 planner는 같은 section 안에서 적절히 분산하기보다 전체 pool에서 반복적으로 높은 점수 fact를 고르기 때문이다.

### 2.5 요약형 section을 정식 경로로 지원하지 못한다

이 구조의 더 근본적인 한계는 `introduction`, `executive_summary`, `cover_summary`, `conclusion` 같은 section을 사실상 예외 취급한다는 점이다.

이런 section은 원래 아래 성격을 가진다.

- 문서의 핵심 논지 여러 개를 압축해야 한다.
- narrative section과 evidence section을 함께 참조해야 한다.
- 하나의 source section만으로는 coverage가 부족한 경우가 많다.

즉 요약형 section은 `cross-section fallback`으로 우연히 처리할 문제가 아니라, 애초에 `multi-section synthesis`를 전제로 설계해야 하는 유형이다.

### 2.6 source 문서가 다양해질수록 불안정해진다

source가 항상 같은 형태의 financial report라면 hand-crafted grouping으로 어느 정도 버틸 수 있다. 하지만 source가 아래처럼 다양해지면 slot-first heuristic의 취약점이 커진다.

- IR note
- memo
- article
- deck
- compliance-heavy report
- heading이 약한 narrative 문서

이럴수록 `slot이 어떤 fact를 고를까`보다 `reference section이 source의 어느 section과 대응되는가`를 먼저 결정해야 한다.

## 3. section-first generation의 핵심 아이디어

section-first generation은 generation의 공식 검색 단위를 `fact`보다 한 단계 위인 `source section`으로 올리는 설계다.

핵심 순서는 아래와 같다.

1. reference section을 본다.
2. 그 reference section의 planning mode를 먼저 결정한다.
3. `local_align`이면 대응되는 source section 후보를 찾고 가장 적합한 source section을 선택한다.
4. `multi-section synthesis`이면 대응되는 source section bundle 후보를 찾고 가장 적합한 source section 집합을 선택한다.
5. 선택 결과를 기반으로 slot을 채운다.
6. 내부에 적합한 fact가 없을 때만 제한적으로 cross-section fallback을 허용한다.

즉 slot-first generation이 `slot -> fact` 직접 매핑이라면, section-first generation은 아래 두 경로를 가진다.

- 본문형 section
  - `reference section -> source section -> slot -> fact`
- 요약형 section
  - `reference section -> source section bundle -> slot -> fact`

이 설계의 장점은 아래와 같다.

- body/heading/data box가 같은 source section 맥락을 공유할 수 있다.
- narrative와 evidence를 더 안정적으로 분리할 수 있다.
- introduction, executive summary, conclusion처럼 여러 논점을 합쳐야 하는 section도 정식 경로로 지원할 수 있다.
- Qwen을 자유 생성기가 아니라 `후보 중 section 선택기`로 제한할 수 있다.
- generation plan이 더 잘 설명되고 디버깅 가능해진다.

## 4. 목표 상태의 전체 흐름

section-first generation에서 공식 흐름은 아래와 같다.

```text
reference_blueprint(v2)
  -> source_fact_pack(v2 with sections)
  -> section matching
  -> section-aware slot assignment
  -> slot drafts
  -> render
```

현재와 비교하면 가장 큰 변화는 `section matching`이 독립 단계로 생긴다는 점이다.

### 4.1 현재 구조

```text
reference slots
  -> source facts 전체 검색
  -> slot assignments
  -> drafts
```

### 4.2 목표 구조

```text
reference sections + slots
  -> source sections + facts
  -> section match decisions
  -> slot assignments within matched section or matched bundle
  -> drafts
```

즉 앞으로 generation에서의 책임 경계는 아래처럼 분리된다.

- blueprint layer
  - reference section과 slot 구조 정의
- source pack layer
  - source section과 fact 구조 정의
- planning layer
  - reference section과 source section 또는 source section bundle을 연결
- assignment layer
  - 선택된 source section 내부 fact 또는 bundle 내부 fact로 slot 채움
- writing layer
  - 이미 정해진 assignment를 문장으로 렌더링

이 구조에서 writing은 마지막 단계이며, 핵심 품질은 planning과 assignment에서 먼저 결정된다.

## 5. reference 입력 계약

현재 `reference_blueprint.json`은 사실상 `slots` 중심 구조다. section-first generation에서는 이것만으로는 부족하다. blueprint가 section-aware contract를 가져야 section 매핑이 가능하다.

### 5.1 왜 `sections + slots` 구조가 필요한가

slot만 나열된 구조에서는 아래를 안정적으로 표현하기 어렵다.

- 어떤 slot들이 하나의 reference section에 속하는가
- 이 section의 대표 purpose가 무엇인가
- 이 section이 narrative인지 non-narrative인지
- 이 section이 어떤 `composition_strategy`를 가져야 하는지
- section 단위로 source section 후보를 찾을 때 필요한 shape 정보가 무엇인가

따라서 `reference_blueprint(v2)`는 최소한 `sections[] + slots[]`를 동시에 가져야 한다.

### 5.2 `ReferenceSectionPlan`

v1에서 reference section은 아래 구조를 가진다고 정의한다.

```python
ReferenceSectionPlan = {
    "reference_section_id": str,
    "title": str | None,
    "purpose": str | None,
    "composition_strategy": str,
    "slot_ids": list[str],
    "slot_role_histogram": dict[str, int],
    "page_start": int,
    "page_end": int,
    "is_narrative": bool,
}
```

각 필드의 의미는 아래와 같다.

- `reference_section_id`
  - blueprint 안에서 reference section을 식별하는 고유 ID
- `title`
  - section 대표 title 또는 heading
- `purpose`
  - section 목적. 예: `thesis`, `investment_summary`, `evidence_panel`, `metadata`, `unknown`
- `composition_strategy`
  - 이 reference section이 어떤 planning strategy를 쓰는지
  - v1 권장 값은 `local_align`, `bounded_synthesis`, `global_synthesis`, `global_lookup`
- `slot_ids`
  - 이 section에 속한 slot ID 목록
- `slot_role_histogram`
  - section 안에서 어떤 slot role이 얼마나 있는지 요약한 histogram
- `page_start`, `page_end`
  - reference section이 걸치는 page 범위
- `is_narrative`
  - generation에서 narrative 흐름을 구성하는 section인지 여부

### 5.2A `purpose`와 `composition_strategy`는 다른 축이다

여기서 가장 중요한 원칙은 `purpose`와 `composition_strategy`를 같은 것으로 취급하지 않는 것이다.

- `purpose`
  - 이 section이 문서 안에서 무엇을 의미하는가
  - 예: `thesis`, `summary`, `comparison`, `background`, `evidence_panel`, `metadata`
- `composition_strategy`
  - 이 section을 source에서 어떻게 구성할 것인가
  - 예: `local_align`, `bounded_synthesis`, `global_synthesis`, `global_lookup`

즉 같은 `summary`라도 어떤 템플릿에서는 하나의 source section을 압축하는 `local_align`일 수 있고, 어떤 템플릿에서는 문서 전반을 종합하는 `global_synthesis`일 수 있다. 반대로 본문 section도 비교형/원인-결과형 문단이면 `bounded_synthesis`가 필요할 수 있다.

기능 설계 차원에서 이 분리가 중요한 이유는, 문서 구조의 의미 계층과 generation planning 계층을 섞지 않아야 later-stage 변형에 더 잘 버티기 때문이다. 따라서 `purpose -> composition_strategy`를 단순 1:1 규칙으로 고정하지 않고, reference layout/slot shape/template intent를 함께 보고 결정하는 것이 맞다.

### 5.3 slot 필드 확장

기존 `ReferenceSlot`은 유지하되, section 내부 위치를 더 잘 표현하기 위해 아래 필드를 추가 설명한다.

- `slot_index_in_section: int`
  - 같은 section 안에서 slot의 상대 순서

즉 slot은 여전히 아래 정보를 가진다.

- `slot_id`
- `section_id`
- `section_title`
- `generic_role`
- `domain_role`
- `target_word_count`
- `preserve_reference_text`
- `slot_index_in_section`

이 필드는 section 내부에서 heading -> summary -> body -> data box 같은 미세 순서를 유지할 때 중요하다.

### 5.4 `reference_blueprint(v2)` 예시 스켈레톤

```json
{
  "reference_template_id": "tmpl-001",
  "document_family": "financial_report",
  "language": "ko",
  "style_tokens": {},
  "page_specs": [],
  "sections": [
    {
      "reference_section_id": "ref-sec-001",
      "title": "투자포인트 1",
      "purpose": "thesis",
      "composition_strategy": "local_align",
      "slot_ids": ["slot-005", "slot-006", "slot-007"],
      "slot_role_histogram": {"body": 2, "section_heading": 1},
      "page_start": 2,
      "page_end": 2,
      "is_narrative": true
    },
    {
      "reference_section_id": "ref-sec-000",
      "title": "Executive Summary",
      "purpose": "executive_summary",
      "composition_strategy": "global_synthesis",
      "slot_ids": ["slot-001", "slot-002", "slot-003"],
      "slot_role_histogram": {"summary": 1, "body": 1, "plain": 1},
      "page_start": 1,
      "page_end": 1,
      "is_narrative": true
    }
  ],
  "slots": []
}
```

## 6. source 입력 계약

section-first generation에서는 source도 단순 fact 목록만으로는 부족하다. source fact pack 역시 `source_sections[] + global_fact_pool + facts[]` 구조를 가져야 한다.

### 6.1 왜 source section view가 필요한가

slot-first 구조에서는 source fact 전체를 평면적으로 검색했다. 하지만 section-first에서는 먼저 source section을 고른 뒤, 그 안에서 fact를 선택해야 한다.

즉 planner는 아래 질문에 답할 수 있어야 한다.

- 이 source section은 narrative section인가
- purpose가 thesis인지 summary인지 evidence인지
- heading/body/evidence/meta 비중이 어떤가
- 어떤 fact ID들이 이 section에 속하는가
- 이 section의 confidence가 충분한가

### 6.2 `SourceSectionView`

v1에서 source section은 아래 구조를 가진다고 정의한다.

```python
SourceSectionView = {
    "source_section_id": str,
    "title": str | None,
    "purpose": str | None,
    "fact_ids": list[str],
    "heading_fact_ids": list[str],
    "body_fact_ids": list[str],
    "evidence_fact_ids": list[str],
    "meta_fact_ids": list[str],
    "confidence": float,
    "is_narrative": bool,
    "page_start": int,
    "page_end": int,
    "section_summary_text": str,
    "source_doc_id": str | None
}
```

각 필드의 의미는 아래와 같다.

- `source_section_id`
  - source section의 고유 식별자
- `title`
  - 대표 heading 또는 section title
- `purpose`
  - `thesis`, `supporting_argument`, `investment_summary`, `evidence_panel`, `metadata`, `disclaimer`, `unknown` 등
- `fact_ids`
  - section 전체에 속한 fact 목록
- `heading_fact_ids`
  - heading 성격 fact 목록
- `body_fact_ids`
  - narrative body 성격 fact 목록
- `evidence_fact_ids`
  - table, metric, evidence panel fact 목록
- `meta_fact_ids`
  - metadata, author, date 등 meta 성격 fact 목록
- `confidence`
  - section detection 결과에 대한 신뢰도
- `is_narrative`
  - generation 서술 흐름에서 narrative로 취급할지 여부
- `page_start`, `page_end`
  - source section의 page 범위
- `section_summary_text`
  - candidate rerank와 trace에 쓰기 위한 짧은 section 요약
- `source_doc_id`
  - v1에서는 optional이지만, 이후 multi-source 확장을 위해 필드를 예약해 둔다

### 6.3 facts와 section의 관계

`SourceFact`는 기존 타입을 유지하되, `section_id` 해석을 강화한다.

- 기존: section_id는 보조 필드
- 앞으로: section artifact 또는 source section view 기준의 authoritative field

즉 `facts[]`는 여전히 generation이 직접 쓰는 근거 단위지만, 어떤 fact가 어느 section에 속하는지는 `source_sections[]`가 우선 정의한다.

### 6.3A document-global evidence channel

다만 모든 근거를 section에 억지로 귀속시키는 것은 좋지 않다. 실제 문서에는 상단 메타, cover 수치 박스, 전역 valuation 표, 공통 시장 환경처럼 특정 section 소속으로 보기 어려운 정보가 많다. 이런 정보는 fallback이 아니라 `document-global evidence channel`로 별도 모델링하는 편이 더 자연스럽다.

권장 개념은 아래와 같다.

- `global_metadata`
  - date, author, publisher, ticker, company name 같은 전역 메타
- `global_evidence_cards`
  - cover metric, valuation snapshot, key market data처럼 여러 section에서 재사용 가능한 전역 수치 카드
- `global_fact_pool`
  - section 소속이 약하지만 generation에서 공식적으로 참조 가능한 fact 집합

즉 source 입력 계약은 개념적으로 `source_sections[] + global_fact_pool + facts[]`를 가져야 한다. 이후 generation planner는 section-local channel과 document-global channel을 분리해서 사용해야 한다. narrative body는 section-local을 우선 쓰고, metadata/cover summary/key metric slot은 global channel을 정식 경로로 사용할 수 있어야 한다.

  "source_sections": [

## 7. 공식 generation plan 계약

현재 `generation_plan`은 flat `SlotAssignment[]` 중심이다. 하지만 section-first generation에서는 먼저 section 매핑이 일어나고, 그 위에서 slot assignment가 일어나므로 plan 자체가 2단계 의사결정을 드러내야 한다.

따라서 v1 공식 `generation_plan.json` 루트는 아래 구조를 가진다고 정의한다.

```json
{
  "reference_sections": [],
  "source_sections": [],
  "section_matches": [],
  "slot_assignments": [],
  "summary": {}
}
```

### 7.1 왜 flat plan이 부족한가

flat `SlotAssignment[]`만 있으면 아래를 설명할 수 없다.

- 어떤 reference section이 어떤 source section에 대응됐는가
- 이 section 선택이 rule인지 Qwen rerank인지
- section match confidence가 얼마인가
- slot assignment가 in-section 선택인지 cross-section fallback인지

즉 section-first 구조에서는 section-level rationale이 공식 artifact에 남아야 한다.

또한 모든 section match가 단일 source section 선택이라는 가정을 버려야 한다. introduction, executive summary, conclusion처럼 문서 전반을 요약하는 section은 `source section bundle`을 선택하는 것이 정식 경로여야 한다.

### 7.2 `SectionMatchDecision`

v1에서 section match 결과는 아래 구조를 가진다고 정의한다.

```python
SectionMatchDecision = {
    "reference_section_id": str,
    "selected_source_section_id": str | None,
    "selected_source_section_ids": list[str],
    "primary_source_section_id": str | None,
    "candidate_source_section_ids": list[str],
    "strategy": str,
    "aggregation_strategy": str | None,
    "confidence": float,
    "reason": str,
    "section_contributions": list[dict],
    "fallback_used": bool,
    "needs_review": bool,
}
```

각 필드의 의미는 아래와 같다.

- `reference_section_id`
  - 어떤 reference section에 대한 결정인지
- `selected_source_section_id`
  - `local_align`에서 사용하는 대표 source section. 매칭 실패 시 `null` 허용
- `selected_source_section_ids`
  - `multi-section synthesis`에서 선택된 source section 집합
- `primary_source_section_id`
  - bundle이 있는 경우 대표 narrative spine 역할을 하는 source section
- `candidate_source_section_ids`
  - rule prefilter가 남긴 후보 목록 또는 bundle 구성에 사용된 base candidate ID 목록
- `strategy`
  - `rule_top1`, `qwen_rerank`, `rule_bundle_select`, `qwen_bundle_rerank`, `fallback_none` 등
- `aggregation_strategy`
  - `local_align`, `coverage_bundle`, `thesis_plus_evidence`, `document_global_summary` 등
- `confidence`
  - 최종 section 선택에 대한 confidence
- `reason`
  - 선택 이유 요약
- `section_contributions`
  - 어떤 source section이 어떤 역할로 bundle에 기여하는지 요약
- `fallback_used`
  - 원래 선택 경로가 실패해 fallback이 들어갔는지
- `needs_review`
  - 사람 검수가 필요한지

### 7.3 `SectionAwareSlotAssignment`

slot assignment도 section-aware contract를 가져야 한다.

```python
SectionAwareSlotAssignment = {
    "slot_id": str,
    "reference_section_id": str,
    "selected_source_section_id": str | None,
    "selected_source_section_ids": list[str],
    "selected_fact_ids": list[str],
    "selected_global_fact_ids": list[str],
    "strategy": str,
    "confidence": float,
    "rationale": str,
    "cross_section_fallback_used": bool,
    "preserve_reference_text": bool,
}
```

이 객체는 기존 `SlotAssignment`를 완전히 폐기한다기보다, v1에서는 확장형 대체 계약으로 이해하면 된다.

여기서 `selected_global_fact_ids`는 section-local 선택과 document-global 선택을 분리해 기록하기 위한 필드다. 이 구분이 있어야 나중에 
- 이 슬롯이 section 정합성 위에서 채워졌는지
- 아니면 전역 metadata/evidence channel을 참조했는지
를 artifact 차원에서 바로 추적할 수 있다.

추가로 아래 필드는 summary/trace 레벨에서 매우 유용하다.

- `section_match_confidence`
- `section_match_reason`
- `needs_review`
- `bundle_coverage_axes`

### 7.4 `summary`

`generation_plan.json`의 `summary`에는 최소한 아래 지표를 넣는다고 정의한다.

- `section_match_count`
- `matched_section_count`
- `unmatched_section_count`
- `slot_assignment_count`
- `cross_section_fallback_count`
- `needs_review_count`
- `synthesis_section_count`
- `successful_synthesis_match_count`

이 요약이 있어야 artifact 리뷰 시 “이번 run에서 section matching이 얼마나 안정적이었는가”를 빠르게 볼 수 있다.

## 8. public interface와 타입 확장 방향

### 8.1 `GenerationConfig` 확장 방향

현재 `GenerationConfig`는 generation mode, sampling, fact count 정도만 가진다. section-first generation에서는 아래 개념을 추가해야 한다.

```python
GenerationConfig(
    matching_mode: Literal["slot_first", "section_first"] = "section_first",
    max_section_candidates: int = 3,
    max_synthesis_sections: int = 4,
    max_facts_per_synthesis_section: int = 2,
    use_global_evidence_channel: bool = True,
    max_global_facts_per_slot: int = 2,
    section_match_threshold: float = 0.7,
    allow_cross_section_fallback: bool = True,
    max_cross_section_facts_per_slot: int = 1,
    use_qwen_for_section_rerank: bool = mode == "qwen",
    use_qwen_for_fact_rerank: bool = False,
)
```

각 필드의 의도는 아래와 같다.

- `matching_mode`
  - 기존 slot-first와 새 section-first를 공존시키는 안전장치
- `max_section_candidates`
  - Qwen 또는 rule이 검토할 source section 후보 수
- `max_synthesis_sections`
  - 요약형 section bundle에 포함할 최대 source section 수
- `max_facts_per_synthesis_section`
  - bundle 안의 개별 source section에서 가져올 fact 상한
- `section_match_threshold`
  - section match를 신뢰 가능한 것으로 볼 최소 점수
- `allow_cross_section_fallback`
  - in-section fact가 부족할 때 외부 fact를 쓸지 여부
- `max_cross_section_facts_per_slot`
  - body slot 오염을 막기 위한 상한
- `use_qwen_for_section_rerank`
  - Qwen이 section candidate rerank에 참여할지 여부
- `use_qwen_for_fact_rerank`
  - v1에서는 `False`로 고정. Qwen을 fact-level 자유 선택기로 쓰지 않는다
- `use_global_evidence_channel`
  - section-local retrieval 외에 document-global evidence channel을 정식 경로로 허용할지 여부
- `max_global_facts_per_slot`
  - slot이 전역 evidence를 몇 개까지 참조할 수 있는지에 대한 상한

### 8.2 `GenerationRunSummary` 확장 방향

현재 `GenerationRunSummary`는 slot 중심 지표가 강하다. section-first generation에서는 최소한 아래 필드가 추가되어야 한다.

- `section_match_count`
- `matched_section_count`
- `unmatched_section_count`
- `cross_section_fallback_count`
- `synthesis_section_count`
- `successful_synthesis_match_count`

이 필드가 있어야 quality regression을 slot 생성 개수만이 아니라 section 매핑 안정성 기준으로 볼 수 있다.

## 9. 상세 데이터 흐름과 동작 설계

## 9.0 planning mode 분기

section-first generation은 모든 reference section에 동일한 매핑 규칙을 적용하지 않는다. 먼저 각 reference section의 `composition_strategy`를 보고 planning path를 결정한다.

- `local_align`
  - 일반 thesis/body 중심 section
  - `reference section 1개 -> source section 1개`
- `bounded_synthesis`
  - 비교형/분석형 본문처럼 제한된 수의 source section을 묶는 section
  - `reference section 1개 -> source section 2~3개`
- `global_synthesis`
  - executive summary, cover summary, conclusion처럼 문서 전반을 요약하는 section
  - `reference section 1개 -> source section 여러 개`
- `global_lookup`
  - metadata, author, date, key metric처럼 전역 evidence channel을 우선 쓰는 section

### 9.0.1 언제 synthesis 계열 전략을 쓰는가

아래 조건 중 하나라도 강하게 만족하면 synthesis section으로 분류하는 것이 좋다.

- `purpose`가 `introduction`, `executive_summary`, `cover_summary`, `conclusion`
- slot role histogram이 `summary + short body + metric box`처럼 문서 전체 압축형 구조
- reference section이 문서 앞부분에 있으면서 여러 thesis를 한 번에 요약하는 레이아웃
- 한 source section만으로는 coverage가 부족하고 여러 section의 핵심을 모아야 자연스러운 경우

즉 synthesis는 fallback이 아니라 정식 planning mode다.

### 9.0.2 strategy ladder

초안에서는 사실상 `local_align`와 `multi_section_synthesis`의 2분법으로 읽히지만, 기능 설계상 더 자연스러운 전략 체계는 아래와 같다.

- `local_align`
  - 하나의 source section을 중심으로 채움
- `bounded_synthesis`
  - 2~3개 source section을 제한적으로 묶음
  - 비교형 본문, pros/cons, 원인-결과형 분석 section에 적합
- `global_synthesis`
  - 문서 전반의 핵심 축을 모아 요약
  - executive summary, cover summary, conclusion에 적합
- `global_lookup`
  - 특정 section보다 document-global evidence channel을 우선 사용
  - date, author, ticker, cover metric, firm info 같은 슬롯에 적합

즉 `multi_section_synthesis`는 더 정확히 말하면 `bounded_synthesis`와 `global_synthesis`로 쪼개는 편이 좋다. 이 구분이 있어야 synthesis가 summary 전용 예외가 아니라, 다양한 복합 section을 표현하는 일반 전략이 된다.

### 9.0.3 왜 summary 전용 synthesis로 두면 안 되는가

실제 문서에서는 본문도 여러 source section을 묶어야 하는 경우가 많다. 예를 들면 sector comparison, catalyst vs risk, before/after, cause/effect 같은 section은 body이지만 `bounded_synthesis`가 더 자연스럽다. 따라서 `summary = synthesis`, `body = local` 같은 이분법은 기능적으로 너무 강한 제약이다.

## 9.1 `reference_blueprint(v2)` 생성 단계

현재 `build_reference_blueprint()`는 slot-only 구조에 가깝다. section-first generation에서는 blueprint가 section-aware 구조를 만들어야 한다.

입력은 여전히 `reference_template.json`이지만, section source의 우선순위는 아래처럼 정의한다.

1. `section_artifact.json`이 있으면 그것을 authoritative source로 사용
2. 없으면 `reference_template.section_order`를 narrative projection/fallback으로 사용

즉 공식 경로는 `section_artifact` 기반이고, `section_order` 기반은 호환성 fallback이다.

이 단계의 출력은 아래를 포함해야 한다.

- `sections[]` as `ReferenceSectionPlan[]`
- `slots[]` as section-aware `ReferenceSlot[]`
- slot과 section의 일관성 검증 결과

### 9.1.1 reference section grouping 규칙

reference section은 완전히 새로 만들어낸다기보다, 기존 template 또는 section artifact가 제공하는 section 구조를 generation-friendly하게 projection한다.

이 projection에서 중요한 것은 아래다.

- 모든 slot은 정확히 하나의 reference section에 속해야 한다.
- `preserve_reference_text` 슬롯도 section에는 속한다.
- narrative section과 non-narrative section을 구분하되, generation 대상이 아닌 section도 artifact에는 남길 수 있다.
- synthesis section은 purpose와 slot shape를 기준으로 명시적으로 표기해야 한다.

## 9.2 `source_fact_pack(v2)` 생성 단계

현재 `build_source_fact_pack()`는 `facts[]` 중심이다. section-first generation에서는 여기에 `source_sections[]`가 추가되어야 한다.

이 단계의 역할은 두 가지다.

1. 기존처럼 generation용 `SourceFact[]`를 만든다.
2. 같은 fact들을 authoritative source section 아래로 묶은 `SourceSectionView[]`를 만든다.

### 9.2.1 source section summary

`SourceSectionView.section_summary_text`는 Qwen rerank와 trace 설명을 위해 필요하다. v1에서는 section 내 대표 heading과 대표 body fact를 조합한 짧은 요약으로 정의한다.

예:

- heading이 있으면 heading 우선
- body fact가 있으면 첫 1~2개 body를 뒤에 붙임
- evidence/meta만 있는 section은 해당 성격이 드러나게 요약

즉 summary는 human-facing 설명이자 model payload 요약이다.

### 9.2.1A section summary cache의 역할과 한계

`section_summary_text`는 필요하지만, 이것만으로 Qwen의 최종 판단을 맡기면 summary-only blind spot이 생긴다. 즉 핵심 논지가 요약에서 빠지면 모델은 잘못된 section을 고를 수밖에 없다. 따라서 summary는 최종 판단 입력이 아니라 coarse retrieval용 index로 취급해야 한다.

이 문제를 줄이기 위해 section detection 단계에서는 단일 3줄 요약문만 저장하기보다 `section card`를 캐시하는 것이 좋다.

권장 캐시 항목은 아래와 같다.

- `section_summary_text`
- `heading_text` 또는 `raw_heading`
- `key_claims[]`
- `evidence_snippets[]`
- `risk_snippets[]`
- `summary_confidence`

즉 section detection에서 Qwen이 동작하는 경우, Qwen이 section을 나누는 것과 함께 짧은 summary를 생성하는 것은 유용하다. 다만 그 결과물은 generation의 최종 판단 근거가 아니라, 이후 generation 단계에서 reference-conditioned candidate packet을 만들기 위한 cached section card로 이해해야 한다.

정리하면 detection 단계의 summary 생성은 `좋은 최적화`이지만 `단독 해법`은 아니다. 중요한 것은 summary를 저장하는 것보다, generation 단계에서 확장 가능한 다중 뷰 카드로 저장해 두는 것이다.

### 9.2.2 section grouping의 우선성

source fact extraction은 계속 유지하지만, source section이 존재하는 순간 이후 planning에서는 `fact.section_id`보다 `source_sections[]`의 membership이 authoritative하다.

## 9.3 local section candidate generation

`local_align` reference section마다 source section 전체를 다 보게 하면 noisy하다. 따라서 먼저 rule-based prefilter로 후보 집합을 작게 만든다.

v1에서 prefilter 기준은 아래와 같이 정의한다.

- `is_narrative`
- `purpose compatibility`
- `quality`
- `section length`
- `evidence_ratio`
- `heading presence`
- `page order proximity`
- `document_family prior`

후보 수는 `max_section_candidates = 3`으로 고정한다.

### 9.3.1 prefilter 예시

- reference thesis section
  - narrative이고 body density가 있는 source section을 우선 남긴다.
- reference summary section
  - investment_summary, opinion box, price snapshot 계열 section을 우선 남긴다.
- reference metadata section
  - metadata 또는 author/date 성격 section을 우선 남긴다.
- reference heading-only section
  - heading presence가 strong한 section을 우선 남긴다.

즉 prefilter의 목적은 정답을 결정하는 것이 아니라, `말이 안 되는 후보를 먼저 제거하는 것`이다.

## 9.3A multi-section synthesis candidate generation

`bounded_synthesis`와 `global_synthesis`에서는 candidate generation의 목표가 달라진다. 이 단계는 “가장 비슷한 source section 1개”를 찾는 것이 아니라, “요약 section이 덮어야 할 핵심 축을 충분히 커버하는 source section 집합”을 찾는 것이다.

핵심 아이디어는 아래와 같다.

1. reference summary section에서 필요한 coverage axis를 추출한다.
2. 각 axis를 가장 잘 대표하는 source section 후보를 찾는다.
3. 중복이 심한 section을 줄이고, narrative와 evidence가 균형 있게 섞인 bundle 후보를 만든다.
4. bundle 후보를 score/rerank해서 최종 source section 집합을 선택한다.

### 9.3A.1 coverage axis

v1에서 synthesis section이 보통 요구하는 축은 아래와 같다.

- `core_thesis`
- `secondary_thesis`
- `numeric_evidence`
- `risk_or_caveat`
- `market_context`

모든 요약 section이 이 축을 다 필요로 하지는 않는다. planner는 reference section의 slot role histogram과 purpose를 보고 필요한 축만 활성화한다.

### 9.3A.2 bundle 후보 생성

bundle 후보 생성은 아래처럼 설명할 수 있다.

- thesis 목적 source section 중 salient score가 높은 것 1~2개
- evidence/support 목적 source section 중 metric coverage가 높은 것 1개
- 필요하면 risk/caveat 목적 source section 1개

즉 bundle은 보통 `2~4개 section`으로 제한하는 것이 좋다. v1 기본 상한은 `max_synthesis_sections = 4`로 두는 것이 안전하다.

### 9.3A.3 bundle 점수화 기준

synthesis bundle의 점수는 단일 section 점수와 달리 set-level objective를 가져야 한다.

- `coverage_score`
  - 필요한 축을 얼마나 고르게 덮는가
- `diversity_score`
  - 같은 thesis를 반복하지 않는가
- `salience_score`
  - 문서의 핵심 주장과 대표 숫자를 포함하는가
- `evidence_support_score`
  - 주장만 있고 숫자 근거가 비지 않는가
- `redundancy_penalty`
  - 비슷한 section만 여러 개 뽑지 않는가
- `bundle_size_penalty`
  - 너무 많은 section을 섞어 summary coherence를 해치지 않는가

즉 synthesis planning은 `best single match` 문제가 아니라 `best section set` 문제다.

## 9.4 rule section scoring

`local_align` prefilter를 통과한 후보들에 대해 section-level score를 계산한다. v1에서 공식 feature는 아래와 같다.

- `purpose match`
- `heading/body shape similarity`
- `narrative density match`
- `evidence density penalty`
- `metadata-only penalty`
- `section confidence`
- `page/order monotonicity bonus`
- `reuse penalty`
- `family prior bonus`

이 중 가장 중요한 두 feature는 아래로 고정한다.

- `purpose match`
- `narrative density match`

### 9.4.1 feature 설명

- `purpose match`
  - reference section purpose와 source section purpose의 호환성
- `heading/body shape similarity`
  - heading 비중, body 비중, 짧은 박스/긴 본문 구조가 유사한지
- `narrative density match`
  - narrative body fact 비율이 reference 기대치와 유사한지
- `evidence density penalty`
  - evidence-only section이 narrative target에 들어오는 것을 막기 위한 감점
- `metadata-only penalty`
  - metadata section이 thesis/body target에 들어오는 것을 막기 위한 감점
- `section confidence`
  - section detection 자체의 confidence 반영
- `page/order monotonicity bonus`
  - 이전에 선택한 source section 순서와 너무 역행하지 않는지
- `reuse penalty`
  - 동일 source section이 과도하게 여러 reference section에 재사용되는지
- `family prior bonus`
  - 같은 family면 약한 가산점. hard gate는 아님

### 9.4.2 점수식 예시

문서에서는 exact weight를 고정할 필요는 없지만, score 구조는 아래 수준으로 설명한다.

```text
section_score =
  w1 * purpose_match
+ w2 * heading_body_shape_similarity
+ w3 * narrative_density_match
+ w4 * section_confidence
+ w5 * page_order_monotonicity_bonus
+ w6 * family_prior_bonus
- w7 * evidence_density_penalty
- w8 * metadata_only_penalty
- w9 * reuse_penalty
```

핵심은 `purpose match`와 `narrative density`가 가장 큰 영향력을 가져야 한다는 점이다.

## 9.5 Qwen section rerank/select

v1에서 Qwen은 section-aware generation에 참여하지만, 역할은 엄격히 제한된다.

### 9.5.1 Qwen의 역할

Qwen은 아래 역할만 맡는다.

- rule prefilter가 만든 `top-k source section candidates`를 재정렬한다.
- synthesis section에서는 `top-k source section bundles`를 재정렬한다.
- 필요하면 그중 하나를 최종 선택한다.
- 선택 이유를 구조화된 rationale로 남긴다.

반대로 Qwen은 아래 역할을 맡지 않는다.

- 새로운 source section invent
- candidate set 밖 source section 선택
- candidate set 밖 bundle invent
- fact-level 자유 검색
- writing 단계와 매핑 단계를 섞는 일

즉 v1의 Qwen은 `candidate-constrained section selector`이며, synthesis section에서는 `candidate-constrained section bundle reranker`다.

### 9.5.2 입력 payload

Qwen payload에는 아래가 포함된다.

- `reference section summary`
- `required slot roles`
- `top-k source section candidates` 또는 `top-k source section bundles`
- `candidate summaries`
- `candidate fact snapshots`
- `style/language hint`

### 9.5.2A summary-only blind spot 대응 원칙

Qwen이 최적의 section 또는 bundle을 고를 때 `section_summary_text`만 보고 판단하면, 실제로는 룰 기반 요약기의 편향을 재평가하는 것에 가까워진다. 따라서 v1에서도 summary-only rerank는 피해야 한다.

권장 흐름은 아래와 같다.

1. static summary로 coarse shortlist를 만든다.
2. shortlist에 대해서만 raw evidence가 포함된 expanded candidate packet을 만든다.
3. Qwen은 expanded packet을 보고 최종 rerank/select를 수행한다.

즉 summary는 검색용이고, 최종 선택은 raw supporting snippets가 포함된 packet 위에서 내려야 한다.

### 9.5.2B expanded candidate packet

각 candidate source section은 최소한 아래와 같은 multi-view packet으로 Qwen에 전달하는 것이 좋다.

- `summary`
- `raw_heading`
- `raw_body_head`
- `raw_body_tail`
- `raw_evidence_sample`
- `raw_risk_sample`
- `role_distribution`
- `section_confidence`
- `summary_coverage_confidence`

이 중 `raw_heading`, `raw_body_head`, `raw_body_tail`은 사용자가 지적한 방식처럼 실제 원문 일부를 보여주는 필드다. 이 방식은 매우 효과적이며, Qwen이 summary 문장만 보지 않고 실제 문맥 조각을 함께 볼 수 있게 해준다.

예를 들면 candidate packet은 아래처럼 확장될 수 있다.

```json
{
  "candidate_source_sections": [
    {
      "source_section_id": "src-sec-005",
      "summary": "AI 서버 전환 관련 논지",
      "raw_heading": "[제목] 글로벌 AI 투자 동향",
      "raw_body_head": "3분기 주요 빅테크들의 AI 설비투자가 전분기 대비 15% 증가했으며...",
      "raw_body_tail": "... 이로 인해 관련 서플라이 체인의 수혜가 예상된다.",
      "raw_evidence_sample": "AI CAPEX +15%, 서버 수요 회복 가시화",
      "summary_coverage_confidence": 0.82
    }
  ]
}
```

핵심은 `summary + raw snippets + structure signals`를 함께 주는 것이다.

### 9.5.2C reference-conditioned candidate view

또한 같은 source section이라도 reference section의 목적에 따라 보여줄 요약이 달라져야 한다. 즉 generation 단계에서는 detection 때 저장한 static summary를 그대로 쓰기보다, reference purpose에 맞춘 decision view를 조립하는 것이 좋다.

예를 들면:

- thesis body section을 고를 때
  - heading + 대표 body snippet 중심
- executive summary용 section을 고를 때
  - 핵심 thesis + 숫자 근거 + risk/caveat snippet 중심
- evidence panel을 고를 때
  - metric/evidence snippet 중심

즉 detection 단계에서 cached section card를 만들고, generation 단계에서는 이를 reference-conditioned candidate packet으로 재구성하는 2단계 설계가 가장 안전하다.

### 9.5.2D adaptive expansion

summary 품질이 낮거나 coverage가 불충분하면 더 많은 원문을 보여주는 적응형 확장도 필요하다.

예를 들면 아래와 같은 조건에서 snippet 확장을 늘릴 수 있다.

- `summary_coverage_confidence`가 낮음
- heading은 약하지만 body가 긴 section
- evidence snippet이 없는 summary section
- synthesis bundle에서 핵심 축 하나가 비어 있음

이 경우 `raw_body_sample` 수를 늘리거나, evidence/risk snippet을 추가 확장한 뒤 Qwen rerank를 다시 수행하는 것이 좋다.

예시 스켈레톤은 아래와 같다.

```json
{
  "reference_section": {
    "reference_section_id": "ref-sec-003",
    "title": "투자포인트 2",
    "purpose": "thesis",
    "required_slot_roles": ["section_heading", "body", "body"],
    "summary": "긴 본문 2개와 heading 1개로 구성된 narrative thesis section"
  },
  "candidate_source_sections": [
    {
      "source_section_id": "src-sec-005",
      "purpose": "thesis",
      "summary": "AI 서버 전환 관련 고객사 회복 논지를 담은 thesis section",
      "fact_snapshots": ["...", "..."]
    }
  ],
  "style_hint": {
    "document_family": "financial_report",
    "language": "ko"
  }
}
```

synthesis section에서는 아래와 같은 bundle payload도 가능하다.

```json
{
  "reference_section": {
    "reference_section_id": "ref-sec-000",
    "title": "Executive Summary",
    "purpose": "executive_summary",
    "required_slot_roles": ["summary", "body", "plain"],
    "summary": "문서 전체 thesis와 숫자 근거를 압축하는 요약 section"
  },
  "candidate_bundles": [
    {
      "bundle_id": "bundle-01",
      "source_section_ids": ["src-sec-003", "src-sec-007", "src-sec-009"],
      "coverage_axes": ["core_thesis", "numeric_evidence", "risk_or_caveat"],
      "summary": "핵심 thesis 1개, 보조 thesis 1개, 숫자 근거 1개를 포함하는 bundle"
    }
  ]
}
```

### 9.5.3 출력 계약

출력은 자유 텍스트가 아니라 아래처럼 구조화된 JSON으로 제한한다.

```json
{
  "selected_source_section_id": "src-sec-005",
  "confidence": 0.88,
  "reason": "Reference thesis section requires a narrative section with heading plus multi-body support, and candidate src-sec-005 best matches that structure."
}
```

synthesis section에서는 아래처럼 bundle 선택 결과를 반환할 수 있다.

```json
{
  "selected_source_section_ids": ["src-sec-003", "src-sec-007", "src-sec-009"],
  "primary_source_section_id": "src-sec-003",
  "confidence": 0.86,
  "reason": "This bundle best covers the main thesis, numeric evidence, and risk context required by the executive summary."
}
```

이 계약의 핵심은 아래 두 가지다.

- Qwen은 후보 집합 밖을 선택할 수 없다.
- Qwen은 section 또는 bundle 선택 이유를 명시적으로 남겨야 한다.

### 9.5.4 fallback 정책

Qwen rerank가 실패하면 아래처럼 처리한다.

- schema mismatch
  - rule top-1 또는 rule top-bundle 사용
  - trace에 fallback reason 기록
- confidence가 너무 낮음
  - rule top-1, rule top-bundle 또는 unmatched 처리
  - `needs_review=true`
- candidate set이 비어 있음
  - `selected_source_section_id=null`
  - synthesis에서는 `selected_source_section_ids=[]`
  - 이후 slot-level fallback은 제한적으로만 허용

즉 v1에서 Qwen은 quality booster이지, deterministic baseline을 대체하는 single point of failure가 아니다.

## 9.5A synthesis section의 slot-to-fact selection

`bounded_synthesis`와 `global_synthesis`에서는 slot-to-fact selection도 단일 section 내부 탐색이 아니라 `selected_source_section_ids`가 만든 bundle 안에서 이뤄진다.

하지만 여기서도 무제한 자유 선택은 허용하지 않는다. 대신 `bundle-aware deterministic selection`을 사용한다.

기본 원칙은 아래와 같다.

- summary slot
  - 서로 다른 thesis source section에서 1개씩 대표 fact를 뽑고, 필요하면 evidence section에서 숫자 fact를 1~2개 추가
- short body slot
  - primary thesis section 또는 secondary thesis section의 대표 body fact 우선
- metric/data slot
  - evidence/numeric source section에서 선택
- caution/risk slot
  - risk/caveat 성격 section이 있으면 그 section을 우선 사용

즉 synthesis section의 목표는 “하나의 section 안에서 놀기”가 아니라 “여러 source section이 기여하되, 각 section이 무엇을 기여하는지 통제된 방식으로 유지하는 것”이다.

### 9.5A.1 `section_contributions`

bundle planning 결과에는 최소한 아래 같은 contribution 요약이 남아야 한다.

```json
[
  {
    "source_section_id": "src-sec-004",
    "role": "core_thesis",
    "planned_fact_budget": 2
  },
  {
    "source_section_id": "src-sec-007",
    "role": "numeric_evidence",
    "planned_fact_budget": 2
  }
]
```

이 정보가 있어야 summary draft가 어떤 section들을 어떻게 종합했는지 추적할 수 있다.

## 9.5B document-global evidence channel 사용 규칙

generation은 section-local retrieval만으로 충분하지 않다. 일부 슬롯은 section보다 document-global evidence channel을 우선 사용해야 더 자연스럽다.

권장 규칙은 아래와 같다.

- narrative body
  - 기본은 section-local 또는 synthesis bundle
- comparative/analytic body
  - `bounded_synthesis`를 우선하고, 필요 시 global evidence를 보조적으로 사용
- executive summary / cover summary
  - `global_synthesis` + global evidence card를 함께 사용 가능
- metadata / author / date / ticker
  - `global_lookup`을 우선 사용
- key metric / valuation / market snapshot box
  - section보다 `global_evidence_cards`를 우선 사용 가능

즉 section 밖 정보는 fallback이 아니라 정식 channel이며, plan artifact에도 이 사용 사실이 남아야 한다.

## 9.6 matched section 내부의 slot-to-fact selection

`local_align`에서 section match가 끝나면 그다음은 slot별 fact selection이다. 이 단계는 `selected_source_section_id` 내부 fact만 우선 대상으로 본다.

### 9.6.1 기본 원칙

- body slot
  - `body_fact_ids`와 `section kind` fact 우선
- heading slot
  - `heading_fact_ids` 우선
- summary slot
  - summary/evidence mix 허용
- metadata slot
  - `meta_fact_ids` 우선
- preserve slot
  - reference text 유지

즉 `_score_fact_for_slot()`의 role matching 철학은 유지하되, scoring의 대상 pool을 먼저 줄인다.

### 9.6.2 v1의 결정

중요한 점은 v1에서 Qwen을 fact-level 자유 선택기로 쓰지 않는다는 것이다.

이유는 아래와 같다.

- section 매핑과 fact 선택을 한 번에 모델에게 맡기면 traceability가 떨어진다.
- 오류가 났을 때 section match 문제인지 fact rerank 문제인지 분리가 어렵다.
- deterministic scorer는 slot role과 table/metadata 안전 규칙을 안정적으로 적용하기 좋다.

따라서 v1에서는:

- section match는 `rule + optional Qwen rerank`
- fact selection은 `deterministic in-section scoring`
- synthesis section은 `deterministic bundle-aware scoring`

으로 역할을 분리한다.

## 9.7 cross-section fallback

`local_align`에서 selected source section 내부에 적합한 fact가 없을 때만, 제한적으로 cross-section fallback을 허용한다.

v1 정책은 아래처럼 고정한다.

- `metadata/author/date` slot
  - section 외부 fallback 허용
- `summary` slot
  - evidence panel 또는 market snapshot fallback 허용
- `body` slot
  - section 외부 fallback 최대 1 fact 허용
- `title/heading` slot
  - section 외부 fallback 기본 금지

fallback이 발생하면 아래를 반드시 남긴다.

- `cross_section_fallback_used = true`
- `needs_review = true`
- rationale에 fallback 이유 명시

### 9.7.1 왜 body fallback을 제한하는가

body slot에 external fact가 여러 개 섞이기 시작하면 section-first의 핵심 장점이 사라진다. 따라서 v1에서는 body slot의 cross-section fallback을 아주 보수적으로 둔다.

단, 이 규칙은 `local_align` section에만 강하게 적용한다. `multi_section_synthesis`는 cross-section이 fallback이 아니라 본질이므로 별도의 budget 규칙을 쓴다.

### 9.7.1A synthesis budget

요약형 section은 `cross-section fallback` 대신 `bundle budget` 개념으로 다루는 것이 맞다.

권장 budget 예시는 아래와 같다.

- summary/introduction/conclusion section
  - source section 최대 4개
  - section당 fact 최대 2개
  - 전체 fact budget 6~8개
- cover summary
  - thesis 1~2개
  - numeric evidence 1~2개
  - 필요 시 risk/context 1개

즉 synthesis section에서는 `body slot당 외부 fact 1개` 같은 고정 규칙보다, `section purpose 기반 bundle budget`이 더 적합하다.

### 9.7.2 매칭 실패 section 처리

reference section 자체가 source section 또는 source section bundle에 매칭되지 않았다면, 이후 slot assignment는 아래 순서로 보수적으로 처리한다.

1. preserve 가능한 slot은 reference text 유지
2. metadata 계열만 제한적 global fallback 허용
3. narrative/body slot은 unmatched로 두거나 review-required fallback 사용

즉 `매칭 실패를 조용히 숨기지 않는 것`이 중요하다.

## 9.8 draft writing 단계

writing 단계는 현재 `_resolve_slot_draft()` 구조를 크게 유지한다. v1의 핵심 변화는 매핑이며, writing은 그 위에 올라가는 단계이기 때문이다.

다만 payload에는 아래 정보를 추가해 더 일관된 writing을 유도한다.

- `reference_section_title`
- `source_section_title`
- `source_section_purpose`
- `section_match_reason`
- synthesis section이면 `selected_source_section_ids`
- synthesis section이면 `section_contributions`

즉 writer는 “무슨 slot인지”뿐 아니라 “어떤 section 대응 또는 bundle 대응 위에서 이 slot이 채워졌는지”도 알게 된다.

### 9.8.1 writing 단계에서 하지 않는 것

v1 writing 단계는 아래를 하지 않는다.

- section match 재판정
- fact set 재검색
- fact-level 자유 재조합

이것들은 planning/assignment 레이어 책임이다.

## 10. artifact 변화

## 10.1 `reference_blueprint.json`

현재는 `slots` 중심 구조지만, v1에서는 `sections + slots`를 함께 담는 section-aware blueprint로 확장한다.

최소 변화:

- `sections[]` 추가
- 각 section에 `composition_strategy` 포함
- 각 slot에 `slot_index_in_section` 포함
- section과 slot의 cross-reference 검증

## 10.2 `source_fact_pack.json`

현재는 `facts` 중심 구조다. v1에서는 아래를 추가한다.

- `source_sections[]`
- 각 section의 summary text
- 각 section의 fact membership 정보
- `global_fact_pool`
- `global_metadata`
- `global_evidence_cards`

즉 `facts[]`는 유지하되, planning의 상위 단위는 `source_sections[]`가 된다.

## 10.3 `generation_plan.json`

현재의 flat slot assignment 리스트에서 아래 구조로 바뀐다.

- `reference_sections`
- `source_sections`
- `section_matches`
- `slot_assignments`
- `summary`

이 artifact가 section-first generation의 핵심 디버깅 지점이다.

또한 synthesis section이 있는 경우 `section_matches` 안에 bundle match 정보가 남아야 한다.

## 10.4 `generation_trace.json`

trace stage는 최소 두 가지로 나눈다.

- `match_section`
- `write_slot`

필요하면 아래를 추가할 수 있다.

- `fallback_slot_assignment`

즉 앞으로는 “Qwen writing trace”뿐 아니라 “Qwen section matching trace”도 공식 artifact에 남아야 한다.

## 10.5 `generation_diagnostics.json`

summary에는 최소한 아래 section-level metric을 넣는다.

- `section_match_count`
- `matched_section_count`
- `unmatched_section_count`
- `cross_section_fallback_count`
- `needs_review_count`
- `synthesis_section_count`
- `successful_synthesis_match_count`

## 11. v1에서 고정하는 결정

이 문서는 구현자가 임의로 흔들리지 않도록 v1 범위를 아래처럼 고정한다.

- generation의 공식 기본 전략은 `section_first`
- v1 source 입력은 `단일 source artifact`
- `local_align`, `bounded_synthesis`, `global_synthesis`, `global_lookup`를 v1의 정식 strategy로 인정
- `introduction`, `executive_summary`, `cover_summary`, `conclusion`은 기본적으로 `global_synthesis` 후보로 본다
- v1 Qwen 매핑은 `section candidate rerank/select` 또는 `bundle rerank/select`까지만 담당
- v1 slot fact selection은 deterministic
- v1 writing은 현재 Qwen writer를 유지
- `document_family`는 gating이 아니라 ranking prior
- `section_artifact`가 있으면 그것이 정식 입력
- `section_artifact`가 없을 때만 `section_order` 기반 section view를 fallback으로 허용

v1에서 제외하는 항목은 아래와 같다.

- multi-source section stitching
- full `only_qwen` generation matching
- Qwen fact-level free selection
- section match와 writing의 end-to-end joint inference
- synthesis bundle의 fact-level free selection

## 12. 테스트 계획

section-first generation은 글이 그럴듯한지보다 `계획이 맞는지`를 먼저 검증해야 한다. 따라서 테스트도 artifact correctness 중심으로 설계한다.

## 12.1 `reference_blueprint v2`

검증 항목:

- section list와 slot list가 일관되게 생성되는가
- 모든 slot이 정확히 하나의 reference section에 속하는가
- `slot_index_in_section`이 reading order와 충돌하지 않는가
- synthesis section에 `composition_strategy=global_synthesis` 또는 `bounded_synthesis`가 제대로 붙는가

## 12.2 `source_fact_pack v2`

검증 항목:

- 모든 narrative fact가 source section에 속하는가
- section 밖 전역 정보가 `global_fact_pool` 또는 `global_metadata`로 분리되는가
- section summary와 fact ids가 일관되는가
- evidence/meta fact가 잘못 narrative section에 과다 포함되지 않는가

## 12.3 rule section matching

검증 항목:

- thesis body slot이 같은 source thesis section 내부 fact를 우선 사용하는가
- evidence panel이 narrative section match로 선택되지 않는가
- metadata slot만 section 외부 fallback을 허용하는가

## 12.3A synthesis section matching

검증 항목:

- executive summary가 source section 2개 이상을 사용하는 bundle로 계획될 수 있는가
- bundle이 `thesis + evidence` 축을 함께 포함하는가
- 동일 논점만 반복하는 redundant bundle이 선택되지 않는가
- `section_contributions`와 fact budget이 artifact에 남는가

## 12.4 Qwen section rerank

검증 항목:

- top-k source section 후보 중 올바른 section을 선택 또는 재정렬하는가
- synthesis mode에서 top-k bundle 후보 중 더 좋은 bundle을 선택 또는 재정렬하는가
- 선택 이유와 confidence가 artifact에 남는가
- schema mismatch 시 rule top-1 또는 rule top-bundle로 되돌아가고 trace에 fallback이 남는가

## 12.5 end-to-end generation

검증 항목:

- `generation_plan.json`에 `selected_source_section_id` 또는 `selected_source_section_ids`가 채워지는가
- section match 실패 시 `cross_section_fallback_used=true`가 남는가
- synthesis section에서 bundle coverage가 artifact에 남는가
- 기존 markdown/html 생성이 계속 가능하고 section grouping이 더 자연스러워지는가

## 12.6 regression 관점의 기준 변경

기존 테스트는 exact fact id matching에 기울기 쉽다. section-first generation에서는 아래를 더 중요한 기준으로 삼는다.

- `selected section correctness`
- `in-section fact preference`
- `cross-section fallback visibility`
- `synthesis bundle coverage`

즉 회귀 기준도 flat retrieval 정확도보다 section-aware correctness 중심으로 바뀌어야 한다.

## 13. 구현 순서 제안

실제 구현은 아래 순서가 가장 안전하다.

1. `reference_blueprint(v2)`에 `sections[]`와 `composition_strategy` 추가
2. `source_fact_pack(v2)`에 `source_sections[]` 추가
3. `generation_plan.json` 루트를 `section_matches + slot_assignments + summary` 구조로 확장
4. rule section-first planner 구현
5. `local_align`/`bounded_synthesis`/`global_synthesis`/`global_lookup` planner path 분기 추가
6. Qwen section/bundle rerank/select 추가
7. in-section 또는 bundle-aware deterministic fact selection으로 slot assignment 연결
8. writer payload에 section match context 추가
9. diagnostics/test를 section-aware 기준으로 갱신

즉 가장 먼저 할 일은 writing 개선이 아니라 `plan artifact를 section-aware하게 바꾸는 것`이다.

## 14. 결론

section-first generation은 단순히 slot scoring을 조금 더 똑똑하게 만드는 작업이 아니다. generation의 공식 검색 단위를 `slot -> fact`에서 `reference section -> source section -> slot -> fact`로 바꾸는 구조 전환이다.

다만 이 전환은 모든 section을 `1:1 section match`로 몰아가는 뜻이 아니다. 본문형 section은 local align이 맞지만, 요약형 section은 애초에 여러 source section의 핵심을 종합해야 한다. 따라서 `multi-section synthesis`를 fallback이 아니라 정식 planning mode로 인정하는 것이 중요하다.

이 전환이 필요한 이유는 분명하다.

- section 흐름을 유지하고
- narrative와 evidence를 더 안정적으로 분리하고
- Qwen의 역할을 traceable하게 제한하고
- source 문서 다양성에 더 잘 대응하고
- introduction이나 executive summary가 여러 논점과 숫자 근거를 함께 종합할 수 있게 하기 위해서다

v1에서는 욕심을 줄이는 것이 중요하다. section 매핑은 `rule + Qwen rerank`로 제한하고, fact selection은 deterministic하게 유지해야 한다. 다만 요약형 section은 예외가 아니라 정식 synthesis path를 가져야 한다. 그래야 introduction이나 executive summary가 여러 source section의 핵심을 종합하는 본래 역할을 수행할 수 있다.

이 문서 기준의 첫 구현 목표는 더 세련된 문장 생성이 아니라, `같은 source section 안에서 더 일관된 slot assignment`와 `요약형 section에서 더 좋은 multi-section coverage`를 만드는 것이다.
