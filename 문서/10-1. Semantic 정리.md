# Semantic 정리

이 문서는 현재 구현된 `allm` 프로젝트에서 `semantic` 단계가 실제로 어떻게 동작하는지, 어떤 기준으로 block의 역할을 판단하는지, 그리고 `section`이 어떤 과정으로 결정되는지를 코드 기준으로 자세히 정리한 문서다.

이 문서의 범위는 크게 3가지다.

- block 하나에 semantic role이 붙는 과정
- `rule`, `qwen`, `shadow` 모드의 차이
- `section_id`와 `section_order`가 결정되는 방식

중요한 전제는 현재 semantic 계층이 `문서 전체를 한 번에 해석하는 시스템`이 아니라, `canonical block 단위로 역할을 부여하고 그 결과를 후처리로 연결하는 구조`라는 점이다. 즉 semantic의 기본 단위는 페이지가 아니라 `block`이다.

---

## 1. semantic 단계가 시작되기 전에 이미 준비되어 있는 것

semantic은 OCR raw output에서 바로 시작하지 않는다. semantic이 시작될 때는 이미 아래 전처리가 끝난 상태다.

```text
OCR raw block
-> canonicalize
-> Dolphin/Paddle fusion
-> FusedPage 생성
-> PageAnalysis 생성
-> semantic 시작
```

즉 semantic은 아래 정보를 입력으로 받는다.

- `FusedPage`
  - canonical block 목록
  - reading order
  - bbox
  - fused text
  - source engine 정보

- `PageAnalysis`
  - page archetype
  - OCR quality score
  - column count
  - layout pattern
  - text/table/visual 비율

이 구조 때문에 semantic은 단순 키워드 분류기가 아니라, `정규화된 block + 페이지 구조 정보`를 보고 판단하는 계층이 된다.

---

## 2. semantic이 최종적으로 block에 채우는 필드

semantic 결과가 반영되면 `CanonicalBlock`에는 아래 필드들이 채워질 수 있다.

- `generic_role`
- `domain_role`
- `role_confidence`
- `section_id`
- `section_purpose`
- `used_for_generation`
- `semantic_source`
- `semantic_reason`
- `semantic_needs_review`
- `semantic_backend`
- `semantic_model_name`
- `semantic_prompt_version`
- `semantic_fallback_reason`
- `generated_role_name`
- `generated_role_description`
- `generated_parent_role_name`
- `generated_role_level`

이때 semantic이 바로 모든 값을 한 번에 확정하는 것은 아니다.

1. block별 1차 role decision을 만든다.
2. 그 decision을 코드가 정규화한다.
3. generation 정책과 section 연결을 후처리한다.
4. 마지막으로 `section_id`를 다시 부여한다.

즉, semantic 결과는 `모델/규칙의 판단 + deterministic post-processing`의 합성 결과다.

---

## 3. semantic 전체 오케스트레이션 흐름

semantic 전체 실행은 `apply_financial_semantic_overlay()`가 담당한다. 실제 흐름을 풀어 쓰면 아래와 같다.

```text
페이지 단위 반복
-> 페이지 분석 정보 조회
-> rule decision 준비
-> safety rule decision 준비
-> block 순회
-> block semantic 필드 초기화
-> 모드(rule/qwen/shadow)에 따라 decision 생성
-> decision 정규화 및 block 반영
-> 모든 block 처리 후 section_id 재부여
-> SemanticRunSummary 집계
```

이 과정에서 semantic은 크게 두 층으로 나뉜다.

- `판단층`
  - rule 또는 Qwen이 block 역할을 제안한다.

- `정규화/후처리층`
  - 허용 가능한 role pair인지 검증하고
  - confidence 기준을 적용하고
  - `used_for_generation`, `section_purpose`, `section_id`를 정리한다.

---

## 4. 먼저 block을 어떤 순서로 처리하는가

semantic은 block을 페이지 안에서 아래 기준으로 정렬한 뒤 순차 처리한다.

- `reading_order`
- `bbox y`
- `bbox x`

즉, semantic은 block 간 앞뒤 관계를 reading order 기반으로 본다. 이 순서는 이후 아래 판단에도 영향을 준다.

- 이전 텍스트 block / 다음 텍스트 block 선택
- nearest heading 계산
- nearest evidence 계산
- rule 모드의 `report_title_seen` 같은 상태 추적
- section_id 부여 순서

---

## 5. semantic이 block마다 가장 먼저 하는 일

각 block을 처리할 때 맨 먼저 `_reset_semantic_fields()`를 호출해 이전 semantic 값을 비운다.

초기화 대상:

- role 관련 필드
- section 관련 필드
- backend/model 정보
- generated role 정보

즉, semantic은 기존 block에 덮어쓰는 방식이 아니라, 매 실행마다 semantic 결과를 새로 계산하는 구조다.

---

## 6. semantic mode별 실제 동작 차이

현재 semantic mode는 3개다.

- `rule`
- `qwen`
- `shadow`

이 셋은 결과가 조금 다른 정도가 아니라, 판단 책임이 다르다.

### 6.1 `rule` 모드

이 모드는 `compute_financial_rule_decisions()`가 semantic owner다.

동작:

1. 문서군이 `financial_report`이면 financial rule decision을 만든다.
2. block마다 해당 rule decision이 있으면 바로 적용한다.
3. rule에 걸리지 않은 block은 semantic 필드가 비어 있을 수 있다.
4. 마지막에 section_id만 후처리로 붙인다.

특징:

- 가장 deterministic하다.
- 금융 리포트 구조에 강하게 맞춰져 있다.
- article, form, general report에서는 coverage가 약하다.

### 6.2 `qwen` 모드

이 모드는 `Qwen-first` 구조다.

동작:

1. block이 모델 호출 대상인지 확인한다.
2. block context payload를 만든다.
3. Qwen에 JSON only 응답을 요청한다.
4. 응답을 schema로 검증한다.
5. confidence와 role pair를 검사한다.
6. 통과하면 Qwen 결과를 적용한다.
7. 실패하면 safety rule 또는 unknown fallback을 적용한다.

특징:

- 가장 풍부한 semantic을 만든다.
- generated role 정보도 자연스럽게 채울 수 있다.
- OCR 품질과 응답 형식 안정성에 영향을 많이 받는다.

### 6.3 `shadow` 모드

이 모드는 `Qwen을 실행하되 final assignment는 rule을 유지`하는 비교 모드다.

동작:

1. Qwen을 실제로 호출한다.
2. trace를 남긴다.
3. 하지만 block에 실제 반영하는 것은 rule 결과다.
4. Qwen과 rule이 다르면 disagreement를 집계한다.

특징:

- migration 검증용이다.
- 현재 rule 체계를 깨지 않고 Qwen 결과를 평가할 수 있다.

---

## 7. rule 기반 semantic은 어떤 기준으로 판단하는가

rule 기반 semantic은 `compute_financial_rule_decisions()`와 `compute_financial_safety_rules()`에 구현되어 있다.

핵심 특징은 아래 세 가지다.

- 금융 리포트 전용에 가깝다.
- text, 위치, canonical label을 함께 본다.
- 일부 규칙은 다음 block까지 연결한다.

### 7.1 rule이 쓰는 대표 관측값

rule은 주로 아래 값을 쓴다.

- `text`
- `canonical_label`
- `raw_label`
- `bbox_norm`
- 위치 zone
  - 왼쪽 사이드바 여부
  - 상단 메타 영역 여부
  - 하단 메타 영역 여부
  - 메인 본문 영역 여부
- 정규식 / keyword
  - ticker
  - analyst / email
  - disclaimer 문구
  - buy/sell/hold
  - key data / consensus data / stock price

### 7.2 safety rule

safety rule은 비교적 high-precision한 것만 담당한다.

대표 대상:

- analyst 정보
- disclaimer / compliance
- top metadata
- research center metadata

여기서의 의도는 semantic 전부를 rule로 다 맞추는 것이 아니라, `틀리면 안 되는 메타/안전성 역할`을 보수적으로 붙이는 것이다.

### 7.3 financial rule의 주요 패턴

현재 구현상 대표 패턴은 아래와 같다.

- 상단 메타 영역에 `기업분석`, `report`, 연도 텍스트가 있으면
  - `generic_role=metadata`
  - `domain_role=report_header_meta`

- 이메일 또는 analyst 표현이 있으면
  - `generic_role=author_info`
  - `domain_role=analyst_info`

- disclaimer / compliance 키워드 또는 `list` raw label이면
  - `generic_role=disclaimer`
  - `domain_role=disclaimer_block`

- 좌측 사이드바에서 `buy`, `sell`, `hold`, `목표주가`, `현재주가` 등 투자 의견 키워드가 보이면
  - `generic_role=summary`
  - `domain_role=investment_opinion_box`

- 좌측 사이드바에서 `key data`가 보이면
  - 현재 block을 `section_heading + key_data_box`
  - 바로 다음 `table` block을 `evidence + key_data_box`로 연결

- 좌측 사이드바에서 `consensus data`가 보이면
  - 현재 block을 `section_heading + consensus_box`
  - 다음 `table` block을 evidence로 연결

- 좌측 사이드바에서 `stock price`가 보이면
  - 현재 block을 `section_heading + price_chart_block`
  - 다음 `chart/image` block을 evidence로 연결

- 메인 본문 영역의 table에 `매출액`, `영업이익`, `eps`, `bps`, `roe`, `per` 같은 금융 수치 단서가 있으면
  - `generic_role=evidence`
  - `domain_role=financial_table_block`

- 메인 본문 상단에서 ticker 패턴이나 document title이 잡히면
  - `generic_role=main_title`
  - `domain_role=report_title`

- report title 이후에 짧은 heading이 나오면
  - `generic_role=section_heading`
  - `domain_role=thesis_heading`

- 메인 본문에서 40자 이상 긴 문단이면
  - `generic_role=body`
  - `domain_role=supporting_argument`

- table/chart/image인데 더 강한 규칙이 없으면
  - `generic_role=evidence`
  - `domain_role=unsupported_evidence`

### 7.4 rule 모드의 중요한 한계

이 규칙들은 명시적으로 `financial_report` 문서군을 상정한다. 따라서 현재 코드에서는 아래 현상이 자연스럽다.

- article 문서에서는 domain role coverage가 줄어든다.
- form 문서에서는 semantic이 빈약할 수 있다.
- 위치와 키워드가 다른 레이아웃에서는 rule 강건성이 낮아질 수 있다.

즉, rule semantic은 `정확하지만 범용적이지는 않은` 구조다.

## 8. Qwen 기반 semantic은 어떤 입력을 보고 판단하는가

Qwen semantic의 핵심은 `build_block_context()`가 만드는 payload다. 모델은 이 payload를 보고 block 하나의 의미를 추론한다.

이 payload는 크게 5조각이다.

- task metadata
- target block 정보
- page context
- local neighbors
- structural relations

### 8.1 target block에 들어가는 정보

Qwen은 target block에 대해 아래 정보를 받는다.

- `block_id`
- `text`
- `canonical_label`
- `raw_label`
- `bbox_norm`
- `reading_order`
- `text_quality_score`
- `engine_sources`
- `tags`
- `flags`
- `zone_tags`
- `primary_zone`
- `ocr_labels`
- `block_index_on_page`
- `total_blocks_on_page`
- `center_x`
- `center_y`
- `width_ratio`
- `height_ratio`
- `area_ratio`

즉, 모델은 텍스트만 받는 것이 아니라 `이 block이 페이지 어디에 있고, 얼마나 크고, OCR label은 무엇이며, 몇 번째 block인지`까지 같이 본다.

### 8.2 page context에 들어가는 정보

page context에는 아래 값이 들어간다.

- `document_family`
- `page_archetype`
- `quality_score`
- `column_count`
- `dominant_layout_pattern`
- `left_sidebar_ratio`
- `meta_area_ratio`
- `text_area_ratio`
- `table_area_ratio`
- `visual_area_ratio`

즉, 모델은 block 하나만 고립해서 보지 않고, `이 페이지가 cover_summary인지 body_narrative인지`, `2단인지`, `비주얼이 많은지`도 함께 본다.

### 8.3 local neighbors는 어떻게 뽑는가

모델에게는 주변 block도 같이 준다. 이때 무작정 많이 주는 것이 아니라, 구조적으로 가까운 후보만 상위 K개를 준다.

현재 수집 대상:

- 이전 text block 최대 2개
- 다음 text block 최대 2개
- 가장 가까운 heading 최대 1개
- 가장 가까운 evidence 최대 2개

그리고 전체 neighbor 수는 `SemanticConfig.max_context_blocks`로 제한된다. 현재 기본값은 `6`이다.

neighbor 선택 시 쓰는 기준:

- reading order 거리
- 블록 중심점 거리
- 같은 zone 여부 보너스

즉, semantic context는 단순 앞뒤 1개가 아니라, `읽기 순서 + 공간 거리`를 함께 고려한 근접 context다.

### 8.4 structural relations에 들어가는 값

Qwen에게는 추가로 block 간 구조 관계도 준다.

- nearest heading과의 거리
- nearest evidence와의 거리
- nearest heading과 같은 column인지
- previous/next text와 같은 column인지
- heading/evidence와의 horizontal overlap
- nearest heading 대비 reading order 차이

이 정보 덕분에 모델은 예를 들어 아래를 더 잘 구분할 수 있다.

- 제목 바로 아래 본문인지
- 표 옆 캡션인지
- 사이드바 summary인지
- 본문 headline인지

### 8.5 OCR label은 어떻게 쓰는가

현재 프롬프트의 핵심 지시는 `OCR label은 low-level hint일 뿐 final semantic answer가 아니다`이다.

즉 모델은 아래를 참고할 수는 있다.

- dolphin `sec_0`
- paddle `doc_title`
- paddle `paragraph_title`

하지만 이것을 그대로 베끼지 말고, 텍스트와 구조를 함께 보고 semantic function을 판단하도록 유도된다.

## 9. Qwen에게 무엇을 출력하게 하는가

Qwen backend는 JSON only 응답을 강제한다. 모델은 block마다 아래 필드를 반환해야 한다.

- `block_id`
- `generated_role_name`
- `generated_role_description`
- `generated_parent_role_name`
- `generated_role_level`
- `generic_role`
- `domain_role`
- `role_confidence`
- `section_purpose`
- `used_for_generation`
- `reason`
- `needs_review`

### 9.1 각 출력 필드는 무엇을 의미하는가

#### `block_id`

- 지금 판단 중인 target block의 식별자다.
- 입력 payload의 `target_block.block_id`와 정확히 같아야 한다.
- 이 값이 다르면 parser에서 `schema_mismatch`로 떨어진다.

#### `generated_role_name`

- Qwen이 자유롭게 생성하는 1차 semantic role 이름이다.
- 현재 semantic 설계에서 가장 중요한 설명 필드다.
- snake_case의 짧은 이름을 기대한다.
- 예: `firm_branding_header`, `market_summary_paragraph`, `summary_heading`

즉 이 값은 `이 block이 실제로 무슨 기능을 하는가`를 가장 직접적으로 표현한다.

#### `generated_role_description`

- `generated_role_name`을 한 문장으로 설명한 값이다.
- downstream이나 사람 리뷰어가 role 이름만 보고 헷갈리지 않도록 붙는 보조 설명이다.
- 비어 있으면 schema 보강 단계에서 generic/domain role 기반 기본 설명으로 메워질 수 있다.

#### `generated_parent_role_name`

- 생성된 role이 더 큰 어떤 역할군에 속하는지를 나타내는 상위 역할 이름이다.
- 예를 들어 `summary`, `metadata`, `document_structure` 같은 상위 분류가 들어갈 수 있다.
- 없으면 `null`도 가능하다.

이 값은 문서 내부 role ontology를 더 계층적으로 만들기 위한 필드다.

#### `generated_role_level`

- 생성된 role이 문서 어느 수준의 역할인지 나타낸다.
- 허용값은 아래 3개다.
  - `document`
  - `section`
  - `block`

예를 들어:

- 문서 전체 제목이면 `document`
- 섹션 제목이면 `section`
- 일반 문단이나 요약 bullet이면 `block`

#### `generic_role`

- 시스템이 미리 정의한 legacy coarse role이다.
- 허용 enum은 다음 범주 안에서만 골라야 한다.
  - `main_title`
  - `section_heading`
  - `summary`
  - `body`
  - `evidence`
  - `metadata`
  - `author_info`
  - `disclaimer`
  - `unknown`

즉 `generic_role`은 open-role 결과를 기존 파이프라인과 호환시키기 위한 큰 분류 축이다.

#### `domain_role`

- `generic_role`보다 더 세분화된 legacy subtype이다.
- 예: `report_title`, `thesis_heading`, `supporting_argument`, `report_header_meta`
- 확신이 부족하면 `null`이어도 된다.
- 중요한 점은 `generic_role`과 임의 조합이 허용되지 않는다는 것이다.

즉 `domain_role`은 자유 태그가 아니라, 시스템이 허용한 subtype이어야 한다.

#### `role_confidence`

- 모델이 자신의 판단을 얼마나 확신하는지 나타내는 0~1 실수값이다.
- 이후 시스템이 이 값을 보고 결과를 수용하거나 축소하거나 거절한다.
- 현재 기본 threshold는 아래와 같다.
  - 0.55 미만: fallback
  - 0.55 이상 0.75 미만: generic만 유지, domain 제거
  - 0.75 이상: 그대로 수용 가능

즉 이 값은 단순 참고용이 아니라, 실제 후처리 분기 기준이다.

#### `section_purpose`

- 이 block이 서사적으로 어떤 용도를 가지는지에 대한 설명 또는 힌트다.
- 예: `main_argument`, `thesis`, `supporting_argument`, `investment_summary`
- 다만 이 값은 최종 authoritative output은 아니다.

현재 구현에서는 `_normalize_generation_policy()`가 role 기준으로 이 값을 다시 정리하므로, 모델 원본 값이 그대로 최종값이 되지 않을 수 있다.

#### `used_for_generation`

- 이 block을 downstream generation에 사용할지 말지에 대한 boolean 힌트다.
- 예를 들어 본문/제목은 보통 `true`, evidence table은 보통 `false`가 된다.
- 하지만 이것도 최종적으로는 코드가 role 기준으로 재정규화한다.

즉 모델이 이 값을 직접 내놓긴 하지만, 실제 의미는 `후처리 전에 주는 제안값`에 가깝다.

#### `reason`

- 왜 그렇게 판단했는지 한 문장으로 설명하는 근거다.
- 현재 구현에서는 Qwen source일 때만 `semantic_reason`으로 block에 남는다.
- 리뷰어가 semantic trace를 읽을 때 가장 직관적으로 보는 필드다.

좋은 `reason`은 보통 아래를 함께 묶는다.

- 텍스트 내용
- 위치 정보
- 주변 heading/evidence와의 관계
- OCR label 힌트

#### `needs_review`

- 모델 스스로도 결과를 사람이 확인해야 한다고 보는지 나타내는 boolean이다.
- 불확실하거나 domain role이 애매하면 `true`로 두는 것이 의도된 사용법이다.
- 그리고 이 값은 모델만 정하는 것이 아니라, 후처리에서도 강제로 `true`로 바뀔 수 있다.

예를 들어 generic/domain pair가 맞지 않으면 parser가 domain role을 지우고 `needs_review=true`로 바꾼다.

### 9.2 이 필드들을 추론할 때 어떤 프롬프트를 쓰는가

실제 프롬프트는 `allm/semantic_backends/qwen_transformers.py`의 `_build_messages()`에서 만든다. 구조는 `system message 1개 + user message 1개`다.

#### system prompt의 역할

system prompt는 Qwen에게 아래 원칙을 강하게 준다.

- OCR block의 semantic role을 추론하라.
- 출력은 JSON만 하라.
- 첫 글자는 `{`, 마지막 글자는 `}`여야 한다.
- thinking, 설명문, markdown fence를 출력하지 마라.
- 먼저 open generated role을 추론하라.
- 그 다음 `generic_role`, `domain_role`은 compatibility hint로만 주어라.
- OCR label은 low-level hint일 뿐 최종 semantic answer가 아니다.
- 불확실하면 보수적으로 답하고 `needs_review=true`를 써라.

즉 system prompt의 핵심은 아래 두 문장으로 요약할 수 있다.

- `자유로운 semantic role 이름은 생성해도 된다.`
- `하지만 legacy role은 허용된 enum 안에서만 써야 한다.`

#### user prompt의 구조

user prompt는 짧은 지시문 뒤에 JSON payload를 그대로 붙이는 형태다.

상단 지시문은 대략 아래 의미를 가진다.

- target OCR block의 semantic function을 추론하라.
- 주변 context, OCR label, geometry, structural relation을 함께 써라.
- JSON object 하나만 반환하라.

그 뒤에 실제 payload가 붙는다. payload의 큰 구조는 아래와 같다.

```json
{
  "task": {
    "prompt_version": "...",
    "document_family": "...",
    "page_archetype": "...",
    "page_quality_score": 0.0
  },
  "allowed_roles": {
    "generic_role": [...],
    "domain_role": [...]
  },
  "target_block": { ... },
  "page_context": { ... },
  "local_neighbors": [ ... ],
  "structural_relations": { ... },
  "response_schema": { ... },
  "instructions": [ ... ]
}
```

#### prompt 안에서 모델이 실제로 보는 정보

Qwen은 프롬프트 안에서 최소한 아래를 함께 본다.

- `task`
  - prompt version
  - document family
  - page archetype
  - page quality

- `allowed_roles`
  - 현재 시스템이 허용하는 generic/domain role 목록

- `target_block`
  - text
  - canonical label
  - raw label
  - bbox_norm
  - reading order
  - OCR labels
  - zone tags
  - page 내 상대적 위치

- `page_context`
  - column 수
  - layout pattern
  - text/table/visual 비율
  - meta/sidebar 비율

- `local_neighbors`
  - 이전/다음 text
  - nearest heading
  - nearest evidence

- `structural_relations`
  - heading/evidence와의 거리
  - 같은 column 여부
  - overlap ratio
  - 상대 순서

즉 프롬프트는 그냥 `이 문장을 분류해라`가 아니라, `이 block이 페이지 구조 안에서 어떤 기능을 하는지 추론해라`에 가깝다.

#### prompt 안의 response schema 지시

프롬프트에는 모델이 따라야 할 응답 schema도 명시된다. 각 필드에 대해 아래 같은 형태의 설명이 들어간다.

- `generated_role_name`: 짧은 snake_case 역할 이름
- `generated_role_description`: 한 문장 설명
- `generated_parent_role_name`: 상위 역할 이름 또는 null
- `generated_role_level`: `document | section | block`
- `generic_role`: 허용된 generic role
- `domain_role`: 허용된 domain role 또는 null
- `role_confidence`: 0~1 float
- `section_purpose`: string 또는 null
- `used_for_generation`: boolean 또는 null
- `reason`: 한 문장
- `needs_review`: boolean

즉 모델은 단순 자연어 지시만 받는 것이 아니라, `정확히 어떤 JSON key를 어떻게 채워야 하는지`까지 프롬프트에서 본다.

#### prompt 안의 추가 instruction 목록

user payload 안에는 instruction list도 들어간다. 의미를 정리하면 아래와 같다.

- JSON 밖의 설명을 하지 마라.
- thinking process를 출력하지 마라.
- markdown code fence를 출력하지 마라.
- JSON object로 바로 시작하라.
- OCR label을 그대로 복사하지 말고 semantic role 이름을 생성하라.
- OCR label, raw text, layout, structural relation을 함께 사용하라.
- `generic_role`, `domain_role`은 compatibility field일 뿐이다.
- domain role이 애매하면 generic만 두고 domain은 null로 하라.
- 전체 role이 애매하면 보수적인 generated role을 쓰고 `needs_review=true`로 하라.

이 instruction 목록 때문에 현재 Qwen prompt는 단순 role classifier보다는 `structured semantic reasoning + constrained JSON emitter`로 동작하도록 설계되어 있다.

### 9.3 이 설계가 의미하는 것

결국 현재 Qwen prompt 설계는 아래 두 축을 동시에 만족시키려는 구조다.

- open-role 추론
  - `generated_role_name`, `generated_role_description`, `generated_parent_role_name`으로 더 자연스러운 semantic 설명을 만든다.

- legacy compatibility
  - `generic_role`, `domain_role`, `section_purpose`, `used_for_generation`으로 기존 pipeline과 연결한다.

그래서 Qwen은 완전히 자유로운 분류기도 아니고, 완전히 닫힌 enum classifier도 아니다. 현재 구조는 `자유 생성형 semantic 설명을 하되, 시스템이 이해 가능한 legacy 축으로도 같이 정렬해 주는 프롬프트`라고 이해하는 것이 가장 정확하다.

---

## 10. 모델 응답은 어떤 기준으로 수용/거절되는가

Qwen이 응답했다고 바로 적용되지는 않는다. `resolve_block_decision()`이 여러 관문을 통과시킨 뒤에만 적용한다.

### 10.1 1차 관문: 페이지 품질

`payload.page_quality_score < quality_threshold`이면 Qwen 판단을 받아들이지 않는다.

현재 기본값:

- `quality_threshold = 0.55`

이 경우:

- `qwen` 모드에서는 `low_page_quality` fallback trace를 남긴다.
- `shadow` 모드에서는 아예 trace 없이 fallback decision으로 간다.

즉, OCR 품질이 낮으면 semantic owner를 모델에게 넘기지 않는 구조다.

### 10.2 2차 관문: backend 호출 성공 여부

모델 호출이 예외를 내면 `backend_error`로 fallback 된다.

이 경우:

- `qwen` 모드에서는 fallback reason이 기록된다.
- `shadow` 모드에서는 trace만 남기고 final assignment는 rule 유지다.

### 10.3 3차 관문: JSON 파싱

응답이 JSON 형식이 아니면 `invalid_json`이 된다.

현재 parser는 아래를 정리해서 읽는다.

- markdown fence 제거
- `<think>...</think>` 제거
- JSON object만 추출

그래도 실패하면 fallback 된다.

### 10.4 4차 관문: schema 검증

Pydantic 모델 검증을 통과해야 한다.

검증 항목 예:

- `generic_role`이 허용 enum인지
- `domain_role`이 허용 enum인지
- `role_confidence`가 0~1 범위인지
- `reason`이 비어 있지 않은지
- generated role name이 snake_case 규칙을 지키는지
- generated role level이 `document|section|block` 중 하나인지

실패하면 `schema_mismatch` fallback이 된다.

### 10.5 5차 관문: generic/domain pair 정합성

응답이 schema는 맞아도, `generic_role`과 `domain_role` 조합이 허용되지 않으면 domain role을 제거하고 `needs_review=true`로 바꾼다.

예를 들어 현재 허용 관계는 대략 아래처럼 고정돼 있다.

- `main_title -> report_title`
- `section_heading -> key_data_box | consensus_box | price_chart_block | thesis_heading`
- `body -> supporting_argument`
- `metadata -> report_header_meta | research_center_meta`

즉, domain role은 generic role 위에 있는 자유로운 추가 태그가 아니라, `허용된 subtype`처럼 다뤄진다.

### 10.6 6차 관문: confidence threshold

confidence는 두 단계 threshold를 가진다.

현재 기본값:

- `confidence_generic_only_threshold = 0.55`
- `confidence_accept_threshold = 0.75`

동작:

- `role_confidence < 0.55`
  - decision 전체를 버리고 fallback 사용
  - fallback reason은 `low_role_confidence`

- `0.55 <= role_confidence < 0.75`
  - generic role은 유지
  - domain role은 제거
  - `needs_review=true`

- `role_confidence >= 0.75`
  - decision을 그대로 수용

즉, 현재 구현은 confidence가 애매하면 `완전 거절` 또는 `generic only 축소 수용`을 한다.

---

## 11. fallback decision은 무엇을 쓰는가

모델이 실패하거나 거절될 때는 무조건 빈 값으로 끝나지 않는다. semantic은 block마다 fallback decision을 하나 준비해 둔다.

우선순위:

1. safety rule decision
2. 없으면 `unknown` decision

`unknown` decision 기본값:

- `generic_role = unknown`
- `domain_role = null`
- `role_confidence = 0.0`
- `used_for_generation = false`

즉, semantic은 실패해도 `판단 불가`라는 명시적 상태를 남길 수 있다.

---

## 12. 실제로 block에 값을 반영할 때 무슨 일이 일어나는가

decision이 수용되면 `_apply_decision_to_block()`가 실제 block 필드를 채운다. 여기서 중요한 일이 몇 가지 더 일어난다.

### 12.1 generated role 보강

`enrich_generated_role()`가 먼저 호출된다.

이 함수는 아래를 보강한다.

- generated role name이 없으면 default name 생성
- generated role description이 없으면 generic/domain role 기반 기본 설명 채움
- parent role이 없으면 generic role 기반 기본 parent 채움
- generated role level이 없으면 `block` 기본값 부여

즉, 모델이 일부 필드를 비워도 downstream에서 쓸 수 있도록 semantic schema가 보수적으로 메꿔준다.

### 12.2 generation 정책 재정규화

그 다음 `_normalize_generation_policy()`가 `used_for_generation`과 `section_purpose`를 role 기준으로 다시 계산한다.

대표 규칙:

- `report_title` -> `used_for_generation=true`, `main_argument`
- `thesis_heading` -> `true`, `thesis`
- `supporting_argument` -> `true`, `supporting_argument`
- `investment_opinion_box` -> `true`, `investment_summary`
- `key_data_box / consensus_box / price_chart_block`
  - heading이면 `true`
  - evidence면 `false`
  - purpose는 `evidence_panel`
- `financial_table_block / unsupported_evidence` -> `false`, `evidence_panel`
- `disclaimer_block` -> `true`, `disclaimer`
- `metadata` generic -> 보통 `false`
- `body` generic -> `true`, `supporting_argument`
- `evidence` generic -> `false`

즉, 모델이 `used_for_generation`을 어떤 값으로 줬는지보다, 최종적으로는 코드가 role 의미에 맞게 다시 정리한 값이 더 중요하다.

### 12.3 semantic_source는 어떻게 기록되는가

`semantic_source`는 최종 적용된 판단의 출처를 기록한다.

대표 값:

- `rule`
- `qwen`
- `safety_rule`
- `unknown_fallback`

그리고 `semantic_reason`은 현재 `source == qwen`일 때만 저장된다. 즉 rule의 reason은 내부 decision에는 있어도 block 필드에는 남기지 않는다.

---

## 13. safety rule이 항상 켜져 있는 것은 아니다

이 부분은 현재 구현에서 꽤 중요하다.

`_safety_rules_enabled()`는 아래처럼 동작한다.

- `rule` 모드면 항상 켠다.
- 그 외 모드에서는 `model_name`에 `qwen` 문자열이 들어가면 끈다.

즉 기본 설정인 아래 조합에서는:

- `mode = qwen`
- `model_name = Qwen/Qwen3.5-9B`

현재 코드상 safety rule이 비활성화된다.

의미:

- Qwen 실패 시 fallback이 safety rule이 아니라 `unknown`이 될 가능성이 높다.
- `qwen` 모드는 생각보다 더 강한 Qwen-first 구조다.

이 부분은 문서를 읽는 사람이 놓치기 쉬운 구현 디테일이다.

---

## 14. semantic trace는 무엇을 남기는가

`qwen` 또는 `shadow` 모드에서는 trace가 남을 수 있다. trace entry 하나에는 대략 아래가 들어간다.

- 입력 payload 전체
- raw model response
- parsed_decision
- applied_decision
- fallback_reason
- latency_ms

즉 trace는 단순 로그가 아니라, 아래를 모두 비교할 수 있는 디버그 artifact다.

- 모델이 뭘 봤는지
- 모델이 뭐라고 답했는지
- parser가 어떻게 해석했는지
- 최종적으로 시스템이 무엇을 적용했는지

실제 `financial1-00` trace를 보면, 예를 들어 상단 좌측의 `MIRAE ASSET` block에 대해 모델은 아래처럼 판단했다.

- generated role: `firm_branding_header`
- generic role: `metadata`
- domain role: `report_header_meta`
- reason: top-left meta zone의 금융기관 branding header

이 예시가 보여주는 것은, 현재 Qwen semantic이 `텍스트 내용 + 위치 zone + 주변 문맥`을 함께 사용한다는 점이다.

---

## 15. section은 두 종류가 있다: `section_id`와 `section_order`

이 부분은 semantic을 읽을 때 가장 많이 헷갈리는 지점이다. 현재 구현에는 section과 관련된 개념이 2개 있다.

- block-level `section_id`
- template-level `section_order`

둘은 생성 시점도 다르고 목적도 다르다.

---

## 16. `section_id`는 어떻게 결정되는가

`section_id`는 semantic 후처리의 일부로 `_assign_section_ids()`에서 결정된다. 즉 이것은 semantic 계층의 산물이다.

규칙은 매우 단순하다.

### 16.1 시작점

page의 block을 reading order 순으로 순회한다.

### 16.2 새 section 시작 조건

아래 role을 만나면 새 `section_id`를 발급한다.

- `domain_role == report_title`
- `domain_role == thesis_heading`

새 id 형식:

- `sec-001`
- `sec-002`
- `sec-003`

### 16.3 같은 section에 붙는 block

아래 role은 현재 살아 있는 section에 붙는다.

- `domain_role == supporting_argument`

즉 본문 supporting argument는 가장 최근의 `report_title` 또는 `thesis_heading` 아래로 연결된다.

### 16.4 orphan supporting block 처리

`supporting_argument`인데 앞에서 section 시작점이 한 번도 안 나왔다면:

- `section_id`는 부여되지 않는다.
- 대신 `semantic_needs_review = true`가 된다.

즉, `section_id`는 매우 semantic role 중심의 간단한 연결 구조다.

### 16.5 중요한 특징

`section_id`는 아래 block에는 보통 붙지 않는다.

- metadata
- disclaimer
- evidence
- summary panel 다수

즉 `section_id`는 문서 전체의 narrative spine을 표현하려는 값에 가깝다.

---

## 17. `section_order`는 어떻게 결정되는가

`section_order`는 semantic 함수에서 만들지 않는다. 이것은 template 조립 단계의 `build_section_order()`가 만든다.

즉 `section_order`는 semantic 후처리라기보다, `semantic 결과를 참고하는 template-level narrative ordering`이다.

### 17.1 먼저 anchor page를 고른다

`section_order`는 모든 페이지에서 만들지 않는다. 아래 archetype의 page만 anchor page가 된다.

- `cover_summary`
- `body_narrative`

즉 evidence page나 compliance page는 section ordering의 주 대상이 아니다.

### 17.2 narrative block만 고른다

anchor page에서 아래 조건을 만족하는 block만 section 후보가 된다.

포함 기준:

- text가 있어야 한다.
- canonical label이 text-like이거나
- semantic generic role이 `main_title`, `section_heading`, `body`여야 한다.

제외 기준:

- generic role이 `metadata`, `author_info`, `disclaimer`, `evidence`이면 제외
- raw label이 `header`, `foot`, `fnote`, `anno`, `list`이면 제외

즉 `section_order`는 semantic 결과를 참고하지만, purely semantic role만으로 만들지는 않는다. canonical label과 raw label도 같이 쓴다.

### 17.3 어떤 block을 새 section title로 보는가

`build_section_order()`는 block가 title-like인지 heuristic하게 판단한다.

새 section 시작 조건:

- `domain_role`이 `report_title` 또는 `thesis_heading`
- 또는 canonical label이 `document_title`, `section_heading`, `subheading`
- 또는 text 길이가 80자 이하이고 이전 block과의 세로 간격이 충분히 크면

즉, semantic role이 명확하지 않아도 heading처럼 보이면 새 section을 만들 수 있다.

### 17.4 title이 아니면 어떻게 처리하는가

title-like가 아니면 현재 section의 `block_ids`에 이어 붙인다.

만약 아직 section이 하나도 없는데 body block이 먼저 나오면:

- 그 body block 첫 줄을 title처럼 써서 새 section을 만든다.
- 이후 이어지는 block을 같은 section에 넣는다.

즉 `section_order`는 `사람이 읽는 문단 흐름`을 만들기 위한 heuristic grouping이다.

### 17.5 `section_order`의 section id 형식

여기서 생성되는 section id는 semantic의 `sec-001`과 다르다.

형식:

- `section-001`
- `section-002`
- `section-003`

즉 아래 두 값은 서로 다른 계층이다.

- block field `section_id = sec-002`
- template section `section_id = section-003`

이 둘은 이름이 비슷하지만 직접 매핑되는 같은 값이 아니다.

---

## 18. 왜 `section_id`와 `section_order`가 둘 다 필요한가

현재 구조에서 둘은 역할이 다르다.

### 18.1 `section_id`

- semantic 후처리 결과
- role 중심
- report title / thesis heading / supporting argument 연결용
- block-level attribution

### 18.2 `section_order`

- template 조립 결과
- narrative reading order 중심
- 사람이 읽을 section list 생성용
- template-level summary

즉 `section_id`는 `semantic relation`, `section_order`는 `narrative presentation order`라고 이해하는 것이 가장 정확하다.

---

## 19. 실제 artifact에서 어떻게 보이는가

실제 `financial1-00` artifact를 보면 아래처럼 나타난다.

- block들에는 `section_id = sec-001`, `sec-002` 등이 붙는다.
- template의 `section_order`에는 `section-001`, `section-002`, `section-003`가 따로 기록된다.

이 예시는 아래를 잘 보여준다.

- semantic이 block-level section spine을 만든다.
- template는 그 위에 사람 친화적인 서술 순서를 다시 만든다.
- 두 값은 목적이 다르기 때문에 개수가 완전히 같을 필요도 없다.

특히 `financial1-00`의 `section_order`에는 사이드바의 `Market Insight`도 하나의 section으로 잡혀 있다. 즉 template section은 꼭 `supporting_argument` 같은 semantic narrative spine과 1:1 대응하지 않는다.

## 20. semantic summary는 무엇을 집계하는가

semantic 실행이 끝나면 `SemanticRunSummary`를 만든다. 대표 필드는 아래와 같다.

- `mode`
- `backend`
- `model_name`
- `prompt_version`
- `page_count`
- `block_count`
- `attempted_count`
- `accepted_count`
- `fallback_count`
- `needs_review_count`
- `avg_role_confidence`
- `fallback_reasons`
- `applied_source_counts`
- `shadow_disagreement_count`

이 값은 `parser_diagnostics.json` 안에 들어가며, 한 번의 semantic run이 얼마나 안정적으로 동작했는지 보는 요약판이다.

### 20.1 `attempted_count` 해석 시 주의할 점

이 값은 모든 block 수와 항상 같지 않다.

예를 들어 아래는 attempted로 집계되지 않을 수 있다.

- low page quality라서 모델 호출을 건너뛴 경우
- Qwen 대상 block이 아닌 경우

즉 `attempted_count`는 `실제로 모델 응답을 시도했거나, 호출/파싱 실패가 기록된 횟수`에 가깝다.

## 21. 현재 semantic 설계의 핵심 성격

현재 구현을 요약하면 semantic은 아래 성격을 가진다.

### 21.1 deterministic rule과 generative model의 혼합 구조

- rule은 금융 리포트에 강하다.
- Qwen은 더 유연한 block 의미 추론을 한다.
- 최종 결과는 schema, confidence, fallback 규칙으로 다시 정제된다.

### 21.2 block 중심이지만 page 구조를 강하게 활용한다

semantic 단위는 block이지만, 실제 판단은 아래를 같이 본다.

- page archetype
- zone
- 주변 heading/evidence
- column 관계
- 위치와 읽기 순서

즉 사실상 `page-structured block classification`이다.

### 21.3 section도 전적으로 모델에게 맡기지 않는다

- block semantic role은 모델/규칙이 만든다.
- `section_id`는 코드가 deterministic하게 부여한다.
- `section_order`는 template 단계가 heuristic하게 다시 만든다.

즉 section은 `모델이 다 정한다`가 아니라, semantic 결과를 바탕으로 시스템이 다시 조립하는 구조다.

---

## 22. 결론

현재 `allm`의 semantic 단계는 단순히 block에 라벨 하나 붙이는 단계가 아니다. 실제로는 아래를 모두 수행한다.

- block semantic function 추론
- generated role 생성 및 보강
- confidence 기반 수용/거절
- fallback 및 review flag 관리
- generation 사용 여부 정규화
- block-level section spine 부여
- template-level section ordering을 위한 기반 제공

가장 중요한 포인트는 아래 두 가지다.

- semantic의 1차 판단 주체는 `rule` 또는 `Qwen`이지만, 최종 결과는 코드가 정규화한다.
- `section_id`와 `section_order`는 같은 개념이 아니며, 전자는 semantic spine, 후자는 template narrative ordering이다.

즉 현재 프로젝트의 semantic 계층은 `OCR block meaning inference + deterministic narrative structuring`을 함께 담당하는 중간 해석 레이어라고 보는 것이 가장 정확하다.

## 23. `financial1-00` artifact 기준 케이스 스터디

위 설명이 전체 규칙이라면, 이 절은 실제 artifact를 보고 `한 block이 어떤 단서를 바탕으로 어떤 semantic 결과로 이어지는지`를 사례형으로 풀어 쓴 부분이다.

이번 케이스 스터디는 아래 artifact를 기준으로 읽는다.

- `artifacts/financial1-00/01_reference/reference_template.json`
- `artifacts/financial1-00/01_reference/semantic_trace.json`

이 샘플의 전역 조건은 다음과 같다.

- `document_family = article`
- `page_archetype = body_narrative`
- `quality_score = 0.7733`
- semantic 실행 모드: `qwen`

즉, 이 샘플은 전형적인 `financial_report`로 판정되지는 않았지만, Qwen이 block 단위로 더 세밀한 역할을 추론하는 사례로 보기 좋다.

### 23.1 사례 A: 좌측 상단 `MIRAE ASSET` 블록은 왜 metadata가 되었는가

대상 block:

- `block_id = dolphin-financial1-00-0`
- text: `MIRAE ASSET / 미래에셋증권`
- canonical label: `paragraph`

이 block에서 중요한 입력 단서:

- 페이지 좌측 상단에 있다.
- `zone_tags`에 `left_sidebar_zone`, `top_meta_zone`가 같이 붙는다.
- Paddle 쪽 raw label은 `paragraph_title`이었다.
- 가장 가까운 큰 heading은 메인 제목 block `dolphin-financial1-00-6`이다.
- text 자체가 회사명/브랜드명 형태다.

Qwen raw response의 핵심:

- `generated_role_name = firm_branding_header`
- `generic_role = metadata`
- `domain_role = report_header_meta`
- confidence `0.95`

이 판단이 나온 이유:

- 텍스트가 기사 본문이 아니라 기관명/브랜드명처럼 보인다.
- 페이지 위치가 좌상단 메타 영역이다.
- 문서 헤더를 구성하는 작은 브랜딩 블록의 전형적 형태다.

최종 artifact 반영 결과:

- `generic_role = metadata`
- `domain_role = report_header_meta`
- `generated_role_name = firm_branding_header`
- `used_for_generation = true`
- `section_id`는 없음

여기서 중요한 포인트:

- canonical label은 `paragraph`였지만 semantic은 metadata로 뒤집혔다.
- 즉 semantic은 OCR label보다 `위치 + 텍스트 기능`을 더 강하게 본다.
- metadata 계열 block이라 `section_id`는 부여되지 않는다.

### 23.2 사례 B: `Market Insight`는 왜 section_order에는 들어가지만 `section_id`는 없는가

대상 block:

- `block_id = paddle-financial1-00-1`
- text: `Market Insight`
- canonical label: `paragraph_or_meta`
- flag: `secondary_only`

이 block에서 중요한 입력 단서:

- 좌측 사이드바에 독립적으로 존재한다.
- 텍스트 길이가 짧고 heading처럼 보인다.
- 주변 문맥상 sidebar content를 소개하는 제목 역할이다.
- trace의 raw response에서는 `generic_role = section_heading`, `domain_role = report_header_meta`로 제안됐다.

하지만 최종 적용 결과는 다음과 같다.

- `generic_role = section_heading`
- `domain_role = null`
- `semantic_needs_review = true`
- `generated_role_name = sidebar_section_header`

왜 domain role이 사라졌는가:

- 현재 허용 관계에서 `section_heading -> report_header_meta`는 유효한 pair가 아니다.
- 그래서 parser/normalizer가 domain role을 제거하고 `needs_review=true`로 바꿨다.

그럼에도 `section_order`에는 왜 들어가는가:

- `build_section_order()`는 semantic role만 보는 것이 아니라
- 짧은 텍스트
- heading처럼 보이는 형태
- narrative block 조건
- 공간적 간격
- 를 함께 본다.

그래서 최종 template에는 아래가 생성된다.

- `section_order[0].section_id = section-001`
- `title = Market Insight`

반면 `section_id`는 없다.

즉 이 사례는 아래를 동시에 보여준다.

- semantic의 block-level section spine에는 들어가지 않을 수 있다.
- 하지만 template-level narrative ordering에는 section처럼 보일 수 있다.

### 23.3 사례 C: 메인 제목은 어떻게 `report_title`과 `sec-001`이 되는가

대상 block:

- `block_id = dolphin-financial1-00-6`
- canonical label: `document_title`
- text: `AI 데일리 글로벌 마켓 브리핑 ...`

이 block에서 중요한 입력 단서:

- canonical label 자체가 `document_title`이다.
- Paddle raw label도 `doc_title`이다.
- 페이지 상단의 넓은 영역을 차지한다.
- reading order 상 앞부분에 위치한다.
- 주변에 이를 상위 heading으로 볼 만한 더 큰 제목이 없다.

Qwen raw/parsed/applied decision:

- `generated_role_name = report_title`
- `generic_role = main_title`
- `domain_role = report_title`
- confidence `0.95`

최종 artifact 반영 결과:

- `generic_role = main_title`
- `domain_role = report_title`
- `section_purpose = main_argument`
- `used_for_generation = true`
- `section_id = sec-001`

왜 `section_id = sec-001`이 붙는가:

- `_assign_section_ids()`는 `domain_role == report_title`를 만나면 새 section을 시작한다.
- 이 block이 페이지에서 처음 만나는 narrative spine 시작점이므로 `sec-001`이 된다.

그리고 template 관점에서는:

- `section_order`에서 `section-002`의 title이 된다.

즉 이 block 하나가 동시에 두 역할을 한다.

- semantic 관점: 문서의 첫 narrative spine 시작점
- template 관점: 사람이 읽는 주요 section title

### 23.4 사례 D: `Summary` heading은 왜 `thesis_heading`이 되는가

대상 block:

- `block_id = dolphin-financial1-00-8`
- text: `Summary`
- canonical label: `section_heading`
- tag: `paper_abstract`

이 block에서 중요한 입력 단서:

- 메인 제목 뒤에 나온다.
- 텍스트가 짧고 명확한 section heading이다.
- `paper_abstract` tag가 있다.
- 위치상 본문 상단 section opener처럼 보인다.

Qwen raw response의 핵심:

- `generated_role_name = summary_heading`
- `generic_role = section_heading`
- `domain_role = thesis_heading`
- confidence `0.95`

최종 artifact 반영 결과:

- `generic_role = section_heading`
- `domain_role = thesis_heading`
- `section_id = sec-002`
- `section_purpose = thesis`
- `generated_role_level = section`

왜 `sec-002`인가:

- `report_title`가 이미 `sec-001`을 시작했다.
- 그 다음 `thesis_heading`을 만나면 `_assign_section_ids()`가 새 section을 시작한다.
- 그래서 Summary block부터는 `sec-002` 계열 서사가 시작된다.

template 관점에서는:

- `section_order`에서 `section-003`의 title이 된다.
- 그리고 그 아래에 이어지는 여러 body/summary block들이 이 section에 묶인다.

즉 `thesis_heading`은 semantic과 template 양쪽에서 강한 section anchor 역할을 한다.

### 23.5 사례 E: 긴 summary 문단은 어떻게 `supporting_argument`로 연결되는가

대상 block:

- `block_id = dolphin-financial1-00-9`
- 긴 시장 요약 문단
- canonical label: `paragraph`

이 block에서 중요한 입력 단서:

- 길이가 충분히 긴 본문 텍스트다.
- 바로 앞에 `Summary` heading이 있다.
- 내용이 지수 하락과 원인을 설명하는 핵심 문단이다.
- nearest heading이 `dolphin-financial1-00-8`이다.

Qwen raw/parsed/applied decision:

- `generated_role_name = market_summary_paragraph`
- `generic_role = body`
- `domain_role = supporting_argument`
- confidence `0.92`

최종 artifact 반영 결과:

- `generic_role = body`
- `domain_role = supporting_argument`
- `section_id = sec-002`
- `section_purpose = supporting_argument`

왜 `sec-002`로 붙는가:

- 바로 앞에서 `Summary`가 `thesis_heading`으로 `sec-002`를 시작했다.
- `_assign_section_ids()`는 이후 나오는 `supporting_argument`들을 현재 section에 붙인다.

즉 이 block은 `thesis_heading -> supporting_argument` 연결이 실제 artifact에서 어떻게 나타나는지 보여주는 가장 전형적인 예다.

### 23.6 사례 F: 모델이 맞힌 것 같아도 시스템이 일부를 버리는 경우

대상 block:

- `block_id = paddle-financial1-00-10`
- text: `- 미국 증시 주요 지수 일제히 하락: 다우 -1.56%, 나스닥 -1.78%`
- canonical label: `paragraph_or_meta`
- flag: `secondary_only`

Qwen raw response의 핵심:

- `generated_role_name = market_movement_summary`
- `generic_role = summary`
- `domain_role = key_data_box`
- confidence `0.92`

하지만 parsed/applied decision에서는 다음처럼 바뀐다.

- `generic_role = summary`
- `domain_role = null`
- `needs_review = true`

왜 이런 일이 일어나는가:

- 현재 스키마에서 `summary -> key_data_box`는 허용되지 않는다.
- `key_data_box`는 `section_heading` 또는 `evidence` 계열에만 허용된다.
- 그래서 시스템은 모델의 domain role을 버리고 generic role만 유지했다.

최종 artifact 반영 결과:

- `generic_role = summary`
- `domain_role` 없음
- `semantic_needs_review = true`
- `used_for_generation = true`

이 사례가 중요한 이유:

- 모델 응답이 그럴듯해 보여도 시스템은 schema를 우선한다.
- 현재 semantic은 `모델 자유 추론`이 아니라 `허용된 ontology 안에서 정규화된 추론`이다.

### 23.7 사례 G: 후반부 짧은 headline/body는 왜 review가 붙는가

대상 block 예:

- `paddle-financial1-00-12`
- `dolphin-financial1-00-12`
- `dolphin-financial1-00-13`

공통 특징:

- Qwen은 이 block들을 시장 충격, 자금 이탈, 지수 요약 같은 body/supporting 계열로 읽었다.
- raw response 안에는 때때로 `unsupported_evidence` 또는 `key_data_box` 같은 domain role이 제안되기도 한다.
- 하지만 최종 artifact에서는 domain role이 제거되고 `generic_role = body`만 남은 block이 있다.

대표 이유:

- generic/domain pair가 허용되지 않는 경우
- 내용은 요약/지표 리스트인데 canonical label은 paragraph 계열이라 role 해석이 애매한 경우
- semantic spine에는 들어갈 수 있지만 세부 subtype은 확신하기 어려운 경우

그래서 최종 artifact에는 아래처럼 남는다.

- `generic_role = body`
- `section_purpose = supporting_argument`
- `semantic_needs_review = true`

즉 현재 시스템은 애매한 세부 subtype을 억지로 유지하지 않고, generic role만 남긴 채 review 대상으로 돌리는 쪽을 선택한다.

### 23.8 이 케이스 스터디에서 읽어야 할 핵심 포인트

`financial1-00` 사례를 종합하면 현재 semantic은 아래 방식으로 동작한다.

1. OCR label을 그대로 믿지 않는다.
2. 위치, 주변 heading, zone, 문장 길이, 태그를 함께 본다.
3. Qwen은 block 기능을 자유롭게 설명한다.
4. 하지만 시스템은 generic/domain ontology에 맞지 않으면 잘라낸다.
5. `section_id`는 narrative spine만 따라간다.
6. `section_order`는 사람이 읽을 흐름을 다시 묶는다.

즉 실제 artifact를 보면 semantic의 본질은 `모델이 의미를 제안하고, 시스템이 그것을 구조화 가능한 형태로 다듬는 과정`이라는 점이 가장 잘 드러난다.
