# AI 판단 추적 기능 가이드

## 목적

이 문서는 `llm-to-document`에서 AI가 어떤 근거를 찾고, 어떤 판단을 거쳐 최종 문서를 만들었는지 추적할 수 있도록 추가한 기능을 설명한다.

이번 변경의 목표는 다음과 같다.

- 문서 생성 파이프라인이 어떤 source 문서를 실제로 사용했는지 확인
- 검색기가 어떤 후보를 찾았고 왜 특정 블록을 선택했는지 확인
- 최종 문서 생성 직전에 모델이 스스로 정리한 의도/근거/불확실성을 `analysis.json`으로 확인
- 실패 시 어느 단계에서 멈췄는지 `events.jsonl`로 빠르게 추적

## 활성화 방법

판단 추적 기능은 기본적으로 꺼져 있다.

### 1. 환경변수로 켜기

```powershell
$env:LLM2DOC_DEBUG_TRACE = "1"
```

허용되는 참 값은 다음과 같다.

- `1`
- `true`
- `yes`
- `on`

### 2. 함수 인자로 켜기

```python
create_document(query, src_docs, target_doc, debug_trace=True)
create_document_ver2(query, src_docs, target_doc, debug_trace=True)
```

함수 인자를 주면 환경변수보다 우선한다.

## 저장 위치와 디렉터리 구조

판단 추적이 켜져 있으면 실행마다 별도 run 폴더가 생긴다.

경로 형식:

```text
output/<결과폴더>/trace/<run_id>/
```

예시:

```text
output/contextual_ver2/trace/20260415T104500123456Z-ab12cd34/
```

기본 파일 구조는 다음과 같다.

```text
trace/<run_id>/
├─ analysis.json
├─ events.jsonl
├─ llm/
│  ├─ tool_loop_response_1.json
│  ├─ tool_loop_response_2.json
│  ├─ analysis_response.json
│  └─ final_document_response.json
└─ search/
   ├─ semantic.jsonl
   └─ contextual_ver2.jsonl
```

기존의 `debug_write_input.txt`, `debug_write_output.txt`, `debug_write_reason.txt`, `debug_finish_*.png`는 그대로 유지된다. 새 trace 산출물만 옵션으로 추가된다.

## analysis.json 스키마

`analysis.json`은 tool 사용이 끝난 뒤, 최종 문서를 다시 출력하기 직전에 모델이 스스로 정리한 판단 요약이다.

최상위 필드는 고정이다.

```json
{
  "query": "string",
  "user_intent": "string",
  "source_documents": [
    {
      "document_id": "string",
      "used": true,
      "reason": "string"
    }
  ],
  "evidence": [
    {
      "document_id": "string",
      "page_id": 1,
      "block_id": "string",
      "matched_via": "semantic",
      "used_for": "string",
      "why_selected": "string"
    }
  ],
  "writing_strategy": ["string"],
  "uncertainties": ["string"]
}
```

의미는 다음과 같다.

- `query`: 사용자 요청 원문
- `user_intent`: 모델이 해석한 실제 작성 목적
- `source_documents`: 각 source 문서의 사용 여부와 사용 이유
- `evidence`: 최종 문서에 반영된 근거 블록 목록
- `matched_via`: 검색에서 어떤 방식으로 잡혔는지
  - semantic 검색기: `semantic`, `fallback`
  - contextual ver2 검색기: `block`, `window`, `section`
- `writing_strategy`: 어떤 방식으로 타깃 레이아웃에 내용을 배치할지
- `uncertainties`: 모델이 확신하지 못한 부분

모델이 JSON을 완전히 정확하게 내지 못하더라도, 코드에서 필드를 정규화해서 저장한다. 즉 `analysis.json`은 항상 같은 형태를 유지하도록 만들었다.

## events.jsonl 형식

`events.jsonl`은 실행 중 발생한 주요 이벤트를 시간순으로 저장하는 단일 스트림 로그다.

각 줄은 하나의 JSON 객체다.

```json
{
  "ts": "2026-04-15T10:45:01.234567+00:00",
  "run_id": "20260415T104500123456Z-ab12cd34",
  "component": "create_document_ver2",
  "event": "tool_requested",
  "payload": {
    "tool": "search_source_document",
    "call_id": "call_123",
    "arguments": "{\"query\": \"KMW 투자 포인트는?\"}"
  }
}
```

이번 구현에서 필수로 남기는 이벤트는 다음과 같다.

- `run_started`
- `layout_loaded`
- `tool_requested`
- `tool_result`
- `analysis_requested`
- `analysis_saved`
- `final_document_requested`
- `final_document_saved`
- `render_started`
- `render_completed`
- `run_failed`
- `run_completed`

디버깅할 때는 보통 아래 순서대로 보면 된다.

1. `run_failed`가 있는지 확인
2. 마지막 `tool_requested` / `tool_result` 확인
3. `analysis_saved`가 만들어졌는지 확인
4. `final_document_requested` 이후 `final_document_saved`가 나왔는지 확인
5. `render_started` 이후 `render_completed`가 나왔는지 확인

## 검색기 점수 로그 보는 방법

### 1. semantic 검색기

파일:

```text
search/semantic.jsonl
```

대상 코드:

- `llm2doc/tool_search_source_document.py`

각 query마다 다음 정보를 남긴다.

- `query`
- `docs`
- `candidates`
- `selected`

`candidates` 안에는 다음 필드가 있다.

- `record_id`
- `document`
- `page`
- `display_block_id`
- `distance`
- `generic_role`
- `generated_role_name`
- `section_purpose`
- `role_confidence`
- `semantic_needs_review`
- `rerank_score`
- `selected`

해석 기준:

- `distance`: Chroma 원시 거리값
- `rerank_score`: 역할 신뢰도와 review penalty를 반영한 최종 정렬 점수
- `selected=true`: 최종 top-3에 포함된 후보

### 2. contextual ver2 검색기

파일:

```text
search/contextual_ver2.jsonl
```

대상 코드:

- `llm2doc/tool_search_source_document_ver2.py`

각 query마다 다음 정보를 남긴다.

- `retrieved_candidates`
- `grouped_anchors`
- `selected`

`retrieved_candidates`는 개별 record 기준 로그다.

- `record_kind`: `block`, `window`, `section`
- `distance`
- `lexical_score`
- `metadata_score`
- `role_confidence`
- `multi_kind_count`
- `final_score`
- `selected`

`grouped_anchors`는 같은 anchor block으로 묶은 뒤의 로그다.

- `anchor_key`
- `best_record`
- `matched_kinds`
- `best_distance`
- `multi_kind_count`
- `final_score`
- `selected`

해석 기준:

- `record_kind`가 `window`나 `section`이면 주변 문맥을 포함한 레코드다.
- `multi_kind_count`가 크면 같은 anchor가 여러 방식으로 동시에 검색에 걸린 것이다.
- `selected=true`인 anchor가 최종적으로 LLM에 보여준 결과다.

## 어디를 어떻게 바꿨는가

### 1. `llm2doc/debug_trace.py`

새 공통 모듈을 추가했다.

핵심 역할:

- `LLM2DOC_DEBUG_TRACE` 환경변수 해석
- `run_id` 생성
- `trace/<run_id>/` 디렉터리 생성
- JSON/JSONL 저장
- 디버그가 꺼져 있을 때 no-op 동작

핵심 함수/객체:

- `env_flag()`
- `resolve_debug_trace()`
- `DecisionTracer.create()`
- `DecisionTracer.event()`
- `DecisionTracer.save_json()`
- `DecisionTracer.append_jsonl()`

### 2. `llm2doc/create_document.py`

기본 파이프라인을 다음과 같이 바꿨다.

- `debug_trace: bool | None = None` 인자 추가
- 기본 검색기를 `plain`에서 semantic 검색기로 교체
- 실행 시작 시 tracer 생성
- tool loop 응답 raw JSON 저장
- `tool_requested`, `tool_result` 이벤트 저장
- trace가 켜져 있으면 별도 `analysis.json` 생성
- 최종 문서는 별도 final-document 요청 단계에서 다시 받도록 변경
- 렌더링 시작/완료와 전체 run 완료/실패 이벤트 저장

추가 helper:

- `response_to_jsonable()`
- `extract_json_object()`
- `normalize_analysis_payload()`
- `maybe_generate_analysis()`
- `request_final_document()`

### 3. `llm2doc/create_document_ver2.py`

ver2 파이프라인도 base와 같은 방식으로 맞췄다.

- `debug_trace` 인자 추가
- tracer 생성 및 이벤트 저장
- tool loop raw response 저장
- `analysis.json` 생성
- final document 재요청 로직 추가
- `create_document.py`의 공통 helper 재사용

즉, 이전에는 base 쪽에만 있던 “최종 문서 형식 재검증” 개념을 ver2에도 동일하게 적용했다.

### 4. `llm2doc/tool_search_source_document.py`

semantic 검색기에 tracer 인자를 추가했다.

변경 내용:

- 생성자에 `tracer: DecisionTracer | None = None`
- query마다 semantic 후보 전체를 `search/semantic.jsonl`에 저장
- 최종 선택 결과를 이벤트로 저장
- `rerank_score` 계산 함수를 분리해 로그와 실제 정렬이 같은 값을 쓰도록 정리

### 5. `llm2doc/tool_search_source_document_ver2.py`

contextual ver2 검색기에 tracer 인자를 추가했다.

변경 내용:

- 생성자에 `tracer: DecisionTracer | None = None`
- 후보별 `distance`, `lexical_score`, `metadata_score`, `final_score` 저장
- anchor 기준 그룹화 결과 저장
- 최종 선택 결과를 이벤트로 저장

## 기본 검색기를 semantic으로 바꾼 이유와 영향

기본 파이프라인인 `create_document.py`는 원래 `ToolSearchSourceDocumentPlain`을 사용하고 있었다.

이번 변경에서는 `ToolSearchSourceDocument`로 바꿨다. 이유는 다음과 같다.

- 기본 파이프라인과 ver2 모두에서 “왜 그 블록이 선택됐는지”를 같은 수준으로 추적해야 했기 때문
- `plain` 검색기는 단순 텍스트 유사도 결과만 보여 주고 역할 기반 로그가 부족했기 때문
- semantic 검색기는 `generic_role`, `section_purpose`, `role_confidence`를 포함해 더 해석 가능한 로그를 남길 수 있기 때문

영향은 다음과 같다.

- 기존 `plain` 대비 검색 결과가 일부 달라질 수 있다.
- 대신 기본 파이프라인에서도 선택 근거를 더 풍부하게 볼 수 있다.
- trace가 꺼져 있어도 검색기 자체는 semantic 버전으로 동작한다.

## 권장 확인 순서

판단 추적을 켠 뒤 문제가 생기면 아래 순서로 보면 된다.

1. `events.jsonl`의 마지막 이벤트 확인
2. `llm/tool_loop_response_*.json`에서 tool loop 종료 시점 확인
3. `analysis.json`에서 모델이 스스로 정리한 근거 확인
4. `search/*.jsonl`에서 어떤 후보가 선택됐는지 확인
5. `debug_write_output.txt`와 최종 이미지(`debug_finish_*.png`) 비교

이 흐름대로 보면 “검색이 잘못됐는지”, “모델 판단이 잘못됐는지”, “최종 문서 형식이 깨졌는지”를 빠르게 분리할 수 있다.
