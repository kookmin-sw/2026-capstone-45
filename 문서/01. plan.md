# 한국어 증권/기업 리포트 생성 MVP 상세 설계

## 1. 목표

이 프로젝트의 목표는 사용자가 제공한 `레퍼런스 문서`의 형식과 구조를 따라, 별도로 제공한 `생성용 자료`를 기반으로 새로운 문서를 생성하는 것이다.

현재 설계는 다음 MVP 범위를 기준으로 한다.

- 단일 도메인: 증권/기업 리포트
- 언어 범위: 한국어 우선
- 형식 목표: 픽셀 복제가 아니라 구조 유사
- 콘텐츠 범위: 글과 이미지 중심
- 검수 방식: 초안 생성 후 사람 검수

이 문서는 실행 가능한 수준의 상세 설계를 제공한다. 구현자는 이 문서를 기준으로 저장소 구조, 데이터 스키마, 파이프라인, 수용 기준을 그대로 구현할 수 있어야 한다.

## 2. 범위와 비범위

### 2.1 MVP 포함 범위

- 레퍼런스 PDF 또는 이미지 입력
- 생성용 자료로 PDF, 이미지, 텍스트, 마크다운 입력
- 레퍼런스의 섹션 구조, 제목 체계, 본문 흐름, 이미지 배치 규칙 추출
- 생성용 자료에서 텍스트와 이미지 근거 추출
- 구조화된 초안 AST 생성
- HTML 렌더링 및 PDF 출력
- provenance 추적
- 사람 검수 후 final 산출물 승격

### 2.2 MVP 제외 범위

- 표의 구조적 생성과 복원
- 차트 재생성
- 실시간 웹 크롤링
- 편집 UI
- 다국어 품질 보장
- 완전 자동 발행
- 픽셀 단위 레퍼런스 복제

### 2.3 범위 외 기능의 처리 방식

- 레퍼런스에 표가 포함되어 있더라도 MVP는 이를 직접 생성 대상으로 삼지 않는다.
- 생성용 자료에 표가 있어도 안전하게 본문화할 수 있는 경우에만 사용한다.
- 표 중심 페이지는 `unsupported_for_mvp`로 마킹하고 생성 대상에서 제외한다.

## 3. 설계 원칙

- 내용 정확도와 형식 유사도를 분리해서 다룬다.
- 모든 생성 문단은 source 근거를 가져야 한다.
- LLM에는 자유 생성이 아니라 구조화된 출력만 허용한다.
- 레이아웃은 템플릿 규칙으로 재현하고, 픽셀 단위 복제는 하지 않는다.
- 사람이 최종 승인하기 전에는 결과를 final로 취급하지 않는다.
- OCR 결과는 오염될 수 있으므로 후처리와 신뢰도 관리가 필요하다.

## 4. 전체 파이프라인

파이프라인은 아래 9단계로 고정한다.

1. Job 생성 및 입력 수집
2. 레퍼런스 문서 파싱
3. 생성용 자료 정규화
4. 사실 추출 및 콘텐츠 브리프 생성
5. 구조화 초안 AST 생성
6. HTML 렌더링
7. PDF 생성
8. 검수 패키지 생성 및 승인
9. 평가 및 산출물 보관

## 5. 저장소 구조

```text
inputs/
  jobs/
    <job_id>/
      job_manifest.json
      reference/
      sources/

artifacts/
  <job_id>/
    01_reference/
      reference_template.json
      layout_preview/
    02_sources/
      normalized_sources.jsonl
      source_images/
      source_previews/
    03_brief/
      content_brief.json
      unsupported_items.json
    04_draft/
      draft_document.json
      provenance.json
      draft.html
      draft.pdf
    05_review/
      review_checklist.json
      review_decision.json
    06_final/
      final.pdf
      final_metadata.json
```

## 6. 단계별 상세 설계

## 6.1 1단계: Job 생성 및 입력 수집

### 목적

입력 자료를 일관된 구조로 수집하고, 후속 단계가 항상 같은 인터페이스를 사용할 수 있게 한다.

### 입력

- reference 디렉터리의 단일 레퍼런스 문서 1개
- sources 디렉터리의 생성용 자료 N개
- job 메타데이터

### 필수 메타데이터

- `job_id`
- `target_doc_type`
- `language`
- `target_title`
- `intended_audience`
- `must_include_topics`
- `must_exclude_topics`
- `notes`

### 처리

- 입력 파일 확장자를 검증한다.
- reference는 정확히 1개만 허용한다.
- source가 0개면 실패 처리한다.
- 허용되지 않은 포맷은 reject한다.
- 입력 파일의 sha256을 계산해 중복 여부를 기록한다.
- 모든 입력 파일에 `source_id`를 부여한다.

### 출력

- `inputs/jobs/<job_id>/job_manifest.json`
- 정규화된 입력 디렉터리 구조

### 실패 처리

- reference가 2개 이상이면 실패
- source가 비어 있으면 실패
- 손상된 PDF나 이미지면 실패
- 파일 인코딩 문제 발생 시 해당 파일을 `invalid_source`로 기록하고 계속 진행하지 않는다

### 수용 기준

- 모든 job은 manifest를 가진다.
- 모든 source는 source_id를 가진다.
- reference는 1개로 고정된다.

## 6.2 2단계: 레퍼런스 문서 파싱

### 목적

레퍼런스 문서를 사람이 따라야 할 양식이 아니라 기계가 이해할 수 있는 템플릿 구조로 바꾼다.

### 사용 기술

- 기본 parser: `dolphin`
- PDF는 이미지로 래스터라이즈한 뒤 페이지 단위 파싱

### 처리 순서

1. 레퍼런스 문서를 페이지 이미지로 변환
2. 문서 레이아웃 파싱
3. 읽기 순서 추출
4. 제목, 본문, 그림 블록 필터링
5. 템플릿 규칙 생성

### 추출 대상

- 페이지 수
- 페이지 크기
- 여백
- 컬럼 수
- 제목 블록 계층
- 섹션 순서
- 본문 블록 평균 길이
- 이미지 블록 위치와 크기 비율
- 캡션 존재 여부

### 무시 또는 제한 대상

- 표 블록
- 수식 블록
- 코드 블록

이들은 `unsupported_blocks`로 기록만 하고 생성 단계에서는 제외한다.

### 출력 스키마

`reference_template.json` 필드:

- `template_id`
- `source_path`
- `domain`
- `language`
- `page_specs`
- `section_order`
- `style_tokens`
- `image_slots`
- `unsupported_blocks`

### style_tokens 정의

- `title_font_scale`
- `subtitle_font_scale`
- `body_font_scale`
- `line_height_ratio`
- `paragraph_spacing`
- `section_spacing`
- `column_count`
- `body_width_ratio`
- `image_alignment_rules`

### 실패 처리

- parser가 레이아웃을 거의 못 잡으면 `distorted_reference`로 마킹
- 섹션 구조 추출 실패 시 manual review required
- 페이지별 결과 편차가 크면 템플릿 신뢰도를 낮춘다

### 수용 기준

- 레퍼런스에서 section_order를 추출할 수 있어야 한다.
- 이미지 슬롯 정보가 1개 이상 있으면 위치와 크기 비율이 기록되어야 한다.
- 표 중심 페이지는 unsupported로 명시되어야 한다.

## 6.3 3단계: 생성용 자료 정규화

### 목적

서로 다른 입력 포맷을 공통 스키마로 변환해 이후 LLM과 렌더러가 같은 입력 규격을 쓰도록 만든다.

### 입력 유형별 처리

#### 텍스트와 마크다운

- 제목, 문단, 리스트를 추출한다.
- 문단 길이를 기준으로 chunk 분할한다.
- 너무 짧은 노이즈 문단은 제거한다.

#### PDF

- 페이지를 이미지로 변환한다.
- OCR과 레이아웃 파싱을 수행한다.
- paragraph와 figure만 유지한다.
- 표는 직접 사용하지 않고 unsupported로 남긴다.

#### 이미지

- OCR 수행
- 이미지 내 텍스트와 시각 요소를 구분
- 문서 이미지면 텍스트 source로 사용
- 기사 캡처나 차트 이미지면 figure candidate로 사용

### 공통 후처리

- 연속 줄바꿈 정리
- OCR 깨짐 패턴 치환
- 지나치게 짧은 잡음 제거
- 중복 문단 제거
- 이미지 후보 중 해상도 하한 미달 제거
- 같은 사실의 중복 chunk 연결

### 출력 스키마

`normalized_sources.jsonl` 각 레코드 필드:

- `chunk_id`
- `source_id`
- `source_type`
- `page`
- `text`
- `language`
- `tags`
- `confidence`
- `is_noise`
- `is_supported`

`source_images/` 이미지 메타 필드:

- `image_id`
- `source_id`
- `path`
- `page`
- `caption`
- `ocr_text`
- `usable`

### 실패 처리

- OCR 실패 chunk는 `is_supported=false`
- confidence가 낮은 chunk는 brief 단계에서 우선순위를 낮춘다
- 이미지가 너무 작거나 손상되면 unusable 처리

### 수용 기준

- 모든 source는 텍스트 chunk 또는 image candidate 중 하나 이상으로 변환된다.
- unsupported와 noise 여부가 명시된다.

## 6.4 4단계: 사실 추출 및 콘텐츠 브리프 생성

### 목적

source 전체를 LLM에 그대로 던지지 않고, 생성할 문서의 핵심 메시지와 섹션별 근거를 정리한 브리프를 만든다.

### 처리 순서

1. source chunk 점수화
2. 핵심 주제 군집화
3. 상충 숫자와 주장 탐지
4. must_include_topics 반영
5. 섹션 후보 할당
6. 이미지 후보 매칭

### chunk 점수화 기준

- 문서 제목과의 관련성
- 반복 출현 여부
- 수치 포함 여부
- 섹션 키워드 일치도
- OCR confidence
- 출처 신뢰도

### 상충 정보 처리

- 같은 항목에 여러 숫자가 있으면 `conflict_set` 생성
- conflict는 초안 생성 시 보수적으로 생략하거나 review required로 표시

### 출력 스키마

`content_brief.json` 필드:

- `job_id`
- `target_title`
- `document_goal`
- `audience`
- `tone`
- `section_plan`
- `selected_facts`
- `selected_images`
- `conflicts`
- `unsupported_items`

`section_plan` 각 항목 필드:

- `section_id`
- `title_hint`
- `target_length`
- `required_source_refs`
- `optional_source_refs`
- `image_slots`

### 실패 처리

- must_include_topics를 뒷받침하는 source가 없으면 경고 생성
- conflict가 너무 많으면 draft 단계 전에 review required 상태로 둔다

### 수용 기준

- 모든 섹션에는 최소 1개 이상의 required_source_refs가 존재해야 한다.
- must_include_topics가 브리프에 반영되어야 한다.

## 6.5 5단계: 구조화 초안 AST 생성

### 목적

문서를 바로 markdown으로 쓰지 않고 구조화된 문서 AST로 생성해 이후 렌더링과 provenance 추적을 안정화한다.

### 생성 규칙

- 허용 블록 타입만 생성한다.
- 표 생성 금지
- 근거 없는 숫자 금지
- source에 없는 주장 금지
- 이미지 삽입은 selected_images만 사용
- 레퍼런스의 섹션 순서를 따른다

### 허용 블록 타입

- `document_title`
- `section_heading`
- `paragraph`
- `image`
- `caption`
- `page_break_hint`

### DraftDocument 스키마

- `document_id`
- `language`
- `title`
- `blocks`

`blocks` 각 항목 필드:

- `block_id`
- `type`
- `text`
- `image_id`
- `source_refs`
- `layout_role`

### 생성 프롬프트 정책

- source_refs가 없는 문단은 만들지 말 것
- 숫자와 고유명사는 source 문구를 최대한 보존할 것
- unsupported topic은 언급하지 말 것
- 한 섹션의 길이는 brief의 target_length 범위를 벗어나지 말 것

### 실패 처리

- source_refs 없는 블록이 나오면 draft reject
- 금지된 블록 타입이 나오면 draft reject
- 이미지 슬롯보다 이미지가 많으면 우선순위가 낮은 이미지 제거

### 수용 기준

- 모든 paragraph와 image는 source_refs를 가진다.
- section 순서가 brief와 일치한다.
- 표 블록이 존재하지 않는다.

## 6.6 6단계: HTML 렌더링

### 목적

초안 AST와 레퍼런스 템플릿을 이용해 사람이 검수 가능한 HTML을 만든다.

### 렌더링 방식

- AST를 HTML semantic block으로 매핑
- CSS는 template의 style_tokens를 반영
- page_break_hint는 CSS 페이지 분할 힌트로 반영
- 이미지 슬롯은 비율과 정렬 규칙을 우선 적용

### HTML 매핑 규칙

- `document_title` -> `<h1>`
- `section_heading` -> `<h2>` 또는 `<h3>`
- `paragraph` -> `<p>`
- `image` -> `<figure><img>`
- `caption` -> `<figcaption>`

### CSS 규칙

- 폰트는 한국어 가독성이 좋은 기본 서체 사용
- body width는 template의 body_width_ratio 반영
- 섹션 간 간격은 section_spacing 반영
- 이미지는 슬롯 비율에 맞춰 `max-width`와 `float` 또는 block 정렬 적용
- 다단 문서는 CSS column 또는 명시적 grid로 처리

### 출력

- `draft.html`
- 레이아웃 preview가 필요한 경우 추가 preview PNG

### 실패 처리

- 이미지 경로 누락 시 placeholder를 넣지 않고 해당 블록 제거
- HTML 변환 오류 시 block 단위 fallback rendering 사용

### 수용 기준

- HTML이 브라우저에서 정상 렌더링된다.
- 이미지와 캡션이 서로 멀리 떨어지지 않는다.
- 레퍼런스의 큰 구조가 유지된다.

## 6.7 7단계: PDF 생성

### 목적

검수 및 최종 납품에 사용할 PDF를 만든다.

### 처리

- `draft.html`을 headless 브라우저 또는 동등한 HTML-to-PDF 렌더러로 변환
- 페이지 여백과 페이지 크기는 template의 page_specs를 따른다
- PDF 메타데이터에 job_id와 generated_at 기록

### 출력

- `draft.pdf`

### 실패 처리

- PDF 렌더링 실패 시 HTML까지는 유지하고 job 상태를 `render_failed`로 표시

### 수용 기준

- PDF가 열려야 한다.
- 본문과 이미지가 페이지 밖으로 잘리지 않아야 한다.

## 6.8 8단계: 검수 패키지 생성 및 승인

### 목적

초안의 사실성과 가독성을 사람이 빠르게 판단할 수 있도록 검수 패키지를 만든다.

### 검수 패키지 구성

- `draft.pdf`
- `draft.html`
- `content_brief.json`
- `provenance.json`
- `review_checklist.json`

### review_checklist 항목

- 제목이 target_title과 부합하는가
- must_include_topics가 모두 반영되었는가
- must_exclude_topics가 제외되었는가
- 숫자와 주장이 근거 source에 존재하는가
- 이미지가 문맥과 맞는가
- 섹션 흐름이 자연스러운가
- 레퍼런스 구조 유사성이 충분한가

### review_decision 스키마

- `job_id`
- `status`
- `reviewer`
- `checked_at`
- `issues`
- `notes`

허용 status:

- `approved`
- `rejected`
- `needs_regeneration`

### 승인 규칙

- `approved`일 때만 final.pdf 생성
- `rejected`면 job 종료
- `needs_regeneration`이면 4단계 또는 5단계부터 재실행

### 수용 기준

- 승인 전에는 final 산출물이 없어야 한다.
- 검수 결과는 반드시 JSON으로 저장되어야 한다.

## 6.9 9단계: 평가 및 산출물 보관

### 목적

품질을 정량적으로 추적하고 회귀를 막는다.

### MVP 평가 지표

- section order match rate
- must_include topic coverage
- unsupported leakage rate
- source-backed paragraph ratio
- render success rate
- reviewer approval rate

### 평가 데이터셋 구성

- 표 의존도가 낮은 한국어 증권/기업 리포트 샘플 10건
- 각 샘플에 reference 1개, source 3~10개
- 최소 3개는 이미지 포함 source

### 보관 정책

- final 산출물과 provenance는 함께 저장
- rejected 결과도 원인 분석용으로 유지

## 7. CLI 인터페이스

다음 CLI를 공개 인터페이스로 정의한다.

```bash
python -m allm.prepare_reference --job-id <id> --reference <path>
python -m allm.prepare_sources --job-id <id> --sources <dir>
python -m allm.build_brief --job-id <id>
python -m allm.generate_draft --job-id <id>
python -m allm.render_draft --job-id <id>
python -m allm.finalize_review --job-id <id> --decision <path>
```

### CLI 계약

- 모든 명령은 job_id를 입력받는다.
- 모든 명령은 artifacts 하위에만 결과를 쓴다.
- 실패 시 0이 아닌 exit code를 반환한다.
- 각 단계는 선행 아티팩트가 없으면 실패한다.

## 8. 구현 순서

### Phase 1

- 입력 디렉터리 규격과 manifest 구현
- reference parser wrapper 구현
- source normalizer 구현

### Phase 2

- draft AST generator 구현

### Phase 3

- PDF renderer 구현

### Phase 4

- 대표 샘플셋 실험

## 9. 테스트 시나리오

### 기능 테스트

- reference 1개와 source 3개로 전체 파이프라인 실행
- source 이미지가 없는 경우 텍스트만으로 draft 생성
- OCR 노이즈가 있는 PDF source 처리
- must_include topic 누락 시 경고 생성
- unsupported topic이 draft에 새어 나오지 않는지 검증
- review가 approved가 아닐 때 final.pdf가 생성되지 않는지 검증

### 회귀 테스트

- 동일 입력에서 AST 스키마가 유지되는지 검증
- renderer가 section 순서를 바꾸지 않는지 검증
- provenance가 모든 paragraph에 붙는지 검증

## 10. 주요 리스크와 대응

### OCR 품질 저하

- confidence 기반 필터링
- 짧은 잡음 제거
- source별 신뢰도 차등 반영

### 표 의존 리포트

- MVP에서는 제외
- unsupported_for_mvp로 명시

### hallucination

- source_refs 없는 블록 금지
- draft reject 규칙 적용

### 이미지 과다 또는 부족

- image_slots 기준으로 삽입 수 제한
- 이미지가 없으면 텍스트 전용으로 렌더

## 11. 이후 확장 포인트

MVP 이후 아래 순서로 확장한다.

1. 표 블록 지원
2. 차트 재생성 또는 chart-to-image 파이프라인
3. 한영 병행 지원
4. 편집 UI
5. 자동 검수 보조

표 지원은 이 MVP가 안정화된 이후 별도 Phase로 추가한다. 그 전까지는 `task.md`의 전체 비전 중 `글/이미지 중심 생성`만 구현 대상으로 본다.
