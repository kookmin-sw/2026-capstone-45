백업4-2의 표 구현

- OCR이 label == "table" 블록을 만듭니다.
표 내용은 ```<table>...</table> HTML```로 보존됩니다.

- table_retrieval.py가 target 문서의 표를 찾습니다.

- 표의 첫 행을 헤더, 첫 열을 row label로 가정합니다.

- row label들로 source 문서를 검색합니다.
관련 source page를 LLM 입력에 미리 붙입니다.

- LLM이 표를 채웁니다.

- number_guard.py가 최종 표 안 숫자를 검사해서 source 근거에 없는 숫자는 -로 바꿉니다.

----

쉽게 말하면 지금 표 처리 구조는 이렇습니다.

**핵심 요약**

표를 직접 “계산해서 채우는” 시스템은 아니고,  
**OCR이 뽑아준 `<table>` HTML을 유지한 다음, LLM이 source 문서를 보고 표를 채우게 돕는 구조**입니다.

흐름은 이렇게 갑니다.

1. **OCR이 표를 찾음**
   PaddleOCR-VL이 문서 이미지를 보고 `label == "table"`인 블록을 만듭니다. 표 내용은 보통 HTML `<table>...</table>` 형태로 들어옵니다.

2. **표 HTML을 그대로 보존**
   일반 문단은 `<p>`로 감싸지만, 표는 escape하지 않고 그대로 둡니다.

   ```html
   <div id="target-page-1-block-3">
     <table>...</table>
   </div>
   ```

3. **target 문서에서 표 구조를 분석**
   `table_retrieval.py`가 target 문서 안의 표를 찾고,
   첫 행은 헤더, 첫 열은 row 이름이라고 가정합니다.

   예를 들면:

   ```text
          2023   2024
   매출액    -      -
   영업이익  -      -
   순이익    -      -
   ```

   여기서 row label은 `매출액`, `영업이익`, `순이익`입니다.

4. **row label로 source 문서 검색**
   target 표에 `매출액`, `영업이익`, `순이익`이 있으면, 이 단어들을 검색어로 써서 source 문서에서 관련 페이지를 미리 찾습니다.

5. **찾은 source 페이지를 LLM에게 추가로 줌**
   LLM에게 이렇게 알려주는 셈입니다.

   ```text
   이 target 표에는 매출액/영업이익/순이익 row가 있다.
   source 문서 중 이 페이지들이 관련 있어 보인다.
   찾은 row만 채우고, 못 찾은 row는 '-'로 둬라.
   표 구조는 바꾸지 마라.
   ```

6. **LLM이 최종 HTML 표를 생성**
   LLM이 target 표 모양을 유지하면서 source 내용으로 표를 채웁니다.

7. **마지막에 숫자 검증**
   생성된 표 안 숫자가 source 근거에 없으면 해당 셀을 `-`로 바꿉니다.  
   즉 LLM이 근거 없는 숫자를 만들어내는 걸 막으려는 후처리입니다.

**한 문장으로 정리하면**

현재 구현은 **표를 HTML로 보존하고, target 표의 row 이름으로 source 문서를 미리 찾아준 뒤, LLM이 표를 채우게 하고, 마지막에 근거 없는 숫자는 지우는 방식**입니다.

**잘 되는 경우**

- 첫 행이 헤더인 표
- 첫 열이 항목명인 표
- `매출액`, `영업이익`, `순이익`처럼 row 이름이 명확한 표
- source 문서에도 비슷한 row 이름이 있는 표

**약한 경우**

- 병합 셀이 많은 표
- 헤더가 여러 줄인 표
- 첫 열이 항목명이 아닌 표
- row 이름은 비슷하지만 단위/연도/맥락이 다른 표
- 표 안 숫자의 정확한 위치까지 엄격히 맞춰야 하는 경우

즉 지금은 **“표 전용 파서/매핑 엔진”이라기보다는 “LLM이 표를 잘 채우도록 도와주는 보조 장치”**에 가깝습니다.

----

조금 더 구체적으로 보면, 이 코드의 표 처리 구현은 **“표를 직접 계산/편집하는 엔진”이라기보다는, 표 HTML을 보존하고 LLM이 채우도록 보조 근거를 붙이는 방식**입니다.

**1. OCR에서 표를 어떻게 잡는가**

시작점은 [artifact/ocr/_pipeline.py](<C:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/artifact/ocr/_pipeline.py:39>)입니다.

여기서 PaddleOCR-VL이 페이지 이미지에서 layout block을 감지합니다. 이후:

`merge_tables=False`라서 여러 페이지/여러 블록의 표를 하나로 합치지는 않습니다. OCR이 감지한 표 블록 단위 그대로 유지하는 쪽입니다.


형태가 됩니다.

[artifact/ocr/_artifact.py](<C:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/artifact/ocr/_artifact.py:23>)에서 표 여부를 이렇게 봅니다.

그리고 `to_structured_html()`에서 표 블록은 escape하지 않고 그대로 삽입합니다.


그래서 타겟 문서나 소스 문서가 LLM에게 전달될 때 표는 이런 식이 됩니다.

중요한 점은 `data-bbox`는 표의 위치/크기를 유지하는 데 쓰이고, `<table>` 내부 구조는 OCR 결과 HTML을 그대로 따릅니다.

**5. target 표에서 row label, column label을 뽑는 방식**

표 파싱은 [_parse_table_block](<C:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/table_retrieval.py:494>)에서 합니다.


헤더 행 선택 방식은 단순합니다.

```python

    if len(cells) == max_cell_count and max_cell_count > 1:

```

즉 **셀 개수가 가장 많은 첫 번째 행을 헤더로 간주**합니다.

그 다음:

```python

```

헤더 행의 첫 번째 셀은 제외하고 나머지를 column label로 봅니다.

row label은 헤더 아래 행들의 첫 번째 셀입니다.

```python

        row_labels.append(label)
```

즉 현재 구현의 기본 가정은 이겁니다.

```text

이후 첫 열:  [매출액]     [...]
             [영업이익]   [...]
             [순이익]     [...]
```

이런 전형적인 “첫 열이 항목명, 첫 행이 헤더”인 표에 잘 맞습니다.

대략 이런 검색어가 됩니다.


**9. 후보 페이지에서 row coverage를 계산**

[_inspect_candidate_pages](<C:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/table_retrieval.py:365>)는 검색 결과 block 자체만 보는 게 아니라, 그 block이 속한 source page 전체 HTML을 가져옵니다.

```python

```

coverage 계산은 단순합니다.


즉 source page visible text 안에 target 표의 row label이 들어 있으면 matched로 봅니다.

```python

    status = "partial"
else:
    status = "missing"
```

**13. 생성 후 숫자 검증에서 표 셀을 다시 검사**

LLM이 최종 HTML을 만들면 바로 저장하지 않고 [request_final_document](<C:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/create_document.py:525>)에서 숫자 guard를 돌립니다.

```python

    sanitize_unsupported_numbers(normalized_document, source_evidence_text)
```

표는 [number_guard.py](<C:/Users/echin/Desktop/ALLLM/llm-to-document/llm2doc/number_guard.py:291>)의 `_sanitize_table()`에서 처리합니다.

방식은:

1. `<table>` 찾기
2. header row 찾기
3. 첫 번째 컬럼은 row label로 보고 제외
4. 나머지 데이터 셀에서 숫자 추출
5. source evidence에 없는 숫자는 `-`로 대체

코드상 핵심은 이 부분입니다.

```python
if row_index <= header_row_index or col_index == 0:
    continue
```

헤더와 첫 컬럼은 검사하지 않습니다.

```python
unsupported_numbers = [
    number for number in detected_numbers
    if number not in source_numbers
]
```

지원되지 않는 숫자가 있으면:

```python
cell.clear()
cell.append("-")
```

즉 표에서 LLM이 임의 숫자를 넣으면 최종 HTML에서 그 셀을 비워버립니다.

**현재 구현의 강점**

- OCR 표 HTML을 최대한 보존합니다.
- target 표 구조를 그대로 유지하도록 prompt와 context가 설계되어 있습니다.
- source page를 표 row label 기준으로 미리 찾아 붙여서, LLM이 표를 채울 때 근거를 더 쉽게 봅니다.
- 최종 단계에서 unsupported number를 `-`로 막는 방어 로직이 있습니다.

**현재 구현의 약점**

- 셀 단위 데이터 모델이 없습니다. `Table`, `Row`, `Cell` 같은 내부 구조로 관리하지 않고 HTML 문자열을 BeautifulSoup으로 그때그때 파싱합니다.
- 헤더 행 판단이 “셀 개수가 가장 많은 첫 행”이라 다중 헤더, 병합 셀, 복잡한 재무표에는 취약합니다.
- row label은 “첫 번째 컬럼”이라고 가정합니다. 첫 컬럼이 기간/구분값이거나 row header가 여러 열에 걸쳐 있으면 coverage가 부정확해질 수 있습니다.
- source page coverage는 row label 포함 여부만 봅니다. 실제 값이 맞는지, 단위가 같은지, 연도가 맞는지는 preflight에서 직접 검증하지 않습니다.
- 숫자 guard는 source evidence에 숫자가 존재하는지만 봅니다. 같은 숫자가 다른 맥락에 있어도 통과할 수 있고, 반대로 OCR 표기 차이 때문에 맞는 숫자가 `-` 처리될 수도 있습니다.

결론적으로 지금 구조는 **“표를 인식하고 HTML 구조를 유지한 채 LLM에게 채우게 하되, 생성 전에 관련 source page를 더 붙이고, 생성 후 숫자 hallucination을 지우는 구현”**입니다. 표 내용을 deterministic하게 재구성하는 시스템은 아니고, LLM 생성 품질을 보조하는 retrieval + validation 레이어에 가깝습니다.
